# Урок 9: Тюнинг Ядра (Kernel Tuning & Limits)

## Цель
Понять, как лимиты ядра Linux могут сломать приложение, даже если ресурсов (CPU/RAM) достаточно, и научиться управлять параметрами через `sysctl`.

## Основные команды
- `sysctl -a` — показать все текущие параметры ядра.
- `sysctl <параметр>` — посмотреть конкретный параметр (например, `sysctl fs.file-max`).
- `sysctl -w <параметр>=<значение>` — изменить параметр "на лету" (до перезагрузки).
- `sysctl -p` — применить параметры из файла `/etc/sysctl.conf` (сохранение после перезагрузки).
- `dmesg -T` — логи ядра (часто ядро ругается туда, если упирается в лимиты сетей или inotify).

## Проблема: Inotify Watches
В современных системах (особенно где работают базы данных, NodeJS, React или Kubernetes) приложения часто "подписываются" на изменения файлов через механизм ядра `inotify`. По умолчанию лимит подписок (`fs.inotify.max_user_watches`) очень мал (обычно 8192). Если приложение пытается следить за бóльшим количеством файлов, оно падает с ошибкой `No space left on device` или `Too many open files`, хотя на диске полно места!

## Задание
1. Запустите `sudo ./simulate.sh` (требует root). Скрипт понизит лимит до 8192, создаст 15000 файлов и попытается повесить на них `inotify`.
2. Скрипт упадёт с ошибкой `No space left on device` после ~8к watches — лимит исчерпан.
3. Проверьте текущий лимит: `sysctl fs.inotify.max_user_watches`.
4. Увеличьте лимит временно: `sudo sysctl -w fs.inotify.max_user_watches=524288`.
5. Чтобы лимит сохранился навсегда, добавьте строку `fs.inotify.max_user_watches=524288` в файл `/etc/sysctl.d/99-tuning.conf` и выполните `sudo sysctl -p /etc/sysctl.d/99-tuning.conf`.
6. Перезапустите тест с флагом, чтобы скрипт не понизил лимит обратно:
   ```bash
   sudo KEEP_LIMIT=1 ./simulate.sh
   ```
   Теперь увидите `УСПЕХ! Удалось подписаться на 15000 файлов.`
