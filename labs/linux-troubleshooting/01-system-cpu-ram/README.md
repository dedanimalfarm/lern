## Теория для изучения перед модулем

### 1. Что такое Load Average (LA)?
Команда `uptime` или `top` возвращает три числа Load Average (за 1, 5 и 15 минут):
```
load average: 2.50, 1.20, 0.40
```
- **Load Average** показывает среднее число процессов, которые находятся в состоянии выполнения (Running, используют CPU) или ожидают очереди на выполнение (Runnable), а также процессов в состоянии непрерываемого сна (Uninterruptible Sleep, обычно ждут дискового ввода-вывода).
- **Сравнение с ядрами CPU:** 
  * Если у вас 2 ядра процессора, то LA = 2.00 означает полную утилизацию (100% нагрузка).
  * LA = 4.00 означает двукратную перегрузку (процессы проводят половину времени в очереди).
  * LA = 0.50 означает, что процессор загружен всего на 25%.

### 2. Сигналы процессов: SIGTERM vs SIGKILL
Для остановки процессов в Linux используются системные сигналы:
- **`SIGTERM` (сигнал 15, `kill -15 <PID>`):** вежливое завершение. Процесс ловит этот сигнал, завершает текущие операции, закрывает открытые файлы, сетевые соединения и освобождает память. Используется по умолчанию.
- **`SIGKILL` (сигнал 9, `kill -9 <PID>`):** грубое убийство. Ядро Linux мгновенно уничтожает процесс, не давая ему возможности сохранить данные или корректно закрыть дескрипторы файлов. Применяйте только тогда, когда процесс завис и не реагирует на `SIGTERM`.

## Практическое задание

### Шаг 1: Запуск симуляции сбоя

Запустите скрипт симуляции нагрузки:

```bash
bash simulate.sh
```

**Что происходит:** Скрипт запускает в фоне утилиту `stress-ng`, которая нагружает ядра процессора и занимает часть оперативной памяти. Система начинает работать медленнее.

---

### Шаг 2: Экспресс-диагностика

Оценим общую загрузку системы:

```bash
# Проверяем общую загрузку и Load Average
uptime

# Смотрим использование оперативной памяти
free -h
```

**Что смотрим / Как это работает:**
Вы увидите возрастающий Load Average. `free -h` покажет уменьшение объема свободной (`free`) оперативной памяти и рост используемой (`used`).

---

### Шаг 3: Поиск процесса-виновника

Найдем процесс, потребляющий максимальное количество ресурсов.

**Вариант А: Интерактивный (`top` / `htop`)**
Запустите команду `top`. 
- По умолчанию процессы отсортированы по использованию CPU.
- Нажмите клавишу **`M`** (заглавную), чтобы отсортировать процессы по потреблению оперативной памяти.
- Нажмите клавишу **`P`**, чтобы вернуть сортировку по CPU.
- Найдите PID процесса, который создает аномальную нагрузку (он будет называться `stress-ng`).

**Вариант Б: Быстрый однострочник через `ps`**
Если система сильно тормозит, интерактивный `top` может долго открываться. Быстрее использовать неинтерактивный `ps`:

```bash
# Топ-5 процессов по потреблению CPU
ps aux --sort=-%cpu | head -n 6

# Топ-5 процессов по потреблению RAM (сортировка по RSS)
ps aux --sort=-%mem | head -n 6
```

Запишите PID (обычно второй столбец) виновного процесса.

---

### Шаг 4: Остановка сбоя

Корректно завершите процесс, используя его PID:

```bash
# Сначала пробуем отправить вежливый SIGTERM
kill -15 <PID>

# Проверяем через 2 секунды, исчез ли процесс.
# Если он все еще жив (завис), отправляем жесткий SIGKILL:
kill -9 <PID>
```

Убедитесь через `uptime` и `free -h`, что нагрузка упала, а свободная память вернулась в систему.

## Вопросы на собеседовании

<details>
<summary><b>1. На сервере 4 ядра CPU. uptime показывает load average: 8.00 4.00 2.00. Что это значит?</b></summary>
Это означает деградацию производительности. За последнюю 1 минуту нагрузка в два раза превышала вычислительную способность процессора (очередь из процессов). Однако за 15 минут нагрузка составляла всего 2.00 (половина возможностей CPU). Система испытывает краткосрочный всплеск активности.
</details>

<details>
<summary><b>2. В чем разница между kill -9 и kill -15?</b></summary>
<code>kill -15</code> отправляет сигнал <code>SIGTERM</code>, позволяя процессу освободить ресурсы и корректно закрыть файлы. <code>kill -9</code> отправляет <code>SIGKILL</code>, который перехватывается ядром и мгновенно уничтожает процесс без очистки ресурсов и сохранения данных.
</details>

<details>
<summary><b>3. Как найти PID процесса, зная только его имя?</b></summary>
С помощью утилит <code>pgrep &lt;имя&gt;</code> или <code>pidof &lt;имя&gt;</code>.
</details>

## Шпаргалка

- `uptime` — базовая загрузка хоста.
- `ps aux --sort=-%cpu` — сортировка процессов по CPU.
- `free -h` — статус оперативной памяти.
- `pgrep -f "pattern"` — поиск PID по шаблону запуска процесса.
- `kill -15 <PID>` — вежливая остановка.
- `kill -9 <PID>` — принудительное завершение.
