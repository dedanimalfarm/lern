## Часть 1: Ресурсная модель REST

### Теория для изучения перед частью

- **REST строится вокруг ресурсов, а не действий.** URL называет «вещь»
  (тикет, пользователь), метод HTTP говорит, что с ней сделать. Сравните:

  | Стиль | Пример | Комментарий |
  |---|---|---|
  | RPC (глаголы в URL) | `POST /createTicket`, `GET /getTicketById?id=5` | так делали раньше, встречается в legacy |
  | REST (существительные) | `POST /tickets`, `GET /tickets/5` | действие задаёт метод |

- **Коллекция и элемент:** `/tickets` — коллекция (множественное число),
  `/tickets/5` — конкретный элемент. GET к коллекции — список, GET к
  элементу — один объект; POST делается К КОЛЛЕКЦИИ (id назначит сервер).
- **Версия в пути** (`/api/v1/`) — контракт: внутри v1 формат не ломают.
  Несовместимые изменения = новая версия. Старую сначала помечают
  deprecated, затем отключают (наш стенд отвечает за v0 редиректом 301 —
  мягкий вариант; жёсткий — 410 Gone).
- **Representation:** по сети ходит не «сам тикет», а его представление —
  JSON-снимок. Это объясняет семантику PUT в Части 4: «вот НОВОЕ
  представление целиком».

### 1.1 Ресурсы, коллекции, версии

```bash
# Полный CRUD тикета — обратите внимание, URL везде называет РЕСУРС:
curl -s 'http://localhost:8080/api/v1/tickets?per_page=2' | jq '.items | length'   # Read (коллекция)
# 2
curl -s http://localhost:8080/api/v1/tickets/1 | jq .id                             # Read (элемент)
# 1
curl -s -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' -d '{"title":"Тест ресурсной модели"}' | jq .id   # Create
# 9

  -H 'Content-Type: application/json' -d '{"priority":"low"}' | jq .priority        # Update
# "low"
curl -s -o /dev/null -w '%{http_code}\n' -X DELETE http://localhost:8080/api/v1/tickets/9   # Delete
# 204
```

## Часть 2: Чтение документации — OpenAPI

### Теория для изучения перед частью

- **OpenAPI (бывш. Swagger)** — стандарт описания REST API в одном
  YAML/JSON-файле. Это и есть «техническая документация», которую
  требуют уметь читать. Из неё генерируются и Swagger-UI странички,
  и Postman-коллекции (модуль 04), и клиентские SDK.
- **Карта спецификации — четыре места, где искать ответы:**

  | Секция | Вопрос, на который отвечает |
  |---|---|
  | `info`, описание | как аутентифицироваться, общие правила, формат ошибок |
  | `paths` | какие есть URL, какие методы, какие параметры |
  | `paths…parameters` | имена query-параметров, обязательность, `enum`, `default`, `maximum` |
  | `components/schemas` | состав объектов: какие поля, типы, какие обязательны (`required`) |

- **Чему верить при расхождении:** живой API прав, дока может отставать.
  Но сначала убедитесь, что вы читаете доку ТОЙ ЖЕ версии API, в которую
  шлёте запросы (классика: дока v2, прод на v1).
- **Как читать быстро:** не подряд, а поиском. `grep -n` по YAML или
  Ctrl+F в Swagger-UI: имя поля → его schema → required/enum/default.

## Часть 3: Пагинация и фильтры

### Теория для изучения перед частью

- **Никто не отдаёт миллион записей одним ответом.** Коллекции отдаются
  страницами. Наш стенд использует самый распространённый вариант —
  page/per_page; в других API встретите limit/offset и cursor-пагинацию
  (Jira в модуле 07 — `startAt`/`maxResults`, новые API — `nextPageToken`).
- **Метаданные страницы** живут в теле (`page`, `per_page`, `total`,
  `pages`) и/или заголовках (`X-Total-Count`). Признак «надо листать
  дальше»: `page < pages` (или непустая следующая страница).
- **Фильтры — это query-параметры**, и они работают НА СЕРВЕРЕ. Фильтровать
  jq'ом на клиенте — тащить лишние данные; правильный порядок: сервер
  фильтрует (`?status=open`), клиент дофильтровывает мелочь.
- **Кириллица в query** должна быть URL-кодирована. curl делает это флагом
  `--data-urlencode` вместе с `-G` (перенести данные в query при GET).

---

**Цель:** уверенно листать коллекции и не «терять» данные за первой страницей.

---

### 3.1 Страницы, total, pages

```bash
curl -s 'http://localhost:8080/api/v1/tickets?per_page=3&page=2' \
  | jq '{page, per_page, total, pages, ids: [.items[].id]}'
# {
#   "page": 2,
#   "per_page": 3,
#   "total": 8,       <- всего подходит
#   "pages": 3,       <- страниц при per_page=3
#   "ids": [4, 5, 6]  <- вторая тройка
# }

# Поиск с кириллицей: -G + --data-urlencode кодируют за вас
curl -s -G http://localhost:8080/api/v1/tickets --data-urlencode 'q=принтер' \
  | jq -r '.items[].title'
# Принтер на 3 этаже печатает пустые листы
```

### 3.2 Обойти все страницы циклом

```bash
# Заготовка «выгрузить ВСЁ» — пригодится в любой системе с пагинацией
PAGES=$(curl -s 'http://localhost:8080/api/v1/tickets?per_page=3' | jq '.pages')
for p in $(seq 1 "$PAGES"); do
  curl -s "http://localhost:8080/api/v1/tickets?per_page=3&page=$p" \
    | jq -r '.items[].id'
done
# 1 2 3 4 5 6 7 8   (по одному на строку)
```

## Часть 4: PUT против PATCH

### Теория для изучения перед частью

- **PATCH** = «измени ВОТ ЭТИ поля» — частичное обновление. Все остальные
  поля не трогаются.
- **PUT** = «вот новое representation ЦЕЛИКОМ» — полная замена. Поля,
  которые вы не прислали, сервер сбрасывает к дефолтам (наша дока честно
  предупреждает об этом в описании PUT — вы нашли это в практикуме 2.2).
- Отсюда правило эксплуатации: **для «поменять статус/назначить агента» —
  только PATCH.** PUT — когда вы действительно собираете объект заново.
- Инцидент-классика: «после синхронизации пропали назначения тикетов» —
  интеграция обновляла тикеты PUT'ом, отправляя только изменённые поля.

---

**Цель:** один раз потерять данные PUT'ом в лаборатории, чтобы никогда —
в проде.

---

### 4.1 PATCH — точечное изменение

```bash
# Берём тикет 2 в работу: меняем ровно два поля
curl -s -X PATCH http://localhost:8080/api/v1/tickets/2 \
  -H 'Content-Type: application/json' \
  -d '{"status":"in_progress","assignee":"support"}' | jq '{id,status,assignee,priority}'
# {
#   "id": 2,
#   "status": "in_progress",
#   "assignee": "support",
#   "priority": "low"        <- не трогали — и он не изменился
# }
```

### 4.2 PUT — полная замена (и потеря данных)

```bash
# До: тикет 3 в работе, назначен на support
curl -s http://localhost:8080/api/v1/tickets/3 | jq '{id,status,assignee,requester}'
# {"id":3,"status":"in_progress","assignee":"support","requester":"maria.ivanova@corp.example"}

# «Обновим заголовок» PUT'ом, прислав только title…
curl -s -X PUT http://localhost:8080/api/v1/tickets/3 \
  -H 'Content-Type: application/json' \
  -d '{"title":"Не приходят письма от Jira"}' | jq '{id,status,assignee,requester}'
# {
#   "id": 3,
#   "status": "open",        <- ПОТЕРЯЛИ: сброшен к дефолту
#   "assignee": null,        <- ПОТЕРЯЛИ
#   "requester": null        <- ПОТЕРЯЛИ
# }
```

> Это не баг стенда — это семантика PUT по HTTP-стандарту, и многие API так
> работают. Прежде чем «просто обновить поле» в незнакомой системе, найдите
> в её доке, как ведёт себя PUT, и есть ли PATCH.

```bash
# Вернём тикет 3 в исходное состояние (заодно потренируем PATCH)
curl -s -X PATCH http://localhost:8080/api/v1/tickets/3 \
  -H 'Content-Type: application/json' \
  -d '{"status":"in_progress","assignee":"support"}' >/dev/null
```

1. Сформулируйте разницу PUT и PATCH одним предложением каждое.

## Часть 5: Валидация против документации

### Теория для изучения перед частью

- **422 — это разговор сервера с вами на языке документации:** в `details`
  он называет поле и правило, которое нарушено. Сопоставьте с `schemas` в
  спеке — и правка очевидна.
- **Имена полей не угадывают, а читают в доке.** Половина ошибок интеграций
  — «перепутали имя поля» (`subject` вместо `title`, `prio` вместо
  `priority`): другая система — другой словарь.
- **Read-only поля** (`id`, `status` при создании, `created_at`) задаёт
  сервер. Хорошие API отвергают попытку прислать их (как наш — 422),
  плохие — молча игнорируют (что хуже: вы думаете, что задали статус,
  а он не задался).

---

## Шпаргалка

```bash
# === CRUD-мэппинг ===
GET    /api/v1/tickets        # список (страницей!)
POST   /api/v1/tickets        # создать (201 + Location)
GET    /api/v1/tickets/5      # прочитать
PATCH  /api/v1/tickets/5      # изменить ЧАСТЬ полей
PUT    /api/v1/tickets/5      # заменить ЦЕЛИКОМ (несданное — к дефолтам!)
DELETE /api/v1/tickets/5      # удалить (204)

# === Пагинация и фильтры ===
curl -s 'URL/tickets?status=open&priority=high&per_page=50&page=2'
curl -s -G URL/tickets --data-urlencode 'q=кириллица или спецсимволы'
... | jq '{page, pages, total}'          # есть ли ещё страницы
PAGES=$(... | jq .pages); for p in $(seq 1 $PAGES); do ...page=$p...; done

# === Документация ===
grep -n '^  /' docs/openapi.yaml         # все пути
grep -n 'enum:' docs/openapi.yaml        # допустимые значения
grep -n -A3 'required' docs/openapi.yaml # обязательные поля
curl -s http://localhost:8080/openapi.yaml   # дока с самого сервера

# === Типовой workflow тикета ===
curl -s -X POST ... -d '{"title":"...","priority":"critical"}'        # создать
curl -s -X PATCH .../tickets/N -d '{"status":"in_progress","assignee":"support"}'
curl -s -X PATCH .../tickets/N -d '{"status":"resolved"}'
```
