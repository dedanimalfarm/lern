> ⏱ время ~90 мин · сложность 2/5 · пререквизиты: модули 01–03; Postman на хосте

Цель: уверенно работать в Postman — инструменте №1 из требований вакансии:
импортировать и собирать коллекции, гонять их Runner'ом, писать автотесты
на ответы и передавать данные между запросами через переменные. К концу
модуля у вас есть СВОЯ экспортированная коллекция со своим запросом и
тестами — готовый артефакт для собеседования.

> **Почему после curl.** Postman делает то же самое, что curl (HTTP-запросы),
> но добавляет то, что нужно команде: сохранённые наборы запросов
> (коллекции), окружения (dev/stage/prod), автотесты и человекочитаемую
> историю. curl — для серверов и скриптов, Postman — для исследования API и
> регрессионных проверок. L2 пользуется обоими.

## Предварительные требования

- **Postman** установлен на хосте (Windows: <https://www.postman.com/downloads/>;
  аккаунт необязателен — «Skip and go to the app» / lightweight API client).
- Стенд поднят в WSL: `cd labs/api && scripts/api.sh up`.
- **WSL2 ↔ Windows:** Windows-приложения видят WSL-сервисы по `localhost` —
  Postman достучится до `http://localhost:8080` без настроек. Если нет
  (старые сборки WSL) — `ip addr show eth0` в WSL и используйте этот IP.

## Стартовая проверка

```bash
# в WSL: стенд жив и в дефолтном режиме

# {"status":"ok","uptime_s":...}
# {"fault":"none","auth_mode":"off","rate_limit":0,...,"tickets":8}
```

В Postman: **New → HTTP Request** → `GET http://localhost:8080/health` →
**Send**. Внизу должно появиться `{"status":"ok",...}` и бейдж
`200 OK` с временем ответа.

## Часть 1: Подключение Postman к стенду

### Теория для изучения перед частью

- **Окно запроса** повторяет всё, что вы делали в curl, по вкладкам:
  строка метода+URL (= `-X` и URL), **Params** (query-параметры — то, что
  после `?`), **Headers** (= `-H`), **Body** (= `-d`), кнопка Send.
- **Окно ответа:** Body (с автоформатированием JSON — то, для чего в curl
  нужен jq), Headers, бейджи: статус-код, время, размер.
- **Body → raw → JSON** — главный выбор для REST: Postman сам поставит
  `Content-Type: application/json`. Проверить можно в Headers запроса —
  автозаголовки показываются серым (toggle «hidden»).

---

**Цель:** воспроизвести в Postman цикл из модуля 01 и увидеть соответствие
«вкладка ↔ флаг curl».

---

### 1.1 Первый запрос руками

В Postman:

1. `GET http://localhost:8080/api/v1/tickets`, во вкладке **Params**
   добавьте `status = open`, `per_page = 50` — смотрите, как URL дополняется
   сам. **Send** → `200 OK`, в Body — JSON с подсветкой (сравните с голым
   curl-выводом: вот зачем людям Postman).
2. `POST http://localhost:8080/api/v1/tickets`: **Body → raw → JSON**,
   вставьте `{"title": "Первый из Postman", "priority": "low"}` → **Send** →
   `201 Created`. Во вкладке **Headers** ОТВЕТА найдите `Location`.
3. Повторите POST с Body-типом **Text** вместо JSON → `415` — та же
   «лестница ошибок», что в модуле 01, только теперь видно, что тип Body
   управляет заголовком.

## Часть 2: Импорт коллекции и environment

### Теория для изучения перед частью

- **Коллекция** — сохранённый набор запросов с папками, описаниями и
  тестами. Хранится как обычный JSON (формат v2.1) — поэтому коллекции
  можно ревьюить в git и читать глазами (вы уже умеете — модуль 02).
- **Environment** — набор переменных (`base_url`, токены) для подстановки
  через `{{var}}`. Одна коллекция + разные environment = dev/stage/prod
  без правки запросов. **Секреты живут в environment, а не в коллекции** —
  коллекцией делятся, окружением нет (подробнее в модуле 05).
- Файлы курса: `common/postman/helpdesk.postman_collection.json` и
  `common/postman/local.postman_environment.json`.

## Часть 3: Переменные и цепочки запросов

### Теория для изучения перед частью

- **`{{var}}`** подставляется из активного environment (или коллекции/
  глобальных) в URL, заголовки, тело — куда угодно.
- **Записать переменную из ответа** можно скриптом в Tests:
  `pm.environment.set('ticket_id', pm.response.json().id)`.
  Следующие запросы читают её как `{{ticket_id}}` — получается **цепочка**:
  создал → получил → изменил, без ручного копирования id.
- Это решает реальную задачу смены: «прогнать сквозной сценарий по тестовой
  системе одним кликом» (создание заявки → взятие в работу → проверка).

---

**Цель:** проследить, как `ticket_id` течёт по коллекции.

---

1. Выполните по очереди: «Создать тикет» → «Получить созданный тикет» →
   «Взять в работу (PATCH)». Каждый следующий работает с тем же тикетом,
   хотя id вы нигде не вводили.
2. Откройте глазик 👁 рядом с выбором environment: переменная `ticket_id`
   появилась после первого запроса (её записал скрипт).
3. В WSL убедитесь, что цепочка реально дошла до API:

```bash
curl -s -G http://localhost:8080/api/v1/tickets --data-urlencode 'q=Создано из Postman' \
  | jq -r '.items[-1] | "\(.id) \(.status) \(.assignee)"'
# 9 in_progress support
```

## Часть 4: Тесты и Collection Runner

### Теория для изучения перед частью

- **Тест** — JavaScript во вкладке Tests, выполняется ПОСЛЕ получения
  ответа. Базовый словарь:

  ```javascript
  pm.test('имя проверки', function () {            // одна проверка
      pm.response.to.have.status(200);             // статус-код
      pm.response.to.have.header('Location');      // заголовок есть
      const body = pm.response.json();             // распарсить JSON
      pm.expect(body.status).to.eql('open');       // равенство
      pm.expect(body.items).to.be.an('array');     // тип
  });
  ```

- **Зачем тесты L2:** коллекция с тестами = smoke-набор «жив ли API»
  после релиза/инцидента. Один клик — и видно, ЧТО именно сломалось,
  вместо ручного перещёлкивания двадцати запросов.
- **Collection Runner** прогоняет коллекцию по порядку (можно с задержкой,
  итерациями, данными из CSV). **Newman** — тот же прогон из CLI: его
  ставят в CI или на сервер, где нет GUI.

---

**Цель:** прочитать готовые тесты, прогнать коллекцию целиком и добавить
свой запрос с тестами.

---

### 4.1 Как устроен тест

Откройте Tests у «404 — тикет не существует»: негативная проверка
*ожидает* 404 — для smoke-набора «API корректно отвечает на мусор» не менее
важно, чем 200 на нормальные запросы.

### 4.2 Прогон всей коллекции

Правый клик по коллекции → **Run collection** → **Run**. Ожидаемый итог:
**7 запросов, 16 проверок, 0 failed** (все строки зелёные).

### 4.3 Newman — коллекция из командной строки

```bash
# в WSL (node уже есть; -y скачает newman при первом запуске)
cd labs/api
npx -y newman@6 run common/postman/helpdesk.postman_collection.json \
  -e common/postman/local.postman_environment.json
# ↳ Получить созданный тикет
#   GET http://localhost:8080/api/v1/tickets/9 [200 OK, 383B, 45ms]
#   ✓  статус 200
#   ✓  это именно созданный тикет
# ...
# │ requests     │ 7  │ 0 │
# │ assertions   │ 16 │ 0 │      <- те же тесты, что в GUI
```

> Newman превращает вашу коллекцию в smoke-проверку для cron/CI: «каждые
> 5 минут прогоняй коллекцию, при failed — алерт». Это уже готовый
> мини-мониторинг API силами L2.

## Часть 5: Импорт OpenAPI и экспорт коллекции

### Теория для изучения перед частью

- **Postman умеет генерировать коллекцию из OpenAPI-спеки** (Import →
  файл `docs/openapi.yaml`): получится каркас всех запросов API с
  параметрами из документации. Generated-коллекция — черновик: тел и
  тестов в ней нет, но как «оглавление чужого API» — бесценно.
- **Экспорт** (правый клик → Export → v2.1) — то, как коллекциями
  обмениваются: в тикет, в git, коллеге. Экспортированный файл — обычный
  JSON, который вы умеете валидировать (`python3 -m json.tool`).

---

**Цель:** собрать свой запрос, добавить тесты и экспортировать коллекцию —
финальный артефакт модуля.

---

1. Импортируйте `docs/openapi.yaml` (выберите «Postman Collection» при
   импорте) — изучите сгенерированный каркас, сравните с ручной коллекцией.
2. В коллекцию «Helpdesk API — учебная коллекция», папку `10 Tickets CRUD`,
   добавьте запрос **«Решить тикет (PATCH resolved)»**:
   - `PATCH {{base_url}}/api/v1/tickets/{{ticket_id}}`;
   - Body raw JSON: `{"status": "resolved"}`;
   - Tests: статус 200 **и** `body.status === 'resolved'` (минимум два

## Шпаргалка

```text
# === Postman UI ===
Params   = query-параметры (?key=value)
Headers  = curl -H            Body raw JSON = curl -d + Content-Type
Tests (Scripts→Post-response) = проверки после ответа
{{var}}  = переменная environment;  глазик 👁 = текущие значения
Import   = коллекция / environment / OpenAPI-спека
Export   = Collection v2.1 (JSON-файл)

# === Скрипты ===
pm.test('имя', function () { ... });
pm.response.to.have.status(201);
pm.response.to.have.header('Location');
const body = pm.response.json();
pm.expect(body.status).to.eql('open');
pm.expect(body.items).to.be.an('array');
pm.environment.set('ticket_id', body.id);   // цепочка
pm.environment.get('ticket_id');

# === Newman (CLI) ===
npx -y newman@6 run collection.json -e environment.json
```
