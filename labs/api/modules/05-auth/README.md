## Часть 1: Карта способов аутентификации

### Теория для изучения перед частью

- **Аутентификация** («кто ты?») и **авторизация** («что тебе можно?») —
  разные этапы. Ошибка первого — 401, второго — 403. Это та же пара
  AuthN/AuthZ, что в kubectl/kube-apiserver (модуль 01 k8s-курса).
- **Способы, которые встретите в интеграциях** (от простого к сложному):

  | Способ | Как передаётся | Где встречается | Особенности |
  |---|---|---|---|
  | API key | заголовок `X-API-Key: <key>` (имя бывает любое) | вебхуки, простые сервисы, Grafana | ключ долгоживущий; «кто» = владелец ключа |
  | Basic | `Authorization: Basic base64(user:pass)` | старые API, Jira Cloud (email+API-token, модуль 07) | base64 — НЕ шифрование; только поверх HTTPS |
  | Bearer (часто JWT) | `Authorization: Bearer <token>` | большинство современных API | токен короткоживущий, получают отдельным запросом |
  | OAuth2 | Bearer + протокол ВЫДАЧИ токена | «Войти через Google», интеграции SaaS | для L2: итог всё равно Bearer-токен |
  | mTLS | клиентский сертификат на уровне TLS | банки, межсервисное | видно по ошибкам handshake, не по 401 |

- **Всё передаётся в заголовках.** Токен в query-параметре (`?token=…`) —
  антипаттерн: URL попадают в логи прокси и истории браузеров.
- **`WWW-Authenticate`** в ответе 401 — сервер сообщает, КАКОЙ способ ждёт.
  Читайте его прежде, чем гадать.

## Часть 2: API key

### Теория для изучения перед частью

- **API key** — простейшая схема: выдали строку, прикладываешь её к
  каждому запросу. Сервер сравнивает со своей — всё. Никаких сессий,
  сроков, ролей (если не прикрутили отдельно).
- Минусы прямо вытекают из простоты: ключ не истекает (украли — пользуются,
  пока не заметите), отзыв = смена ключа у ВСЕХ интеграций на нём.
- На стенде режим включается `AUTH_MODE=apikey`, ключ —
  `lab-secret-key-2026` (env `API_KEY`).

---

**Цель:** пройти цикл «нет ключа → не тот ключ → тот ключ» и вернуть стенд
в режим token.

---

```bash
# ВРЕМЕННО переключаем стенд в режим apikey
AUTH_MODE=apikey scripts/api.sh up
# [helpdesk-api] listening on :8080 auth=apikey ...

# Без ключа: 401 + подсказка способа
curl -s -i http://localhost:8080/api/v1/tickets | grep -E 'HTTP|WWW-Auth'
# HTTP/1.1 401 Unauthorized
# WWW-Authenticate: ApiKey
curl -s http://localhost:8080/api/v1/tickets
# {"error":{"code":"missing_api_key","message":"нужен заголовок X-API-Key"}}

# Не тот ключ — тоже 401, но код ошибки ДРУГОЙ (читайте тело!)
curl -s -H 'X-API-Key: wrong' http://localhost:8080/api/v1/tickets
# {"error":{"code":"invalid_api_key","message":"ключ не подходит"}}

# Правильный ключ — работаем. Ключ — в переменной, НЕ в тексте команды:
export HELPDESK_KEY='lab-secret-key-2026'
curl -s -H "X-API-Key: $HELPDESK_KEY" http://localhost:8080/api/v1/whoami
# {"sub":"apikey-client","role":"admin","auth_mode":"apikey"}

# Возвращаем режим модуля:
AUTH_MODE=token scripts/api.sh up
```

> `missing_api_key` vs `invalid_api_key` — одинаковый статус 401, разные
> `error.code`. Первый чинится «добавь заголовок», второй — «проверь/смени
> ключ». Привычка читать тело ошибки, а не только код, экономит итерации.

## Часть 3: Basic → Bearer: токены и JWT

### Теория для изучения перед частью

- **Типовой флоу современных API — двухшаговый:**
  1. один раз: `POST /auth/token` с Basic-кредитами → получить
     короткоживущий **access token**;
  2. каждый запрос: `Authorization: Bearer <token>`.
- **Basic — это просто base64(login:pass)**, никакого шифрования:
  `echo -n 'support:support123' | base64` → `c3VwcG9ydDpzdXBwb3J0MTIz`,
  и обратно так же легко. Конфиденциальность даёт только HTTPS вокруг.
- **JWT (JSON Web Token)** — формат токена из трёх base64url-частей через
  точки: `header.payload.signature`:
  - header — алгоритм подписи (`{"alg":"HS256","typ":"JWT"}`);
  - payload — **читаемые** клеймы: `sub` (кто), `role`, `iat` (выдан),
    `exp` (истекает, unix-время);
  - signature — подпись header+payload секретом сервера.
- **Следствие, которое все забывают:** содержимое JWT видно ЛЮБОМУ, кто
  его получил (это не шифрование!), но подделать нельзя — изменишь хоть
  байт payload, подпись не сойдётся.

---

**Цель:** пройти полный токен-флоу и научиться отвечать на вопрос «почему
интеграция получает 401» чтением самого токена.

---

### 3.1 Получить токен

```bash
# Шаг 1: Basic-кредиты (-u в curl) -> токен
curl -s -X POST -u support:support123 http://localhost:8080/api/v1/auth/token | jq .
# {
#   "access_token": "eyJhbGciOiAiSFMyNTYiLCAidHlwIjogIkpXVCJ9.eyJzdWIiOiAic3VwcG9ydCIs...",
#   "token_type": "Bearer",
#   "expires_in": 3600          <- через час токен умрёт
# }

# Сохраняем в переменную (на смене вы будете делать это постоянно)
TOKEN=$(curl -s -X POST -u support:support123 \
  http://localhost:8080/api/v1/auth/token | jq -r .access_token)

# Шаг 2: токен в каждый запрос
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:8080/api/v1/whoami
# {"sub":"support","role":"agent","auth_mode":"token"}

# Неправильные кредиты на шаге 1 -> 401 ещё ДО всякого токена
curl -s -X POST -u support:wrongpass http://localhost:8080/api/v1/auth/token | jq -c .error
# {"code":"invalid_credentials","message":"логин или пароль не подходят"}
```

### 3.2 Заглянуть внутрь JWT

```bash
# Токен = три части через точку. Возьмём вторую (payload) и декодируем.
# Нюанс: base64url без паддинга — добиваем '==' перед base64 -d.
PAYLOAD=$(echo "$TOKEN" | cut -d. -f2)
echo "${PAYLOAD}==" | base64 -d 2>/dev/null | jq .
# {
#   "sub": "support",
#   "role": "agent",
#   "iat": 1781139713,
#   "exp": 1781143313     <- date -d @1781143313 покажет, когда умрёт
# }

# Первая часть (header) — каким алгоритмом подписан
echo "$TOKEN" | cut -d. -f1 | base64 -d 2>/dev/null | jq -c .
# {"alg":"HS256","typ":"JWT"}

# А теперь подделаем payload (role: admin), оставив старую подпись:
FAKE=$(echo -n '{"sub": "support", "role": "admin", "iat": 1781100000, "exp": 1999999999}' \
  | base64 -w0 | tr '+/' '-_' | tr -d '=')
HDR=$(echo "$TOKEN" | cut -d. -f1); SIG=$(echo "$TOKEN" | cut -d. -f3)
curl -s -H "Authorization: Bearer $HDR.$FAKE.$SIG" http://localhost:8080/api/v1/whoami
# {"error":{"code":"invalid_signature","message":"токен не принят"}}
#  ^ ЧИТАТЬ payload может кто угодно, ИЗМЕНИТЬ — никто: подпись не сошлась
```

> Зачем это L2: «интеграция вдруг ловит 401» — декодируйте её токен и
> посмотрите `exp`. В половине случаев инцидент решается за минуту:
> токен протух, а ретрай-логика не перевыпускает его.

### 3.3 Срок жизни токена

```bash
# Поднимем стенд с временем жизни токена 1 секунда и посмотрим на смерть токена
AUTH_MODE=token TOKEN_TTL=1 scripts/api.sh up
T=$(curl -s -X POST -u support:support123 http://localhost:8080/api/v1/auth/token | jq -r .access_token)
sleep 2
curl -s -H "Authorization: Bearer $T" http://localhost:8080/api/v1/tickets | jq -c .error
# {"code":"token_expired","message":"токен не принят"}

# Возвращаем нормальный TTL
AUTH_MODE=token scripts/api.sh up
```

## Часть 4: 401 против 403 — роли и права

### Теория для изучения перед частью

- **401 Unauthorized** = «я не понял, КТО ты» (нет/битый/протухший токен).
  Чинится на стороне КЛИЕНТА: получить/обновить токен, поправить заголовок.
- **403 Forbidden** = «я знаю, кто ты, и тебе НЕЛЬЗЯ». Чинится на стороне
  СЕРВЕРА/администратора: выдать роль, расширить права. Слать тот же запрос
  повторно бессмысленно.
- На стенде роли встроены в токен (`role: agent|admin`), и DELETE требует
  admin — ровно как в реальных системах «агент не может удалять заявки».

---

**Цель:** получить оба кода в соседних командах и навсегда зафиксировать
разницу.

---

```bash
# Агент может читать и создавать:
TOKEN=$(curl -s -X POST -u support:support123 http://localhost:8080/api/v1/auth/token | jq -r .access_token)
curl -s -o /dev/null -w '%{http_code}\n' -H "Authorization: Bearer $TOKEN" \
  http://localhost:8080/api/v1/tickets
# 200

# ...но НЕ удалять: аутентификация прошла, авторизация — нет
curl -s -i -X DELETE -H "Authorization: Bearer $TOKEN" \
  http://localhost:8080/api/v1/tickets/5 | grep -E 'HTTP|error'
# HTTP/1.1 403 Forbidden
# {"error":{"code":"forbidden","message":"роль 'agent' не может удалять тикеты (нужна роль admin)"}}

# Совсем без токена — 401: сервер даже не знает, кто пришёл
curl -s -o /dev/null -w '%{http_code}\n' -X DELETE http://localhost:8080/api/v1/tickets/5
# 401

# Админ — может (мы не удаляем 5-й по-настоящему, он нужен дальше; проверим на 404-м)
ADM=$(curl -s -X POST -u admin:admin123 http://localhost:8080/api/v1/auth/token | jq -r .access_token)
curl -s -o /dev/null -w '%{http_code}\n' -X DELETE -H "Authorization: Bearer $ADM" \
  http://localhost:8080/api/v1/tickets/999
# 404    <- права есть (не 403!), просто такого тикета нет
```

> Финальный штрих: для админа сервер дошёл до проверки существования
> ресурса (404), для агента — отбился раньше (403). По коду видно, НА КАКОМ
> ЭТАПЕ запрос умер — та же логика этапов, что в каталоге ошибок apiserver
> из k8s-модуля 01.

## Часть 5: Rate limiting — 429

### Теория для изучения перед частью

- **Rate limit** защищает API от перегрузки: больше N запросов в окно —
  получай **429 Too Many Requests** + заголовок **`Retry-After: <сек>`**
  (когда можно вернуться).
- Правильная реакция клиента — **подождать и повторить** (backoff),
  неправильная — долбить дальше (некоторые системы за это банят дольше).
- Инцидент-классика: «интеграция работала, после расширения филиала
  отвалилась» = выросло число запросов, упёрлись в лимит.

---

**Цель:** упереться в лимит руками и прочитать `Retry-After`.

---

```bash
# Стенд с лимитом 5 запросов / 10 секунд
AUTH_MODE=token RATE_LIMIT=5 scripts/api.sh up
TOKEN=$(curl -s -X POST -u support:support123 http://localhost:8080/api/v1/auth/token | jq -r .access_token)

for i in $(seq 1 7); do
  printf '%s ' "$(curl -s -o /dev/null -w '%{http_code}' \
    -H "Authorization: Bearer $TOKEN" http://localhost:8080/api/v1/tickets)"
done; echo
# 200 200 200 200 200 429 429

curl -s -i -H "Authorization: Bearer $TOKEN" http://localhost:8080/api/v1/tickets \
  | grep -E 'HTTP|Retry-After'
# HTTP/1.1 429 Too Many Requests
# Retry-After: 10                 <- через сколько секунд приходить

# Вернуть стенд без лимита:
AUTH_MODE=token scripts/api.sh up
```

## Часть 6: Гигиена секретов

### Теория для изучения перед частью

- **Секреты не живут в:** коде и git-репозиториях, текстах тикетов,
  скриншотах, логах, Postman-КОЛЛЕКЦИЯХ (helpers: коллекцией делятся —
  секрет уедет вместе с ней).
- **Секреты живут в:** переменных окружения, секрет-хранилищах (Vault,
  модуль 07 screening-курса), Postman-ENVIRONMENT (поле типа secret).
- **В команды — только через переменные:** `-u support:$PASS`, ключ из
  `$HELPDESK_KEY`. Прямой пароль в команде остаётся в `~/.bash_history`.
- **В тикет/эскалацию** токен не вставляют НИКОГДА; если случилось —
  токен считается скомпрометированным и перевыпускается.

## Шпаргалка

```bash
# === Режимы стенда ===
AUTH_MODE=apikey scripts/api.sh up      # X-API-Key: lab-secret-key-2026
AUTH_MODE=token  scripts/api.sh up      # Basic -> Bearer (support/admin)
AUTH_MODE=token RATE_LIMIT=5 scripts/api.sh up

# === API key ===
curl -s -H "X-API-Key: $KEY" URL

# === Токен-флоу ===
TOKEN=$(curl -s -X POST -u support:support123 \
  http://localhost:8080/api/v1/auth/token | jq -r .access_token)
curl -s -H "Authorization: Bearer $TOKEN" URL

# === JWT внутри ===
echo "$TOKEN" | cut -d. -f2             # payload (base64url)
P=$(echo "$TOKEN" | cut -d. -f2); echo "${P}==" | base64 -d 2>/dev/null | jq .
date -d @<exp>                          # когда истекает

# === Карта кодов ===
# 401 missing_token       -> нет заголовка Authorization: Bearer
# 401 invalid_signature   -> токен битый/чужой/подделанный
# 401 token_expired       -> перевыпустить токен (см. exp)
# 401 invalid_credentials -> логин/пароль на /auth/token
# 403 forbidden           -> прав не хватает; чинит АДМИН, не клиент
# 429 + Retry-After       -> подождать и повторить (backoff)
```
