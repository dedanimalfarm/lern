## Часть 1: Синтаксис и типы JSON

### Теория для изучения перед частью

- **JSON (JavaScript Object Notation)** — текстовый формат обмена данными.
  Им говорят почти все современные API. Весь JSON собирается из **шести
  типов**:

  | Тип | Пример | Замечания |
  |---|---|---|
  | object | `{"key": "value"}` | пары «ключ: значение», ключ — ВСЕГДА строка в двойных кавычках |
  | array | `[1, 2, 3]` | упорядоченный список любых типов |
  | string | `"текст"` | ТОЛЬКО двойные кавычки; `\"` — кавычка внутри, `\n` — перевод строки |
  | number | `42`, `3.14`, `-1e9` | без кавычек; нет отдельного int/float |
  | boolean | `true`, `false` | строго в нижнем регистре |
  | null | `null` | «значение отсутствует» |

- **Вложенность не ограничена:** объект в массиве в объекте — норма.
  Ответ списка тикетов — это объект, внутри которого массив объектов:
  `{"items": [{...}, {...}], "total": 8}`.
- **Чего в JSON НЕТ** (в отличие от Python/JS): комментариев, одинарных
  кавычек, висячей запятой после последнего элемента, `True/False/None`.
  Большинство «битых JSON» в тикетах — ровно эти четыре вещи.
- **`"строка" "123" против 123`:** число в кавычках — это строка. API,
  ожидающий число, ответит 422 на `"per_page": "5"` — тип не тот, хотя
  глазами выглядит похоже.

---

**Цель:** увидеть все типы JSON в живом ответе API и перестать путать
null/пустоту/отсутствие.

---

### 1.1 Шесть типов на одном тикете

```bash
# Тикет 3 содержит сразу несколько типов: числа, строки, null нет — есть строка
curl -s http://localhost:8080/api/v1/tickets/3 | jq '{id, title, assignee, description}'
# {
#   "id": 3,                                  <- number
#   "title": "Не приходят письма от Jira",    <- string
#   "assignee": "support",                    <- string
#   "description": ""                         <- ПУСТАЯ строка — это не null!
# }

# А весь список — объект с массивом объектов внутри + числа пагинации
curl -s 'http://localhost:8080/api/v1/tickets?per_page=2' | jq 'keys'
# [
#   "items",      <- array
#   "page",       <- number
#   "pages",
#   "per_page",
#   "total"
# ]
```

### 1.2 null ≠ пустая строка ≠ отсутствие поля

```bash
# У тикета 1 нет назначенного агента: assignee = null (поле ЕСТЬ, значение — null)
curl -s http://localhost:8080/api/v1/tickets/1 | jq '.assignee, .nosuchfield, (.assignee == null), (has("assignee")), (has("nosuchfield"))'
# null      <- значение поля assignee
# null      <- jq отдаёт null и для НЕСУЩЕСТВУЮЩЕГО поля!
# true
# true      <- но has() различает: assignee есть...
# false     <- ...а nosuchfield — нет
```

> Три разных состояния — `"assignee": null`, `"assignee": ""` и отсутствие
> ключа — для API могут означать разное («не назначен», «назначен пустым»,
> «поле не передавали»). При разборе инцидентов «почему поле не сохранилось»
> это различие — половина ответа. jq маскирует разницу между null и
> отсутствием — проверяйте `has()`.

## Часть 2: Типовые ошибки JSON и как их читать

### Теория для изучения перед частью

- **Парсер всегда говорит, ГДЕ споткнулся** (`line N column M`) и often
  ЧЕГО ожидал. Читать сообщения парсера — навык, который экономит часы.
- **Каталог типовых поломок** (90% битого JSON в реальных тикетах):

  | Симптом в файле | Сообщение парсера (python) | Починка |
  |---|---|---|
  | `{'key': 'v'}` — одинарные кавычки | `Expecting property name enclosed in double quotes` | заменить на `"` |
  | `{"a": 1,}` — висячая запятая | `Illegal trailing comma before end of object` | убрать запятую |
  | `// комментарий` | `Expecting value` / `Expecting property name` | удалить комментарии |
  | `"text: "abc"def"` — неэкранированная кавычка | `Expecting ',' delimiter` | `\"` внутри строки |
  | `True`, `None` — питонизмы | `Expecting value` | `true`, `null` |
  | обрезанный файл | `Unexpected EOF` / `Unfinished string` | дослать конец файла |

- **Два валидатора всегда под рукой:** `python3 -m json.tool file.json`
  (стандартная библиотека, отличные сообщения) и `jq . file.json`
  (заодно красиво печатает). CI-проверки JSON строятся на них же.

---

**Цель:** по сообщению парсера находить место и причину поломки, не
вычитывая весь файл глазами.

---

### 2.1 Каталог ошибок

```bash
# Одинарные кавычки
echo "{'a': 1}" | python3 -m json.tool
# Expecting property name enclosed in double quotes: line 1 column 2 (char 1)

# Висячая запятая — заметьте, насколько точен питоновский парсер
echo '{"a": 1,}' | python3 -m json.tool
# Illegal trailing comma before end of object: line 1 column 8 (char 7)

# Тот же файл глазами jq — формулировка другая, смысл тот же
echo '{"a": 1,}' | jq .
# jq: parse error: Expected another key-value pair at line 1, column 9
```

### 2.2 Валидация: python -m json.tool и jq

```bash
# Валидно? Молчит и форматирует = да. Ругается = нет, и говорит где.
echo '{"a": [1, 2, {"b": null}]}' | python3 -m json.tool
# {
#     "a": [
#         1,
#         2,
#         {
#             "b": null
#         }
#     ]
# }

# Быстрый «валидатор» для скриптов: код возврата
echo '{"ok": true}' | jq -e . >/dev/null && echo VALID || echo BROKEN
# VALID
```

**Контрольные вопросы:**
1. Парсер сказал `Expecting property name enclosed in double quotes: line 1
   column 2`. Какая ошибка почти наверняка в файле?
2. Почему JSON с комментариями `//` не парсится, хотя «в VS Code работало»?
   *(подсказка: JSONC ≠ JSON)*
3. Какими двумя командами можно проверить валидность JSON-файла без
   установки чего-либо?
4. Файл валиден для `jq`, но API всё равно отвечает 422. Где теперь искать
   проблему? *(вспомните «лестницу» из модуля 01)*

## Часть 3: jq — читать ответы API

### Теория для изучения перед частью

- **jq — это grep/awk для JSON.** На вход — JSON, на выход — то, что
  выбрал ваш фильтр. Базовые фильтры:
  - `.` — весь документ (заодно pretty-print);
  - `.field` — поле объекта; `.a.b.c` — вложенное;
  - `.items[]` — РАЗВЕРНУТЬ массив в поток элементов;
  - `|` — конвейер фильтров (как в bash);
  - `select(условие)` — пропустить только подходящие элементы;
  - `{a, b}` — собрать новый объект из полей.
- **`-r` (raw)** — печатать строки без кавычек: обязателен, когда результат
  уходит в bash-переменные или другие утилиты.
- **`-c` (compact)** — каждый результат одной строкой: удобно для grep/wc.
- **`-e`** — код возврата по результату (null/false → exit 1): для if'ов
  в скриптах.

---

**Цель:** вытащить из ответа API ровно то, что нужно, одной командой.

---

### 3.1 Pretty-print и навигация по полям

```bash
# Просто красиво напечатать (главное применение №1)
curl -s http://localhost:8080/api/v1/tickets/1 | jq .
# {
#   "id": 1,
#   "title": "VPN не подключается из дома",
#   ...
# }

# Достать одно поле
curl -s http://localhost:8080/api/v1/tickets/1 | jq -r '.title'
# VPN не подключается из дома

# Несколько полей -> новый маленький объект
curl -s http://localhost:8080/api/v1/tickets/1 | jq '{id, status, priority}'
# {
#   "id": 1,
#   "status": "open",
#   "priority": "high"
# }
```

### 3.2 Массивы, пайпы, select

```bash
# .items[] разворачивает массив, дальше работаем с каждым элементом
curl -s 'http://localhost:8080/api/v1/tickets?per_page=3' | jq -r '.items[].title'
# VPN не подключается из дома
# Принтер на 3 этаже печатает пустые листы
# Не приходят письма от Jira

# select: только критичные тикеты (-c: по одному на строку)
curl -s 'http://localhost:8080/api/v1/tickets?per_page=50' \
  | jq -c '.items[] | select(.priority=="critical") | {id, title}'
# {"id":4,"title":"Ошибка 500 при выгрузке отчёта из CRM"}

# Сколько элементов пришло НА ЭТОЙ странице (не путать с .total!)
curl -s 'http://localhost:8080/api/v1/tickets?per_page=50' | jq '.items | length'
# 8
```

> `.total` — сколько всего нашлось на сервере; `.items | length` — сколько
> приехало в этом ответе. Расхождение = пагинация: данные есть, но на
> следующих страницах. Инцидент «API теряет тикеты» часто оказывается
> «клиент читает только первую страницу».

**Контрольные вопросы:**

## Часть 4: jq-рецепты для смены в поддержке

### Теория для изучения перед частью

- На смене L2 типовые вопросы к тикет-системе звучат так: «сколько открытых?»,
  «что в работе по критичным?», «выгрузи список для отчёта». Всё это — одна
  команда curl + jq, если знать три приёма:
  - **`group_by(.field)`** — сгруппировать массив (после него обычно
    `map({(.[0].field): length}) | add` — посчитать по группам);
  - **`[.a, .b] | @tsv`** — собрать табличную строку (для Excel/отчёта);
  - **`--arg name value`** — передать bash-переменную внутрь фильтра
    БЕЗОПАСНО (без подстановки строк в фильтр).

---

**Цель:** собрать сводку по тикетам и табличную выгрузку — заготовки для
ежедневного отчёта смены.

---

```bash
# Рецепт 1: сводка «сколько в каком статусе» (основа отчёта смены)
curl -s 'http://localhost:8080/api/v1/tickets?per_page=50' \
  | jq '.items | group_by(.status) | map({(.[0].status): length}) | add'
# {
#   "closed": 1,
#   "in_progress": 2,
#   "open": 4,
#   "resolved": 1
# }

# Рецепт 2: табличная выгрузка открытых тикетов (id, приоритет, тема)
curl -s 'http://localhost:8080/api/v1/tickets?status=open&per_page=50' \
  | jq -r '.items[] | [.id, .priority, .title] | @tsv'
# 1	high	VPN не подключается из дома
# 2	low	Принтер на 3 этаже печатает пустые листы
# 5	medium	Нужен доступ к репозиторию gitlab
# 7	medium	Медленно открывается корпоративный портал

# Рецепт 3: параметр из bash-переменной — только через --arg
WANTED=critical
curl -s 'http://localhost:8080/api/v1/tickets?per_page=50' \
  | jq -r --arg p "$WANTED" '.items[] | select(.priority==$p) | .title'
# Ошибка 500 при выгрузке отчёта из CRM
```

> Почему `--arg`, а не `select(.priority==\"$WANTED\")` строкой? Если в
> переменной окажутся кавычки или скобки — фильтр сломается (а в худшем
> случае выполнит не то). `--arg` передаёт значение как данные, не как код —
> та же идея, что подготовленные выражения в SQL.

## Часть 5: Писать JSON для POST-запросов

### Теория для изучения перед частью

- **Три способа собрать JSON-тело**, от простого к надёжному:
  1. **строка в `-d '...'`** — для простых случаев; помните: снаружи
     одинарные кавычки bash, внутри — двойные JSON;
  2. **файл**: `-d @body.json` — тело лежит в файле, его можно
     валидировать заранее;
  3. **`jq -n --arg`** — генерация: jq сам экранирует кавычки, переводы
     строк и кириллицу. Единственный надёжный способ, когда в значениях
     данные пользователя.

---

**Цель:** научиться собирать корректное JSON-тело даже из «грязных» данных.

---

```bash
# Способ 3 в действии: текст с кавычками внутри — руками такое не наэкранируешь
jq -n --arg title 'Кавычки "внутри" и кириллица' --arg prio high \
  '{title: $title, priority: $prio}'
# {
#   "title": "Кавычки \"внутри\" и кириллица",
#   "priority": "high"
# }

# Сразу в API: jq генерирует тело, curl читает его со stdin (-d @-)
jq -n --arg title 'Сообщение об ошибке: disk "sda" full' \
  '{title: $title, priority: "high"}' \
  | curl -s -X POST http://localhost:8080/api/v1/tickets \
      -H 'Content-Type: application/json' -d @- | jq '{id, title}'
# {
#   "id": 9,
#   "title": "Сообщение об ошибке: disk \"sda\" full"
# }
```

**Контрольные вопросы:**
1. Почему тело `-d '{"title": "..."}'` оборачивают в одинарные кавычки bash?
2. В каком случае ручная сборка JSON-строки гарантированно сломается и чем
   её заменить?
3. Что означает `-d @body.json` и `-d @-`?

## Шпаргалка

```bash
# === Валидация ===
python3 -m json.tool file.json        # валидно? + где ошибка
jq . file.json                        # то же + pretty-print
jq -e . >/dev/null && echo VALID      # для if'ов в скриптах

# === Чтение ===
... | jq .                            # красиво напечатать
... | jq -r '.field'                  # одно поле, без кавычек
... | jq '.items[]'                   # развернуть массив
... | jq '.items[] | select(.status=="open")'
... | jq '{id, title}'                # выбрать подмножество полей
... | jq '.items | length'            # размер массива
... | jq 'keys'                       # какие поля есть у объекта
... | jq 'has("field")'               # поле есть? (vs null!)

# === Отчёты ===
... | jq '.items | group_by(.status) | map({(.[0].status): length}) | add'
... | jq -r '.items[] | [.id, .priority, .title] | @tsv'
... | jq -r --arg p high '.items[] | select(.priority==$p) | .title'

# === Генерация ===
jq -n --arg t "любой текст" '{title: $t}'          # безопасная сборка
jq -n ... | curl -s -X POST URL -H 'Content-Type: application/json' -d @-
curl ... -d @body.json                              # тело из файла
```
