## Часть 1: Как системы интегрируются

### Теория для изучения перед частью

- **Два базовых паттерна обмена** между системами:

  | | Polling (опрос) | Webhook (уведомление) |
  |---|---|---|
  | Кто инициирует | потребитель (`GET /tickets?updated_since=…` по cron) | источник (POST на URL потребителя при событии) |
  | Задержка | до интервала опроса | секунды |
  | Нагрузка | постоянная, даже без изменений | только при событиях |
  | Надёжность | потребитель сам контролирует | если потребитель лежал — событие можно ПОТЕРЯТЬ |
  | Типовой сбой | упёрлись в rate limit (м05) | «вебхуки молча перестали приходить» |

- **Webhook — это «API наоборот»:** Helpdesk сам делает POST на URL внешней
  системы (`{"event": "ticket.created", "ticket": {...}}`). Для приёмника
  это обычный входящий HTTP-запрос — всё, что вы умеете (статус-коды,
  Content-Type, тела), работает и тут.
- **Кто отвечает за недоставку** — главный вопрос при разборе: хорошие
  источники ретраят с backoff и ведут журнал доставок; плохие — шлют один
  раз и забывают (наш стенд — нарочно «плохой»: одна попытка + строка в
  логе).
- **Сцепка систем всегда асимметрична:** при разборе инцидента сначала
  установите, КТО инициатор и ГДЕ лог доставки — на источнике, а лог
  приёма — на приёмнике.

---

**Контрольные вопросы:**
1. Polling vs webhook: кто инициатор, какая задержка, что с надёжностью?
2. Приёмник вебхуков лежал 2 часа. Что произошло с событиями за это время
   (два возможных мира — от чего зависит)?
3. На какой стороне искать «вебхук не дошёл»: логи кого смотреть первыми?

## Часть 2: Вебхуки на практике

### Теория для изучения перед частью

- Наш стенд шлёт события `ticket.created`, `ticket.status_changed`,
  `ticket.deleted` POST'ом на `WEBHOOK_URL` — асинхронно, чтобы не
  тормозить ответ API (как делают реальные системы).
- Приёмник `webhook_sink.py` сохраняет всё принятое и отдаёт журнал:
  `GET :9100/received` — по нему доказываем доставку.
- Лог отправителя — `scripts/api.sh logs`: строки `[webhook] ... 200`
  (доставлено) либо `[webhook] DELIVERY FAILED ...` (нет).

---

**Цель:** увидеть полный путь события и оба лога — доставки и приёма.

---

### 2.1 Доставка события

```bash
# Создаём тикет — Helpdesk должен уведомить «внешнюю систему»
curl -s -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' -d '{"title":"Вебхук-тест"}' | jq -c '{id,title}'
# {"id":9,"title":"Вебхук-тест"}

# Журнал приёмника: событие дошло
curl -s http://localhost:9100/received | jq '.deliveries[-1]'
# {
#   "received_at": "2026-06-11T01:12:04Z",
#   "path": "/hook",
#   "payload": {
#     "event": "ticket.created",
#     "ticket": {"id": 9, "title": "Вебхук-тест", ..., "status": "open", ...},
#     "sent_at": "2026-06-11T01:12:04Z"
#   }
# }

# Лог отправителя подтверждает доставку:
scripts/api.sh logs 3
# [webhook] ticket.created -> http://127.0.0.1:9100/hook 200

# Смена статуса -> второе событие
curl -s -X PATCH http://localhost:8080/api/v1/tickets/9 \
  -H 'Content-Type: application/json' -d '{"status":"in_progress"}' >/dev/null
curl -s http://localhost:9100/received | jq -r '.count, .deliveries[-1].payload.event'
# 2
# ticket.status_changed
```

### 2.2 Сломанная доставка

```bash
# «Внешняя система упала»
scripts/api.sh sink-down

# Helpdesk при этом работает — пользователь создаёт тикет как обычно:
curl -s -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' -d '{"title":"Уйдёт ли вебхук?"}' | jq .id
# 10        <- API ответил 201, пользователь ничего не заметил!

# ...а вот интеграция — нет. Доказательство в логе ОТПРАВИТЕЛЯ:
scripts/api.sh logs 3
# [webhook] DELIVERY FAILED ticket.created -> http://127.0.0.1:9100/hook:
#           <urlopen error [Errno 111] Connection refused>

# Поднимаем приёмник обратно. ВАЖНО: пропущенное событие НЕ приедет —
# наш источник не ретраит. Тикет 10 для внешней системы не существует.
scripts/api.sh sink-up
curl -s http://localhost:9100/received | jq .count
# 0         <- журнал приёмника пуст (процесс перезапущен, событие потеряно)
```

> Это анатомия инцидента «в мессенджер перестали падать заявки»: основная
> система живёт, пользователи довольны, а интеграция тихо мертва. Поэтому
> доставку вебхуков мониторят отдельно, а потерянное досылают вручную или
> синхронизацией (polling как страховка).

**Контрольные вопросы:**
1. Почему API ответил 201, хотя вебхук не доставился? Хорошо это или плохо?
2. Какие две строки лога доказывают доставку и недоставку?
3. Приёмник подняли — придут ли пропущенные события? От чего это зависит
   и как «долить» пропущенное?

## Часть 3: Ретраи, дубли и идемпотентность

### Теория для изучения перед частью

- **Таймаут ≠ неуспех.** Если клиент отвалился по таймауту на POST, он
  НЕ ЗНАЕТ, выполнился ли запрос (мог выполниться после его ухода).
  Наивный ретрай в этой ситуации создаёт **дубль** — отсюда инциденты
  «каждая заявка заводится дважды».
- **Идемпотентность спасает:** клиент шлёт заголовок **`Idempotency-Key:
  <уникальная-строка>`**; сервер запоминает ответ по ключу и на повтор
  возвращает ЕГО ЖЕ, не создавая второй ресурс. Ключ генерирует клиент —
  один на ЛОГИЧЕСКУЮ операцию (не на попытку!).
- Где встречается: платёжные API (Stripe), создание заказов/заявок.
  Если API идемпотентность не поддерживает — дубли разгребает поддержка
  (дедупликация по полям + период).

---

**Цель:** руками создать дубль, а потом победить его идемпотентным ключом.

---

```bash
# БЕЗ ключа: два одинаковых POST = два тикета (классический дубль)
for i in 1 2; do
  curl -s -X POST http://localhost:8080/api/v1/tickets \
    -H 'Content-Type: application/json' \
    -d '{"title":"Дубль без ключа"}' | jq .id
done
# 11
# 12        <- две заявки об одном и том же

# С ключом: повтор возвращает ПЕРВЫЙ ответ, дубль не создаётся
for i in 1 2; do
  curl -s -i -X POST http://localhost:8080/api/v1/tickets \
    -H 'Content-Type: application/json' \
    -H 'Idempotency-Key: retry-demo-1' \
    -d '{"title":"Дубль при ретрае?"}' | grep -E 'HTTP|Idempotency-Replayed|"id"'
done
# HTTP/1.1 201 Created
# {"id":13,"title":"Дубль при ретрае?",...}
# HTTP/1.1 201 Created
# Idempotency-Replayed: true      <- сервер говорит: это повтор, отдаю сохранённое
# {"id":13,...}                   <- ТОТ ЖЕ id

curl -s 'http://localhost:8080/api/v1/tickets?q=Дубль при ретрае' --get \
  --data-urlencode 'q=Дубль при ретрае' | jq .total
# 1
```

**Контрольные вопросы:**
1. Почему после таймаута POST нельзя просто повторить запрос?
2. Как работает Idempotency-Key с обеих сторон — клиента и сервера?
3. Один ключ — на операцию или на попытку? Что будет, если перепутать?
4. API без идемпотентности дублирует заявки при ретраях. Какие меры
   доступны на стороне поддержки/потребителя?

## Часть 4: Боевые инциденты

### Теория для изучения перед частью

- **Метод:** каждый инцидент гоняем по циклу
  **Воспроизведение → Диагностика → Решение → Профилактика** (как в
  k8s-модуле 01). Поломка включается ручкой стенда
  `POST /api/v1/_lab/fault {"mode": "..."}` и выключается `{"mode":"none"}`.
- **Главная таблица модуля — сортировка инцидента по симптому:**

  | Симптом | Слой | Первая команда | Типовая причина |
  |---|---|---|---|
  | curl exit 7, code=000 | сеть | `curl -v` | сервис не слушает / firewall |
  | curl exit 28 (таймаут) | сервис | `curl --max-time + /health` | перегрузка, блокировки, медленный бэкенд |
  | 5xx | сервер | логи сервера | баг/деплой/зависимость упала |
  | 200, но парсер падает | данные | `jq .` на тело | обрезанный/битый ответ |
  | 200, но клиент отказывается | контракт | `-D -` (заголовки) | Content-Type, кодировка |
  | 4xx | клиент | модулі 01/05 | запрос/токен/права |

- `/health` остаётся зелёным при всех наших поломках — это нарочно:
  **частичная деградация** («мониторинг ок, пользователи жалуются») —
  самый частый и самый коварный класс инцидентов.

---

### Инцидент 1: API «висит» — таймауты

**Жалоба:** «выгрузка заявок зависает, иногда по минуте, потом отваливается».

**Воспроизведение:**

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"slow"}'
# {"fault":"slow"}
```

**Диагностика:**

```bash
# 1) Ограничиваем ожидание и смотрим exit-код
time curl -s --max-time 5 http://localhost:8080/api/v1/tickets ; echo "exit=$?"
# real    0m5.009s
# exit=28        <- ТАЙМАУТ: сервис слушает, но не отвечает вовремя

# 2) А health при этом мгновенный:
curl -s -o /dev/null -w '%{http_code} %{time_total}s\n' http://localhost:8080/health
# 200 0.001s     <- частичная деградация: жив, но конкретный путь тормозит

# 3) Доказательная база для эскалации: тайминги по фазам
curl -s -o /dev/null --max-time 20 \
  -w 'connect=%{time_connect}s ttfb=%{time_starttransfer}s total=%{time_total}s\n' \
  http://localhost:8080/api/v1/tickets
# connect=0.000s ttfb=15.003s total=15.003s
#         ^ соединение мгновенное, ВСЁ время — ожидание ответа: тормозит приложение, не сеть
```

**Решение/обход:** на стороне L2 — выставить клиентам разумный
`--max-time`, собрать тайминги, эскалировать с цифрами (Часть 5).
Здесь — выключаем поломку:

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"none"}'
```

**Профилактика:** таймауты везде (клиент без таймаута виснет вечно);
разделять connect-timeout и read-timeout; мониторить не только /health,
но и латентность боевых путей.

### Инцидент 2: 500 Internal Server Error

**Жалоба:** «при любом обращении к заявкам — ошибка».

**Воспроизведение:**

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"error500"}' >/dev/null
```

**Диагностика:**

```bash
curl -s -i http://localhost:8080/api/v1/tickets | head -1
# HTTP/1.1 500 Internal Server Error
curl -s http://localhost:8080/api/v1/tickets | jq -c .error
# {"code":"internal_error","message":"unexpected error, see server logs"}
#                                     ^ сервер сам говorит: смотрите ЕГО логи

# 5xx => вина сервера. Запрос менять бессмысленно. Идём в логи:
scripts/api.sh logs 5
# [2026-06-11T01:14:10Z] 127.0.0.1 "GET /api/v1/tickets HTTP/1.1" 500 -
# Это access-лог; в реальной системе рядом был бы stacktrace приложения.

# Фиксируем масштаб: ВСЕ ли операции? (важно для эскалации)
curl -s -o /dev/null -w 'tickets: %{http_code}\n' http://localhost:8080/api/v1/tickets
curl -s -o /dev/null -w 'health:  %{http_code}\n' http://localhost:8080/health
# tickets: 500
# health:  200
```

**Решение/обход:** 5xx чинит владелец сервиса (L3) — задача L2 собрать
запрос/ответ/время/масштаб и эскалировать. Выключаем:

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"none"}' >/dev/null
```

**Профилактика:** алерт на долю 5xx; время начала 500-х сверять с
деплоями/изменениями (первый вопрос L3: «что меняли?»).

### Инцидент 3: 200 OK, но клиент падает

**Жалоба:** «у вас всё работает (код 200!), а наша интеграция падает с
parse error».

**Воспроизведение:**

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"badjson"}' >/dev/null
```

**Диагностика:**

```bash
# Код действительно 200:
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8080/api/v1/tickets
# 200

# Но прогоните тело через валидатор:
curl -s http://localhost:8080/api/v1/tickets | jq .
# jq: parse error: Unfinished string at EOF at line 1, column 39

# Смотрим сырое тело — ответ ОБРЕЗАН на полуслове:
curl -s http://localhost:8080/api/v1/tickets
# {"items":[{"id":1,"title":"VPN ne podkl
```

**Урок инцидента:** **«200 OK» — про транспорт, не про данные.** Клиент
обязан валидировать тело; диагност — смотреть сырое тело, а не только код.
Причины в реальности: обрыв соединения на прокси, переполненный буфер,
баг сериализации. Выключаем (`{"mode":"none"}`).

**Профилактика:** в smoke-наборах (модуль 04) проверять не только статус,
но и парсинг тела + ключевые поля — наши Postman-тесты так и написаны.

### Инцидент 4: «у нас всё зелёное», а интеграция не парсит

**Жалоба:** «ваш API отдаёт данные (мы глазами видим JSON!), а наша
библиотека говорит unsupported content type».

**Воспроизведение:**

```bash
curl -s -X POST http://localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"wrongct"}' >/dev/null
```

**Диагностика:**

```bash
# Тело выглядит нормальным JSON'ом... а ЗАГОЛОВОК?
curl -s -i http://localhost:8080/api/v1/tickets | grep -E 'HTTP|Content-Type'
# HTTP/1.1 200 OK
# Content-Type: text/html; charset=utf-8     <- вот и причина!

# jq «прощает» (он смотрит только на тело), а строгие HTTP-клиенты
# (Java/C#/Python requests с raise_for_status+json) — нет:
# они доверяют заголовку и отказываются парсить text/html как JSON.
```

**Урок инцидента:** расхождение «тело vs заголовок» ломает строгих
клиентов. Если глазами «всё ок», а у интеграции — нет, сверяйте
ЗАГОЛОВКИ (`-D -`), а не только тело. Выключаем (`{"mode":"none"}`).

**Профилактика:** контрактные проверки в smoke: и код, и Content-Type,
и парсинг (см. тест «есть заголовок X-Total-Count» в нашей коллекции —
та же идея).

**Контрольные вопросы (по всем инцидентам):**
1. Восстановите таблицу «симптом → слой → первая команда» по памяти.
2. Чем exit-код 28 отличается от 7 и какие гипотезы у каждого?
3. Почему «200 OK» не гарантирует пригодных данных? Два примера из лабы.
4. health 200, боевой путь 500 — как называется это состояние и почему
   оно опасно для мониторинга?
5. Какие фазы показывает `-w 'connect=… ttfb=…'` и как по ним отличить
   «тормозит сеть» от «тормозит приложение»?

## Часть 5: Эскалация в L3

### Теория для изучения перед частью

- **Эскалация — это передача дела, а не жалобы.** Хорошая эскалация
  позволяет L3 начать чинить, не переспрашивая. Шаблон:

  ```markdown
  ## Симптом
  Что и у кого не работает, с какого времени (TZ!), масштаб (все/часть).

  ## Воспроизведение
  Точная команда (curl с заголовками, без секретов!) и фактический ответ.

  ## Ожидаемо / Факт
  Что должно быть (по доке) и что наблюдаем.

  ## Что уже проверено
  health, другие эндпоинты, токен/права, с какого момента сломалось,
  что менялось. Каждый пункт — с доказательством (вывод команды).

  ## Workaround
  Есть/нет; если есть — какой и для кого включён.
  ```

- **Чего НЕ должно быть:** секретов (модуль 05!), «не работает, посмотрите»,
  скриншотов текста вместо текста, выводов без команд, которыми получены.
- L2 закрывает 4xx-классы сам; эскалирует обоснованные 5xx, таймауты
  серверной стороны и расхождения контракта (инциденты 2–4).

## Шпаргалка

```bash
# === Вебхуки ===
scripts/api.sh sink-up; WEBHOOK_URL=http://127.0.0.1:9100/hook scripts/api.sh up
curl -s http://localhost:9100/received | jq '.count, .deliveries[-1].payload.event'
scripts/api.sh logs 20 | grep webhook        # 200 = доставлено, DELIVERY FAILED = нет

# === Идемпотентность ===
curl -X POST URL -H 'Idempotency-Key: <op-id>' ...   # повтор НЕ создаст дубль
# ключ: один на ОПЕРАЦИЮ, не на попытку

# === Управление поломками стенда ===
curl -s -X POST localhost:8080/api/v1/_lab/fault -H 'Content-Type: application/json' \
  -d '{"mode":"slow|error500|badjson|wrongct|none"}'

# === Дерево диагностики ===
curl -s -o /dev/null -w '%{http_code}\n' --max-time 5 URL ; echo $?
#   exit 7  -> не слушает (сеть/процесс)        exit 28 -> таймаут (перегрузка)
#   000     -> HTTP не состоялся                5xx     -> логи сервера, эскалация
#   200 + jq parse error -> битое тело          200 + не тот Content-Type -> -D -
curl -s -o /dev/null -w 'connect=%{time_connect}s ttfb=%{time_starttransfer}s\n' URL
#   connect большой -> сеть; ttfb большой -> приложение

# === Эскалация (разделы) ===
# Симптом / Воспроизведение / Ожидаемо-Факт / Что уже проверено / Workaround
```
