> ⏱ время ~60 мин · сложность 1/5 · пререквизиты: базовый Linux-терминал

Цель: научиться видеть HTTP-обмен целиком — метод, путь, заголовки, тело,
статус-код — и уверенно отправлять/проверять GET/POST-запросы из curl.
К концу модуля вы самостоятельно чините скрипт, который «должен создавать
заявку, но не работает», по одним только ответам сервера.

Это фундамент всей работы L2: тикеты «интеграция не работает» почти всегда
сводятся к вопросу «какой запрос ушёл и какой ответ пришёл?».

## Часть 1: Анатомия HTTP-обмена

### Теория для изучения перед частью

- **HTTP — это текстовый протокол «запрос → ответ»** поверх TCP. Клиент
  (curl, браузер, Postman, чужой бэкенд) открывает соединение, пишет текст
  запроса, сервер отвечает текстом ответа. Никакой магии — всё можно
  прочитать глазами.
- **Запрос** состоит из: *строки запроса* (`GET /api/v1/tickets HTTP/1.1` —
  метод, путь, версия), *заголовков* (`Host`, `User-Agent`, …; по одному на
  строку), *пустой строки* и опционального *тела* (для POST/PUT/PATCH).
- **Ответ** состоит из: *статус-строки* (`HTTP/1.1 200 OK` — версия, код,
  текст), *заголовков* (`Content-Type`, `Content-Length`, …), *пустой
  строки* и *тела* (обычно JSON).
- **URL раскладывается на части:**
  `http://localhost:8080/api/v1/tickets?status=open&page=1`
  = схема `http` + хост `localhost` + порт `8080` + путь `/api/v1/tickets`
  + query-параметры `status=open&page=1` (пары `ключ=значение` через `&`).
- **Ключевые флаги curl:** `-v` (показать ВЕСЬ обмен: `>` — что ушло,
  `<` — что пришло), `-s` (убрать прогресс-бар), `-i` (тело + заголовки
  ответа), `-o /dev/null` (выбросить тело).

---

**Цель:** один раз увидеть обмен целиком, чтобы дальше читать любой HTTP
не задумываясь.

---

### 1.1 Полный обмен глазами curl -v

```bash
# -v печатает служебную информацию: > строки запроса, < строки ответа
curl -sv http://localhost:8080/api/v1/tickets/1 -o /dev/null 2>&1 | grep -E '^(>|<)'
# > GET /api/v1/tickets/1 HTTP/1.1     <- СТРОКА ЗАПРОСА: метод, путь, версия
# > Host: localhost:8080               <- обязательный заголовок HTTP/1.1
# > User-Agent: curl/8.14.1            <- кто спрашивает (видно в логах сервера!)
# > Accept: */*                        <- какой формат ответа клиент готов принять
# >                                    <- пустая строка = конец заголовков, тела нет
# < HTTP/1.1 200 OK                    <- СТАТУС-СТРОКА: версия, код, расшифровка
# < Server: HelpdeskAPI/1.0 Python/3.13.5
# < Date: Wed, 10 Jun 2026 20:43:13 GMT
# < Content-Type: application/json; charset=utf-8   <- формат тела ответа
# < Content-Length: 245                <- длина тела в байтах
# <                                    <- дальше идёт тело (мы выбросили его в -o /dev/null)

# Само тело — JSON одного тикета
curl -s http://localhost:8080/api/v1/tickets/1
# {"id":1,"title":"VPN не подключается из дома","description":"","status":"open",
#  "priority":"high","requester":"anna.petrova@corp.example","assignee":null,
#  "created_at":"2026-06-01T09:00:00Z","updated_at":"2026-06-01T09:00:00Z"}
```

> Реальные API отдают JSON **одной строкой без отступов** (так меньше байт).
> Читать это глазами тяжело — поэтому в модуле 02 появится `jq`, а в
> модуле 04 — Postman, которые форматируют ответ за вас.

### 1.2 Только заголовки ответа: -i, -I, -D

```bash
# -i: заголовки + тело вместе (быстрый взгляд)
curl -s -i http://localhost:8080/health
# HTTP/1.1 200 OK
# Server: HelpdeskAPI/1.0 Python/3.13.5
# Date: Wed, 10 Jun 2026 20:43:13 GMT
# Content-Type: application/json; charset=utf-8
# Content-Length: 29
#
# {"status":"ok","uptime_s":61}

# -I: HEAD-запрос — ТОЛЬКО заголовки, сервер не шлёт тело вовсе
curl -s -I http://localhost:8080/health | head -3
# HTTP/1.1 200 OK
# Server: HelpdeskAPI/1.0 Python/3.13.5
# Date: Wed, 10 Jun 2026 20:43:13 GMT

# -D <файл>: сохранить заголовки в файл, тело — отдельно. Главный приём,
# когда нужно приложить доказательства к тикету/эскалации.
curl -s -D /tmp/api-lab/m01-headers.txt -o /dev/null http://localhost:8080/api/v1/tickets
head -2 /tmp/api-lab/m01-headers.txt
# HTTP/1.1 200 OK
# Server: HelpdeskAPI/1.0 Python/3.13.5
```

**Контрольные вопросы:**
1. Из каких четырёх частей состоит HTTP-запрос? Что отделяет заголовки от тела?
2. В выводе `curl -v` что означают префиксы `>` и `<`?
3. Разложите URL `http://api.corp.ru:8443/v2/users?active=true&page=2` на
   схему, хост, порт, путь и query-параметры.
4. Чем `curl -I` отличается от `curl -i`? Какой метод реально уходит на
   сервер при `-I`?
5. Зачем в HTTP/1.1 обязателен заголовок `Host`, если адрес сервера и так
   известен из URL? *(подсказка: на одном IP может жить много сайтов)*

## Часть 2: Методы HTTP

### Теория для изучения перед частью

- **Метод — это глагол:** что сделать с ресурсом по данному пути.
  В REST-API сложилось соответствие с CRUD:

  | Метод | CRUD | Семантика | Тело запроса | Безопасный* | Идемпотентный** |
  |---|---|---|---|---|---|
  | GET | Read | прочитать ресурс | нет | да | да |
  | POST | Create | создать ресурс | да | нет | **нет** |
  | PUT | Update | полностью заменить | да | нет | да |
  | PATCH | Update | изменить часть полей | да | нет | нет |
  | DELETE | Delete | удалить | нет | нет | да |
  | HEAD | Read | как GET, но без тела | нет | да | да |

  \* безопасный = не меняет состояние сервера;
  \*\* идемпотентный = повтор запроса даёт тот же результат, что и один запрос.
- **Почему GET не должен ничего менять:** GET-запросы кэшируются, повторяются
  прокси и «предзагружаются» браузерами. Если `GET /delete?id=5` удаляет
  запись — её удалит и поисковый робот, прошедший по ссылке.
- **Почему POST не идемпотентен:** два одинаковых POST = два созданных
  тикета. Отсюда классический инцидент L2 «интеграция дублирует заявки»
  (разберём в модуле 06).
- **В curl метод задаётся `-X`,** а `-d '<данные>'` добавляет тело
  (и автоматически переключает метод на POST, если `-X` не указан).

---

**Цель:** отправить GET и POST (ядро требования вакансии) и увидеть разницу
их семантики на живой тикет-системе.

---

### 2.1 GET — прочитать

```bash
# Коллекция ресурсов: список тикетов (стенд отдаёт постранично, по 5)
curl -s http://localhost:8080/api/v1/tickets | head -c 300; echo
# {"items":[{"id":1,"title":"VPN не подключается из дома",...

# Один ресурс: тикет по id
curl -s http://localhost:8080/api/v1/tickets/4
# {"id":4,"title":"Ошибка 500 при выгрузке отчёта из CRM","description":"",
#  "status":"in_progress","priority":"critical",...}

# Query-параметры — фильтрация без изменения данных (GET безопасен!)
curl -s 'http://localhost:8080/api/v1/tickets?status=open&per_page=50' | head -c 200; echo
# {"items":[{"id":1,...  <- только открытые
```

> Кавычки вокруг URL с `&` обязательны: без них shell разорвёт команду —
> `&` в bash означает «запустить в фоне». Классическая ошибка новичка:
> «фильтр не работает», а на самом деле параметр просто не дошёл до сервера.

### 2.2 POST — создать

```bash
# Создаём тикет: метод POST, заголовок о формате тела, само тело — JSON
curl -s -i -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' \
  -d '{"title":"Не работает второй монитор","priority":"high","requester":"user@corp.example"}'
# HTTP/1.1 201 Created                 <- НЕ просто 200: создан новый ресурс
# Location: /api/v1/tickets/9          <- ГДЕ он теперь живёт
# Content-Type: application/json; charset=utf-8
#
# {"id":9,"title":"Не работает второй монитор","description":"","status":"open",
#  "priority":"high","requester":"user@corp.example","assignee":null,
#  "created_at":"2026-06-10T20:43:13Z","updated_at":"2026-06-10T20:43:13Z"}

# Проверяем результат POST'а GET'ом — по пути из Location
curl -s http://localhost:8080/api/v1/tickets/9 | head -c 120; echo
# {"id":9,"title":"Не работает второй монитор",...
```

> Цикл «**POST → 201 → Location → GET**» — это и есть «уметь отправлять и
> проверять POST/GET-запросы» из вакансии. Отправили, получили подтверждение
> создания, прочитали обратно — убедились, что данные легли как надо.

### 2.3 DELETE и HEAD

```bash
# Удаляем тикет 8 («Слетела лицензия Office», закрыт)
curl -s -i -X DELETE http://localhost:8080/api/v1/tickets/8
# HTTP/1.1 204 No Content              <- успех, но телу взяться неоткуда
# Content-Length: 0

# Повтор DELETE: ресурса уже нет
curl -s -X DELETE http://localhost:8080/api/v1/tickets/8
# {"error":{"code":"not_found","message":"тикет 8 не существует"}}
```

> DELETE идемпотентен по *состоянию* (после одного и после пяти DELETE
> тикета одинаково нет), хотя *ответ* на повтор другой (404). Это нормальная
> и частая реализация.

**Контрольные вопросы:**

2. Что обязательно нужно добавить к POST-запросу с JSON-телом, кроме самого
   тела? *(два пункта: флаг метода не в счёт)*
3. POST вернул `201 Created`. Как узнать, по какому URL теперь доступен

4. Какие методы из таблицы идемпотентны? Почему для интеграций это важно?
5. Вы выполнили `curl 'http://api/tickets?status=open&page=2'` без кавычек

## Часть 3: Статус-коды

### Теория для изучения перед частью

- **Первая цифра кода = класс ответа.** Это первое, на что смотрит инженер
  поддержки в логах:

  | Класс | Смысл | Чья «вина» | Типичные коды |
  |---|---|---|---|
  | 2xx | успех | — | 200 OK, 201 Created, 204 No Content |
  | 3xx | иди в другое место | — | 301 Moved Permanently, 302 Found, 304 Not Modified |
  | 4xx | ошибка **клиента** | запрос составлен неверно | 400, 401, 403, 404, 405, 415, 422, 429 |
  | 5xx | ошибка **сервера** | запрос нормальный, сервер сломан | 500, 502, 503, 504 |

- **Каталог 4xx, который нужно знать наизусть:**

  | Код | Название | Когда возникает | Что проверять |
  |---|---|---|---|
  | 400 | Bad Request | тело запроса не разобрано (битый JSON) | синтаксис тела |
  | 401 | Unauthorized | сервер не понял, КТО вы | токен/ключ (модуль 05) |
  | 403 | Forbidden | понял кто, но НЕЛЬЗЯ | права/роль (модуль 05) |
  | 404 | Not Found | нет такого пути или ресурса | опечатка в URL, id |
  | 405 | Method Not Allowed | путь есть, метод — нет | метод; заголовок `Allow` в ответе |
  | 415 | Unsupported Media Type | не тот `Content-Type` | заголовки запроса |
  | 422 | Unprocessable Entity | JSON валиден, но данные не прошли проверку | поля тела против документации |
  | 429 | Too Many Requests | слишком часто | rate limit (модуль 05) |

- **4xx против 5xx — главный сортировочный вопрос L2.** 4xx чинится на
  стороне клиента (исправить запрос/токен/URL), 5xx — повод смотреть логи
  сервера и эскалировать (модуль 06).

---

**Цель:** получить руками каждый важный код и запомнить его «триггер».

---

### 3.1 Получить каждый класс кодов руками

```bash
# 200: обычное чтение
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8080/api/v1/tickets
# 200

# 404: опечатка в пути (/ticket вместо /tickets)
curl -s http://localhost:8080/api/v1/ticket
# {"error":{"code":"not_found","message":"нет такого пути: /api/v1/ticket"}}

# 404: путь верный, ресурса нет
curl -s http://localhost:8080/api/v1/tickets/999
# {"error":{"code":"not_found","message":"тикет 999 не существует"}}

# 405: на коллекции тикетов DELETE не определён; смотрите заголовок Allow!
curl -s -i -X DELETE http://localhost:8080/api/v1/tickets | grep -E 'HTTP|Allow'
# HTTP/1.1 405 Method Not Allowed
# Allow: GET, POST                     <- сервер сам говорит, что МОЖНО

# 415: забыли Content-Type при POST
curl -s -i -X POST http://localhost:8080/api/v1/tickets -d '{"title":"abc"}' | head -1
# HTTP/1.1 415 Unsupported Media Type

# 400: Content-Type есть, но JSON битый (одинарные кавычки!)
curl -s -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' -d "{'title': 'тест'}"
# {"error":{"code":"invalid_json","message":"тело запроса — не валидный JSON",
#  "details":{"position":"line 1 column 2","parser":"Expecting property name enclosed in double quotes"}}}

# 422: JSON корректный, но данные не проходят валидацию
curl -s -X POST http://localhost:8080/api/v1/tickets \
  -H 'Content-Type: application/json' -d '{"priority":"urgent"}'
# {"error":{"code":"validation_failed","message":"тело запроса не прошло валидацию",
#  "details":[{"field":"priority","message":"допустимые значения: low, medium, high, critical"},
#             {"field":"title","message":"обязательное поле"}]}}
```

> Запомните лестницу ошибок POST: **415** (не тот формат) → **400** (формат
> тот, но синтаксис битый) → **422** (синтаксис ок, но данные неверные) →
> **201** (всё ок). Сервер проверяет запрос именно в этом порядке, и каждый
> код точно говорит, на какой ступени вы споткнулись.

### 3.2 3xx и curl -L

```bash
# Старая версия API отвечает редиректом на новую
curl -s -i http://localhost:8080/api/v0/tickets/1 | grep -E 'HTTP|Location'
# HTTP/1.1 301 Moved Permanently
# Location: /api/v1/tickets/1          <- куда переехало

# По умолчанию curl за редиректом НЕ идёт (и это хорошо для диагностики).
# -L = follow redirects: сходить по Location автоматически
curl -s -L http://localhost:8080/api/v0/tickets/1 | head -c 80; echo
# {"id":1,"title":"VPN не подключается из дома",...
```

> Браузер и Postman ходят за редиректами молча, поэтому проблему «API
> переехал» из них не видно. curl без `-L` честно показывает 301 — иногда
> это и есть разгадка инцидента «после обновления интеграция отвалилась».

**Контрольные вопросы:**
1. Чем принципиально отличается 4xx от 5xx с точки зрения того, КТО должен
   чинить?
2. POST вернул 400, затем после правки — 422. Что вы исправили и что
   осталось исправить?
3. Какой заголовок ответа при 405 подскажет допустимые методы?

## Часть 4: Заголовки

### Теория для изучения перед частью

- **Заголовки — это метаданные обмена:** формат тела, аутентификация,
  кэширование, служебная информация. Формат: `Имя: значение`, регистр имени
  не важен (`content-type` == `Content-Type`).
- **Заголовки запроса, которые вы будете выставлять чаще всего:**
  `Content-Type: application/json` (формат МОЕГО тела),
  `Accept: application/json` (какой формат ответа я хочу),
  `Authorization: Bearer <токен>` (кто я — модуль 05),
  `User-Agent` (кто клиент — по нему вас найдут в логах сервера).
- **Content-Type против Accept** путают на собеседованиях: Content-Type
  описывает то, что отправляешь ТЫ; Accept — то, что ты готов ПОЛУЧИТЬ.
- **Кастомные заголовки** начинаются с `X-` по старой традиции
  (`X-API-Key`, `X-Total-Count`, `X-Request-Id`) — современный стандарт
  префикс не требует, но на практике `X-*` повсюду.

---

**Цель:** научиться добавлять заголовки к запросу и вычитывать полезное из
заголовков ответа.

---

### 4.1 Заголовки запроса: -H

```bash
# Каждый -H добавляет один заголовок
curl -s http://localhost:8080/api/v1/tickets/1 \
  -H 'Accept: application/json' \
  -H 'User-Agent: l2-diagnostics/1.0' \
  -o /dev/null

# Сервер ВИДИТ ваш User-Agent — найдите себя в access-логе стенда:
scripts/api.sh logs 3
# [2026-06-10T20:43:13Z] 127.0.0.1 "GET /api/v1/tickets/1 HTTP/1.1" 200 -
```

> Привычка ставить говорящий `User-Agent` в свои скрипты — золото для
> коллег: в логах сервера сразу видно, КАКАЯ интеграция шлёт запросы.

### 4.2 Заголовки ответа, которые нужно уметь читать

```bash
curl -s -D - -o /dev/null 'http://localhost:8080/api/v1/tickets?status=open'
# HTTP/1.1 200 OK
# Server: HelpdeskAPI/1.0 Python/3.13.5    <- кто отвечает (важно при разборе 502!)
# Date: Wed, 10 Jun 2026 20:43:13 GMT      <- время СЕРВЕРА (сверяйте с логами)
# Content-Type: application/json; charset=utf-8
# Content-Length: 712                       <- размер тела
# X-Total-Count: 4                          <- кастомный: всего открытых тикетов
```

| Заголовок ответа | Что говорит инженеру |
|---|---|
| `Content-Type` | как парсить тело; `text/html` вместо JSON = первый звонок (модуль 06) |
| `Content-Length` | 0 при 204 — норма; 0 при 200 — подозрительно |
| `Location` | куда идти после 201 (создано) или 3xx (переехало) |
| `Allow` | допустимые методы (приходит с 405) |
| `Retry-After` | через сколько секунд повторять (приходит с 429/503) |
| `Server`, `Date` | кто и когда ответил — критично при разборе цепочек прокси |

**Контрольные вопросы:**
1. Чем `Content-Type` отличается от `Accept`?
2. Зачем интеграционному скрипту нестандартный `User-Agent`?
3. В каких двух разных ситуациях сервер присылает `Location` и что он
   означает в каждой?
4. Ответ имеет `Content-Type: application/json`, а тело начинается с
   `<html>`. Какие выводы?

## Часть 5: curl как инструмент диагностики

### Теория для изучения перед частью

- **`-w '<формат>'`** печатает метрики обмена после завершения:
  `%{http_code}` (код), `%{time_total}` (полное время),
  `%{time_connect}` (установка TCP), `%{size_download}` (байты тела).
  Это «мини-мониторинг» из одной команды.
- **`--max-time N`** — таймаут всей операции в секундах. Без него curl может
  висеть минутами — а вместе с ним вы получаете **exit-код 28**, по которому
  скрипты понимают «сервис не отвечает» (отработаем в модуле 06).
- **Exit-код curl** (`echo $?`): `0` — обмен состоялся (даже если HTTP 500!),
  `7` — connection refused, `28` — таймаут, `6` — DNS не разрешился.
  HTTP-ошибка и сетевая ошибка — разные слои; curl различает их за вас.

---

**Цель:** собрать «болванку» проверки API, которую можно вставить в любой
скрипт мониторинга или приложить к тикету.

---

### 5.1 -w: код, тайминги, размер

```bash
# Одна строка диагностики: жив ли, как быстро, сколько отдал
curl -s -o /dev/null \
  -w 'code=%{http_code} time_total=%{time_total}s size=%{size_download}B\n' \
  http://localhost:8080/api/v1/tickets
# code=200 time_total=0.000752s size=1365B

# То же при недоступном сервисе (порт никем не занят):
curl -s -o /dev/null -w 'code=%{http_code}\n' http://localhost:9999/health
# code=000        <- 000 = HTTP-обмена НЕ БЫЛО (сетевая ошибка, не HTTP!)
echo $?
# 7               <- connection refused
```

### 5.2 --max-time: таймаут на стороне клиента

```bash
# Здоровый сервис укладывается с запасом
curl -s --max-time 5 -o /dev/null -w '%{http_code}\n' http://localhost:8080/health
# 200

# В модуле 06 включим FAULT=slow и увидим exit-код 28 (таймаут) в действии.
```

> `code=000` + exit-код — это ответ на вопрос «сервис лежит или тормозит?»
> ещё ДО чтения логов: 7 = не слушает порт, 28 = слушает, но не отвечает
> вовремя, 6 = имя не резолвится (проблема DNS, а не сервиса).

**Контрольные вопросы:**
1. curl вернул `code=000`. Был ли HTTP-запрос вообще? Где искать причину?
2. Чем отличаются exit-коды curl 7 и 28? Какие гипотезы рождает каждый?
3. HTTP 500 — какой будет exit-код curl и почему?
4. Соберите одну команду: проверить `POST /api/v1/tickets` и напечатать
   только код ответа и время.

## Шпаргалка

```bash
# === Посмотреть обмен ===
curl -v URL                          # весь диалог: > запрос, < ответ
curl -s URL                          # тихо, только тело
curl -s -i URL                       # заголовки ответа + тело
curl -s -I URL                       # HEAD: только заголовки
curl -s -D headers.txt -o body.json URL   # заголовки и тело по файлам

# === Методы ===
curl -s URL                                          # GET (по умолчанию)
curl -s -X POST URL -H 'Content-Type: application/json' -d '{"k":"v"}'
curl -s -X PATCH URL -H 'Content-Type: application/json' -d '{"k":"v"}'
curl -s -X DELETE URL

# === Заголовки и редиректы ===
curl -s -H 'Accept: application/json' -H 'X-API-Key: ...' URL
curl -s -L URL                       # идти за 3xx по Location

# === Диагностика ===
curl -s -o /dev/null -w 'code=%{http_code} t=%{time_total}s\n' URL
curl -s --max-time 5 URL ; echo $?   # 0 ок / 7 refused / 28 timeout / 6 DNS
# Лестница ошибок POST: 415 (Content-Type) -> 400 (битый JSON) -> 422 (валидация) -> 201

# === Стенд ===
scripts/api.sh up|down|status|logs
curl -s -X POST localhost:8080/api/v1/_lab/reset -H 'Content-Type: application/json' -d '{}'
```
