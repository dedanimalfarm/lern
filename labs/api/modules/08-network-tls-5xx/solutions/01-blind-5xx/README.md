# Методичка: как различить 502 / 503 / 504 / flaky вслепую

Один диагностический проход — серия из нескольких запросов с полными
заголовками:

```bash
for i in 1 2 3 4 5 6; do
  curl -s -o /dev/null -w '%{http_code} ct=%{content_type}\n' \
    http://localhost:8080/api/v1/tickets
done
curl -s -i http://localhost:8080/api/v1/tickets | head -20   # заголовки + тело
```

Дерево решений:

| Что видим | Диагноз | Почему |
|---|---|---|
| Каждый раз **502**, тело `text/html` «502 Bad Gateway / nginx» | `error502` | Балансировщик достучался до бэкенда, но получил мусор/обрыв. Ответил **не** наш сервис (формат тела чужой). |
| Каждый раз **504**, тело `text/html` «504 Gateway Time-out / nginx» | `error504` | Балансировщик **не дождался** ответа бэкенда за таймаут. Тоже ответ LB, а не приложения. |
| Каждый раз **503**, тело — наш JSON, есть `Retry-After` | `error503` | Сервис жив и сам говорит «перегружен/обслуживание, вернись через N». Единственный из трёх, кто шлёт `Retry-After`. |
| **То 200, то 500** в серии, тело 500 — наш JSON | `flaky` | Интермиттентный отказ: часть инстансов за LB сломана. На одиночном запросе можно «не поймать» — видно только по серии. |

Ключевые различители:
1. **Стабильность.** Если в серии есть и 200, и 5xx — это `flaky`. Если код
   один и тот же на каждый запрос — один из 502/503/504.
2. **Content-Type.** `text/html` с «nginx» = ответил балансировщик (502/504).
   `application/json` с нашим `{"error":...}` = ответило приложение (503).
3. **Retry-After.** Прислал только 503 — приложение знает, когда вернётся.
   502/504 его не шлют: LB не знает, когда починится бэкенд.

> На собеседовании это и есть ответ на «чем 502 отличается от 503 и 504»:
> **502** — бэкенд ответил мусором/оборвал; **504** — бэкенд не ответил
> вовремя; **503** — бэкенд жив, но временно отказывает и сам зовёт назад.
