## Часть 1: Сеть ниже HTTP — curl exit-коды

### Теория для изучения перед частью

Прежде чем сервер вернёт *любой* статус-код, должны случиться три вещи:
имя резолвится в IP (**DNS**), устанавливается **TCP-соединение**, и при
HTTPS — **TLS-рукопожатие**. Если упало здесь, HTTP-кода **нет вообще** —
есть только код выхода `curl`. L1 часто путает «нет ответа» с «ошибка
сервера»; L2 обязан различать слой.

| exit | значение | где сломалось | первая гипотеза |
|---|---|---|---|
| 6 | Couldn't resolve host | DNS | опечатка в хосте, DNS-сервер, VPN |
| 7 | Failed to connect | TCP | сервис не слушает, firewall, не тот порт |
| 28 | Operation timed out | ожидание | перегрузка, висящий бэкенд, сеть |
| 35 | SSL connect error | TLS-рукопожатие | протокол/шифры, MITM-прокси |
| 60 | SSL cert problem | проверка сертификата | истёк/самоподпись/не тот CN |

> `%{http_code}` при сетевом сбое равен `000` — это и есть маркер «HTTP не
> состоялся». Сам код выхода смотрите через `echo $?` сразу после curl.

### 🎤 Вопрос собеседования
> «curl пишет `Could not resolve host` / `Connection refused` / таймаут —
> в чём разница и куда смотреть в каждом случае?»

### Практика

```bash
# exit 6 — DNS
curl -s -o /dev/null http://no-such-host.invalid/ ; echo "exit=$?"        # 6

# exit 7 — порт никто не слушает
curl -s -o /dev/null http://127.0.0.1:9 ; echo "exit=$?"                  # 7

# exit 28 — таймаут: включаем медленный режим и ставим короткий --max-time
curl -s -X POST localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"slow"}'
curl -s -o /dev/null --max-time 3 localhost:8080/api/v1/tickets ; echo "exit=$?"  # 28
curl -s -X POST localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"none"}'
```

➡️ **Задание:** `tasks/01-network-tls.md`, часть А — соберите каталог
кодов 6/7/28 в `/tmp/api-lab/m08-net.txt`.

**Контрольные вопросы:**
1. Чем `Connection refused` (7) отличается от таймаута (28) по причине?
2. Почему `%{http_code}` показал `000`, хотя сервер «вроде работает»?

## Часть 2: TLS и сертификаты

### Теория для изучения перед частью

HTTPS = HTTP внутри TLS. Перед обменом данными клиент проверяет
**сертификат** сервера: подписан ли доверенным центром (CA), не истёк ли,
совпадает ли имя (CN/SAN) с хостом. Любое «нет» — рукопожатие не
состоялось, данные не пошли.

- **`-k` (`--insecure`)** отключает проверку. В диагностике — на минуту,
  чтобы отделить «проблема в сертификате» от «проблема в сервисе». В
  скрипте/интеграции «навсегда» — дыра: вы соглашаетесь говорить с кем
  угодно, кто притворится сервером (MITM).
- Сертификат смотрят **без браузера**: `openssl s_client` показывает цепочку
  и даты, `openssl x509 -noout -dates` — срок.

### 🎤 Вопрос собеседования
> «Сертификат истёк, интеграция упала. Коллега предлагает добавить `-k` и
> жить дальше — ваш ответ? Как проверить сертификат руками?»

### Практика

```bash
scripts/api.sh tls-up                 # https://127.0.0.1:8443, самоподписанный

# Без -k: ошибка доверия (exit 60) — самоподпись браузеру/CA не известна
curl -s -o /dev/null -w 'code=%{http_code}\n' https://127.0.0.1:8443/health ; echo "exit=$?"
# exit=60

# С -k: соединение идёт, но проверки нет — так в проде НЕ оставляют
curl -sk https://127.0.0.1:8443/health

# Правильный разбор: что за сертификат, чей, до какого числа
echo | openssl s_client -connect 127.0.0.1:8443 2>/dev/null \
  | openssl x509 -noout -subject -dates
# subject=CN=localhost
# notBefore=...   notAfter=...

scripts/api.sh tls-down
```

➡️ **Задание:** `tasks/01-network-tls.md`, часть Б — соберите
`/tmp/api-lab/m08-tls.txt` (поведение без `-k`, CN+срок, риск `-k`).

**Контрольные вопросы:**
1. Три причины, по которым валидный по формату сертификат всё равно
   отклоняется (срок, имя, цепочка доверия).
2. Чем `-k` опасен именно для **интеграции** (а не разового curl)?
3. Одной командой узнать дату истечения сертификата прод-хоста.

## Часть 3: 5xx за балансировщиком — кто виноват

### Теория для изучения перед частью

В проде между клиентом и приложением почти всегда стоит **балансировщик /
обратный прокси** (nginx, ELB, ingress). Часть 5xx генерирует **он**, а не
ваше приложение, — и это меняет, куда смотреть:

| код | кто ответил | смысл | куда смотреть |
|---|---|---|---|
| 500 | приложение | необработанная ошибка в коде | логи приложения |
| 502 | балансировщик | бэкенд ответил мусором/оборвал | жив ли бэкенд, его логи |
| 503 | приложение/LB | временно недоступен (перегрузка, обслуживание) | `Retry-After`, автоскейл |
| 504 | балансировщик | бэкенд не ответил за таймаут | почему бэкенд медленный |

Главный различитель — **кто сформировал ответ**: тело `text/html` с
«nginx» = балансировщик (502/504); наш `application/json` `{"error":...}` =
приложение (500/503). И только **503** шлёт `Retry-After` — приложение
само зовёт вернуться.

### 🎤 Вопрос собеседования
> «Чем 502 отличается от 503 и 504? Кто «виноват» в каждом случае? И как
> диагностировать 500, который воспроизводится у одного запроса из двадцати?»

### Практика

```bash
for m in error502 error503 error504; do
  curl -s -X POST localhost:8080/api/v1/_lab/fault \
    -H 'Content-Type: application/json' -d "{\"mode\":\"$m\"}" >/dev/null
  echo "=== $m ==="; curl -s -i localhost:8080/api/v1/tickets | head -6
done

# Интермиттентный отказ — виден только на СЕРИИ, не на одиночном запросе
curl -s -X POST localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"flaky"}' >/dev/null
for i in $(seq 1 6); do
  curl -s -o /dev/null -w '%{http_code} ' localhost:8080/api/v1/tickets
done; echo                                  # 200 200 500 200 200 500

curl -s -X POST localhost:8080/api/v1/_lab/fault \
  -H 'Content-Type: application/json' -d '{"mode":"none"}' >/dev/null
```

➡️ **Задание:** `tasks/02-5xx-triage.md` — каталог 502/503/504 с «кто
ответил» в `/tmp/api-lab/m08-5xx.txt`.

**Контрольные вопросы:**
1. Как по одному ответу понять, что 5xx сгенерировал балансировщик, а не
   приложение?
2. Почему «500 у каждого 20-го запроса» — это про **один сломанный инстанс
   за LB**, и как это подтвердить?
3. Какой из 5xx уместно ретраить и почему именно `Retry-After` делает

## Часть 4: «В curl работает, в браузере нет» — CORS

### Теория для изучения перед частью

**CORS** — это правило **браузера**, а не сервера. Когда JS со страницы
`https://portal.corp.example` дёргает API на `https://helpdesk.corp.example`
(другой origin), браузер сам решает, отдавать ли ответ скрипту — по заголовку
`Access-Control-Allow-Origin` в ответе. Нет заголовка → браузер **блокирует**
ответ, хотя сервер его честно вернул. Поэтому `curl` (он не браузер) проблему
**не видит** — классическая ловушка диагностики.

Для «не-простых» запросов (JSON-тело, `Authorization`, методы кроме
GET/POST/HEAD) браузер шлёт **preflight** — предварительный `OPTIONS`, и
если в ответ не пришли нужные `Access-Control-*`, основной запрос даже не
отправляется (в DevTools: `net::ERR_FAILED`, статус 0).

### 🎤 Вопрос собеседования
> «Пользователь: "из веб-приложения запрос падает, а вы говорите, что API
> работает". curl и правда отвечает 200. Что происходит и чья это проблема?»

### Практика

```bash
# Стенд по умолчанию CORS не разрешает. Эмулируем браузерный запрос: с Origin
curl -s -i -H 'Origin: https://portal.corp.example' \
  localhost:8080/api/v1/tickets | grep -i 'access-control' \
  || echo "  (нет Access-Control-Allow-Origin -> браузер заблокирует ответ)"

# Preflight на «не-простой» запрос: без разрешённого origin — без заголовков
curl -s -i -X OPTIONS -H 'Origin: https://portal.corp.example' \
  -H 'Access-Control-Request-Method: POST' \
  localhost:8080/api/v1/tickets | head -8

# Теперь поднимем стенд с разрешённым origin — заголовки появляются
CORS_ORIGIN='https://portal.corp.example' scripts/api.sh up >/dev/null
curl -s -i -H 'Origin: https://portal.corp.example' \
  localhost:8080/api/v1/tickets | grep -i 'access-control'
# Access-Control-Allow-Origin: https://portal.corp.example

scripts/api.sh up >/dev/null      # вернуть дефолт (CORS выключен)
```

> Вердикт инцидента: **сервер работает, дело в CORS** — на стороне API не
> хватает `Access-Control-Allow-Origin` для нужного origin (или сломан
> preflight). Это правка конфигурации API/прокси, а не «у вас браузер
> кривой».

и почините через `CORS_ORIGIN`, вердикт в `/tmp/api-lab/m08-cors.txt`.

## Часть 5: Кэш и условные запросы — ETag, 304

### Теория для изучения перед частью

`200 OK` — не единственный «успех». **304 Not Modified** означает «у тебя
уже свежая копия, тело не шлю». Механизм: сервер на `GET` отдаёт `ETag` —
версию ресурса; клиент в следующий раз присылает `If-None-Match: <ETag>`,
и если ресурс не менялся — `304` без тела (экономия трафика и времени).

Кэшировать ответ могут не только клиент, но и прокси/CDN — поэтому
«пользователь видит старые данные, а в API новые» часто решается не на
сервере, а пониманием, **где** засел кэш.

### 🎤 Вопрос собеседования
> «Что такое 304 и почему тела нет — это ошибка? Пользователь видит
> устаревшие данные, в API они уже новые. Где может кэшироваться?»

### Практика

```bash
ETAG=$(curl -s -i localhost:8080/api/v1/tickets/1 \
  | awk -F': ' 'tolower($1)=="etag"{print $2}' | tr -d '\r')
echo "ETag=$ETAG"

# Совпало -> 304 без тела
curl -s -i -H "If-None-Match: $ETAG" localhost:8080/api/v1/tickets/1 | head -1
# HTTP/1.1 304 Not Modified

# Изменим тикет -> ETag станет другим -> снова 200 с телом
curl -s -o /dev/null -X PATCH localhost:8080/api/v1/tickets/1 \
  -H 'Content-Type: application/json' -d '{"priority":"critical"}'
curl -s -i -H "If-None-Match: $ETAG" localhost:8080/api/v1/tickets/1 | head -1
# HTTP/1.1 200 OK   (старый ETag больше не совпадает)
```

**Контрольные вопросы:**

## Часть 6: Конкурентный доступ — 412 и lost update

### Теория для изучения перед частью

Два агента открыли один тикет, оба меняют. Кто сохранил последним — затёр
правки первого: **lost update**. Защита — **optimistic locking** через тот
же `ETag`: клиент присылает `If-Match: <ETag, который он читал>`. Если на
сервере версия уже другая — `412 Precondition Failed`: «перечитай, ты
работал с устаревшей копией».

### 🎤 Вопрос собеседования
> «Два оператора одновременно редактируют одну заявку. Как API не дать им
> затереть правки друг друга? Что такое 409/412?»

### Практика

```bash
OLD=$(curl -s -i localhost:8080/api/v1/tickets/2 \
  | awk -F': ' 'tolower($1)=="etag"{print $2}' | tr -d '\r')

# «Другой оператор» успел сохранить -> версия изменилась
curl -s -o /dev/null -X PATCH localhost:8080/api/v1/tickets/2 \
  -H 'Content-Type: application/json' -d '{"priority":"high"}'

# Мы пишем со СТАРЫМ If-Match -> 412, правки не затёрты
curl -s -i -X PATCH localhost:8080/api/v1/tickets/2 \
  -H "If-Match: $OLD" -H 'Content-Type: application/json' \
  -d '{"priority":"low"}' | head -1
# HTTP/1.1 412 Precondition Failed
```

**Контрольные вопросы:**

## Часть 7: Долгие операции — 202 Accepted и polling

### Теория для изучения перед частью

Не всякая работа влезает в один запрос. Выгрузка отчёта, импорт, генерация
— это минуты. Держать соединение открытым нельзя (таймауты LB, обрывы),
поэтому API отвечает **202 Accepted**: «принял, делаю», и даёт в `Location`
адрес ресурса статуса. Клиент **опрашивает** его (с учётом `Retry-After`),
пока `status` не станет `done`. Это паттерн, а не «API тормозит».

### 🎤 Вопрос собеседования
> «Что значит 202 Accepted и чем он отличается от 200? Как клиент узнаёт
> результат долгой операции?»

### Практика

```bash
LOC=$(curl -s -i -X POST localhost:8080/api/v1/exports \
  -H 'Content-Type: application/json' -d '{"format":"csv"}' \
  | awk -F': ' 'tolower($1)=="location"{print $2}' | tr -d '\r')
echo "status_url=$LOC"

# Сразу после запроса — ещё processing
curl -s "localhost:8080$LOC" | jq -c .
# {"id":1,"status":"processing"}

# Опрос до готовности
until curl -s "localhost:8080$LOC" | jq -e '.status=="done"' >/dev/null; do sleep 1; done
curl -s "localhost:8080$LOC" | jq .
# {"id":1,"status":"done","download_url":"/api/v1/exports/1/download",...}
```

➡️ **Задание:** `tasks/04-async-export.md` — довести экспорт до `done`,
сохранить в `/tmp/api-lab/m08-export.json`.

**Контрольные вопросы:**
1. Почему долгую операцию не делают синхронно одним запросом?
2. Что подсказывает клиенту, как часто опрашивать статус?
3. Чем 202 отличается от 201?

## Часть 8: Ретраи правильно — идемпотентность методов

### Теория для изучения перед частью

После таймаута/5xx хочется повторить запрос. Безопасность ретрая зависит от
**идемпотентности метода**: повтор не должен создавать новый эффект.

| метод | идемпотентен | ретрай без оглядки? |
|---|---|---|
| GET, HEAD | да | да |
| PUT, DELETE | да (результат тот же) | да |
| POST | **нет** | только с `Idempotency-Key` (модуль 06) |

Правильный ретрай — **exponential backoff + jitter**: 1с, 2с, 4с… + случайная
добавка, чтобы все клиенты не ударили одновременно (**retry storm**). Тупой
цикл «повторять сразу» добивает уже перегруженный сервис.

### 🎤 Вопрос собеседования
> «Какие методы можно ретраить не задумываясь, а какие — нет, и почему? Что
> такое exponential backoff и retry storm?»

### Практика

```bash
# POST дважды БЕЗ ключа -> два тикета (ретрай создал дубль)
curl -s -o /dev/null localhost:8080/api/v1/tickets -X POST \
  -H 'Content-Type: application/json' -d '{"title":"retry без ключа"}'
curl -s -o /dev/null localhost:8080/api/v1/tickets -X POST \
  -H 'Content-Type: application/json' -d '{"title":"retry без ключа"}'
curl -s -G localhost:8080/api/v1/tickets --data-urlencode 'q=retry без ключа' | jq '.total'  # 2

# POST дважды С Idempotency-Key -> один тикет (повтор «проигран», не создан)
K="op-$(date +%s)"
curl -s -o /dev/null localhost:8080/api/v1/tickets -X POST -H "Idempotency-Key: $K" \
  -H 'Content-Type: application/json' -d '{"title":"retry с ключом"}'
curl -s -o /dev/null localhost:8080/api/v1/tickets -X POST -H "Idempotency-Key: $K" \
  -H 'Content-Type: application/json' -d '{"title":"retry с ключом"}'
curl -s -G localhost:8080/api/v1/tickets --data-urlencode 'q=retry с ключом' | jq '.total'  # 1
```

2. Зачем в backoff добавляют jitter?
3. Интеграция после таймаута повторяет POST и плодит дубли — что предложить?

## Часть 9: Версионирование и вывод из эксплуатации — Deprecation/Sunset

### Теория для изучения перед частью

API живёт версиями (`/api/v1`, `/api/v2`). Старую версию не выключают
внезапно — её **депрекейтят**: сервер начинает слать заголовки
`Deprecation` (помечен устаревшим) и `Sunset` (дата выключения, RFC 8594),
плюс `Link: rel="successor-version"`. Интеграции, которые мониторят эти
заголовки, узнают о выводе **заранее** и успевают мигрировать.

## Часть 10: Соседние протоколы за пять минут

L2 не обязан быть экспертом, но обязан **узнавать** не-REST API и знать их
ловушки.

- **SOAP/XML** — энтерпрайз-легаси: тело в XML (`<soap:Envelope>`), один
  URL, действие — в теле/заголовке `SOAPAction`. Ошибки — в `<soap:Fault>`,
  часто при **HTTP 200**.
- **GraphQL** — один POST-эндпоинт, клиент сам описывает, какие поля нужны.
  **Главная ловушка: ошибки приходят в `200 OK`** внутри массива `errors`,
  поле `data` при этом `null`. Мониторинг «по кодам ответа» такие ошибки
  **не видит**.
- **gRPC** — бинарный протокол поверх HTTP/2, свои статус-коды (не HTTP),
  обычно server-to-server.
- **WebSocket** — постоянное двустороннее соединение (начинается с HTTP
  `Upgrade`); для real-time (чаты, нотификации).

### 🎤 Вопрос собеседования
> «Интеграция с GraphQL: мониторинг говорит "всё зелёное, 200", а
> пользователи жалуются на ошибки. Как такое возможно?»

### Практика

```bash
cat modules/08-network-tls-5xx/assets/graphql-response.json | jq '{data, errors}'
# data.ticket == null, а в errors[] — реальная ошибка NOT_FOUND,
# при том что HTTP-статус ответа был бы 200.
```

> Вывод: для GraphQL «здоровье» меряют не по HTTP-коду, а по наличию
> `errors[]` в теле. Это типовой разрыв между «мониторинг зелёный» и
> «пользователю плохо».

**Контрольные вопросы:**
1. Почему мониторинг по HTTP-кодам слеп к ошибкам GraphQL и SOAP?
2. Как с одного взгляда отличить REST-ответ от GraphQL и от SOAP?
3. Что такое `Upgrade` и зачем он WebSocket?

## Часть 11: Чтение HAR

### Теория для изучения перед частью

**HAR** (HTTP Archive) — JSON-дамп сетевой вкладки браузера (DevTools →
Network → Export HAR). Это первое, что просят у пользователя с жалобой
«в вебе не работает»: внутри — каждый запрос с заголовками, кодами и
таймингами. L2 должен уметь его читать (хотя бы `jq`).

### 🎤 Вопрос собеседования
> «Пользователь жалуется на веб-интерфейс. Что попросите и как будете
> анализировать?»

### Практика

```bash
HAR=modules/08-network-tls-5xx/assets/incident.har
# Сводка: метод, URL, статус по каждому запросу
jq -r '.log.entries[] | "\(.request.method) \(.request.url) -> \(.response.status) \(.response._error // "")"' "$HAR"
# GET  .../tickets?status=open -> 200
# POST .../tickets             -> 0  net::ERR_FAILED
```

Разбор: GET (простой запрос) прошёл `200`, а POST (не-простой, с JSON+Auth)
— `status 0`, `net::ERR_FAILED` и присутствует заголовок `Origin`. Признаки
указывают на **CORS-preflight, который не разрешён** для POST (см. часть 4):
сервер «работает», но браузер блокирует кросс-доменный POST.

## Шпаргалка

```bash
# === Слой ниже HTTP: коды выхода curl ===
curl -s -o /dev/null -w '%{http_code}\n' URL ; echo $?
#  exit 6  DNS не резолвится     exit 7  порт не слушает / firewall
#  exit 28 таймаут               exit 60 сертификат не доверен
#  http_code 000 = HTTP не состоялся (смотри exit-код!)

# === TLS без браузера ===
echo | openssl s_client -connect HOST:443 2>/dev/null | openssl x509 -noout -subject -dates
curl -sk https://HOST/...        # -k только для диагностики, не в интеграции!

# === 5xx: кто ответил ===
curl -s -i URL | head -6         # text/html «nginx» = LB (502/504); наш JSON = приложение (500/503)
# 503 шлёт Retry-After; flaky виден только на СЕРИИ запросов

# === CORS (правило браузера, не сервера) ===
curl -i -H 'Origin: https://app.example' URL | grep -i access-control
# нет Access-Control-Allow-Origin -> браузер блокирует (curl не увидит)

# === Условные запросы и конкуренция ===
ETAG=$(curl -sI URL | awk -F': ' 'tolower($1)=="etag"{print $2}' | tr -d '\r')
curl -i -H "If-None-Match: $ETAG" URL          # совпало -> 304
curl -i -X PATCH -H "If-Match: $ETAG" ... URL  # устарел -> 412

# === Длинная операция ===
LOC=$(curl -sI -X POST URL/exports ... | awk -F': ' 'tolower($1)=="location"{print $2}' | tr -d '\r')
until curl -s "$BASE$LOC" | jq -e '.status=="done"' >/dev/null; do sleep 1; done

# === Ретраи ===
# GET/PUT/DELETE — идемпотентны, ретрай ок; POST — только с Idempotency-Key
# backoff: 1,2,4,8с + jitter (против retry storm)

# === Версии ===
curl -sI URL/api/v0/... | grep -iE 'Deprecation|Sunset|Link'   # дата выключения

# === HAR ===
jq -r '.log.entries[] | "\(.request.method) \(.request.url) -> \(.response.status)"' file.har
```
