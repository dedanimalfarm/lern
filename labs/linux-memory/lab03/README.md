## Модуль 1: Что такое Hugepages и как работает THP

### Теория для изучения перед модулем

По умолчанию ядро Linux работает со стандартным размером страницы памяти — **4 КБ**.
Для процессов с огромным объемом памяти (например, СУБД с буфером 64 ГБ) это означает миллионы страниц. Для перевода виртуальных адресов в физические процессор обращается к кэшу трансляции адресов — **TLB (Translation Lookaside Buffer)**.
- Если страниц миллионы, TLB-кэш переполняется. Возникают частые промахи (TLB misses), заставляющие процессор тратить драгоценное время на обход таблиц страниц в оперативной памяти (page table walks).
- **Hugepages** (большие страницы) решают эту проблему: размер одной страницы увеличивается с 4 КБ до **2 МБ** (или даже 1 ГБ). Это сокращает размер таблицы страниц в 512 раз!

**Transparent Hugepages (THP)** — это фоновая служба ядра, которая пытается автоматически и прозрачно для приложений группировать стандартные 4 КБ страницы в 2 МБ Hugepages.

#### Почему THP вредит базам данных?
1. **Compaction latency (Задержки сжатия):** если приложению срочно нужна страница памяти 2 МБ, а свободного непрерывного куска памяти нет, ядро запускает процесс дефрагментации (`khugepaged`), блокируя поток приложения. Это вызывает внезапные скачки задержки (latency spikes/jitter).
2. **Copy-on-Write (CoW) Write Amplification:** базы данных (например, Redis при `BGSAVE` или PostgreSQL при форке процессов) активно используют механизм оптимизации Copy-on-Write.
   - Если страница равна **4 КБ**, при изменении одного байта ядро скопирует только 4 КБ.
   - Если страница равна **2 МБ (THP)**, при изменении одного байта ядро будет вынуждено скопировать **все 2 МБ**! Это приводит к 512-кратному увеличению объема копирования, потреблению процессора и лавинообразному росту RSS процесса.

## Модуль 2: Исследование текущего состояния THP в системе

### Часть 2.1: Просмотр флагов ядра

**Что делаем:** посмотрим текущий режим работы THP в нашей операционной системе.

```bash
# Проверяем режим работы THP
cat /sys/fs/cgroup/../kernel/mm/transparent_hugepage/enabled
cat /sys/fs/cgroup/../kernel/mm/transparent_hugepage/defrag
```
*(Пути к управляющим файлам находятся в `/sys/kernel/mm/transparent_hugepage/`)*

```bash
# Или напрямую:
cat /sys/kernel/mm/transparent_hugepage/enabled
cat /sys/kernel/mm/transparent_hugepage/defrag
```

**Что смотрим / Как это работает:**
Вывод команды содержит варианты: `always [madvise] never`. Квадратные скобки показывают текущий активный режим.
- `always`: THP включен глобально для всех процессов.
- `madvise`: THP включается только для процессов, которые явно запросили это через системный вызов `madvise(..., MADV_HUGEPAGE)`.
- `never`: THP полностью отключен.

### Часть 2.2: Просмотр Hugepages в памяти процессов

**Что делаем:** посмотрим, сколько «огромных» страниц уже выделено в системе.

```bash
# Проверяем общую статистику анонимных огромных страниц
cat /proc/meminfo | grep -i huge
```

**Что смотрим / Как это работает:**
Вы увидите параметры:
- `AnonHugePages`: объем анонимной памяти, выделенной Hugepages (в КБ).
- `Hugepagesize`: стандартный размер Hugepage в системе (обычно `2048 kB`).

## Модуль 3: Эффект Copy-on-Write и Write Amplification (Тест вживую)

### Часть 3.1: Запуск теста с включенным THP

**Что делаем:** включим глобальный режим THP `always` и запустим демонстрационный скрипт `cow-thp-demo.py`. Скрипт выделяет 128 МБ памяти, делает `fork()` и заставляет дочерний процесс изменить по 1 байту в разных частях памяти.

```bash
# Включаем THP глобально
echo "always" > /sys/kernel/mm/transparent_hugepage/enabled

# Запускаем скрипт-демонстрацию
python3 cow-thp-demo.py
```

**Что смотрим / Как это работает:**
Обратите внимание на вывод скрипта в терминале:
1. Родительский процесс выделил 128 МБ (`Parent RSS: ~131000 KB`).
2. В строке `Parent AnonHugePages` вы увидите значение, близкое к 128 МБ (около `126976 KB`). Это значит, что ядро выдало нам память 2-мегабайтными страницами.
3. Дочерний процесс изменил по одному байту в разных частях памяти (всего 2048 изменений, по 1 на каждые 64 КБ).
4. **Смотрим `Child RSS after modification`:** он будет практически равен 128 МБ! 
Ядро скопировало **все 128 МБ** памяти для дочернего процесса, потому что при изменении даже одного байта в 2-мегабайтной странице приходилось копировать всю страницу целиком (CoW Write Amplification).

### Часть 3.2: Запуск теста с отключенным THP

**Что делаем:** отключим THP (установим `never`) и запустим этот же скрипт еще раз.

```bash
# Полностью отключаем THP
echo "never" > /sys/kernel/mm/transparent_hugepage/enabled

# Снова запускаем тест
python3 cow-thp-demo.py
```

**Что смотрим / Как это работает:**
Сравним результаты:
1. `Parent RSS` по-прежнему около 128 МБ.
2. `Parent AnonHugePages` теперь равен **`0 KB`** (огромные страницы не используются!).
3. **Смотрим `Child RSS after modification`:** он составит всего около **`8-12 MB`**!
Поскольку огромные страницы отключены, ядро использует стандартные страницы размером 4 КБ. Дочерний процесс при изменении разрозненных байтов скопировал только те 4 КБ страницы, которые были изменены. Вместо 128 МБ памяти мы потратили всего ~8 МБ. 

Для базы данных Redis (которая делает снимки `BGSAVE` через форк) включенный THP при интенсивной записи означает лавинообразный рост потребления памяти и внезапный OOM-килл из-за избыточного копирования 2-мегабайтных страниц.

## Модуль 4: Отключение THP в системе (временное и постоянное)

### Часть 4.1: Временное отключение (до перезагрузки)

**Что делаем:** запишем `never` в файлы конфигурации ядра для отключения автоматического выделения огромных страниц и их дефрагментации.

```bash
# Отключаем автовыделение
echo "never" > /sys/kernel/mm/transparent_hugepage/enabled

# Отключаем дефрагментацию под огромные страницы
echo "never" > /sys/kernel/mm/transparent_hugepage/defrag
```

### Часть 4.2: Персистентное отключение (после перезагрузки)

Поскольку файлы в `/sys/` виртуальные и сбрасываются при перезапуске системы, в продакшене их отключают на постоянной основе.

**Вариант А: Через systemd-service (Рекомендуемый для продакшена)**
Мы создаем системную службу, которая будет отключать THP при старте ОС.

```bash
# Создаем файл юнита службы
cat <<'EOF' > /etc/systemd/system/disable-thp.service
[Unit]
Description=Disable Transparent Huge Pages (THP)
DefaultDependencies=no
After=sysinit.target local-fs.target
Before=mongod.service redis-server.service postgresql.service

[Service]
Type=oneshot
ExecStart=/bin/sh -c 'echo never > /sys/kernel/mm/transparent_hugepage/enabled && echo never > /sys/kernel/mm/transparent_hugepage/defrag'
RemainAfterExit=yes

[Install]
WantedBy=basic.target
EOF

# Перечитываем конфигурацию системд и добавляем службу в автозапуск
systemctl daemon-reload
systemctl enable disable-thp.service
```

## Вопросы для самопроверки

<details>
<summary><b>1. Почему Transparent Hugepages вредны для СУБД вроде Redis?</b></summary>
Redis делает снимки данных на диск методом fork() (Copy-on-Write). Если THP включен, любое изменение данных во время сохранения заставляет ядро копировать целую огромную страницу 2 МБ вместо стандартной 4 КБ. Это ведет к избыточному потреблению оперативной памяти (Write Amplification) и риску аварийного завершения процесса (OOM-kill).
</details>

<details>
<summary><b>2. В чем разница между режимами always, madvise и never в файле enabled?</b></summary>
- <code>always</code>: ядро пытается автоматически выделять Hugepages для любых процессов.
- <code>madvise</code>: Hugepages выделяются только если процесс сам этого потребовал через специальный системный вызов <code>madvise()</code>.
- <code>never</code>: Hugepages полностью отключены на уровне ядра.
</details>

<details>
<summary><b>3. Как зафиксировать отключение THP после перезагрузки сервера?</b></summary>
Через создание кастомного systemd-юнита (например, <code>disable-thp.service</code>), выполняющего запись "never" в <code>/sys/kernel/mm/transparent_hugepage/enabled</code> и <code>/sys/kernel/mm/transparent_hugepage/defrag</code> на раннем этапе загрузки ОС.
</details>

## Шпаргалка

- `/sys/kernel/mm/transparent_hugepage/enabled` — главный файл переключения режимов THP.
- `/sys/kernel/mm/transparent_hugepage/defrag` — файл настройки дефрагментации страниц.
- `/proc/meminfo` (строка `AnonHugePages`) — просмотр объема выделенной огромной памяти.
- `/proc/PID/smaps` — детальный просмотр выделения Hugepages для конкретного процесса.
