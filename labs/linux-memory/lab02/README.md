## Модуль 1: Архитектура cgroup v2 и проверка окружения

### Теория для изучения перед модулем

В cgroup v1 контроллеры памяти, процессора и дисков жили в независимых иерархиях. Это мешало их координации (например, процесс мог быть ограничен по памяти в одной группе, но писать на диск без лимитов в другой). 

В **cgroup v2** используется **единая унифицированная иерархия** (Unified Hierarchy). Все контроллеры привязаны к одному и тому же дереву директорий в `/sys/fs/cgroup`.

Ключевые файлы управления в каждой cgroup-директории:
- `cgroup.controllers`: список контроллеров, доступных для этой группы.
- `cgroup.subtree_control`: включает или выключает контроллеры для дочерних групп.
- `cgroup.procs`: PID процессов, находящихся в этой группе.

### Часть 1.1: Проверка доступности cgroup v2 и контроллера memory

**Что делаем:** убедимся, что cgroup v2 смонтирована и ядро поддерживает лимиты памяти.

```bash
# Проверим тип файловой системы для cgroup
mount | grep cgroup2

# Посмотрим доступные глобальные контроллеры
cat /sys/fs/cgroup/cgroup.controllers
```

**Что смотрим / Как это работает:**
Вывод `mount` должен показать:
`cgroup2 on /sys/fs/cgroup type cgroup2 ...`

В выводе `cgroup.controllers` вы должны увидеть слово `memory` (а также `cpu`, `io`, `pids` и др.). Это означает, что подсистема ограничения памяти поддерживается ядром и активна.

## Модуль 2: Жесткие лимиты памяти — memory.max и OOM-killer

### Теория для изучения перед модулем

- **`memory.max`**: задает жесткую (hard) верхнюю границу потребления памяти для группы (анонимная память + page cache + swap). Если группа достигает этого лимита и ядро не может освободить память путем очистки кэша страниц (page reclaim), внутри группы срабатывает OOM-killer.
- **`memory.current`**: текущее потребление памяти процессами группы в байтах.
- **`memory.events`**: статистика OOM-киллов и других событий в группе.
- **`memory.oom.group`**: если установлен в `1`, OOM-killer убьет **все** процессы в cgroup при падении хотя бы одного. Если `0` (по умолчанию), пострадает только один процесс-виновник.

### Часть 2.1: Создание cgroup и установка лимита

**Что делаем:** создадим тестовую группу `mem-test-lab` и ограничим её лимитом в 50 МБ.

```bash
# Создаем директорию cgroup (ядро автоматически наполнит её управляющими файлами!)
mkdir -p /sys/fs/cgroup/mem-test-lab

# Устанавливаем лимит в 50 МБ
echo "50M" > /sys/fs/cgroup/mem-test-lab/memory.max

# Проверяем установленный лимит и текущее использование
cat /sys/fs/cgroup/mem-test-lab/memory.max
cat /sys/fs/cgroup/mem-test-lab/memory.current
```

**Что смотрим / Как это работает:**
Любая новая папка внутри `/sys/fs/cgroup` превращается в контрольную группу. Файл `memory.max` теперь содержит `52428800` (50 МБ в байтах). Текущее использование `memory.current` равно `0`, так как в группе еще нет процессов.

### Часть 2.2: Провокация OOM-killer внутри cgroup

**Что делаем:** запустим процесс, который попытается выделить 100 МБ анонимной памяти внутри нашей cgroup с лимитом 50 МБ.

```bash
# Запускаем фоновый процесс stress-ng и переносим его в cgroup.
# Для переноса пишем PID процесса в файл cgroup.procs группы.
stress-ng --vm 1 --vm-bytes 100M --vm-hang 5 -t 15 &
STRESS_PID=$!
sleep 0.5

# Перемещаем процесс в нашу cgroup
echo "$STRESS_PID" > /sys/fs/cgroup/mem-test-lab/cgroup.procs

# Ждем завершения процесса
wait "$STRESS_PID" 2>/dev/null || true
```

**Что смотрим / Как это работает:**
Процесс завершается практически мгновенно. Ядро пресекло выделение памяти. 
Посмотрим, что произошло в системных логах и статистике cgroup:

```bash
# Проверяем счетчик OOM внутри cgroup
cat /sys/fs/cgroup/mem-test-lab/memory.events

# Ищем запись об убийстве процесса в dmesg
dmesg | tail -n 50 | grep -i oom
```

**Пример вывода `memory.events`:**
```
low 0
high 0
max 1
oom 1
oom_kill 1
```
Файл `memory.events` четко фиксирует `oom_kill 1` — OOM-killer убил именно процесс внутри cgroup. Глобальный хост при этом никак не пострадал и не завис, так как OOM был изолирован рамками лимита группы!

## Модуль 3: Мягкие лимиты — memory.high, троттлинг и reclaim

### Теория для изучения перед модулем

- **`memory.high`**: мягкий (soft) лимит. В отличие от `memory.max`, превышение `memory.high` **не вызывает OOM-killer**. вместо этого ядро начинает принудительно отбирать память у процесса (page reclaim) и искусственно замедляет (throttles) его работу при выделении новых страниц, давая планировщику время на очистку.

Это критически важно для поддержания стабильности систем: память сжимается и чистится, процесс тормозит, но не падает сразу.

### Часть 3.1: Наблюдение за троттлингом при превышении memory.high

**Что делаем:** настроим мягкий лимит в 40 МБ, при этом жесткий оставим в 200 МБ. Запустим процесс, потребляющий 80 МБ.

```bash
# Сбрасываем жесткий лимит на высокий уровень
echo "200M" > /sys/fs/cgroup/mem-test-lab/memory.max

# Устанавливаем мягкий лимит в 40 МБ
echo "40M" > /sys/fs/cgroup/mem-test-lab/memory.high

# Запускаем stress-ng на выделение 80 МБ памяти
stress-ng --vm 1 --vm-bytes 80M --vm-hang 5 -t 10 &
STRESS_PID=$!
sleep 0.5

# Переносим в cgroup
echo "$STRESS_PID" > /sys/fs/cgroup/mem-test-lab/cgroup.procs

# Ждем завершения
wait "$STRESS_PID" 2>/dev/null || true

# Смотрим события
cat /sys/fs/cgroup/mem-test-lab/memory.events
```

**Что смотрим / Как это работает:**
В выводе `memory.events` вы увидите увеличение счетчика `high`. Процесс успешно отработал 10 секунд и завершился штатно (без OOM-kill), несмотря на то, что запрашивал 80 МБ при лимите `memory.high=40M`. Ядро урезало его кэш и замедлило аллокацию страниц.

## Модуль 4: Влияние на OOM-killer через oom_score_adj и QoS

### Теория для изучения перед модулем

Когда в Linux заканчивается свободная память (глобально или внутри cgroup), ядро выбирает жертву на основе показателя **OOM Score** (`oom_score`).
- Процесс с наибольшим `oom_score` будет убит первым.
- Мы можем влиять на этот выбор с помощью параметра **OOM Score Adjust** (`oom_score_adj`), который принимает значения от `-1000` (полностью запретить OOM-killer трогать процесс) до `1000` (убить процесс в первую очередь).

Именно так **Kubernetes** реализует свои классы обслуживания (QoS Classes):
1. **Guaranteed**: контейнеры имеют равные `requests` и `limits`. Kubernetes ставит им `oom_score_adj = -997` (высокая защита от OOM).
2. **Burstable**: `requests` меньше `limits`. Kubernetes рассчитывает `oom_score_adj` динамически в зависимости от запрашиваемой памяти (обычно от 2 до 999).
3. **BestEffort**: лимиты и запросы отсутствуют. Kubernetes ставит `oom_score_adj = 1000` (убиваются первыми при дефиците памяти).

### Часть 4.1: Ручная корректировка приоритета процесса

**Что делаем:** запустим фоновый процесс и изменим его `oom_score_adj` до `1000` (сделаем его приоритетной мишенью OOM).

```bash
# Запустим бесконечный легкий процесс sleep
sleep 300 &
SLEEP_PID=$!

# Посмотрим его текущий oom_score и oom_score_adj
cat /proc/$SLEEP_PID/oom_score
cat /proc/$SLEEP_PID/oom_score_adj

# Меняем приоритет OOM-killer на максимальную уязвимость
echo "1000" > /proc/$SLEEP_PID/oom_score_adj

# Снова проверяем oom_score
cat /proc/$SLEEP_PID/oom_score

# Убираем процесс за собой
kill "$SLEEP_PID" 2>/dev/null || true
```

**Что смотрим / Как это работает:**
По умолчанию у нового процесса `oom_score_adj` равен `0`. После записи `1000` значение `oom_score` резко возрастает. При нехватке памяти этот процесс `sleep` будет уничтожен ядром мгновенно, даже если он потребляет всего несколько килобайт памяти.

## Вопросы для самопроверки

<details>
<summary><b>1. В чем главное отличие cgroup v1 от cgroup v2?</b></summary>
В cgroup v1 каждый контроллер (память, процессор, сеть) управлялся в своей независимой иерархии папок. В cgroup v2 используется единая унифицированная иерархия (все контроллеры привязаны к одному общему дереву директорий), что позволяет им эффективно координировать действия.
</details>

<details>
<summary><b>2. Что произойдет с процессом при достижении memory.max и memory.high?</b></summary>
При достижении <b>memory.max</b> процесс будет немедленно убит OOM-killer'ом (если ядро не сможет освободить кэш страниц). При достижении <b>memory.high</b> процесс не убивается, но ядро начинает активно отбирать у него неиспользуемые страницы памяти и искусственно замедляет (дросселирует) выполнение его системных вызовов аллокации.
</details>

<details>
<summary><b>3. Как Kubernetes защищает поды класса Guaranteed от OOM-killer?</b></summary>
Kubernetes прописывает процессам подов класса Guaranteed значение <code>oom_score_adj = -997</code>. Это делает их практически неуязвимыми для OOM-killer по сравнению с подами Burstable и BestEffort (у которых oom_score_adj равен 1000).
</details>

## Шпаргалка

- `mkdir -p /sys/fs/cgroup/GROUP_NAME` — создание новой контрольной группы.
- `echo PID > /sys/fs/cgroup/GROUP_NAME/cgroup.procs` — перенос процесса в группу.
- `echo LIMIT > /sys/fs/cgroup/GROUP_NAME/memory.max` — установка жесткого лимита памяти.
- `echo LIMIT > /sys/fs/cgroup/GROUP_NAME/memory.high` — установка мягкого лимита памяти.
- `cat /sys/fs/cgroup/GROUP_NAME/memory.current` — проверка текущего потребления памяти.
- `echo VALUE > /proc/PID/oom_score_adj` — изменение OOM-приоритета процесса (от -1000 до 1000).
