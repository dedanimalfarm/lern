## Предварительная установка пакетов

**Что делаем:** ставим утилиты, которыми будем смотреть и нагружать память. Без них
часть команд просто не найдётся.

```bash
sudo apt-get update
sudo apt-get install -y \
  stress-ng strace procps util-linux \
  bsdmainutils sysstat python3
```

**Как это работает:** `procps` даёт `free`, `vmstat`, `pmap`, `ps`; `util-linux` —
`ipcs`, `ipcmk`, `ipcrm`, `lsipc` (управление SysV IPC); `stress-ng` создаёт
контролируемую нагрузку на память; `strace` показывает системные вызовы; `sysstat`
добавляет инструменты статистики. На не-Debian (RHEL/Fedora) пакеты называются
похоже: `procps-ng`, `util-linux`, `strace`, `stress-ng`.

## Стартовая проверка

**Что делаем:** сначала прогоняем автоматическую проверку готовности стенда — она
скажет, каких утилит не хватает, хватает ли RAM и места на диске, работает ли PSI.
Скрипт ничего не меняет в системе, только читает.

```bash
bash check.sh
```

**Как это работает:** `check.sh` проверяет наличие всех нужных бинарей, объём
`MemTotal`, активность swap, свободное место под swap-файл (нужно для модуля 4),
права (root/sudo) и доступность `/proc/pressure/memory`. Если есть `[FAIL]` —
поправьте до начала лабы.

Затем посмотрите картину памяти вручную, чтобы знать стартовое состояние:

```bash
free -h
cat /proc/meminfo | head -10
getconf PAGESIZE
swapon --show
```

**Что смотрим:** типичный вывод `free -h` —

```
              total        used        free      shared  buff/cache   available
Mem:           1.9Gi       180Mi       1.4Gi       2.0Mi       320Mi       1.6Gi
Swap:            0B          0B          0B
```

`getconf PAGESIZE` почти всегда вернёт `4096` (размер страницы памяти). Если `Swap`
уже подключён — это нормально: в модуле 4 мы добавим **свой** swap-файл и снимем
его, не трогая существующий.

## Модуль 1: Виртуальная и физическая память

### Теория для изучения перед модулем

- **Виртуальная память:** у каждого процесса своё адресное пространство; MMU и
  таблицы страниц переводят виртуальные адреса в физические.
- **Демандное выделение (demand paging):** ядро выдаёт физическую страницу не в
  момент `malloc`/`mmap`, а при первом **обращении** к ней. Поэтому `mmap(1GB)` не
  занимает 1 ГБ RAM сразу.
- **Метрики процесса:** `VSZ` (виртуальная, «обещано»), `RSS` (резидентная, реально
  в RAM), `SHR` (разделяемая), `PSS` (proportional set size — RSS с честным делением
  общих страниц между процессами).
- **`free`:** `total`, `used`, `free`, `buff/cache`, `available` — и как это связано
  с полями `/proc/meminfo`.
- **Page cache:** копии файлов в свободной RAM для скорости; «съеденная» им память —
  это **не** утечка, ядро отдаст её под нагрузку.
- **`vmstat`:** колонки `si/so` (swap in/out), `bi/bo` (block in/out), `wa` (ожидание IO).

**Цель модуля:** научиться читать `free -h`, `vmstat`, `/proc/meminfo` и понимать,
почему `top` показывает у процесса VIRT=10G, хотя реально он занимает 200 МБ.

---

### 1.1 Базовый снапшот памяти

**Что делаем:** снимаем три среза одной и той же картины — человекочитаемый,
сырой и параметр размера страницы — и сопоставляем их между собой.

```bash
# Память в человекочитаемом виде
free -h

# То же в килобайтах + сырые поля
cat /proc/meminfo | head -20

# Размер страницы памяти на этой системе (обычно 4096)
getconf PAGESIZE
```

**Что смотрим:** найдите и сопоставьте соответствия между `/proc/meminfo` и `free -h`:
- `MemTotal` = `total`
- `MemAvailable` = `available`
- `Buffers + Cached + SReclaimable ≈ buff/cache`

**Как это работает / вывод для практики:** главная цифра для алёртов — **`MemAvailable`**,
а не `MemFree`. Высокое `used` без учёта `available` — пугалка: page cache всегда
занимает «свободную» память и моментально освобождается под нагрузку.

---

### 1.2 Наблюдение за памятью в динамике

**Что делаем:** снимаем 5 сэмплов состояния памяти/IO с интервалом 1 секунда, чтобы
видеть не статичный снимок, а тренд.

```bash
# 5 сэмплов с интервалом 1 сек
vmstat 1 5
```

**Что смотрим:** типичная строка —

```
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 0  0      0  1450M    12M   320M    0    0     5     2   45   80  1  0 99  0  0
```

**Как это работает:** ключевые колонки —
- `free` — свободная RAM;
- `buff/cache` — page cache;
- `si/so` — swap in/out: если **стабильно ненулевые**, система свопается — плохой знак;
- `wa` — процент времени, что CPU ждёт I/O: высокий `wa` = диск/память узкое место.

### 1.3 VIRT vs RES — где обман

**Что делаем:** запускаем процесс, который запрашивает 1 ГБ виртуальной памяти, но
**не трогает** её. Сразу сохраняем его PID в переменную — так мы гарантированно
работаем со своим процессом, а не ищем его по шаблону (и не рискуем задеть чужой).

```bash
python3 -c '
import mmap, os, time
m = mmap.mmap(-1, 1024*1024*1024)  # 1 ГБ анонимной памяти, к ней не обращаемся
print("PID:", os.getpid())
time.sleep(300)
' &
LAZY_PID=$!          # PID только что запущенного фонового процесса
sleep 1
```

```bash
# Смотрим VSZ (виртуальная) и RSS (резидентная) именно у нашего процесса
ps -o pid,vsz,rss,comm -p "$LAZY_PID"
# PID    VSZ  RSS  COMMAND
# 1234   1.0G  8M   python3       <- VIRT гигабайт, RES почти ничего
```

**Что смотрим:** `VSZ ≈ 1 ГБ`, но `RSS` — единицы мегабайт. Память «обещана», но
физически ещё не выдана.

**Что делаем дальше:** запускаем второй процесс, который реально пишет по 1 байту
в каждую страницу — заставляем ядро выдать физические страницы:

```bash
kill "$LAZY_PID" 2>/dev/null      # убираем первый процесс ТОЧЕЧНО по его PID
python3 -c '
import mmap, os, time
m = mmap.mmap(-1, 1024*1024*1024)
for i in range(0, len(m), 4096):
    m[i] = 1                       # касаемся каждой страницы (PAGESIZE = 4096)
print("PID:", os.getpid())
time.sleep(300)
' &
TOUCH_PID=$!
sleep 3
ps -o pid,vsz,rss,comm -p "$TOUCH_PID"
# RSS теперь ~1 ГБ — страницы реально выделились ядром
```

**Вывод:** `VSZ` — это «обещание», `RSS` — это правда. Алёрты строим по `RSS` и
`MemAvailable`, а не по `VSZ`.

> ⚠️ **Почему не `killall python3` и не `pgrep -f 'mmap(...'`:** `killall` снесёт
> **весь** Python на хосте (в т.ч. чужой), а шаблон с круглой скобкой `(` — это
> ещё и невалидное регулярное выражение (`pgrep` ответит `Unmatched (`). Поэтому
> везде в лабе мы убиваем процессы по сохранённому `$!`.

```bash
kill "$TOUCH_PID" 2>/dev/null
```

### 1.4 Page cache: «съел» всю RAM — это норма

**Что делаем:** создаём большой файл, смотрим, как ядро прокеширует его в RAM, и
вручную сбрасываем кеш, чтобы убедиться — это не утечка.

```bash
free -h
# запомни значение buff/cache

# Создаём файл на 1 ГБ — ядро прокеширует его при записи
dd if=/dev/zero of=/tmp/bigfile bs=1M count=1024 status=progress
free -h
# buff/cache вырос на ~1 ГБ, а available почти не изменился
```

**Что смотрим:** `buff/cache` подскочил, но `available` почти не упал — ядро готово
отдать этот кеш под нагрузку в любой момент.

**Что делаем:** сбрасываем page cache (только ради демонстрации):

```bash
sync && echo 3 | sudo tee /proc/sys/vm/drop_caches
free -h
# buff/cache упал
```

**Как это работает:** `sync` сбрасывает «грязные» страницы на диск, `echo 3 >
drop_caches` очищает pagecache+dentries+inodes. В **production это делать не нужно** —
ядро само вытеснит кеш, когда понадобится память; ручной сброс лишь замедлит систему.

```bash
rm -f /tmp/bigfile
```

**Контрольные вопросы:**
1. У процесса VSZ=8 ГБ, RSS=200 МБ. Сколько физической памяти он реально занимает?
2. `free -h` показывает `used=7G`, `available=12G` при `total=16G`. Стоит паниковать?
3. Чем отличаются колонки `free` и `available` в `free -h`? На какую алёртить?
4. Зачем в `vmstat` колонки `si` и `so`? Что значит, если они стабильно ненулевые?

## Модуль 2: Аллокация памяти — malloc, mmap, brk, sbrk

### Теория для изучения перед модулем

- **Адресное пространство процесса:** text/data/bss/heap/mmap-region/stack.
- **Системные вызовы выделения:** `brk`/`sbrk` двигают вершину heap; `mmap` создаёт
  отдельный регион.
- **glibc `malloc()`:** маленькие куски — через `brk`; большие (по умолчанию
  `M_MMAP_THRESHOLD`, старт 128 КБ, порог динамически растёт) — через `mmap`.
- **Почему `free()` мелких блоков не уменьшает RSS:** память помечается свободной в
  пуле glibc, но ядру не возвращается (порог `M_TRIM_THRESHOLD`).
- **`malloc_trim()`:** принудительно вернуть свободные участки heap ядру.
- **`/proc/<pid>/maps` и `pmap -x`:** показывают text, библиотеки (`mmap` файла),
  `[heap]`, `[stack]`, анонимные `mmap`.

**Цель модуля:** увидеть глазами, какие системные вызовы делает `malloc` под капотом,
и понять, почему «процесс утёк память» часто означает «glibc не вернул её ядру».

---

### 2.1 Карта памяти процесса

**Что делаем:** смотрим карту памяти текущего шелла — из чего вообще состоит
адресное пространство процесса.

```bash
# $$ — PID текущего шелла, любой долгоживущий процесс подойдёт
pmap -x $$ | head -30
```

**Что смотрим / как это работает:** в выводе найдите —
- блок `[ heap ]` — сюда уходит `brk`/`sbrk` для мелких аллокаций;
- блок `[ stack ]` — стек главного потока;
- строки с путями на `.so` — это `mmap()` библиотек, **общая** память между процессами;
- безымянные блоки `anon` — `mmap()` для больших malloc или ручной `mmap()`.

```bash
# То же сырьём, без агрегации pmap
cat /proc/$$/maps | head -20
```

---

### 2.2 strace: видим brk и mmap живьём

**Что делаем:** трассируем системные вызовы выделения памяти при маленькой и
большой аллокации — и видим разницу между `brk` и `mmap`.

```bash
# Малый объект — пойдёт через brk (расширение heap)
strace -e trace=brk,mmap python3 -c 'x = "a" * 1000' 2>&1 | tail -15

# Большой объект (10 МБ) — пойдёт через отдельный mmap
strace -e trace=brk,mmap python3 -c 'x = "a" * (10 * 1024 * 1024)' 2>&1 | tail -15
```

**Что смотрим:** в первом случае — несколько `brk(...)`; во втором — отдельный
`mmap(NULL, 10489856, ...)` под 10 МБ. Это и есть граница «малый/большой malloc».

**Как это работает:** `-e trace=brk,mmap` фильтрует strace только по этим вызовам;
`2>&1` нужен потому, что strace пишет в stderr; `tail -15` отрезает финал, где и
происходит аллокация.

---

### 2.3 strace по живому процессу

**Что делаем:** запускаем процесс, который постоянно аллоцирует блоки, и трассируем
его «снаружи» по PID — как при разборе боевого инцидента.

В одном терминале:

```bash
python3 -c '
import time
data = []
for i in range(2000):
    data.append("x" * 1024 * 50)   # копим блоки по 50 КБ
    time.sleep(0.02)
time.sleep(60)
' &
ALLOC_PID=$!
echo "PID: $ALLOC_PID"
```

В другом терминале (или том же, в новом окне):

```bash
sudo strace -e trace=brk,mmap,munmap -p "$ALLOC_PID" 2>&1 | head -40
```

**Что смотрим:** поток `brk(...)` (heap растёт) и периодические `mmap(...)` для
больших блоков. Это и есть «как работает malloc» с точки зрения ядра.

```bash
kill "$ALLOC_PID" 2>/dev/null      # точечно, без killall
```

### 2.4 «Фантомная утечка»: память освобождена, RSS не падает

**Что делаем:** аллоцируем 50 тысяч мелких блоков, освобождаем их и смотрим RSS
до/после `free()` и после `malloc_trim()`. Это воспроизводит классический кейс
«сервис не вернул память после пика».

```bash
python3 <<'EOF' &
import ctypes, time, os
libc = ctypes.CDLL("libc.so.6")
# Важно объявить сигнатуры: иначе ctypes считает, что malloc возвращает 32-битный
# int, и на 64-бит адрес усечётся — memset запишет по битому адресу (segfault).
libc.malloc.restype = ctypes.c_void_p
libc.malloc.argtypes = [ctypes.c_size_t]
libc.free.argtypes = [ctypes.c_void_p]
libc.malloc_trim.argtypes = [ctypes.c_size_t]
PID = os.getpid()

def rss_kb():
    with open(f"/proc/{PID}/status") as f:
        for line in f:
            if line.startswith("VmRSS:"): return int(line.split()[1])

# 50 тысяч аллокаций по 1 КБ — пойдут через brk (мелкие)
ptrs = [libc.malloc(1024) for _ in range(50_000)]
for p in ptrs:
    ctypes.memset(p, 0xAB, 1024)         # касаемся, чтобы страницы реально выделились
print(f"PID={PID}  После malloc: {rss_kb()} KB")

for p in ptrs: libc.free(p)
print(f"PID={PID}  После free:   {rss_kb()} KB   <- RSS почти не упал")

libc.malloc_trim(0)                       # просим glibc вернуть heap ядру
print(f"PID={PID}  После trim:   {rss_kb()} KB   <- вот теперь упал")

time.sleep(60)
EOF
LEAK_PID=$!
wait "$LEAK_PID"
```

**Что смотрим:** после `free()` RSS почти не уменьшился, и только `malloc_trim(0)`
вернул память ядру.

**Как это работает / вывод:** на уровне библиотеки память свободна, но glibc держит
её в своём пуле для повторного использования (быстрее, чем снова просить у ядра).
Ядру она возвращается только при `malloc_trim` или когда glibc сам решит. Поэтому
«Python/Java-сервис не отдаёт память после нагрузки» — часто не утечка, а это поведение.

```bash
kill "$LEAK_PID" 2>/dev/null
```

### 2.5 Сколько RAM реально занимают разделяемые библиотеки

**Что делаем:** берём самый «тяжёлый» по RSS процесс и смотрим, какая часть его RSS
приходится на разделяемые `.so` — то есть на **общую** память.

```bash
# Топ процессов по RSS
ps -eo pid,rss,comm --sort=-rss | head -5

# Берём PID самого жирного и смотрим только .so в его карте
PID=$(ps -eo pid,rss --sort=-rss | awk 'NR==2{print $1}')
pmap -x $PID | grep -E '\.so' | head -10
```

**Что смотрим / как это работает:** `RSS` у `.so` ненулевой, но это **общая** память:
одна копия библиотеки в RAM на все процессы, которые её используют. Поэтому простой
суммой `RSS` всех процессов занятую RAM считать нельзя — общие страницы посчитаются
многократно. Честную оценку даёт `PSS` (`/proc/<pid>/smaps_rollup`).

**Контрольные вопросы:**
1. Чем `brk`/`sbrk` отличается от `mmap` при `malloc`? Когда используется каждый?
2. Почему `free()` мелкого блока не уменьшает RSS?
3. Какой системный вызов используется для загрузки `.so`?
4. У процесса RSS=2 ГБ, но в `pmap` 80% — это `/usr/lib/.../some.so`. Это утечка?
5. Зачем нужен `malloc_trim()`? Что он делает с точки зрения ядра?

## Модуль 3: Страницы памяти и swappiness

### Теория для изучения перед модулем

- **Страница** — единица работы MMU, типично 4 КБ на x86_64.
- **Huge pages** (2 МБ / 1 ГБ) уменьшают промахи TLB (упоминание, без углубления).
- **Вытеснение страницы:** anon-страницы → в swap; file-backed → просто отбрасываются
  (на диске уже есть копия).
- **`vm.swappiness` (0..200 на ядрах ≥5.8, иначе 0..100):** насколько агрессивно
  ядро вытесняет anon-страницы вместо очистки page cache.
- **PSI** (`/proc/pressure/memory`) — современная замена «смотреть на `free`» для алёртов.
- **OOM-killer** включается, когда не помогают ни swap, ни вытеснение кеша.

**Цель модуля:** понять, что такое страница, как `swappiness` управляет поведением
ядра под давлением, и где смотреть pressure-метрики — безопасно, не уронив хост.

---

### 3.1 Размер страницы и общая картина

**Что делаем:** смотрим размер страницы, общее число страниц и ключевые поля
`/proc/meminfo`, которые понадобятся при диагностике.

```bash
getconf PAGESIZE          # размер страницы (почти всегда 4096)
getconf _PHYS_PAGES       # всего физических страниц
getconf _AVPHYS_PAGES     # свободных физических страниц
grep -E 'Dirty|Writeback|AnonPages|Mapped|Shmem' /proc/meminfo
```

**Что смотрим:**

| Поле | Что это |
|------|---------|
| `AnonPages` | анонимные страницы процессов (heap, stack, anon mmap) — могут уйти в swap |
| `Mapped` | страницы из файлов (`.so`, mmap файлов) — на диске уже есть, swap не нужен |
| `Shmem` | разделяемая память (`/dev/shm`, tmpfs, SysV shm) |
| `Dirty` | модифицированные страницы, которые надо записать на диск |
| `Writeback` | прямо сейчас записываются на диск |

---

### 3.2 Swappiness — что это и где живёт

**Что делаем:** читаем текущее значение, понимаем шкалу, временно меняем и
возвращаем обратно.

```bash
cat /proc/sys/vm/swappiness
sysctl vm.swappiness
```

**Что смотрим:**

| Значение | Что означает |
|----------|--------------|
| 0 | Не свопить anon, пока есть что вытеснить из page cache. Для БД, low-latency сервисов |
| 1 | То же, но не отключает swap полностью (есть нюансы с cgroups v2) |
| 10..30 | Сервера с большим page cache (БД, файлсерверы) |
| 60 | Дефолт большинства дистрибутивов — баланс |
| 100..200 | Агрессивный swap; очень редко, на десктопах с медленным HDD |

```bash
# Изменить временно (до перезагрузки)
sudo sysctl -w vm.swappiness=10

# Сделать постоянным (переживёт reboot)
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-swappiness.conf
sudo sysctl --system

# Вернуть как было
sudo sysctl -w vm.swappiness=60
```

**Как это работает (важно!):** `swappiness` — это **направление** выбора «кого
вытеснить», а **не** «при каком % памяти начинать свопить». Ядро начнёт вытеснять,
когда ему реально понадобится память; `swappiness` лишь влияет на выбор: anon-страница
или file-backed.

---

### 3.3 Pressure Stall Information — современный мониторинг

**Что делаем:** читаем PSI — сколько времени процессы простаивали из-за нехватки памяти.

```bash
cat /proc/pressure/memory
```

**Что смотрим:**

```
some avg10=0.00 avg60=0.00 avg300=0.00 total=0
full avg10=0.00 avg60=0.00 avg300=0.00 total=0
```

- `some` — хотя бы один процесс ждёт памяти;
- `full` — **все** процессы ждут памяти (фактически полная остановка);
- `avg10/60/300` — процент времени за последние 10/60/300 секунд.

**Как это работает / вывод:** `full avg10 > 10` — уже плохо, это лучший индикатор
реального голодания памяти, чем «just look at `free`». Аналогично есть
`/proc/pressure/cpu` и `/proc/pressure/io`. Если файла нет — ядро без `CONFIG_PSI`.

---

### 3.4 Создаём давление и смотрим, как реагирует ядро — БЕЗОПАСНО

**Что делаем:** в одном терминале наблюдаем за памятью и PSI, в другом создаём
нагрузку. Нагрузку запускаем **внутри cgroup-scope с лимитом памяти** — чтобы при
нехватке OOM-killer убил именно наш `stress-ng`, а не случайную жертву на всём хосте.

Терминал 1 — наблюдатель:

```bash
watch -n 1 'free -h; echo "---"; cat /proc/pressure/memory'
```

Терминал 2 — нагрузка в изолированном scope:

```bash
free -h            # сколько свободно — например 1.4G
# Просим больше, чем лимит scope. systemd-run создаёт временный cgroup v2
# с memory.max=1G: вся нагрузка и любой OOM остаются ВНУТРИ него.
sudo systemd-run --scope -p MemoryMax=1G --slice=memlab \
  stress-ng --vm 1 --vm-bytes 2G --vm-keep -t 60
```

**Что смотрим:**
- пока scope жив — `available` проседает, `pressure/memory` даёт ненулевые `avg10`;
- если в системе есть swap — стресс будет вытесняться, в `vmstat` появятся `si/so`;
- если swap нет — **внутри scope** сработает OOM-killer и убьёт именно `stress-ng`,
  хост не пострадает. Подтверждаем:

```bash
journalctl -k | grep -iE 'oom|killed process' | tail
dmesg | grep -i 'killed process' | tail
```

**Как это работает / вывод:** `systemd-run --scope -p MemoryMax=…` — это и есть
механизм, которым k8s/systemd ограничивают память (`memory.max` в cgroup v2). Без
лимита (`stress-ng …` напрямую) OOM-killer выбирал бы жертву по **всему** хосту по
`oom_score` — на общем сервере так делать нельзя.

---

### 3.5 Huge pages — упоминание

**Что делаем:** просто смотрим, настроены ли huge pages и включён ли THP.

```bash
grep -i huge /proc/meminfo
cat /sys/kernel/mm/transparent_hugepage/enabled    # always / madvise / never
```

**Как это работает:** тюнинг hugepages — отдельная большая тема. Для типичных
DevOps/поддержки задач важно знать: некоторые сервисы (PostgreSQL, JVM, Oracle,
Redis) могут просить huge pages — тогда `HugePages_Total > 0` это норма, а не аномалия.

## Модуль 4: Swap — назначение, настройка, /etc/fstab

### Теория для изучения перед модулем

- **Назначение swap:** «overflow» для anon-страниц, когда не хватает RAM.
- **Когда полезен:** пиковая нагрузка, гибернация, защита от OOM на не-критичных сервисах.
- **Когда вреден:** low-latency сервисы (БД, kafka, kvm-хосты), SSD с малым ресурсом записи (TBW).
- **Файл vs раздел:** на современных SSD разница в скорости пренебрежимо мала; файл гибче.
- **Структура swap-области:** signature + битмап страниц (создаёт `mkswap`).
- **Приоритеты:** можно иметь несколько swap-областей, ядро выбирает по `pri`.

**Цель модуля:** создать swap-файл, подключить, прописать в `/etc/fstab`, снять и
удалить — не сломав уже существующий swap, если он есть.

---

### 4.1 Текущее состояние swap

**Что делаем:** фиксируем, что есть сейчас, чтобы потом не перепутать свой swap с системным.

```bash
swapon --show
swapon --show=NAME,TYPE,SIZE,USED,PRIO
cat /proc/swaps
free -h | grep Swap
```

**Что смотрим:** если `swapon --show` пуст — swap не подключён, сейчас создадим.

---

### 4.2 Создание swap-файла на 1 ГБ

**Что делаем:** создаём файл, защищаем правами, размечаем как swap и подключаем.

```bash
# Создаём пустой файл нужного размера (fallocate выделяет место мгновенно)
sudo fallocate -l 1G /swapfile-lab
# Альтернатива, медленнее, но работает везде:
# sudo dd if=/dev/zero of=/swapfile-lab bs=1M count=1024 status=progress

# Права: swap-файл должен быть доступен ТОЛЬКО root (иначе mkswap/swapon предупредит)
sudo chmod 600 /swapfile-lab
ls -la /swapfile-lab

# Размечаем файл как swap-область (пишем заголовок)
sudo mkswap /swapfile-lab
```

```bash
# Подключаем и проверяем
sudo swapon /swapfile-lab
swapon --show
free -h
```

**Как это работает:** `fallocate` резервирует место, `mkswap` пишет служебный
заголовок (signature + битмап), `swapon` сообщает ядру использовать этот файл.

> Если видите `swapon: ... Invalid argument` — чаще всего файл на ФС, которая не
> поддерживает swap (btrfs без отключённого CoW), либо забыли `mkswap`.

---

### 4.3 Снять swap

**Что делаем:** отключаем наш swap, не удаляя файл.

```bash
sudo swapoff /swapfile-lab
swapon --show
# sudo swapoff -a   # снять ВСЁ — НЕ делать на проде без надобности
```

**Как это работает:** `swapoff` переносит все страницы из swap обратно в RAM, поэтому
может занять время; если RAM не хватает — вернёт ошибку (это нормально, ничего не сломается).

---

### 4.4 Постоянное монтирование через /etc/fstab

**Что делаем:** прописываем swap в `/etc/fstab`, чтобы он поднимался после перезагрузки,
и проверяем строку **до** ребута.

```bash
sudo swapon /swapfile-lab        # подключаем заново
cat /etc/fstab                   # текущее содержимое
echo "/swapfile-lab  none  swap  sw  0  0" | sudo tee -a /etc/fstab
tail -3 /etc/fstab
```

**Что смотрим:** поля строки —
- `/swapfile-lab` — путь (или `UUID=...` для раздела);
- `none` — нет точки монтирования;
- `swap` — тип; `sw` — опции; `0 0` — dump и fsck отключены.

**Что делаем (критическая проверка перед перезагрузкой):**

```bash
sudo swapoff /swapfile-lab
sudo swapon -a                   # читает /etc/fstab и поднимает всё оттуда
swapon --show                    # наш /swapfile-lab должен снова появиться
```

**Как это работает / зачем:** если в `fstab` ошибка и `swapon -a` ругается — её надо
исправить **до** ребута. Для swap битая строка не критична, но привычка проверять
fstab до перезагрузки спасает от незагружающихся серверов.

> ⚠️ На управляемых хостах (cloud-init/Ansible/образы) запись в `/etc/fstab` могут
> перезатереть при следующем прогоне — держите swap-конфиг в том же инструменте.

---

### 4.5 Несколько swap-областей и приоритеты

**Что делаем:** добавляем второй swap с более высоким приоритетом и смотрим, как
ядро выбирает, какой использовать раньше.

```bash
sudo fallocate -l 512M /swapfile-lab2
sudo chmod 600 /swapfile-lab2
sudo mkswap /swapfile-lab2
sudo swapon -p 10 /swapfile-lab2     # -p задаёт приоритет
swapon --show                        # колонка PRIO: больше = раньше используется
```

```bash
sudo swapoff /swapfile-lab2          # убираем второй
sudo rm /swapfile-lab2
```

---

### 4.6 Удаление swap-файла полностью

**Что делаем:** снимаем, убираем строку из fstab и удаляем файл — возвращаем систему как было.

```bash
sudo swapoff /swapfile-lab
sudo sed -i '\#^/swapfile-lab[[:space:]]#d' /etc/fstab   # удаляем нашу строку
tail -3 /etc/fstab
sudo rm /swapfile-lab
swapon --show
free -h
```

**Контрольные вопросы:**
1. Зачем нужен `chmod 600` для swap-файла перед `mkswap`?
2. Что произойдёт, если `swapoff` запустить на хосте, где swap почти полный, а RAM на пределе?
3. Чем поле `sw` в `/etc/fstab` отличается от `defaults`?
4. На SSD с малым TBW — стоит ли держать swap? Какие альтернативы (`zram`)?
5. Сервер с PostgreSQL — больше RAM без swap или RAM + swap с `swappiness=1`?

## Модуль 5: Анонимная и общая память

### Теория для изучения перед модулем

- **Анонимная память:** heap, stack, anon-mmap — принадлежит одному процессу.
- **Разделяемая память — три «вкуса»:**
  - **SysV shm** — древний API (`shmget`/`shmat`/`shmctl`), объекты видны в `ipcs`;
  - **POSIX shm** — современный (`shm_open()` создаёт файлы в `/dev/shm`);
  - **`mmap` с `MAP_SHARED`** — самый универсальный (через файл или anon-shared).
- **`tmpfs` и `/dev/shm`:** где живут POSIX shm; `df /dev/shm` показывает лимит из RAM.
- **Утилиты:** `ipcs`, `ipcmk`, `ipcrm`, `lsipc`.
- **Утечки SysV shm:** процесс умер, сегмент остался — типичный «исчезающий гигабайт».

**Цель модуля:** различать анонимную и разделяемую память, видеть SysV/POSIX shm в
системе, уметь найти и **безопасно** удалить осиротевшие сегменты.

---

### 5.1 Анонимная vs разделяемая в /proc/meminfo

**Что делаем:** смотрим две непересекающиеся метрики и где живёт tmpfs.

```bash
grep -E 'AnonPages|Shmem|MemFree|MemAvailable' /proc/meminfo
```

**Что смотрим:** `AnonPages` (heap/stack/anon mmap всех процессов) и `Shmem`
(tmpfs, `/dev/shm`, SysV shm, anon-shared mmap) — **не пересекаются** и обе не входят
в `MemFree`. Растёт `Shmem` без причины — ищите tmpfs/`/dev/shm` или утечку SysV shm.

```bash
df -h -t tmpfs        # точки tmpfs едят память из Shmem
mount | grep tmpfs
```

---

### 5.2 POSIX shared memory через /dev/shm

**Что делаем:** создаём POSIX shm-объект (это просто файл в tmpfs `/dev/shm`) и
наблюдаем рост `Shmem`.

```bash
mount | grep /dev/shm        # /dev/shm — это tmpfs
df -h /dev/shm

sudo dd if=/dev/zero of=/dev/shm/posix_demo bs=1M count=100 status=none
ls -lh /dev/shm/
grep Shmem /proc/meminfo     # вырос на ~100 МБ

sudo rm /dev/shm/posix_demo  # удаление
grep Shmem /proc/meminfo
```

**Как это работает:** любой процесс может сделать `shm_open("/имя", ...)` и получить
тот же объект — это и есть POSIX shared memory. Chrome/Firefox/PostgreSQL активно его используют.

---

### 5.3 SysV shared memory — древний API

**Что делаем:** смотрим существующие SysV-объекты и лимиты, создаём свой сегмент,
находим и удаляем его.

```bash
ipcs -m              # сегменты shared memory
ipcs -q              # очереди сообщений
ipcs -s              # семафоры
ipcs                 # всё разом

ipcs -l              # лимиты системы на SysV shm
sysctl kernel.shmmax kernel.shmall kernel.shmmni
```

**Что делаем (создаём сегмент):**

```bash
ipcmk -M 50M
# Shared memory id: 12345

ipcs -m
# key        shmid   owner  perms  bytes      nattch  status
# 0x12abcd   12345   root   644    52428800   0
```

**Что смотрим:** `key` — числовой ключ (по нему процессы находят сегмент); `shmid` —
ID в ядре; `nattch` — сколько процессов подключено (`shmat`); `bytes` — размер.

```bash
grep Shmem /proc/meminfo

# Удаление по shmid (берём свой)
SHMID=$(ipcs -m | awk '/^0x/ && $5=="52428800"{print $2}' | head -1)
ipcrm -m "$SHMID"
ipcs -m
```

---

### 5.4 Утечка SysV shm — «куда делся гигабайт RAM»

**Что делаем:** воспроизводим классику — приложение упало, сегмент остался занятым,
но никому не принадлежит. Создаём такие сегменты, **сохраняя их shmid**, чтобы потом
удалить именно свои.

```bash
# Создаём «осиротевшие» сегменты и сразу запоминаем их shmid
LAB_SHMIDS=()
for i in 1 2 3; do
  out=$(ipcmk -M 100M); echo "$out"
  LAB_SHMIDS+=("$(echo "$out" | awk '{print $NF}')")
done
ipcs -m
grep Shmem /proc/meminfo
```

**Что смотрим (как распознать утечку в реальной системе):**

```bash
# 1) Shmem большой
grep Shmem /proc/meminfo
# 2) Никто не сидит на сегментах (nattch = 0)
ipcs -m | awk '$6==0 && $2 ~ /[0-9]+/{print $0}'
```

**Что делаем (безопасная чистка — только свои):**

```bash
for id in "${LAB_SHMIDS[@]}"; do ipcrm -m "$id"; done
ipcs -m
lsipc -m        # современный аналог просмотра (util-linux >= 2.27)
```

> ⚠️ **Почему не массово `ipcs … | xargs ipcrm`:** удаление **всех** сегментов с
> `nattch=0` снесёт и те, к которым приложение ещё не успело подключиться (`shmat`).
> На проде сносите только то, про что точно знаете, что это мусор.

**Как это работает / вывод:** в современных приложениях SysV shm почти вытеснен
POSIX shm и mmap. Видите утечку SysV shm — почти наверняка это легаси на C/C++ или
старый Oracle/Sybase.

---

### 5.5 mmap с MAP_SHARED — самый универсальный механизм

**Что делаем:** два процесса разделяют память через `mmap` одного файла: один пишет,
другой читает.

```bash
dd if=/dev/zero of=/tmp/mmap-shared bs=1M count=10 status=none   # файл-бэкенд

# Процесс 1: пишет в файл через общий маппинг
python3 -c '
import mmap, time
f = open("/tmp/mmap-shared", "r+b")
m = mmap.mmap(f.fileno(), 10*1024*1024, mmap.MAP_SHARED)
m[0:5] = b"HELLO"
print("writer wrote HELLO, sleeping")
time.sleep(60)
' &
WRITER_PID=$!
sleep 1

# Процесс 2: читает из того же файла — видит то, что записал первый
python3 -c '
import mmap
f = open("/tmp/mmap-shared", "r+b")
m = mmap.mmap(f.fileno(), 10*1024*1024, mmap.MAP_SHARED)
print("reader sees:", m[0:5])
'
# reader sees: b'HELLO'

kill "$WRITER_PID" 2>/dev/null     # точечно, без killall
rm /tmp/mmap-shared
```

**Как это работает / вывод:** `MAP_SHARED` означает, что изменения видны всем, кто
замапил тот же файл, и пишутся обратно в файл. На этом механизме построены и POSIX
shm (это mmap файла в `/dev/shm`), и shared buffers PostgreSQL.

2. Где физически живут POSIX shm-объекты? Как посмотреть их список?

## Модуль 6: Боевые инциденты (troubleshooting)

Память — главный источник инцидентов в поддержке. Здесь три кейса в формате
«симптом → диагностика → root cause → решение». Это ровно то, что спрашивают на
собеседовании фразой «сервис убивают по ночам, ваши действия».

---

### Инцидент 1: сервис рестартует по ночам — кто его убивает?

**Симптом:** процесс периодически перезапускается, в его логе обрыв без ошибки.

**Что делаем (диагностика):**

```bash
# Воспроизведём OOM безопасно — в изолированном scope (хост не пострадает)
sudo systemd-run --scope -p MemoryMax=256M --slice=memlab \
  stress-ng --vm 1 --vm-bytes 1G --vm-keep -t 20 || true

# Ищем след OOM-killer в журнале ядра
journalctl -k | grep -iE 'out of memory|killed process|oom' | tail
dmesg | grep -iE 'killed process|oom' | tail
```

**Что смотрим:** строка вида `Out of memory: Killed process <PID> (stress-ng) … 
total-vm:…, anon-rss:…`. Ядро называет жертву и причину.

**Как это работает:** при нехватке памяти ядро выбирает жертву по `oom_score`
(грубо — кто больше всех занял). Подкрутить «кого жалеть» можно через `oom_score_adj`:

```bash
cat /proc/self/oom_score
cat /proc/self/oom_score_adj      # -1000..1000; меньше = реже убивают
```

**Root cause / решение:** либо у сервиса утечка (см. инцидент 2), либо ему банально
мало RAM/лимита. Решение — поднять лимит (`MemoryMax`/k8s `limits.memory`), починить
утечку или защитить критичный процесс через `oom_score_adj`.

### Инцидент 2: «память течёт» — а течёт ли?

**Симптом:** RSS сервиса растёт под нагрузкой и не падает после неё.

**Что делаем (диагностика):** воспроизводим §2.4 и смотрим, помогает ли `malloc_trim`:

```bash
python3 <<'EOF'
import ctypes, os
libc = ctypes.CDLL("libc.so.6")
libc.malloc.restype = ctypes.c_void_p          # иначе адрес усечётся до 32 бит
libc.malloc.argtypes = [ctypes.c_size_t]
libc.free.argtypes = [ctypes.c_void_p]
libc.malloc_trim.argtypes = [ctypes.c_size_t]
def rss():
    return int(open(f"/proc/{os.getpid()}/status").read().split("VmRSS:")[1].split()[0])
ptrs=[libc.malloc(1024) for _ in range(100_000)]
for p in ptrs: ctypes.memset(p,1,1024)
print("после malloc:", rss(), "KB")
for p in ptrs: libc.free(p)
print("после free:  ", rss(), "KB")
libc.malloc_trim(0)
print("после trim:  ", rss(), "KB")
EOF
```

**Что смотрим:** если после `free` RSS большой, а после `trim` упал — это **не утечка**,
а удержание heap аллокатором glibc. Если `trim` не помогает и RSS растёт бесконечно —
это **настоящая** утечка (объекты не освобождаются).

**Root cause / решение:** настоящую утечку ищут профайлером (`valgrind --leak-check`,
`heaptrack`, для Python `tracemalloc`). «Ложную» лечат настройкой аллокатора
(`MALLOC_TRIM_THRESHOLD_`, `MALLOC_ARENA_MAX`) или периодическим `malloc_trim`.

### Инцидент 3: хост «колом» — всё тормозит, CPU простаивает

**Симптом:** высокая latency, `load average` большой, но CPU `id` высокий — система
ждёт не процессор.

**Что делаем (диагностика):**

```bash
vmstat 1 5                        # смотрим si/so и wa
cat /proc/pressure/memory         # full avg10 — главный индикатор
swapon --show                     # сколько swap занято
free -h
```

**Что смотрим:** стабильно ненулевые `si/so` в `vmstat` + `pressure/memory full
avg10 > 10` = **swap-thrashing**: ядро без конца гоняет страницы RAM↔swap, и CPU
простаивает в ожидании диска.

**Root cause / решение:** памяти не хватает, и swap лишь маскирует это, замедляя всё.
Лечение — добавить RAM / уменьшить рабочий набор; для low-latency сервисов снизить
`swappiness` (1..10) или вынести их из-под свопа; найти и ограничить «прожорливый»
процесс (`ps -eo pid,rss,comm --sort=-rss | head`).

```bash
# уберём scope из инцидента 1, если остался
sudo systemctl stop memlab.slice 2>/dev/null || true
```

## Теоретические вопросы (итоговые)

> Ответы свёрнуты — сначала ответьте сами, потом разверните и сверьтесь.

### Блок 1: Виртуальная и физическая память

1. Что такое виртуальная память и зачем она, если есть физическая RAM?
2. Разница между VSZ, RSS, SHR и PSS. Какую цифру брать для алёрта «процесс жрёт память»?
3. Что такое demand paging? Почему `mmap(1GB)` не выделяет 1 ГБ сразу?
4. Что означает `MemAvailable` в `/proc/meminfo`? Чем отличается от `MemFree`?
5. Зачем `vmstat` показывает `si` и `so`? Что плохого, если они стабильно ненулевые?

<details><summary>Ответы (блок 1)</summary>

1. Абстракция: у каждого процесса своё непрерывное адресное пространство, изоляция,
   возможность дать «больше, чем есть» (overcommit) и подгружать по требованию.
2. VSZ — обещано (вся виртуалка), RSS — реально в RAM, SHR — разделяемая часть,
   PSS — RSS с честным делением общих страниц. Для алёрта — **RSS** (и `MemAvailable` по хосту).
3. Выдача физической страницы при первом обращении, а не при аллокации. Поэтому
   `mmap(1GB)` даёт только VSZ; RSS растёт по мере касания страниц.
4. `MemAvailable` — сколько реально можно отдать без свопа (с учётом освобождаемого
   кеша); `MemFree` — только совсем свободная RAM. Алёртить по `MemAvailable`.
5. `si/so` = swap in/out. Стабильно ненулевые → система свопается, вероятен дефицит RAM.
</details>

### Блок 2: Аллокация памяти

6. Какие системные вызовы стоят за `malloc()` в glibc? Когда какой?
7. Почему `free()` маленького блока не уменьшает RSS? Что это меняет?
8. Что такое `M_MMAP_THRESHOLD`? Значение по умолчанию?
9. Почему `pmap -x | grep \.so` показывает много памяти, но это не утечка?
10. У процесса в `/proc/<pid>/maps` много безымянных `anon`-блоков. Что это?

<details><summary>Ответы (блок 2)</summary>

6. `brk`/`sbrk` для мелких (двигают вершину heap), `mmap` для крупных (отдельный регион).
7. glibc держит освобождённую память в своём пуле для повторного использования;
   ядру не возвращает до `malloc_trim`/порога `M_TRIM_THRESHOLD`.
8. Порог, выше которого malloc идёт через `mmap`; старт 128 КБ, в glibc растёт
   динамически (до ~32 МБ) в зависимости от паттерна освобождений.
9. `.so` — разделяемые: одна копия в RAM на все процессы; суммировать RSS нельзя.
10. Анонимные `mmap` — большие malloc (> threshold), arena-области glibc или ручной `mmap`.
</details>

### Блок 3: Страницы и swappiness

11. Размер страницы по умолчанию на x86_64 и почему такой?
12. Чем `vm.swappiness=0` отличается от `=1` в современных ядрах (cgroups v2)?
13. На каких нагрузках уменьшать swappiness до 1–10, на каких — увеличивать?
14. Что такое PSI? Почему лучший индикатор «плохо от нехватки памяти», чем `free`?
15. Разница между вытеснением anon-страницы и отбрасыванием file-backed страницы.

<details><summary>Ответы (блок 3)</summary>

11. 4 КБ — компромисс между гранулярностью и накладными расходами таблиц страниц/TLB.
12. `0` — не свопить anon, пока есть что вытеснить из кеша (в cgroup v2 может полностью
    запретить anon-своп для группы); `1` — минимально свопить, но не отключать совсем.
13. Уменьшать — для БД/low-latency (анон важнее кеша); увеличивать — почти никогда,
    разве что десктоп с дефицитом RAM.
14. PSI меряет **время простоя из-за нехватки ресурса**, а не «сколько занято» —
    напрямую отражает деградацию; `free` не показывает давление.
15. Anon уникальна → её надо сохранить в swap; file-backed уже есть на диске →
    страницу можно просто выбросить и при нужде перечитать.
</details>

### Блок 4: Swap

16. Зачем swap, если RAM много? Два валидных кейса.
17. На каких системах swap скорее вреден? Почему?
18. Swap-файл vs раздел на современных SSD? Когда что?
19. Что будет, если в `/etc/fstab` указать несуществующий swap-файл и перезагрузиться?
20. Что такое `zram` и когда он лучше обычного swap?

<details><summary>Ответы (блок 4)</summary>

16. Гибернация (нужен swap ≥ RAM); защита от OOM на не-критичных сервисах/пики нагрузки;
    вытеснение редко используемых anon-страниц, чтобы освободить RAM под кеш.
17. Low-latency (БД, kafka, kvm-хост): свопнутая страница = резкий скачок latency;
    плюс износ дешёвых SSD по TBW.
18. Разница в скорости пренебрежима; файл гибче (создать/удалить на лету). Раздел —
    если нужна гибернация или жёсткая изоляция.
19. Для swap не критично: `swapon -a` ругнётся, но система загрузится. (Битый
    **обычный** mount в fstab может уронить загрузку — отсюда привычка проверять.)
20. Сжатый swap в RAM (нет диска). Хорош при дефиците RAM без хорошего SSD/на десктопах,
    контейнерах: «эластичная» память ценой CPU на сжатие.
</details>

### Блок 5: Анонимная и общая память

21. Три механизма shared memory. Чем различаются, какой для нового кода?
22. Где живут POSIX shm-объекты? Почему `df /dev/shm` показывает память?
23. Что такое `Shmem`? Может ли он уйти в swap?
24. Как найти осиротевшие SysV shm? Чем опасны для долгоживущих хостов?
25. PostgreSQL съел 4 ГБ shared memory. Как понять, что это, и учитывается ли это в RSS воркеров?

<details><summary>Ответы (блок 5)</summary>

21. SysV (древний, `ipcs`), POSIX (`shm_open`, файлы в `/dev/shm`), `mmap(MAP_SHARED)`
    (универсальный). Для нового кода — POSIX shm или `mmap`.
22. В tmpfs `/dev/shm`; tmpfs живёт в RAM, поэтому `df` показывает её лимит (обычно 50% RAM).
23. Разделяемая память (tmpfs/SysV/anon-shared). tmpfs-страницы могут уходить в swap
    (это не «отбрасываемый» file-backed, бэкенда на диске у них нет).
24. `ipcs -m` + `nattch=0`. Опасны тем, что занятую ими RAM не видно у процессов —
    «исчезающий гигабайт», копится при падениях легаси-приложений.
25. Это общий сегмент (shared buffers), один на кластер; в RSS каждого воркера он
    учитывается, но физически копия одна — суммировать RSS воркеров нельзя, смотреть PSS.
</details>

## Шпаргалка

```bash
# === Просмотр памяти ===
free -h                              # сводка (алёртить по available!)
cat /proc/meminfo                    # сырые поля
vmstat 1 5                           # динамика (si/so, wa)
grep -E 'AnonPages|Shmem|Mapped|Dirty' /proc/meminfo

# === Память процесса ===
ps -eo pid,vsz,rss,comm --sort=-rss | head
pmap -x <PID>                        # карта памяти
cat /proc/<PID>/maps                 # сырьём
cat /proc/<PID>/status | grep -E 'Vm|Rss'

# === Что под капотом у malloc ===
strace -e trace=brk,mmap,munmap -p <PID>
strace -e trace=brk,mmap python3 -c 'x = "a" * 10000000' 2>&1 | tail

# === Page cache (drop_caches — только для тестов!) ===
sync && echo 3 | sudo tee /proc/sys/vm/drop_caches

# === Swappiness ===
cat /proc/sys/vm/swappiness
sudo sysctl -w vm.swappiness=10
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-swappiness.conf

# === Pressure Stall Information ===
cat /proc/pressure/memory            # full avg10 > 10 — уже плохо
cat /proc/pressure/{cpu,io}

# === Безопасная нагрузка/OOM в изолированном scope ===
sudo systemd-run --scope -p MemoryMax=1G --slice=memlab stress-ng --vm 1 --vm-bytes 2G -t 60
journalctl -k | grep -iE 'oom|killed process' | tail

# === Swap ===
swapon --show
sudo fallocate -l 1G /swapfile && sudo chmod 600 /swapfile
sudo mkswap /swapfile && sudo swapon /swapfile
sudo swapoff /swapfile
echo "/swapfile none swap sw 0 0" | sudo tee -a /etc/fstab
sudo swapon -a

# === Shared memory ===
ipcs                                 # все SysV IPC объекты
ipcs -m                              # только shared memory (nattch=0 — кандидат в утечку)
ipcmk -M 100M                        # создать сегмент 100М
ipcrm -m <shmid>                     # удалить ТОЛЬКО свой shmid
lsipc -m                             # современный просмотр
ls /dev/shm/                         # POSIX shm объекты
```

## Чему вы научились

- Читать память Linux по делу: `free -h`/`MemAvailable`, `vmstat`, `/proc/meminfo` —
  и не пугаться «занятой» page cache.
- Различать VSZ и RSS и понимать demand paging (почему VIRT ≠ занятой RAM).
- Видеть `brk`/`mmap` под `malloc` через `strace` и объяснять «фантомную утечку»
  (glibc не вернул heap) против настоящей.
- Управлять `swappiness` и понимать, что это направление выбора, а не порог.
- Создавать, прописывать в `fstab` и снимать swap, не задев системный.
- Различать SysV/POSIX/`mmap` shared memory и **безопасно** удалять осиротевшие сегменты.
- Диагностировать боевые инциденты: OOM-kill (journalctl/oom_score), утечку
  (`malloc_trim`-тест), swap-thrashing (PSI + `si/so`) — и делать это, не уронив хост
  (нагрузка и OOM в cgroup-scope).
