# FINAL-TEST · Capstone (то самое тестовое из вакансии)

> Это **то же** 60–90-минутное тестовое из `/root/test-job`, но теперь у тебя за плечами 12 лаб и ты можешь его взять и сделать.

## Контекст

Из вакансии:

> План:
> - Часть A — Ansible: базовая конфигурация сервера;
> - Часть B — Observability: «агент» и системные метрики;
> - Часть C — Secrets: имитация Vault-подхода;
> - Часть D — Документация и подход.
>
> Тестовое задание рассчитано примерно на 60–90 минут непрерывной работы.

Этот capstone — **то самое**, разнесённое по чёткому чек-листу с критериями приёмки.

## Стенд

Одна Ubuntu 22.04/24.04 VM (рефер: AWS EC2 t3.micro), доступ по SSH-ключу, NOPASSWD-sudo. Если ты дошёл сюда — стенд у тебя уже есть с лаб [00–10].

## Стартовый шаблон

В этом каталоге есть [`SOLUTION-SKELETON/`](./SOLUTION-SKELETON/) — скелет репозитория, с которого начинаешь работу. Скопируй в свой git-репо:

```bash
cp -r SOLUTION-SKELETON/ ~/junior-test && cd ~/junior-test
git init && git add . && git commit -m "chore: starting from skeleton"
```

## Часть A — Ansible: baseline configuration (25 минут)

Цель: одним `ansible-playbook` привести VM в состояние:

- [ ] существует пользователь `deploy` в группе `sudo`, доступен по ключу из `files/deploy.pub`;
- [ ] ssh запрещает password-auth и root-login;
- [ ] установлены пакеты `curl jq htop chrony unattended-upgrades`;
- [ ] `chrony` и `unattended-upgrades` enabled + started;
- [ ] `ufw` enabled, разрешены `22/tcp` и `9100/tcp` (под node_exporter для Части B);
- [ ] hostname берётся из `inventory_hostname`;
- [ ] **повторный прогон даёт `changed=0`** (идемпотентность).

Опираешься на [лабу 03](../03-ansible-baseline/).

**Что отдать:**
- `ansible.cfg`, `inventory/hosts.ini`, `playbooks/site.yml`, `roles/common/*`.
- В README раздела A — команда прогона и образец вывода с `changed=0`.

## Часть B — Observability: агент и метрики (20 минут)

Цель: на той же VM подняты системные метрики, доступные снаружи.

- [ ] установлен `node_exporter` (через `apt` или systemd-юнит, **не** docker — потому что снаружи это «host metrics»);
- [ ] слушает 9100/tcp на 127.0.0.1 (доступ через ssh-туннель или через ufw для конкретного IP);
- [ ] установлен `prom/prometheus` (можно в docker), scrape job `node` собирает с `localhost:9100`;
- [ ] PromQL-запрос `rate(node_cpu_seconds_total{mode!="idle"}[1m])` возвращает ненулевое значение;
- [ ] (опционально) поднят `grafana/grafana` с дашбордом 1860.

Опираешься на [лабу 05](../05-observability-metrics/).

**Что отдать:**
- `observability/compose.yml` или `roles/observability/*` (на выбор);
- скриншот / `curl :9090/api/v1/targets` где `node` = `up`;
- в README раздела B — как открыть Grafana через ssh-туннель.

## Часть C — Secrets: имитация Vault-подхода (10–15 минут)

Цель: показать **подход**, а не обязательно поднять полный Vault. Допустимые варианты:

**Вариант 1 — Vault dev** (recommended, ~10 минут):
- [ ] запущен `hashicorp/vault` в dev-mode (docker);
- [ ] создан KV v2 mount `kv/`;
- [ ] записан секрет `kv/app/db` (фейковый пароль);
- [ ] создана policy `app-ro` (read on `kv/data/app/*`);
- [ ] выпущен token под этой policy, написана команда читания.

**Вариант 2 — ansible-vault** (минимум):
- [ ] секрет хранится в `group_vars/all/secrets.yml`, зашифрован `ansible-vault`;
- [ ] playbook читает его и кладёт в шаблон (`copy` с `content: "{{ db_password }}"`);
- [ ] **никакой plain-text** пароля в git.

Опираешься на [лабу 07](../07-secrets-vault/).

**Что отдать:**
- `secrets/README.md` с описанием выбранного подхода и команд reproduce.
- Если Vault — `secrets/bootstrap.sh` со всеми командами.

## Часть D — Документация и подход (10 минут)

Цель: каждый этап можно повторить чужому человеку, и человек не сломает себе мозг.

- [ ] корневой `README.md` отвечает на: «как поднять», «как проверить», «как удалить»;
- [ ] раздел «**Что я бы сделал на проде**» с честным списком (HA, Vault не dev, ufw сужает 9100 до Prom-IP, и т. д.);
- [ ] есть `docs/topology.md` (ASCII-схема);
- [ ] есть `ADR-0001.md` (выбор Docker vs systemd, **или** Vault vs ansible-vault — что выбрал и почему);
- [ ] коммиты осмысленные, в стиле Conventional Commits.

Опираешься на [лабу 12](../12-documentation/).

## Критерии оценки (рубрика)

| Что смотрим                                  | вес | где это берут                  |
|----------------------------------------------|-----|--------------------------------|
| Идемпотентность Ansible-плейбука             |  20 | повторный прогон → `changed=0` |
| Безопасность (нет паролей в git, ssh hard'ed) |  15 | grep по репе, sshd конфиг      |
| Метрики работают, target = up                |  15 | `curl :9090/api/v1/targets`    |
| Vault/secrets-подход показан корректно       |  15 | reproduce по `bootstrap.sh`    |
| README — «clone & run»                       |  10 | прогон чужим человеком         |
| Архитектурное мышление (ADR + «на проде»)    |  10 | чтение разделов                |
| Git-гигиена (commits, .gitignore)            |  10 | `git log --graph`              |
| Чистота (как удалить, как погасить, нет утечек диска) | 5 | проверка после teardown |

**Проходной балл** для джуна: ≥70.

## Бюджет времени

| Часть | Что | Время |

| A     | Ansible baseline | 25 мин |
| B     | Observability | 20 мин |
| C     | Secrets | 15 мин |
| D     | Документация | 10 мин |
| —     | Финальный прогон с нуля + чек | 10 мин |

Если на одном из этапов застрял больше чем на +10 мин от бюджета — **переходи к следующему**, потом вернёшься. Лучше иметь все 4 части на 70%, чем 2 части на 100% и 2 пропущенные.

## Как сдать

В формате PR в свой репозиторий (`gh pr create`) или архивом. В описании PR — пройденность чек-листа выше и ссылки на ключевые файлы.

## Полезные напоминания (без подсказок)

- ufw enable **после** allow 22, иначе обрубишь себе ssh — [01](../01-linux-baseline/).
- Не клади `RESTIC_PASSWORD`/`VAULT_TOKEN` в git, даже зашифрованные «временно» — [07](../07-secrets-vault/).
- На t3.micro 1 GB RAM: Vault dev + Prom + Grafana **одновременно** не держи — гаси compose-стеки между частями.
- Документация должна обновляться **в том же коммите**, что и код, не «потом отдельным PR».
