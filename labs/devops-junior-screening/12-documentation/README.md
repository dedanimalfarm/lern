# 12 · Документация и подход

> Тема из вакансии: «Работать строго по архитектурным схемам и инструкциям»; «Часть D — Документация и подход».

## Цель и навыки

Превратить «у меня всё работает локально» в «другой человек может это поднять, поломать и починить, не дёргая меня». Это и есть то, ради чего на собеседовании на тестовом задании вообще существует Часть D.

После лабы ты:

- знаешь четыре жанра инженерной документации (README, RUNBOOK, ADR, postmortem) и когда какой нужен;
- умеешь писать README, который проходит «**clone & run на чужой машине за 10 минут**»;
- пишешь runbook для оперативной ситуации, не для красоты;
- умеешь зафиксировать архитектурное решение в ADR на одну страницу;
- знаешь, как выглядит достойный blameless postmortem (Five Whys, timeline, action items с владельцами).

## Теоретический минимум — четыре жанра

| Жанр       | Кто читает                 | Когда обновляется         | Что обязательно             |
|------------|----------------------------|---------------------------|-----------------------------|
| **README** | новый человек в репе       | каждый PR, если меняется поведение | как поднять, как проверить, как удалить |
| **RUNBOOK**| дежурный в 3 ночи          | после каждого инцидента   | алёрт → проверь → почини → escalate |
| **ADR**    | будущий ты + коллеги       | один раз, при принятии решения | контекст, варианты, решение, последствия |
| **POSTMORTEM** | команда после инцидента | один раз, в течение 5 дней | timeline, root cause, action items, владельцы |

Эти четыре жанра **не взаимозаменяемы**. README ≠ runbook (runbook конкретен про один алёрт). ADR ≠ комментарий в PR (ADR живёт **дольше** PR).

## Базовая отработка

Будешь писать документы **под одну из своих лаб**. Возьми лабу [05](../05-observability-metrics/) — она про observability-стек и удобна для всех 4 жанров.

### Шаг 1. README — клонируем и запускаем

Шаблон в [TEMPLATE-README.md](./TEMPLATE-README.md). Скопируй, заполни под compose-стек из лабы 05. Главные проверки:

- Любой проходящий мимо запускает `docker compose up -d` и через 30 секунд видит метрики. Никаких «о, забыл сказать, надо ещё `chown`».
- Раздел «Как удалить» есть. Без него люди оставляют 3 ГБ мусора и винят тебя.
- Раздел «Что я бы сделал на проде» — отличает джуна от стажёра: ты показываешь, что **знаешь** ограничения учебного стенда (одна нода, dev-mode и т. д.).

### Шаг 2. RUNBOOK — конкретный алёрт

Шаблон в [TEMPLATE-RUNBOOK.md](./TEMPLATE-RUNBOOK.md). Напиши runbook для алёрта «HighCpuUsage» (см. [05](../05-observability-metrics/), расширенную задачу про alertmanager).

Хороший runbook на одном экране отвечает:

1. **Симптом**: какой алёрт, что видно в Grafana.
2. **Импакт**: что не работает с точки зрения пользователя.
3. **Первые 5 минут**: что проверить (`ssh`, `top`, `journalctl -p err -b`).
4. **Что починить**: 2–3 типовые причины и команда фикса для каждой.
5. **Когда эскалировать**: «если за 15 минут не легче — пишем lead'у в …».

### Шаг 3. ADR — почему compose, а не systemd-нативно

Шаблон в [TEMPLATE-ADR.md](./TEMPLATE-ADR.md). Напиши ADR-0001 «Используем docker compose, а не native systemd units, для observability-стека в этой лабе».

В ADR должно быть:

- **Контекст**: на одной маленькой VM, без k8s, без Nomad.
- **Решение**: docker compose.
- **Альтернативы**: чистые systemd units (плюс — нативность; минус — больше boilerplate, без сетевой DNS-магии).
- **Последствия**: завязка на Docker, расход RAM из-за runtime, проще shareability.

ADR — это не «как», это **«почему»**. Через год ты сам не вспомнишь, отчего выбрал compose. ADR — твоя записка-самому-себе.

### Шаг 4. POSTMORTEM — симулированный инцидент

Шаблон в [TEMPLATE-POSTMORTEM.md](./TEMPLATE-POSTMORTEM.md). Возьми реальный инцидент из своей работы по лабам (например: «при включении ufw без allow 22 потерял ssh, восстанавливал через console»).

Главные правила:

- **Blameless**: «процесс позволил X», а не «иванов сделал X».
- **Timeline в UTC** с точностью минут.
- **Root cause ≠ trigger**. Триггер: «нажал ufw enable». Root cause: «не было make-таргета, который автоматически добавляет ssh в allow до enable».
- **Action items с владельцем и датой**. Без владельца — не сделается.

## Расширенная отработка

### Задача 1. Документация в коде

Возьми Ansible-роль из [03](../03-ansible-baseline/) и допиши:

- `meta/main.yml` — авторов, описание, platforms.
- `tasks/main.yml` — `name:` у **каждой** task'и должен читаться человеком в выводе `ansible-playbook`.
- `defaults/main.yml` — каждый параметр с комментарием: что задаёт, дефолт, на что влияет.

После — другой человек, прочитав `defaults/main.yml`, должен понять контракт роли **без чтения tasks/**.

### Задача 2. ASCII-диаграмма стенда

В корень своего лабораторного репо положи `docs/topology.md` с ASCII-диаграммой:

```
+----------------+        ssh        +-----------------+
| control (you)  | <---------------> |   ec2 t3.micro  |
| ~/.ssh/vast    |                   |   ubuntu/sudo   |
+----------------+                   |                 |
                                     | docker:         |
                                     |  ├─ prometheus  |
                                     |  ├─ grafana     |
                                     |  ├─ loki        |
                                     |  └─ vault       |
                                     +-----------------+
```

ASCII лучше PNG: версионируется, диффится, не требует Lucidchart.

### Задача 3. CHANGELOG в стиле Keep a Changelog

Сделай `CHANGELOG.md` с разделами `## [Unreleased]`, `## [0.1.0] — 2026-05-23`. Под каждой версией — категории `Added / Changed / Fixed / Removed / Security`. Это **не лишний труд**, это контракт с любым, кто использует твой код.

## Acceptance criteria

- [ ] Есть `README.md` для лабы 05, по которому **другой человек** поднимет стенд без вопросов.
- [ ] Есть `RUNBOOK-HighCpuUsage.md` на одну страницу.
- [ ] Есть `ADR-0001-compose-vs-systemd.md` с тремя альтернативами и решением.
- [ ] Есть `POSTMORTEM-<дата>.md` про один случай за время прохождения сборника.
- [ ] (Расширенная) `meta/`, `defaults/`, `tasks/` роли документированы; в `docs/` лежит топология.

## Что обсудить на ревью

1. Зачем ADR, если есть commit-message и PR-description?
2. Что делает postmortem **blameless** на самом деле — не «не назвать имя», а что?
3. Можно ли использовать ChatGPT/Claude для написания runbook? (Подсказка: да, как **черновик**, но факты-команды надо сверять с реальностью.)
4. Где у тебя живёт документация? Wiki / Notion / в репе? Какие плюсы/минусы каждого варианта?
5. Что делать с outdated документами — стирать, помечать `DEPRECATED`, переписывать?

## Грабли

| Симптом | Причина | Лечение |
|---------|---------|---------|
| README расходится с кодом | менялся код без README | code review должен это ловить; добавить чек в CI |
| Runbook ссылается на устаревшие команды | не обновили после refactor'а | дата «last verified» в шапке + квартальная ревизия |
| Postmortem превращается в blame-сессию | без модератора и шаблона | используй TEMPLATE-POSTMORTEM с явным разделом «what went well» |
| ADR не пишут «потому что лень» | нет привычки и шаблона | держать TEMPLATE-ADR в репе, накатить «1 ADR в месяц» |
