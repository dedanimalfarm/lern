# 12 — Rootless: контрольные вопросы

**Зачем нужны rootless-контейнеры?**
Запуск контейнеров от настоящего root (uid 0 на хосте) опасен: при уязвимости,
позволяющей выйти из изоляции, атакующий получает root на хосте. Rootless запускает
контейнер от непривилегированного пользователя — побег даёт лишь права этого
пользователя, не root. Так работают podman и rootless Docker.

**Что делают `-U` и `-r` и что будет без `-r`?**
`-U` (`--user`) создаёт новый user namespace. `-r` (`--map-root-user`) отображает
твой uid хоста в uid 0 внутри. Без `-r` ты внутри — неотображённый «overflow» uid
(65534/nobody), и root-операции (mount, chown) падают: ты не root в этом ns.

**Кому на хосте принадлежит файл, созданный «root-ом» внутри rootless?**
Реальному непривилегированному пользователю, создавшему namespace (его host-uid).
Видно в `/proc/<pid>/uid_map`: строка `0 <host_uid> 1`. Внутри uid 0, снаружи —
host_uid. Это и есть гарантия безопасности.

**В чём ограничение single-uid маппинга и чем лечится?**
`--map-root-user` отображает РОВНО один uid (твой → 0). Других uid внутри нет:
`chown 1000` → `Invalid argument`. Для нескольких uid нужен диапазон: `/etc/subuid`
(`user:100000:65536`) + `newuidmap`/`newgidmap`.

**Какие утилиты и файлы нужны для multi-uid (podman/Docker rootless)?**
- `/etc/subuid`, `/etc/subgid` — выделенные пользователю диапазоны под-uid/gid.
- `newuidmap`, `newgidmap` (пакет `uidmap`) — setuid-root помощники, которые
  записывают `uid_map`/`gid_map` с диапазоном (обычный пользователь сам диапазон
  записать не может). podman/rootless-docker вызывают их под капотом.
