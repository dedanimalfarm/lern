# Сценарий 01: новый PID-namespace, но `$$` не 1 / `ps` падает «fatal library error, lookup self»

## Симптом
Создаём PID-namespace, а он ведёт себя «не как контейнер»: то `$$` не равен 1, то
`ps` падает странной ошибкой про библиотеку.
```bash
sudo ./broken/scenario-01/make-broken.sh
# (A) --pid без --fork:
#   PID внутри: 1475428 (ожидали 1)
# (B) --pid --fork БЕЗ --mount-proc — ps падает:
#   $$=1
#   fatal library error, lookup self
#   процессов: 0
```

## Подсказки
1. Случай A: кто на самом деле попадает в новый PID-ns — сам `unshare` или его потомок?
2. Случай B: `$$` уже 1 — значит ns есть. Тогда откуда `ps` берёт данные? Что показывает `/proc`?
3. Какие ровно три флага нужны PID-ns, чтобы «всё как в Docker»?

## Диагностика
- **A — забыт `--fork`.** `unshare(CLONE_NEWPID)` создаёт новый PID-ns, но сам
  процесс `unshare`/`bash` остаётся в СТАРОМ ns (у него уже есть PID). PID 1 в
  новом ns получит только его первый ПОТОМОК. Без `--fork` команда запускается
  тем же процессом → видит свой старый хостовый PID (`1475428`).
- **B — забыт `--mount-proc`.** PID-ns создан (`$$=1`), но `/proc` остался
  хостовым. `ps`/procps читает `/proc`, ищет там себя по PID 1 (своему в новом ns)
  — и не находит соответствия → `fatal library error, lookup self`. `/proc`
  показывает чужой (хостовый) pid-ns.

## Решение
Триада `--pid --fork --mount-proc` (см. `solutions/01-pid-fork-mountproc/fix.sh`):
```bash
sudo ./solutions/01-pid-fork-mountproc/fix.sh
#   $$=1
#   ps -e: 4 процесса (только свои)
```
- `--pid` — создать PID namespace;
- `--fork` — запустить команду как ПОТОМКА (он и станет PID 1);
- `--mount-proc` — заодно сделать mount-ns и перемонтировать `/proc`, чтобы он
  отражал новый pid-ns (тогда `ps` видит только свои процессы).

## Профилактика
- Для PID-ns всегда используй полную триаду `--pid --fork --mount-proc`.
- Помни: `--mount-proc` неявно поднимает и mount-namespace (иначе перемонтировать
  `/proc` без вреда хосту нельзя).
- В Docker/runc эту триаду делает рантайм — поэтому внутри контейнера `ps` сразу
  «правильный». Голый `unshare --pid` без флагов — частый источник путаницы.
