## База

1. Установите необходимые пакеты:
    ```bash
    sudo apt update
    sudo apt install cgroup-tools sysstat htop
    ```

2. Проверьте, что система использует cgroup версии 2:
    ```bash
    mount | grep cgroup2
    ```

## Шаг 1: Определение идентификатора диска

1. Выполните команду `lsblk`, чтобы узнать идентификатор вашего диска:
    ```bash
    lsblk
    ```
    Например, для диска `/dev/sda` идентификатором может быть `8:0`. Для NVMe дисков это может быть `259:0` и т.д. Обязательно обратите внимание на столбец `MAJ:MIN`.

## Шаг 2: Настройка ограничения скорости записи на диск с использованием cgroups

1. Создайте новую группу cgroup для ограничения скорости записи:
    ```bash
    sudo mkdir /sys/fs/cgroup/io_limit_group
    ```
    
Добавьте подсистему `io` если она не присутствует:

    ```bash
    echo "+io" | sudo tee /sys/fs/cgroup/cgroup.subtree_control
    ```

2. Установите ограничение на запись для диска. Например, для диска `sda` с идентификатором `8:0` ограничим запись до 1 MB/с:
    ```bash
    echo "8:0 wbps=1048576" | sudo tee /sys/fs/cgroup/io_limit_group/io.max
    ```

3. Проверьте, что ограничение было установлено:
    ```bash
    cat /sys/fs/cgroup/io_limit_group/io.max
    ```

## Шаг 3: Запуск процесса записи на диск

1. Запустите процесс записи на диск с использованием `dd`, который создаст файл размером 5 GB:
    ```bash
    dd if=/dev/zero of=/tmp/testfile bs=1M count=5000 oflag=direct &
    ```

2. Найдите PID запущенного процесса `dd` (вы также увидите его в консоли после запуска в фоне, например `[1] 12345`):
    ```bash
    ps aux | grep dd
    ```

3. Добавьте процесс в созданную группу cgroup:
    ```bash
    echo <PID> | sudo tee /sys/fs/cgroup/io_limit_group/cgroup.procs
    ```
    Замените `<PID>` на фактический идентификатор процесса.

## Шаг 4: Проверка результата

1. Используйте `iostat` или `htop` для мониторинга производительности диска:
    ```bash
    iostat -dx 2
    ```

    Следите за скоростью записи (wMB/s), она должна быть ограничена до 1 MB/с.

## Шаг 5: Запуск процесса с использованием `systemd-run`

1. Вы можете запустить процесс записи на диск с ограничением скорости записи напрямую через `systemd-run` (не требуется создавать cgroup вручную):
    ```bash
    sudo systemd-run --scope -p "IOWriteBandwidthMax=/dev/sda 1M" dd if=/dev/zero of=/tmp/testfile bs=1M count=100 oflag=direct
    ```
    Эта команда сразу запускает процесс с ограничением записи до 1 MB/с.

## Troubleshooting — частые проблемы

1. **Файл `io.max` отсутствует в созданной группе**  
   - Убедитесь, что вы включили IO-контроллер для потомков в родительской cgroup: `echo "+io" | sudo tee /sys/fs/cgroup/cgroup.subtree_control`.  
   - Если контроллер все еще не появляется, проверьте, поддерживается ли он на вашем ядре и виртуальном окружении.

2. **Ограничение не применяется к процессу (скорость выше ожидаемой)**  
   - В примере мы используем `oflag=direct`. Если убрать этот флаг, `dd` будет писать в page cache ОС (в память), а не напрямую на диск, из-за чего видимая скорость может быть огромной. `io.max` в cgroup v2 для записи без direct IO работает путем ограничения "грязных страниц" (dirty pages) при их сбросе на диск, поэтому в реальном времени команда `dd` завершится быстрее, а процесс сброса кэша ОС (writeback) растянется во времени.
   - Вы указали неверный идентификатор блочного устройства (например, `8:0` вместо правильного значения). Проверьте `lsblk` (столбец `MAJ:MIN`).

3. **Команды возвращают ошибку "Permission denied"**  
   - Убедитесь, что вы выполняете команды записи (`echo ... > ...`) через `sudo tee` (как показано в шагах) или работаете из-под `root`. Прямое перенаправление `echo ... > /sys/...` под обычным пользователем через `sudo` не сработает (sudo применится только к echo).

## Проверка модуля

Для автоматической проверки успешности выполнения задания (после Шага 4) вы можете использовать скрипт `verify.sh`:

```bash
./verify.sh
```

Он проверит правильность создания cgroup и наличия настройки в `io.max`.

## Контрольные вопросы

1. В чем разница применения лимитов с помощью ручного создания cgroups (`mkdir /sys/fs/cgroup/...`) и с помощью `systemd-run`? Какие преимущества дает каждый подход?
2. Почему в Шаге 3 мы используем флаг `oflag=direct` в утилите `dd`? Как изменится поведение, если его не указать, и как cgroups v2 учитывает буферизированный (buffered) ввод-вывод по сравнению с прямым (direct IO)?
3. Если у процесса есть несколько потоков, нужно ли добавлять в `cgroup.procs` все TID (идентификаторы потоков), или достаточно добавить только PID основного процесса? Почему?
