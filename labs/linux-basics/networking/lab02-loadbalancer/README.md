## Часть 1: Теория L4 vs L7 балансировки и алгоритмы

### L4 Балансировка (Транспортный уровень)
* Работает на уровне TCP/UDP (4 уровень модели OSI).
* Балансировщик принимает решение о маршрутизации запроса на основе IP-адресов и TCP/UDP портов отправителя и получателя.
* Он **не заглядывает внутрь пакета** (не разбирает HTTP-заголовки, куки или тело запроса).
* **Плюсы:** Экстремально высокая производительность, низкое потребление CPU и памяти.
* **Минусы:** Отсутствие интеллектуального роутинга (нельзя направить запрос `/api` на один пул серверов, а `/static` на другой; нельзя реализовать авторизацию на уровне балансировщика).

### L7 Балансировка (Прикладной уровень)
* Работает на уровне прикладных протоколов, таких как HTTP, HTTPS, gRPC (7 уровень модели OSI).
* Балансировщик полностью устанавливает TCP-соединение с клиентом, парсит HTTP-запрос и принимает решение на основе URI, заголовков, Cookies, метода HTTP и т.д.
* **Плюсы:** Огромная гибкость. Поддержка "липких сессий" (sticky sessions) по Cookie, распределение по путям (маршрутизация URL), терминация SSL/TLS, фильтрация вредоносных запросов (WAF).
* **Минусы:** Выше задержка (latency), требуется значительно больше CPU и RAM для разбора трафика.

### Основные алгоритмы распределения нагрузки:
1. **Round Robin (циклический):** Запросы последовательно направляются на серверы по кругу (Web1 -> Web2 -> Web3 -> Web1...). Используется по умолчанию, подходит для одинаковых по мощности серверов.
2. **Least Connections (минимум соединений):** Новый запрос отправляется серверу с наименьшим количеством активных TCP-соединений. Эффективен при неравномерном времени обработки запросов (например, тяжелые SQL-запросы или долгие скачивания файлов).
3. **Source / IP Hash (по хэшу источника):** Хэширует IP-адрес клиента и сопоставляет его с определенным сервером. Гарантирует, что один и тот же клиент всегда будет попадать на один и тот же бэкенд (пока тот жив). Помогает сохранять состояние сессии (Session State) без общего кэша сессий.

### 2.1 Тестирование балансировки Round Robin

Отправьте несколько запросов к HAProxy и посмотрите, кто отвечает:
```bash
curl http://localhost:8080
curl http://localhost:8080
curl http://localhost:8080
```
Вы должны увидеть циклическую смену ответов:
* `Hello from Server 1`
* `Hello from Server 2`
* `Hello from Server 3`

Это чистая демонстрация L7-балансировки по алгоритму `Round Robin`.

### 2.2 Работа с панелью статистики HAProxy (Stats)

Мы настроили встроенную веб-панель статистики HAProxy на порту `8404`.

Вы можете запросить сырые метрики через CLI:
```bash
curl -s http://localhost:8404/ | head -n 20
```

А если вы пробросите порт `8404` на свою рабочую станцию, вы сможете открыть панель в браузере: `http://<IP-хоста>:8404/`.
Там в режиме реального времени отображается:
* Статус каждого бэкенда (зеленый — UP, красный — DOWN).
* Объем переданных байт (Sessions -> Bytes -> In/Out).
* Результаты периодических проверок здоровья (Health check -> LastChk).

## Часть 3: Troubleshooting — боевой инцидент (Бэкенд флапает)

**Флапание бэкенда (Flapping / Bouncing)** — состояние, при котором сервер то отвечает на проверки здоровья, то проваливает их. Это один из самых коварных инцидентов, так как он вызывает периодические ошибки 502/503 у клиентов и "штормит" логи балансировщика.

### Симптомы инцидента
Клиенты периодически жалуются, что сайт выдает ошибку "503 Service Unavailable" или долго зависает при загрузке. При этом бэкенд-серверы выглядят запущенными в системе (`pgrep` их видит).

### Шаг 1: Воспроизведение инцидента
Сымитируем нестабильное сетевое соединение с бэкендом `web1` (порт `8081`). Воспользуемся модулем `statistic` фаервола `iptables`, чтобы отбрасывать 50% входящих TCP-пакетов на этот порт:

```bash
sudo iptables -A INPUT -p tcp --dport 8081 -m statistic --mode random --probability 0.5 -j DROP
```

### Шаг 2: Диагностика инцидента
Запустите непрерывный опрос балансировщика:
```bash
for i in {1..20}; do curl -s --connect-timeout 1 http://localhost:8080 || echo "ERROR 503"; sleep 0.5; done
```
Вы увидите, что периодически запросы подвисают или балансировщик дольше переключается на рабочие серверы.

Теперь заглянем в логи. HAProxy отправляет свои логи в `syslog`. Посмотрим логи системы:
```bash
# Если используется systemd-journald:
sudo journalctl -u haproxy -n 30 --no-pager || sudo tail -n 50 /var/log/messages || sudo tail -n 50 /var/log/syslog
```
*(В зависимости от конфигурации syslog вашей ОС логи будут в одном из этих мест).*

Вы должны увидеть записи о падении и восстановлении сервера `web1`:
```text
[WARNING] ... : Health Check for member server web1 failed
[ALERT] ... : backend http_back has no server available! (если падали и другие)
[WARNING] ... : Server http_back/web1 is DOWN, reason: Layer7 timeout, info: " stroke"
[WARNING] ... : Server http_back/web1 is UP, reason: Layer7 check passed
```
HAProxy зафиксировал флапание!

### Как это работает под капотом (Алгоритм Health Check)?
Посмотрите на параметры в `haproxy.cfg`:
`server web1 127.0.0.1:8081 check`
По умолчанию HAProxy выполняет проверку здоровья каждые 2000 мс (параметр `inter`). 
Если проверка проваливается 3 раза подряд (параметр `fall`), сервер помечается как `DOWN` и трафик на него больше не идет. 
Как только проверка успешно проходит 2 раза подряд (параметр `rise`), сервер возвращается в статус `UP`.

При потере 50% пакетов бэкенд успевает провалить 3 проверки подряд, а затем успешно пройти 2 подряд. Это заставляет HAProxy бесконечно переключать состояние бэкенда.

### Шаг 3: Устранение инцидента
Удалите правило `iptables`, блокирующее трафик:
```bash
sudo iptables -D INPUT -p tcp --dport 8081 -m statistic --mode random --probability 0.5 -j DROP
```
После этого проверьте панель статистики на порту `8404` или посмотрите логи — статус сервера `web1` окончательно стабилизируется в `UP`.

## Практические задания (отработка)

1. **Изменение алгоритма балансировки:** Отредактируйте `haproxy.cfg` и замените алгоритм балансировки `balance roundrobin` на `balance source`. Перезапустите HAProxy (`sudo killall haproxy && sudo haproxy -f ./haproxy.cfg -D`). Выполните серию curl-запросов и убедитесь, что вам отвечает всегда только один и тот же бэкенд.
2. **Настройка параметров Health Check:** Настройте для серверов более жесткие параметры проверки здоровья: интервал в 1 секунду, отключение после 2 неудач, включение после 2 успехов. Формат строки бэкенда:
   `server web1 127.0.0.1:8081 check inter 1s fall 2 rise 2`

## Шпаргалка

```bash
# === Управление HAProxy ===
haproxy -f ./haproxy.cfg -c       # проверить валидность конфигурационного файла
haproxy -f ./haproxy.cfg -D       # запустить HAProxy в режиме демона (в фоне)
killall haproxy                   # остановить все процессы HAProxy
pkill -HUP haproxy                # мягкая перезагрузка (graceful reload) без обрыва соединений

# === Анализ соединений и портов ===
ss -tlnp | grep haproxy           # показать порты, которые слушает HAProxy
curl -s http://localhost:8404/    # запросить HTML-страницу статистики

# === Имитация сетевых сбоев (iptables) ===
# Блокировка порта полностью:
sudo iptables -A INPUT -p tcp --dport 8081 -j DROP
sudo iptables -D INPUT -p tcp --dport 8081 -j DROP
# Эмуляция нестабильного канала (потери пакетов):
sudo iptables -A INPUT -p tcp --dport 8081 -m statistic --mode random --probability 0.3 -j DROP
```
