## Часть 1: Теория VLAN, Access/Trunk портов и 802.1q

**VLAN (Virtual Local Area Network)** — технология, позволяющая на одном физическом сетевом оборудовании (коммутаторе, кабелях) создать несколько изолированных друг от друга логических сетей L2-уровня (широковещательных доменов).

### 1.1 Стандарт IEEE 802.1q (Тегирование)
Для разделения трафика к Ethernet-кадру добавляется специальный 4-байтовый тег (802.1q заголовок), содержащий:
* **TPID (0x8100):** Идентификатор протокола тегирования.
* **VLAN ID (VID):** 12-битное число от 1 до 4094, указывающее, какому именно VLAN принадлежит этот кадр.

### 1.2 Access и Trunk порты
* **Access (доступ):** Порт, к которому подключается конечное устройство (компьютер, принтер, сервер). Хост не знает про VLAN и отправляет обычные кадры. На входе в Access-порт коммутатор вешает на кадр тег соответствующего VLAN. При выходе из Access-порта к хосту тег **снимается**.
* **Trunk (транк):** Порт, предназначенный для передачи трафика нескольких VLAN (например, линк между двумя свитчами или свитчем и роутером). В транковом линке кадры передаются **с тегами**.
* **Native VLAN:** Один из VLAN в транковом порту, трафик которого передается без тега (по умолчанию VLAN 1). Если коммутатор получает на транковом порту нетегированный кадр, он относит его к Native VLAN.

### 1.3 Схема Router-on-a-stick
Если хосты находятся в разных VLAN (например, VLAN 10 и VLAN 20), они физически не могут общаться на L2 уровне, даже будучи подключенными к одному свитчу. Чтобы передать пакет между ними, нужен роутер.
В схеме **Router-on-a-stick** роутер подключается к свитчу одним физическим кабелем (Trunk). На физическом интерфейсе роутера создаются виртуальные саб-интерфейсы (sub-interfaces) для каждого VLAN (например, `eth0.10` и `eth0.20`), которые служат шлюзами по умолчанию для соответствующих подсетей.

## Часть 2: Настройка Inter-VLAN маршрутизации (Router-on-a-stick)

Наш скрипт уже создал схему:
* `host1` (VLAN 10) подключен к `br0`. На его VLAN-интерфейсе `veth-h1.10` настроен IP `10.0.10.10/24`.
* `host2` (VLAN 20) подключен к `br0`. На его VLAN-интерфейсе `veth-h2.20` настроен IP `10.0.20.20/24`.
* `router` имеет виртуальные интерфейсы `veth-r.10` (`10.0.10.1`) и `veth-r.20` (`10.0.20.1`) на физическом линке `veth-r`.

### 2.1 Проверка маршрутизации

Проверьте доступность хоста в другом VLAN:
```bash
sudo ip netns exec host1 ping -c 3 10.0.20.20
```
Пинг должен проходить успешно.

### 2.2 Анализ тегов на канальном уровне (L2)

Чтобы убедиться, что пакеты действительно идут с тегами 802.1q, запустим `tcpdump` на виртуальном свитче `br0` с флагом `-e` (выводить link-level заголовки) и фильтром `vlan`:

```bash
sudo tcpdump -i br0 -e -n -c 4 vlan
```
В соседнем терминале запустите пинг: `sudo ip netns exec host1 ping -c 2 10.0.20.20`.

В выводе `tcpdump` вы увидите детальную структуру кадров:
```text
.. ethertype 802.1Q (0x8100), vlan 10, p 0, ethertype IPv4, 10.0.10.10 > 10.0.20.20: ICMP echo request ...
.. ethertype 802.1Q (0x8100), vlan 20, p 0, ethertype IPv4, 10.0.10.10 > 10.0.20.20: ICMP echo request ...
```
Обратите внимание:
1. Запрос выходит из `host1` во `vlan 10` и доходит до роутера.
2. Роутер перенаправляет его во `vlan 20` в сторону `host2`.
Это доказывает работу Inter-VLAN маршрутизации.

## Часть 3: Troubleshooting — боевой инцидент (Нетегированный трафик / Native VLAN)

### Симптомы инцидента
Администратор перенастроил `host1`. Процессы запущены, IP-адрес `10.0.10.10` назначен, но хост не может пинговать даже свой шлюз `10.0.10.1`. Пинг до соседнего VLAN тем более не идет.

### Шаг 1: Воспроизведение инцидента
Имитируем ошибку конфигурации на `host1`: удалим тегированный интерфейс стандарта 802.1q и назначим IP-адрес напрямую на физический (нетегированный) интерфейс `veth-h1`:

```bash
# Удаляем правильный VLAN-сабинтерфейс
sudo ip netns exec host1 ip link del veth-h1.10

# Назначаем IP прямо на физическую сетевую карту (как будто порт настроен как access/untagged)
sudo ip netns exec host1 ip addr add 10.0.10.10/24 dev veth-h1
sudo ip netns exec host1 ip link set veth-h1 up
sudo ip netns exec host1 ip route add default via 10.0.10.1
```

Проверяем пинг до шлюза:
```bash
sudo ip netns exec host1 ping -c 3 -W 1 10.0.10.1
# Пинг полностью лежит (100% packet loss)!
```

### Шаг 2: Систематическая диагностика

1. **Проверка L1/L2 линков:**
   Поднят ли физический интерфейс?
   ```bash
   sudo ip netns exec host1 ip link show veth-h1
   # veth-h1: <BROADCAST,MULTICAST,UP,LOWER_UP> ...
   # Физический линк UP. Мост br0 на хосте тоже UP.
   ```
2. **Прослушивание трафика (tcpdump):**
   Запустим дамп на мосту `br0` и отфильтруем по IP-адресу `host1`:
   ```bash
   sudo tcpdump -i br0 -e -n host 10.0.10.10
   ```
   В другом терминале запустим пинг: `sudo ip netns exec host1 ping -c 2 10.0.10.1`.
   
   В выводе tcpdump мы увидим:
   ```text
   .. veth-h1-br > veth-r-br, ethertype IPv4 (0x0800), length 98: 10.0.10.10 > 10.0.10.1: ICMP echo request
   ```
   Сравните это с выводом из Части 2.2. Заметили разницу?
   Вместо `ethertype 802.1Q (0x8100), vlan 10` мы видим обычный `ethertype IPv4 (0x0800)`. 
   
   Хост отправляет **нетегированные (untagged)** кадры!
   Роутер на своем физическом интерфейсе `veth-r` не имеет IP-адреса, а его саб-интерфейс `veth-r.10` ожидает кадры строго с тегом `10`. Нетегированные кадры роутером отбрасываются.

### Шаг 3: Устранение инцидента
Восстановите правильную конфигурацию `host1`: удалите IP с физического интерфейса и верните VLAN-сабинтерфейс:

```bash
# Удаляем IP с физической карты
sudo ip netns exec host1 ip addr del 10.0.10.10/24 dev veth-h1

# Создаем VLAN-интерфейс с ID 10
sudo ip netns exec host1 ip link add link veth-h1 name veth-h1.10 type vlan id 10
sudo ip netns exec host1 ip addr add 10.0.10.10/24 dev veth-h1.10
sudo ip netns exec host1 ip link set veth-h1.10 up
sudo ip netns exec host1 ip route add default via 10.0.10.1
```

Проверяем связь:
```bash
sudo ip netns exec host1 ping -c 2 10.0.10.1
# Успешно!
```
Связь восстановлена.

## Шпаргалка

```bash
# === Создание VLAN-интерфейса ===
# Формат: ip link add link <физический_интерфейс> name <имя_виртуального> type vlan id <VLAN_ID>
ip link add link eth0 name eth0.10 type vlan id 10

# === Просмотр информации о VLAN ===
ip -d link show type vlan       # показать детальную информацию обо всех VLAN
cat /proc/net/vlan/config       # список настроенных VLAN в системе
cat /proc/net/vlan/eth0.10      # статистика конкретного VLAN-интерфейса

# === Удаление VLAN-интерфейса ===
ip link del eth0.10

# === Захват VLAN трафика через tcpdump ===
tcpdump -i eth0 -e vlan         # показывать только кадры с VLAN-тегами и выводить заголовки L2
tcpdump -i eth0 -e vlan 10      # фильтровать трафик только по VLAN ID 10
```
