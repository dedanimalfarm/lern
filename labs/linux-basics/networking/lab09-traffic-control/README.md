## Часть 1: Теория Traffic Control и модуля netem

### 1.1 Подсистема Traffic Control (tc) в Linux
Ядро Linux имеет мощную систему управления очередями пакетов. С помощью утилиты `tc` можно контролировать отправку пакетов (на прием трафика `tc` влияет ограниченно, основные правила применяются на **исходящий** трафик интерфейса).

Основные понятия:
* **Qdisc (Queueing Discipline — дисциплина очередей):** Алгоритм, который определяет, как пакеты планируются к отправке. По умолчанию используется простая очередь FIFO (First In, First Out) или FQ-CoDel (Fair Queueing Controlled Delay).
* **Class (класс):** Позволяет разделять трафик по группам внутри qdisc (например, выделять полосу для SSH и ограничивать HTTP).
* **Filter (фильтр):** Определяет, в какой класс направить пакет на основе IP, порта или меток.

### 1.2 Модуль netem (Network Emulation)
**netem** — встроенный в ядро Linux модуль эмуляции сети. Он позволяет тестировать поведение протоколов и приложений в условиях плохих каналов связи (WAN, Wi-Fi, спутник) без покупки дорогого оборудования.

Параметры netem:
1. **Delay (Задержка) и Jitter (Джиттер):**
   * *Задержка* — фиксированное время удержания пакета.
   * *Джиттер* — случайное отклонение от базового времени задержки (имитирует нестабильный маршрут).
2. **Loss (Потери пакетов):** Случайное отбрасывание заданного процента исходящих пакетов.
3. **Corrupt (Повреждение):** Внесение случайных битовых ошибок (ошибок контрольной суммы) в тело пакета.
4. **Duplicate (Дублирование):** Отправка одного и того же пакета дважды.
5. **Reorder (Переупорядочивание):** Нарушение последовательности доставки пакетов (например, 2-й пакет приходит раньше 1-го).

## Часть 2: Практическая эмуляция сетевых сбоев

Все правила эмуляции применяются на интерфейсе отправителя. Будем настраивать qdisc на интерфейсе `veth-cli` в пространстве имен `client`.

### 2.1 Эмуляция задержки и джиттера

Добавим задержку 100 мс со случайным отклонением (джиттером) в 20 мс:
```bash
sudo ip netns exec client tc qdisc add dev veth-cli root netem delay 100ms 20ms
```

Запустим пинг для проверки:
```bash
sudo ip netns exec client ping -c 5 10.9.0.2
# Обратите внимание на время (time): оно будет колебаться в районе 80-120 мс.
```

### 2.2 Эмуляция потери пакетов

Изменим правило на потерю 10% пакетов:
```bash
# Для изменения существующего правила используем 'change' вместо 'add'
sudo ip netns exec client tc qdisc change dev veth-cli root netem loss 10%
```

Проверим пинг:
```bash
sudo ip netns exec client ping -c 10 10.9.0.2
# Посмотрите на статистику: packet loss должен быть около 10%.
```

### 2.3 Эмуляция дублирования и повреждения пакетов

Настроим повреждение 5% пакетов (corrupt) и дублирование 2% (duplicate):
```bash
sudo ip netns exec client tc qdisc change dev veth-cli root netem corrupt 5% duplicate 2%
```

Проверим пинг. Поврежденные ICMP-пакеты отбрасываются ядром получателя из-за неверной контрольной суммы, поэтому ping покажет потери, но в выводе `ip -s link show dev veth-srv` в `server` можно увидеть рост ошибок.

### 2.4 Комбинирование сбоев и ограничение полосы (TBF)

Мы можем комбинировать параметры. Например, настроим задержку 50 мс и потерю 5% пакетов одновременно:
```bash
sudo ip netns exec client tc qdisc change dev veth-cli root netem delay 50ms loss 5%
```

Для просмотра текущих правил выполните:
```bash
sudo ip netns exec client tc qdisc show dev veth-cli
```

Очистим правила:
```bash
sudo ip netns exec client tc qdisc del dev veth-cli root
```

## Часть 3: Troubleshooting — боевой инцидент (Забытые правила tc/netem)

### Симптомы инцидента
Веб-приложение работает крайне медленно. База данных отвечает с огромными задержками. Запросы curl периодически отваливаются по таймауту. При этом CPU и RAM на серверах простаивают, а диски свободны.

### Шаг 1: Воспроизведение инцидента
Сымитируем "забытое" правило эмуляции на клиенте с высокой задержкой и потерями:
```bash
sudo ip netns exec client tc qdisc add dev veth-cli root netem delay 250ms loss 15%
```

### Шаг 2: Систематическая диагностика

1. **Измеряем сетевые задержки (ping):**
   ```bash
   sudo ip netns exec client ping -c 5 10.9.0.2
   # Вы увидите:
   # 64 bytes from 10.9.0.2: icmp_seq=1 ttl=64 time=250 ms
   # ...
   # 5 packets transmitted, 4 received, 20% packet loss, time 4004ms
   # Ужасная задержка (250мс) и потери на ровном месте (veth-кабель).
   ```
2. **Проверяем пропускную способность (iperf3):**
   В пространстве `server` запустим сервер iperf3:
   ```bash
   sudo ip netns exec server iperf3 -s -D
   ```
   С клиента запустим тест:
   ```bash
   sudo ip netns exec client iperf3 -c 10.9.0.2 -t 5
   # Из-за 15% потерь пакетов TCP-окно схлопнется, и скорость упадет до единиц килобит/с!
   ```
3. **Проверка настроек очередей (qdisc) интерфейсов:**
   ```bash
   sudo ip netns exec client tc qdisc show
   # Вывод:
   # qdisc netem 1: dev veth-cli root refcnt 2 limit 1000 delay 250.0ms loss 15%
   # Источник проблемы локализован! На интерфейсе veth-cli настроена деградация netem.
   ```

### Шаг 3: Устранение инцидента
Удалите qdisc-правило с интерфейса `veth-cli`:
```bash
sudo ip netns exec client tc qdisc del dev veth-cli root
```

Проверьте пинг и qdisc повторно:
```bash
sudo ip netns exec client ping -c 3 10.9.0.2
# time < 1ms, 0% loss

sudo ip netns exec client tc qdisc show dev veth-cli
# Должно вернуть стандартный qdisc (например, noqueue или fq_codel)
```
Связь полностью восстановлена.

Остановите iperf3 сервер:
```bash
sudo killall iperf3
```

## Как это звучит на собеседовании

1. **Вопрос:** Что такое `qdisc` (queueing discipline) в Linux и какую роль он играет в сетевом стеке?
   **Ответ:** `qdisc` (дисциплина очереди) — это алгоритм управления планированием и передачей сетевых пакетов из ядра на драйвер сетевого интерфейса. Он определяет, в каком порядке пакеты отправляются, как они группируются и какие из них отбрасываются при перегрузке (например, `pfifo_fast`, `fq_codel`, `tbf`).
2. **Вопрос:** Почему правила `tc` применяются преимущественно на исходящий (egress) трафик? Как можно ограничить входящий (ingress) трафик?
   **Ответ:** Ядро Linux может полностью контролировать отправку пакетов (задерживать их, переупорядочивать или отбрасывать на выходе). Входящий же трафик уже принят физической картой, и его нельзя "замедлить", можно только отбросить. Для полноценного управления входящим трафиком используют виртуальный интерфейс IFB (Intermediate Functional Block): входящий трафик перенаправляется на IFB, где к нему применяются обычные egress-правила шейпинга.
3. **Вопрос:** Что такое джиттер (jitter) и для каких типов приложений он критичен?
   **Ответ:** Джиттер — это статистическая флуктуация (изменение) времени доставки пакетов в сети. Если пакеты приходят с нерегулярными интервалами (например, 10 мс, затем 200 мс, затем 5 мс), это вызывает прерывания в приложениях реального времени: IP-телефонии, видеоконференциях и онлайн-играх.
4. **Вопрос:** Как потеря пакетов влияет на скорость передачи TCP и UDP?
   **Ответ:** TCP является надежным протоколом с контролем перегрузки. При обнаружении потерь пакетов алгоритм TCP уменьшает размер окна отправки (congestion window), что приводит к резкому (экспоненциальному) падению скорости передачи данных. UDP не имеет контроля доставки и перегрузки, поэтому он продолжает отправлять данные на исходной скорости, несмотря на любые потери на канале.

## Практические задания (отработка)

1. **Эмуляция переупорядочивания пакетов (Reordering):**
   Настройте интерфейс `veth-cli` так, чтобы 25% пакетов отправлялись немедленно (без задержки), а остальные задерживались на 50 мс. 
   * Команда: `sudo ip netns exec client tc qdisc add dev veth-cli root netem delay 50ms reorder 25% 50%`
   * Запустите ping и проанализируйте вывод. Вы должны увидеть предупреждения вида `PING 10.9.0.2 (10.9.0.2) ... Out of order packet`.
2. **Ограничение пропускной способности (Shaping):**
   Используйте qdisc `tbf` (Token Bucket Filter) для ограничения скорости интерфейса `veth-cli` ровно до 100 Кбит/с. Запустите тест `iperf3` и убедитесь, что скорость зажата в лимит.

## Шпаргалка

```bash
# === Просмотр конфигурации ===
tc qdisc show                            # показать qdisc на всех интерфейсах
tc -s qdisc show dev eth0                # показать статистику по очередям на eth0

# === Настройка netem (деградация сети) ===
tc qdisc add dev eth0 root netem delay 100ms           # задержка 100 мс
tc qdisc add dev eth0 root netem delay 100ms 10ms      # задержка 100 мс ± 10 мс
tc qdisc add dev eth0 root netem loss 5%               # потери пакетов 5%
tc qdisc add dev eth0 root netem corrupt 2%            # повреждение 2% пакетов
tc qdisc add dev eth0 root netem duplicate 1%          # дублирование 1% пакетов

# === Изменение и удаление правил ===
tc qdisc change dev eth0 root netem delay 50ms         # изменить параметры
tc qdisc del dev eth0 root                             # удалить все правила эмуляции

# === Запуск iperf3 ===
iperf3 -s                                # запустить сервер iperf3
iperf3 -c <IP_сервера>                   # запустить клиент (TCP-тест)
iperf3 -c <IP_сервера> -u -b 10M         # запустить UDP-тест на скорости 10 Мбит/с
```
