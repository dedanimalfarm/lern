**CNI (Container Network Interface)** — это открытый стандарт взаимодействия между средами выполнения контейнеров (Kubernetes, Podman, Nomad) и сетевыми плагинами (Calico, Cilium, Flannel, Weave).

В этой лабораторной работе вы научитесь вручную вызывать CNI-плагины, поймете, как устроены переменные окружения CNI, разберете структуру конфигурационного файла и вывода плагина, а также отследите жизненный цикл выделения IP-адресов.

## Часть 1: Концепция CNI и подготовка стенда

Многие считают CNI сложной и загадочной частью Kubernetes. На самом деле CNI — это просто **набор обычных исполняемых файлов (бинарников или скриптов)**, соответствующих спецификации CNI.

Когда оркестратор (например, `kubelet`) хочет настроить сеть для нового пода, он:
1. Создает изолированный Network Namespace для контейнера.
2. Подготавливает конфигурационный JSON-файл.
3. Вызывает CNI-плагин (например, `/opt/cni/bin/bridge`), передавая ему JSON-конфиг через стандартный поток ввода (`stdin`), а параметры контейнера (идентификатор, путь к namespace, имя интерфейса) — через переменные окружения.
4. CNI-плагин выполняет всю низкоуровневую работу по настройке сетевых интерфейсов (то, что вы делали вручную в Lab 06 и Lab 11) и завершается.

Давайте подготовим наш стенд. Перейдите в директорию лабораторной работы и запустите настройку:
```bash
cd /root/lern/labs/linux-basics/networking/lab12-cni-intro
sudo bash setup.sh
```

### Что делает скрипт `setup.sh`:
1. Скачивает официальные референсные плагины CNI версии `v1.3.0` и распаковывает их в каталог `/opt/cni/bin/`.
2. Создает пустой Network Namespace с именем `cni-ns`.
3. Создает JSON-конфигурацию сети `my-cni-config.json`.
4. Запускает CNI-плагин `/opt/cni/bin/bridge` в режиме `ADD` для настройки сети.

## Часть 2: Ручной вызов CNI-плагина (ADD)

Давайте детально разберем, как именно вызывается CNI-плагин. 

Для добавления интерфейса в контейнер устанавливаются следующие переменные окружения:
- `CNI_COMMAND=ADD`: Действие, которое плагин должен выполнить.
- `CNI_CONTAINERID=my-cni-container-123`: Уникальный идентификатор контейнера.
- `CNI_NETNS=/var/run/netns/cni-ns`: Путь к файлу сетевого пространства имен контейнера.
- `CNI_IFNAME=eth0`: Имя интерфейса, которое будет создано внутри контейнера.
- `CNI_PATH=/opt/cni/bin`: Путь, где искать вспомогательные плагины (например, плагины IPAM).

Команда вызова выглядит так:
```bash
cat my-cni-config.json | CNI_COMMAND=ADD CNI_CONTAINERID=my-cni-container-123 CNI_NETNS=/var/run/netns/cni-ns CNI_IFNAME=eth0 CNI_PATH=/opt/cni/bin /opt/cni/bin/bridge > cni_output.json
```

## Часть 3: Разбор JSON-вывода CNI-плагина

После успешного выполнения плагин возвращает JSON-структуру с описанием настроенной сети. Скрипт сохраняет этот вывод в файл `cni_output.json`. Давайте проанализируем его структуру:

```json
{
    "cniVersion": "0.4.0",
    "interfaces": [
        {
            "name": "cni-br0",
            "mac": "46:4d:0a:3d:53:e8"
        },
        {
            "name": "vetha2dd8ab2",
            "mac": "1a:69:ae:57:e6:1e"
        },
        {
            "name": "eth0",
            "mac": "6a:f6:2a:e2:04:54",
            "sandbox": "/var/run/netns/cni-ns"
        }
    ],
    "ips": [
        {
            "version": "4",
            "interface": 2,
            "address": "10.22.0.2/24",
            "gateway": "10.22.0.1"
        }
    ],
    "routes": [
        {
            "dst": "0.0.0.0/0"
        }
    ],
    "dns": {}
}
```

### Как парсить вывод с помощью `jq`:

1. **Получить выделенный IP-адрес:**
   ```bash
   jq -r '.ips[0].address' cni_output.json
   # Вывод: 10.22.0.2/24
   ```
2. **Получить имя хостового veth-интерфейса:**
   Хостовый интерфейс — это тот, у которого нет поля `sandbox`. Мы можем отфильтровать его:
   ```bash
   jq -r '.interfaces[] | select(.sandbox == null and .name != "cni-br0") | .name' cni_output.json
   # Вывод: vetha2dd8ab2
   ```
3. **Получить MAC-адрес интерфейса внутри контейнера:**
   ```bash
   jq -r '.interfaces[] | select(.sandbox != null) | .mac' cni_output.json
   # Вывод: 6a:f6:2a:e2:04:54
   ```

## Часть 4: Освобождение ресурсов (DEL)

Когда жизненный цикл контейнера завершается, оркестратор обязан вызвать CNI-плагин с командой `DEL`. Это гарантирует, что сетевые интерфейсы будут корректно удалены, а выделенный IP-адрес будет освобожден в базе данных IPAM.

### Шаг 1. Проверка текущей аренды IP-адреса
В нашем конфигурационном файле в качестве IPAM-плагина используется `host-local`. Он сохраняет локальные файлы аренды в каталоге `/var/lib/cni/networks/<имя-сети>/`.
Убедитесь, что аренда для нашего IP-адреса активна:
```bash
ls -l /var/lib/cni/networks/my-cni-network/
# Вы должны увидеть файл с именем "10.22.0.2", содержимое которого — ID контейнера:
cat /var/lib/cni/networks/my-cni-network/10.22.0.2
# Вывод: my-cni-container-123
```

### Шаг 2. Запуск команды DEL вручную
Теперь выполните удаление сетевого интерфейса. Важно передать **точно такие же** переменные окружения, которые использовались при создании:
```bash
export CNI_COMMAND=DEL
export CNI_CONTAINERID=my-cni-container-123
export CNI_NETNS=/var/run/netns/cni-ns
export CNI_IFNAME=eth0
export CNI_PATH=/opt/cni/bin

cat my-cni-config.json | /opt/cni/bin/bridge
```
*Примечание: При успешном выполнении команда `DEL` обычно не возвращает никаких данных и завершается с кодом 0.*

### Шаг 3. Проверка освобождения IP-адреса
Проверьте каталог аренды повторно:
```bash
ls -l /var/lib/cni/networks/my-cni-network/
```
Вы увидите, что файл `10.22.0.2` **исчез**. Это доказывает, что плагин `host-local` успешно освободил IP-адрес для последующих контейнеров.

Также проверьте сетевой интерфейс внутри namespace:
```bash
ip netns exec cni-ns ip link
# Интерфейс eth0 больше не существует! CNI-плагин удалил veth-пару.
```
