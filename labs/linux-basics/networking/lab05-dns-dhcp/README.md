## Часть 1: Теория DNS, DHCP (DORA) и Dnsmasq

### 1.1 Как работает DHCP (Процесс DORA)
DHCP (Dynamic Host Configuration Protocol) работает поверх UDP. Назначение адреса клиенту состоит из четырех шагов, легко запоминаемых по аббревиатуре **DORA**:

```text
 Клиент (порт 68)                                     DHCP-сервер (порт 67)
       |                                                        |
       | ------ DHCP Discover (Broadcast, UDP 67) ------------> | (Ищем сервер)
       | <----- DHCP Offer (Unicast/Broadcast, UDP 68) -------- | (Предложение IP)
       | ------ DHCP Request (Broadcast, UDP 67) -------------> | (Запрос аренды)
       | <----- DHCP ACK (Unicast/Broadcast, UDP 68) ---------- | (Подтверждение)
       v                                                        v
```

1. **Discover (Обнаружение):** Клиент отправляет широковещательный (broadcast) запрос во всю сеть, чтобы найти доступные DHCP-серверы.
2. **Offer (Предложение):** DHCP-сервер получает Discover и отправляет предложение с доступным IP-адресом, маской подсети, временем аренды и IP-адресом шлюза/DNS.
3. **Request (Запрос):** Клиент выбирает первое полученное предложение и отправляет широковещательный запрос, информирующий все DHCP-серверы, что он берет именно этот адрес.
4. **ACK (Подтверждение):** Сервер подтверждает аренду и фиксирует ее в своей базе данных (leases-файле).

### 1.2 DNS и концепция Split-DNS
* **DNS (Domain Name System):** Служба для преобразования человекочитаемых имен (например, `db.internal.company`) в IP-адреса.
* **Split-DNS (разделенный DNS):** Архитектура, в которой для одного и того же доменного имени отдаются разные IP-адреса в зависимости от того, откуда пришел запрос (изнутри корпоративной сети или снаружи из интернета). Это позволяет скрывать внутренние IP-адреса сервисов от внешнего сканирования.

### 1.3 Dnsmasq
**Dnsmasq** — это легковесный, простой в настройке DNS-форвардер и DHCP-сервер. Он идеально подходит для локальных сетей, роутеров, а также широко используется под капотом в виртуализации (libvirt) и Kubernetes (dnsmasq-nanny) для кеширования DNS-запросов на нодах.

## Часть 2: Настройка и тестирование служб DHCP/DNS

Конфигурация Dnsmasq сгенерирована в файле `/tmp/dnsmasq/dnsmasq.conf`. Она привязывает сервер к интерфейсу `veth-srv`, задает диапазон адресов `10.0.0.10 - 10.0.0.50` и объявляет локальную DNS-зону `internal.company`.

### 2.1 Анализ базы выданных адресов (Leases)

Каждый раз, когда клиент арендует адрес, Dnsmasq делает запись в файл `/tmp/dnsmasq/dnsmasq.leases`. Посмотрим его содержимое:
```bash
sudo ip netns exec server cat /tmp/dnsmasq/dnsmasq.leases
```
Вы увидите строку вида:
```text
1781683200 42:fd:16:86:da:7e 10.0.0.10 * 01:42:fd:16:86:da:7e
```
* **Поля записи:** `время_истечения_аренды (epoch) | MAC-адрес | IP-адрес | Имя_хоста | Client-ID`

### 2.2 Прослушивание процесса DHCP DORA

Давайте вживую перехватим DHCP-запрос. 
1. Сбросим IP-адрес на клиенте:
   ```bash
   sudo ip netns exec client ip addr flush dev veth-cli
   ```
2. В первом терминале запустим `tcpdump` в пространстве имен сервера на портах DHCP:
   ```bash
   sudo ip netns exec server tcpdump -i veth-srv -n -vv udp port 67 or udp port 68
   ```
3. Во втором терминале запустим DHCP-клиент `udhcpc`:
   ```bash
   sudo ip netns exec client busybox udhcpc -i veth-cli -s /tmp/udhcpc.script -n -q
   ```

В выводе `tcpdump` вы увидите детальный разбор пакетов:
* `BOOTREQUEST` (Discover) с MAC-адресом клиента.
* `BOOTREPLY` (Offer) от сервера `10.0.0.1` с предложением IP `10.0.0.10`.
* `BOOTREQUEST` (Request) от клиента.
* `BOOTREPLY` (ACK) от сервера, подтверждающий аренду.

## Часть 3: Troubleshooting — боевой инцидент (Rogue DHCP / Два DHCP в сети)

**Rogue DHCP Server (Нелегитимный DHCP-сервер)** — ситуация, при которой в сегменте L2-сети появляется второй, неавторизованный DHCP-сервер (это может быть как атака злоумышленника, так и случайно подключенный домашний Wi-Fi роутер). 

### Симптомы инцидента
Клиент подключается к сети, но у него пропадает доступ к интернету или внутренним ресурсам. При проверке IP-адрес клиента оказывается из совершенно другой подсети (например, `192.168.66.X` вместо `10.0.0.X`), а шлюз по умолчанию ведет на неизвестный адрес.

### Шаг 1: Воспроизведение инцидента
Сымитируем атаку Rogue DHCP. Остановим легитимный `dnsmasq` и запустим "злонамеренный" сервер, который выдает адреса из сети `192.168.66.0/24` со шлюзом злоумышленника `192.168.66.66`:

```bash
# Останавливаем легитимный DHCP
sudo killall dnsmasq

# Создаем конфиг злоумышленника
sudo mkdir -p /tmp/rogue-dnsmasq
sudo cat <<EOF > /tmp/rogue-dnsmasq/rogue.conf
interface=veth-srv
bind-interfaces
dhcp-range=192.168.66.10,192.168.66.50,12h
dhcp-option=option:router,192.168.66.66
dhcp-option=option:dns-server,192.168.66.66
EOF

# Запускаем Rogue DHCP сервер
sudo ip netns exec server dnsmasq -C /tmp/rogue-dnsmasq/rogue.conf -x /tmp/rogue-dnsmasq/rogue.pid
```

На клиенте сбросим старый IP и запросим адрес заново:
```bash
sudo ip netns exec client ip addr flush dev veth-cli
sudo ip netns exec client busybox udhcpc -i veth-cli -s /tmp/udhcpc.script -n -q
```

### Шаг 2: Диагностика инцидента

1. **Проверяем полученный IP-адрес на клиенте:**
   ```bash
   sudo ip netns exec client ip -br a show dev veth-cli
   # veth-cli UP 192.168.66.10/24
   # IP выдан из неверной подсети!
   ```
2. **Проверяем маршруты (шлюз):**
   ```bash
   sudo ip netns exec client ip route
   # default via 192.168.66.66 dev veth-cli
   # Весь трафик теперь перехватывается злоумышленником (атака Man-in-the-Middle).
   ```
3. **Определяем виновника (Rogue DHCP Server IP/MAC):**
   Давайте снова запросим адрес, слушая трафик через `tcpdump` с выводом параметров DHCP, чтобы узнать IP-адрес сервера:
   ```bash
   sudo ip netns exec client ip addr flush dev veth-cli
   
   # Запустим tcpdump в фоне на клиенте
   sudo ip netns exec client tcpdump -i veth-cli -n -vv -c 4 udp port 67 or udp port 68 > /tmp/dhcp_inspect.log &
   tcpdump_pid=$!
   sleep 0.5
   
   # Запрашиваем IP
   sudo ip netns exec client busybox udhcpc -i veth-cli -s /tmp/udhcpc.script -n -q
   wait "$tcpdump_pid" 2>/dev/null || true
   
   # Анализируем лог захвата на предмет Server Identifier
   grep "Server-ID" /tmp/dhcp_inspect.log || grep "Server Identifier" /tmp/dhcp_inspect.log
   # Вывод покажет: Server Identifier (54), length 4: 10.0.0.1
   # (В нашем случае IP-адрес сетевой карты сервера остался 10.0.0.1, 
   # но настройки он выдал чужие. В реальной сети Server-ID показал бы реальный IP злоумышленника).
   ```

### Как защититься от Rogue DHCP в продакшене?
На сетевых коммутаторах L2-уровня настраивается функция **DHCP Snooping**:
* Все порты коммутатора делятся на **Trusted (доверенные)** и **Untrusted (недоверенные)**.
* Порт, куда подключен легитимный DHCP-сервер, помечается как *Trusted*.
* Если на *Untrusted* порт приходит пакет `DHCP Offer` или `DHCP ACK` (ответ сервера), коммутатор **блокирует** этот порт. Злоумышленник физически не сможет раздавать адреса.

### Шаг 3: Устранение инцидента
Остановите Rogue DHCP и перезапустите легитимный сервер с правильными настройками:
```bash
sudo killall dnsmasq
sudo rm -rf /tmp/rogue-dnsmasq

# Запуск правильного Dnsmasq
sudo ip netns exec server dnsmasq -C /tmp/dnsmasq/dnsmasq.conf -x /tmp/dnsmasq/dnsmasq.pid

# Перезапрос IP на клиенте
sudo ip netns exec client ip addr flush dev veth-cli
sudo ip netns exec client busybox udhcpc -i veth-cli -s /tmp/udhcpc.script -n -q
```
Убедитесь, что клиент снова получил правильный IP из сети `10.0.0.0/24`.

## Как это звучит на собеседовании

1. **Вопрос:** Детально опишите фазы процесса DHCP DORA.
   **Ответ:** 
   * **Discover (широковещательный от клиента):** Клиент ищет доступные DHCP-серверы в локальном сегменте.
   * **Offer (уникаст/широковещательный от сервера):** Сервер предлагает клиенту свободный IP-адрес, маску подсети, шлюз по умолчанию и DNS-серверы.
   * **Request (широковещательный от клиента):** Клиент подтверждает выбор конкретного сервера и предложенного им IP-адреса.
   * **Acknowledge (уникаст/широковещательный от сервера):** Сервер окончательно закрепляет IP за клиентом и подтверждает начало аренды.
2. **Вопрос:** В каком файле Dnsmasq сохраняет данные о выданных адресах и что там фиксируется?
   **Ответ:** По умолчанию Dnsmasq сохраняет аренды в файле `/var/lib/misc/dnsmasq.leases` (в нашей лабе путь переопределен в `/tmp/dnsmasq/dnsmasq.leases`). В каждой строке фиксируются: время окончания аренды (Unix Epoch), MAC-адрес клиента, выданный IP-адрес, имя хоста (если передано клиентом) и идентификатор клиента (Client ID).
3. **Вопрос:** Что такое атака Rogue DHCP Server и к каким последствиям она может привести?
   **Ответ:** Это появление в сети нелегитимного DHCP-сервера, запущенного злоумышленником или пользователем по ошибке. Последствия включают в себя: атаку Man-in-the-Middle (если Rogue DHCP выдает свой IP в качестве шлюза по умолчанию или DNS), отказ в обслуживании (Deny of Service) из-за выдачи некорректных IP-настроек, либо кражу учетных данных через поддельные DNS.
4. **Вопрос:** Как технология DHCP Snooping предотвращает появление нелегитимных DHCP-серверов?
   **Ответ:** На L2-коммутаторах порты разделяются на доверенные (Trusted) и недоверенные (Untrusted). Порт легитимного DHCP-сервера настраивается как Trusted. Если на любой Untrusted порт приходит пакет DHCP Offer или DHCP ACK (ответы сервера), коммутатор расценивает это как атаку, отбрасывает пакеты и блокирует порт.

## Практические задания (отработка)

1. **Резервирование статического IP (DHCP Static Lease):** 
   Настройте Dnsmasq так, чтобы клиенту с конкретным MAC-адресом всегда выдавался один и тот же IP-адрес (например, `10.0.0.45`).

   Сделайте так, чтобы при запросе имени `service.internal` из внутренней сети возвращался IP `10.0.0.100`, а при запросе из внешней сети (сымитируем ее) возвращался публичный IP. Ознакомьтесь с параметром `localise-queries` в документации Dnsmasq.

## Шпаргалка

```bash
# === Управление dnsmasq ===
dnsmasq --test                          # проверить синтаксис конфигурационного файла
ip netns exec server dnsmasq -C /conf   # запустить dnsmasq внутри namespace

# === Работа с DHCP клиентом (udhcpc) ===
busybox udhcpc -i eth0 -n -q            # отправить DHCP-запрос на интерфейсе eth0 и сразу выйти
ip addr flush dev eth0                  # очистить все IP-адреса на интерфейсе

# === Мониторинг DHCP/DNS ===
tail -f /var/log/syslog | grep dnsmasq  # смотреть логи dnsmasq в реальном времени
cat /var/lib/misc/dnsmasq.leases        # посмотреть базу выданных DHCP-аренд по умолчанию

# === Тестирование DNS ===
dig @10.0.0.1 db.internal.company       # отправить DNS-запрос к серверу 10.0.0.1
dig -x 10.0.0.100                       # обратный DNS-запрос (PTR-запись)
```
