# Раздел 0. Сквозной фундамент (netns / veth / bridge / путь пакета)

Эти понятия используются во всех лабах. Опрос почти всегда начинается отсюда.

### 0.1 [Б] Что такое network namespace и что он изолирует?

**Ответ.** Network namespace (`netns`) — это изолированный экземпляр сетевого стека ядра Linux.
Каждый namespace имеет **свой** набор: сетевых интерфейсов, таблиц маршрутизации, ARP/neighbor-таблицы,
правил iptables/nftables, таблицы conntrack, сокетов и параметров `sysctl net.*`.

**Разбор.** Именно на namespace построена контейнерная изоляция (Docker/Kubernetes). Когда вы делаете
`ip netns exec red <cmd>`, команда выполняется так, будто вы за терминалом отдельной машины. Доказать
изоляцию просто: на хосте `ip -br link` показывает `eth0`, `docker0` и т.д., а внутри свежего
namespace — только `lo`. У каждого namespace **свои** iptables: правило в хосте не действует внутри
namespace и наоборот.

### 0.2 [Б] В новом namespace `ping 127.0.0.1` не работает. Почему?

**Ответ.** Интерфейс `lo` существует, но находится в состоянии `DOWN`. Ядро не обрабатывает пакеты на
выключенном интерфейсе. Нужно поднять: `ip netns exec red ip link set lo up`.

**Разбор.** Это типичная «ловушка первого шага»: loopback не поднимается автоматически в новом
namespace (в отличие от init namespace, где он поднят при загрузке). Пока `lo` лежит, не работает не
только ping, но и любое локальное взаимодействие сервисов (например, приложение, которое слушает
`127.0.0.1`).

### 0.3 [С] Что такое veth-пара? На каком уровне OSI работает?

**Ответ.** veth (virtual ethernet) — это пара виртуальных интерфейсов, соединённых как «виртуальный
кабель»: пакет, вошедший в один конец, выходит из другого. Работает на **L2** (канальный уровень),
передаёт Ethernet-кадры. Соединение **point-to-point** (две точки, без свитча между ними).

**Разбор.** veth всегда создаётся парой (`ip link add veth-red type veth peer name veth-blue`).
Один конец обычно кладут в namespace контейнера, второй — в bridge на хосте. Поскольку это
точка-точка, ARP-запрос (broadcast), отправленный в один конец, выходит из другого без всякого свитча.

### 0.4 [П] При `ip link set veth-red netns red` интерфейс исчез с хоста. Он скопировался?

**Ответ.** Нет, он **переместился**. Ядро меняет принадлежность интерфейса от `init_net` к целевому
namespace, обновляя указатель на `net namespace` в структуре `net_device`. В исходном namespace
интерфейс перестаёт существовать. Это операция O(1) — копирования данных нет.

**Разбор.** Важное следствие: veth-пару нельзя «разорвать на два хоста» — оба конца всегда в ядре
одной машины, просто в разных namespace. Когда удаляют namespace, конец veth внутри него уничтожается,
а парный конец (в bridge) автоматически удаляется вместе с ним, и bridge чистит связанные записи FDB.

### 0.5 [С] Что такое Linux bridge и чем он отличается от hub?

**Ответ.** Linux bridge — это виртуальный **L2-свитч** в ядре: он объединяет подключённые интерфейсы
в один broadcast-домен и пересылает Ethernet-кадры на основе MAC-адресов. В отличие от hub
(который повторяет каждый кадр на все порты), bridge **обучается**: ведёт FDB (Forwarding DataBase,
аналог CAM-таблицы железного свитча) — соответствие «MAC ↔ порт» — и unicast-кадр шлёт только на нужный
порт.

**Разбор.** Поэтому в эксперименте «green слушает, red пингует blue»: green **видит** ARP-запросы (это
broadcast — рассылается на все порты), но **не видит** ICMP-пакеты (unicast — bridge уже знает MAC blue
и шлёт кадр только на порт veth-blue). Это ключевое отличие свитча от хаба.
`bridge fdb show`, `bridge link show` — инструменты для просмотра.

### 0.6 [С] Bridge работает на L2. Зачем тогда мы дали ему IP `10.0.0.1`?

**Ответ.** IP на bridge нужен для **L3-функций хоста**, а не для пересылки кадров:
1. чтобы хост стал участником сети `10.0.0.0/24` и мог напрямую общаться с namespace;
2. чтобы namespace использовали bridge как **default gateway** для выхода наружу через маршрутизацию
   хоста (хост делает NAT).

**Разбор.** Сам bridge для коммутации кадров IP не требуется. Но как только нужен выход в интернет,
кадр должен «подняться» с L2 на L3 хоста — для этого у bridge должен быть IP-адрес шлюза.
Это ровно роль `docker0` (172.17.0.1) в Docker.

### 0.7 [П] Нарисуй путь транзитного пакета через цепочки netfilter (namespace → интернет).

**Ответ.**
```
Пакет из namespace
   │
   ▼  PREROUTING   (raw → conntrack → mangle → nat)   ← здесь срабатывает DNAT
   │
   ▼  Routing decision: dst не наш → FORWARD
   │
   ▼  FORWARD       (mangle → filter)                  ← здесь фильтрация транзита
   │
   ▼  POSTROUTING  (mangle → nat)                      ← здесь SNAT/MASQUERADE
   │
   ▼  Уходит наружу
```
Пакет, **адресованный самому хосту**, идёт иначе: `PREROUTING → routing decision (для хоста) → INPUT →
приложение`. Пакет, **рождённый хостом**: `OUTPUT → POSTROUTING`.

**Разбор.** Эта схема — каркас всего курса по фильтрации. Из неё следуют все «почему»:
почему DNAT в PREROUTING (до решения о маршрутизации), почему SNAT в POSTROUTING (после),
почему транзит фильтруется в FORWARD, а локальные сервисы — в INPUT.

# Раздел 1. Lab 01 — Маршрутизация и проброс портов (DNAT)

### 1.1 [Б] Чем отличается ошибка `Network is unreachable` от `100% packet loss`?

**Ответ.**
- `Network is unreachable` — в таблице маршрутизации **нет маршрута** до сети назначения и нет `default`.
  Пакет даже не покидает хост: ядро сразу возвращает ошибку.
- `100% packet loss` — маршрут **есть**, пакет ушёл, но ответа нет (его дропнули по дороге, нет обратного
  маршрута, блокирует firewall и т.п.).

**Разбор.** Это первый диагностический разветвитель. `unreachable` → проблема локальной маршрутизации
(чините `ip route`). `loss` → проблема дальше по пути: forwarding, firewall, обратный маршрут.

### 1.2 [Б] Зачем нужен `net.ipv4.ip_forward=1`? Работает ли без него Docker?

**Ответ.** По умолчанию Linux **дропает** пакеты, адресованные не ему (чужой dst IP). `ip_forward=1`
превращает хост в маршрутизатор — он начинает пересылать транзитный трафик между интерфейсами.
Без него Docker/Kubernetes-сети **не работают**: пакеты из контейнера до внешней сети не дойдут.

**Разбор.** Включается `sysctl -w net.ipv4.ip_forward=1` (рантайм) или строкой в `/etc/sysctl.conf`
(персистентно). По умолчанию выключено **из соображений безопасности**: обычная рабочая станция не должна
маршрутизировать чужой трафик — это роль роутера. Проверка: `cat /proc/sys/net/ipv4/ip_forward`.

### 1.3 [С] Почему маршруты настраивали и на клиенте, и на сервере?

**Ответ.** Связь двусторонняя. Клиенту нужен маршрут **туда** (через шлюз к сети сервера), серверу —
маршрут **обратно** (через шлюз к сети клиента). Если настроить только клиента, запрос дойдёт до сервера,
но ответный пакет сервер не сможет отправить — в его таблице нет пути к `10.0.1.0/24`, и ответ
уничтожится.

**Разбор.** Это фундаментальный принцип: **маршрутизация симметрична по требованию к таблицам, но не
обязана быть симметрична по пути**. Большинство «зависаний» в сетях — это отсутствие именно обратного
маршрута (см. вопрос 1.6).

### 1.4 [Б] В какой таблице iptables настраивается DNAT и почему именно в PREROUTING?

**Ответ.** Таблица `nat`, цепочка `PREROUTING`. DNAT меняет **адрес/порт назначения**, поэтому подмена
должна произойти **до** решения о маршрутизации (routing decision), чтобы ядро направило пакет на нужный
интерфейс уже с новым destination.

**Разбор.** Если бы DNAT стоял в POSTROUTING (после маршрутизации), ядро уже приняло бы неверное
решение по старому адресу. Для пакетов, **рождённых локально** на хосте, аналог DNAT делается в цепочке
`OUTPUT` (потому что они не проходят PREROUTING). Команда:
`iptables -t nat -A PREROUTING -d 10.0.1.1 -p tcp --dport 80 -j DNAT --to-destination 10.0.2.2:8080`.

### 1.5 [С] Видит ли внутренний сервер реальный IP клиента при чистом DNAT?

**Ответ.** **Да.** DNAT меняет только destination. Source IP пакета остаётся клиентским (`10.0.1.2`),
поэтому сервер видит реальный адрес клиента.

**Разбор.** Это и порождает проблему обратного маршрута: сервер хочет ответить напрямую клиенту, но не
знает пути к его сети. Два решения: (1) дать серверу обратный маршрут/`default gateway`; (2) добавить
SNAT/MASQUERADE на роутере — тогда серверу будет казаться, что запросы идут от самого роутера, и ответ
он отправит роутеру (который знает клиента). Цена варианта (2) — потеря реального source IP в логах
сервера.

### 1.6 [С] Назови две причины, по которым проброс портов «зависает» по таймауту.

**Ответ.**
1. **Нет обратного маршрута** на бэкенде (сервер не знает, как ответить клиенту → ответ дропается).
2. **Firewall блокирует транзит**: политика `FORWARD DROP` без явного разрешения проброшенного трафика
   (и/или нет правила для `ESTABLISHED,RELATED`).

**Разбор.** Обе причины проявляются одинаково — `curl: (28) Connection timed out` — но лечатся
по-разному. Поэтому при диагностике смотрят и `ip route` на бэкенде, и `iptables -L FORWARD -n -v`
(счётчики покажут, ловит ли DROP пакеты).

### 1.7 [П] Чем DNAT отличается от SNAT и MASQUERADE? Какие цепочки используют?

**Ответ.**
- **DNAT** — меняет **destination**, цепочка `PREROUTING` (или `OUTPUT` для локальных). Применение:
  проброс портов, публикация сервисов.
- **SNAT** — меняет **source** на **фиксированный** IP, цепочка `POSTROUTING`. Применение: выход в
  интернет с известным статическим адресом.
- **MASQUERADE** — частный случай SNAT, source подменяется на IP **исходящего интерфейса**,
  определяемый автоматически на каждый пакет. Цепочка `POSTROUTING`.

**Разбор.** MASQUERADE уместен, когда внешний IP **динамический** (DHCP от облака/провайдера) — не нужно
прописывать адрес. SNAT эффективнее при **статическом** IP: не делается lookup адреса интерфейса на
каждый пакет. Это ровно то, что делает `docker run -p 80:8080`: DNAT (входящий проброс) +
MASQUERADE (исходящий трафик контейнера).

# Раздел 2. Lab 02 — Балансировка нагрузки (HAProxy)

### 2.1 [Б] Что такое алгоритм Round Robin?

**Ответ.** Балансировщик по очереди распределяет входящие запросы между бэкендами: запрос 1 → server1,
запрос 2 → server2, запрос 3 → server3, запрос 4 → server1 и т.д. Поэтому `curl` к балансировщику
несколько раз подряд возвращает ответы от разных серверов.

**Разбор.** Round Robin — простейший и самый частый алгоритм. Он не учитывает текущую нагрузку и не
«прилипает» к серверу (нет session affinity). Альтернативы: `leastconn` (на сервер с минимумом
соединений), `source`/sticky (по хешу клиентского IP — для сессий), взвешенный RR (`weight`).

### 2.2 [С] Как HAProxy узнаёт, что сервер «упал», и перестаёт слать ему трафик?

**Ответ.** Через **health checks**. В конфиге `option httpchk GET /` заставляет HAProxy периодически
отправлять проверочный HTTP-запрос каждому бэкенду. Если сервер не отвечает (мы убили его
`pkill -f "http.server 8081"`), HAProxy помечает его как `DOWN` и исключает из ротации — трафик идёт
только на живые server2/server3.

**Разбор.** Это отказоустойчивость: клиент даже не замечает падения одного бэкенда. Параметры проверки
(`inter` — интервал, `rise`/`fall` — сколько успешных/неудачных проверок для смены статуса) настраивают
чувствительность. Без healthcheck балансировщик слал бы часть запросов в «мёртвый» сервер → ошибки у
пользователей.

### 2.3 [П] В чём разница между L4- и L7-балансировкой?

**Ответ.**
- **L4 (`mode tcp`)** — балансировщик работает на транспортном уровне: видит только IP и порты, не
  разбирает содержимое. Быстро, но без знания о HTTP (нельзя роутить по URL/заголовкам).
- **L7 (`mode http`)** — работает на уровне приложения: разбирает HTTP, может маршрутизировать по пути,
  хосту, кукам, терминировать TLS, переписывать заголовки. Дороже по CPU, но гибко.

**Разбор.** HAProxy умеет оба режима. В этой лабе используется HTTP-проксирование (L7): именно поэтому
доступен `option httpchk` и видны имена серверов в ответах. В Kubernetes L4 ≈ `Service`/`kube-proxy`,
L7 ≈ `Ingress`-контроллер.

# Раздел 3. Lab 03 — VPN-туннели WireGuard

### 3.1 [Б] Что создаёт WireGuard и на каком порту/протоколе работает?

**Ответ.** WireGuard создаёт виртуальный сетевой интерфейс `wg0`, через который идёт зашифрованный
туннель. Транспорт — **UDP** (по умолчанию порт **51820**). Внутри туннеля — отдельная подсеть
(в лабе `192.168.100.0/24`), весь трафик к которой шифруется и инкапсулируется в UDP.

**Разбор.** WireGuard живёт в ядре (или как `wireguard-go`), использует современную криптографию
(Curve25519, ChaCha20-Poly1305, BLAKE2). UDP выбран ради простоты и производительности; «туннель»
поднимается лениво — фактического хендшейка нет, пока не пойдёт первый пакет.

### 3.2 [С] Как устроена аутентификация пиров (ключи)?

**Ответ.** У каждого узла есть пара ключей: **приватный** (`wg genkey`) и **публичный**
(`wg pubkey`). В конфиге узла указывается его приватный ключ и список **пиров (Peer)** с их публичными
ключами. Узел шифрует трафик так, что расшифровать его может только владелец соответствующего приватного
ключа. Это асимметричная схема, как у SSH-ключей.

**Разбор.** Никаких логинов/паролей и сертификатов CA — доверие задаётся явным перечислением публичных
ключей пиров. Если публичного ключа пира нет в конфиге — туннель с ним не поднимется. `wg show`
показывает пиров, их публичные ключи, `endpoint` (например `10.0.0.2:51820`) и счётчики переданных
данных (доказательство, что туннель работает).

### 3.3 [П] Что такое `AllowedIPs` и почему это одновременно маршрутизация и контроль доступа
(«cryptokey routing»)?

**Ответ.** `AllowedIPs` у пира задаёт, **какие адреса назначения** отправлять в туннель к этому пиру
(роль маршрутизации) **и** какие source-адреса **принимать** от него как валидные (роль фильтра).
WireGuard называет это **cryptokey routing**: связка «публичный ключ ↔ список разрешённых IP» — это
одновременно таблица маршрутов и таблица доступа.

**Разбор.** Пример: если у пира `AllowedIPs = 192.168.100.2/32`, то пакет на `192.168.100.2` уйдёт в его
туннель, а пакет, пришедший из этого туннеля с чужим source (не из разрешённого диапазона), будет
отброшен. `AllowedIPs = 0.0.0.0/0` превращает пир в «full-tunnel» шлюз (весь трафик в VPN).

# Раздел 4. Lab 04 — VLAN (802.1q)

### 4.1 [Б] Что такое VLAN и как технически помечается принадлежность кадра?

**Ответ.** VLAN (Virtual LAN) логически делит **один** физический L2-сегмент на несколько
изолированных. По стандарту **802.1q** в Ethernet-кадр вставляется 4-байтовый тег с **VID** (VLAN ID,
12 бит → до 4094 VLAN). Свитч и ОС пересылают кадр только в пределах его VLAN.

**Разбор.** В лабе `host1` в VLAN 10, `host2` в VLAN 20 — оба воткнуты в один bridge (`br0`), но
изолированы на L2: их кадры помечены разными тегами, и свитч не смешивает их. Увидеть теги:
`tcpdump -i br0 -e -n vlan`.

### 4.2 [С] host1 и host2 в одном свитче, но не пингуются. Почему? Как связать?

**Ответ.** Они в разных VLAN (10 и 20). Bridge не пересылает кадры между разными VLAN — это
изоляция на **L2**. Чтобы связать VLAN, нужен **маршрутизатор** (inter-VLAN routing): устройство с
интерфейсом в каждом VLAN, которое перекладывает пакеты между ними на **L3**.

**Разбор.** В лабе это «**router-on-a-stick**»: один роутер с двумя sub-интерфейсами `veth-r.10`
(10.0.10.1) и `veth-r.20` (10.0.20.1) и включённым `ip_forward`. Пакет из host1 приходит к роутеру по
VLAN 10, роутер видит, что назначение в подсети VLAN 20, перекладывает пакет в интерфейс VLAN 20
(меняя тег 10 → 20) и отправляет обратно в свитч; host2 получает его по VLAN 20.

### 4.3 [П] Чем отличается access-порт от trunk-порта? Что такое PVID и untagged?

**Ответ.**
- **Access-порт** — несёт **один** VLAN, кадры идут **без тега** (untagged). К нему подключают конечные
  устройства, которые про VLAN ничего не знают.
- **Trunk-порт** — несёт **несколько** VLAN, кадры идут **с тегами**. Соединяет свитчи/роутер.
- **PVID (Port VLAN ID)** — VLAN, в который попадают **входящие нетегированные** кадры на порту.
- **untagged** — флаг, что **исходящие** кадры этого VLAN отправляются без тега (для access-порта).

**Разбор.** В лабе 06 (Задание 7) это видно явно:
`bridge vlan add vid 10 dev veth-red pvid untagged` делает порт access-портом VLAN 10. Роутер-на-палочке
получает trunk, чтобы видеть оба VLAN сразу. Понимание PVID критично: неверный PVID → кадры уходят не в
тот VLAN, и «всё ломается без видимой причины».

# Раздел 5. Lab 05 — Внутренний DNS и DHCP (dnsmasq)

### 5.1 [Б] Опиши процесс DHCP (DORA).

**Ответ.** Клиент без адреса получает IP за 4 шага **DORA**:
1. **Discover** — клиент шлёт broadcast «есть тут DHCP-сервер?».
2. **Offer** — сервер предлагает свободный IP из пула.
3. **Request** — клиент broadcast'ом подтверждает, что берёт именно этот адрес.
4. **Ack** — сервер подтверждает аренду (lease) и выдаёт параметры: IP, маску, шлюз, DNS.

**Разбор.** В лабе `dnsmasq` выдаёт адреса из пула `10.0.0.10–50` и сообщает клиенту, что DNS на
`10.0.0.1`. Клиент `busybox udhcpc -i veth-cli` запрашивает адрес. Broadcast на шагах D и R нужен потому,
что у клиента ещё нет IP и он не знает, какой сервер ответит (их может быть несколько).

### 5.2 [С] Что делает кэширующий DNS и что такое Service Discovery?

**Ответ.** Кэширующий DNS (dnsmasq/CoreDNS/Bind) принимает запросы «имя → IP», отдаёт ответ из
локальной зоны или из кэша (а внешние — форвардит вышестоящему резолверу и кэширует). **Service
Discovery** — это когда сервисы общаются по «красивым» именам (`db.internal.company`), а внутренний DNS
разрешает их в актуальные IP.

**Разбор.** В лабе `dig +short @10.0.0.1 db.internal.company` → `10.0.0.100`. Это позволяет менять IP
сервисов, не трогая клиентов: поменяли запись в DNS — все нашли новый адрес. В Kubernetes тот же принцип
реализует CoreDNS (имена вида `service.namespace.svc.cluster.local`).

### 5.3 [П] Почему в лабе используют `udhcpc` с кастомным скриптом и флагом `-n`, а не системный клиент?

**Ответ.** Чтобы **не перезаписать** хостовый `/etc/resolv.conf` и не сломать DNS основной машины.
`udhcpc -s /tmp/udhcpc.script` задаёт безопасный обработчик, `-n` — «получить адрес и выйти, не
демонизироваться», `-q` — quiet. Адрес применяется только внутри namespace клиента.

**Разбор.** Это пример «гигиены лабораторного стенда»: DHCP-клиент по умолчанию лезет менять системные
файлы (resolv.conf, маршруты). В изолированном эксперименте это недопустимо — поэтому используют
busybox-клиент с подконтрольным скриптом.

# Раздел 6. Lab 06 — Linux Bridge (виртуальный свитч)

> Базовые вопросы про bridge/veth/ARP см. в Разделе 0. Здесь — углублённые.

### 6.1 [С] Как пакет нашёл MAC соседа, если между namespace нет свитча (прямая veth-пара)?

**Ответ.** Через **ARP**. veth-пара — это point-to-point: ARP-запрос (broadcast `who-has 10.0.0.3`)
входит в один конец и выходит из другого, сосед отвечает (`is-at <MAC>`), адрес попадает в neighbor-
таблицу (`ip neigh show`). Свитч не нужен — это прямой кабель между двумя точками.

**Разбор.** ARP (Address Resolution Protocol) связывает L3 (IP) и L2 (MAC): прежде чем отправить
IP-пакет, узел должен узнать MAC получателя в своём сегменте. Запрос — broadcast (`ff:ff:ff:ff:ff:ff`),
ответ — unicast. `ip neigh flush all` сбрасывает кэш (полезно при отладке VLAN-изоляции).

### 6.2 [С] Почему green видит ARP-запросы от red, но не видит ICMP red→blue?

**Ответ.** ARP-запрос — **broadcast**, bridge рассылает его на **все** порты (включая green). ICMP
red→blue — **unicast**: bridge уже выучил MAC blue (записал в FDB) и шлёт кадр **только** на порт
veth-blue. Поэтому green видит «общие» broadcast'ы, но не чужой unicast-трафик.

**Разбор.** Это практическая демонстрация, что bridge — свитч, а не хаб. На хабе green видел бы и ICMP
(хаб повторяет всё на все порты). Понимание broadcast-домена важно для безопасности: всё в одном bridge
без VLAN видит широковещательный трафик друг друга.

### 6.3 [С] Почему MASQUERADE здесь уместнее SNAT?

**Ответ.** Потому что внешний IP хоста может быть **динамическим** (DHCP от облака). MASQUERADE
автоматически подставляет текущий IP исходящего интерфейса, не требуя прописывать адрес. Если бы IP был
**статическим и известным**, эффективнее SNAT (нет lookup адреса интерфейса на каждый пакет).

**Разбор.** Правило из лабы:
`iptables -t nat -A POSTROUTING -s 10.0.0.0/24 -o ens4 -j MASQUERADE`
плюс разрешение форварда:
`iptables -A FORWARD -i br0 -o ens4 -j ACCEPT` и обратного — `RELATED,ESTABLISHED`. Это полный
эквивалент того, что Docker настраивает для сети `docker0` автоматически.

### 6.4 [П] VLAN-изоляция на bridge: red не пингует green в той же подсети 10.0.0.0/24. На каком уровне блок?

**Ответ.** Блокировка на **L2**: при `vlan_filtering 1` bridge не пересылает кадры между разными VLAN
(red в VLAN 10, green в VLAN 20). IP-подсеть одна, но L2-домены разные. Для связи нужен
**inter-VLAN routing** — маршрутизатор с интерфейсом в каждом VLAN, который перекладывает пакеты на L3.

**Разбор.** Это контринтуитивно: «один IP-сегмент, но не пингуется». Урок: связность определяется
**и** L2 (VLAN/коммутация), **и** L3 (маршрутизация). Одной общей подсети мало, если L2 разрезан на VLAN.

# Раздел 7. Lab 07 — iptables / nftables (фильтрация, NAT, conntrack)

### 7.1 [Б] Перечисли таблицы и базовые цепочки iptables. Почему в `filter` нет PREROUTING?

**Ответ.** Таблицы: `raw` (до conntrack), `mangle` (модификация пакетов), `nat` (трансляция адресов),
`filter` (фильтрация). Цепочки: `INPUT` (на хост), `OUTPUT` (с хоста), `FORWARD` (транзит),
`PREROUTING`, `POSTROUTING`. В `filter` нет PREROUTING/POSTROUTING, потому что каждая таблица подключена
к своим hook-точкам Netfilter: `filter` живёт там, где принимается решение о пропуске (INPUT/FORWARD/
OUTPUT), а PRE/POSTROUTING — точки для NAT/mangle (до и после маршрутизации).

**Разбор.** Транзитный пакет: `PREROUTING(nat) → routing decision → FORWARD(filter) → POSTROUTING(nat)`.
Поэтому фильтрацию транзита пишут в FORWARD, а NAT — в PRE/POSTROUTING. (см. также 0.7)

### 7.2 [Б] Почему порядок правил критичен? Что будет, если `DROP all` стоит выше `ACCEPT ssh`?

**Ответ.** iptables проходит цепочку **сверху вниз** и применяет **первое совпавшее** правило (terminating
target прекращает обход). Если `DROP all` выше `ACCEPT ssh`, SSH-пакет совпадёт с DROP и будет отброшен —
до `ACCEPT ssh` обработка не дойдёт.

**Разбор.** Правило: **специфичные правила — выше, общие — ниже**. Финальный `DROP`/`policy DROP` —
в самом конце. Это причина классической аварии «закрыл себе SSH»: политику `DROP` поставили раньше
разрешающего правила.

### 7.3 [С] Чем DROP отличается от REJECT? Что выбрать для внешнего firewall, а что для внутреннего?

**Ответ.** **DROP** молча отбрасывает пакет (отправитель ждёт таймаут). **REJECT** отправляет ответ
(`tcp-reset` или ICMP unreachable) — отказ приходит мгновенно. Для **внешнего** firewall лучше DROP:
атакующий не получает информации, порт выглядит «filtered». Для **внутренней** сети лучше REJECT:
легитимные клиенты получают быстрый отказ вместо долгого зависания, проблема диагностируется быстрее.

**Разбор.** DROP замедляет сканирование портов (сканеру приходится ждать таймаут на каждый порт),
но и легитимный пользователь страдает от таймаута. Это компромисс «скрытность vs скорость диагностики».

### 7.4 [С] Что такое conntrack и что означают состояния NEW / ESTABLISHED / RELATED / INVALID?

**Ответ.** Conntrack — подсистема ядра, отслеживающая **соединения** (stateful inspection).
- **NEW** — первый пакет нового соединения.
- **ESTABLISHED** — пакет принадлежит уже установленному соединению (в обе стороны).
- **RELATED** — пакет связан с существующим соединением, но не его часть (см. 7.5).
- **INVALID** — пакет не соответствует ни одному известному состоянию (часто его дропают).

**Разбор.** Stateful-правило `-m state --state ESTABLISHED,RELATED -j ACCEPT` пропускает ответный трафик
автоматически, без отдельных правил «для ответов». Инструменты: `conntrack -L` (таблица),
`conntrack -E` (события NEW/UPDATE/DESTROY), `/proc/sys/net/netfilter/nf_conntrack_{count,max}`.

### 7.5 [П] Приведи пример RELATED-трафика, который не ESTABLISHED.

**Ответ.** Примеры:
- **ICMP-ошибка** (`destination unreachable`, `time-exceeded`) в ответ на TCP/UDP-пакет — это ICMP,
  но связан с TCP/UDP-соединением.
- **FTP data-соединение** (порт 20), связанное с control-соединением (порт 21) — отдельное TCP-соединение,
  но conntrack через хелпер знает об их связи.
- **ICMP «fragmentation needed»** при Path MTU Discovery.
- **traceroute** — ICMP/UDP-ответы, связанные с исходной трассировкой.

**Разбор.** RELATED требует протокольных **хелперов** conntrack (например, `nf_conntrack_ftp`), которые
«заглядывают» в полезную нагрузку и регистрируют ожидаемое связанное соединение. Без RELATED многие
протоколы (FTP active, traceroute, корректная доставка ICMP-ошибок) ломаются за NAT/firewall.

### 7.6 [С] Почему stateless-правило `--sport 80 -j ACCEPT` опасно?

**Ответ.** Оно пропускает **любой** пакет с source port 80, даже если HTTP-соединения никогда не было.
Атакующий просто формирует пакет с `source port 80` и проходит сквозь firewall. Stateful-подход
(`-m state --state ESTABLISHED,RELATED`) принимает пакет только если он принадлежит соединению,
**инициированному изнутри**.

**Разбор.** Это исторический способ «разрешать ответы» до эпохи conntrack — и его фундаментальная дыра:
source port подделывается тривиально. Stateful inspection закрыл этот класс обходов.

### 7.7 [С] LOG — terminating target? Чем он отличается от ACCEPT/DROP?

**Ответ.** Нет, **LOG не terminating**: он записывает информацию о пакете (в kernel log / `dmesg`) и
**возвращает управление следующему правилу**. ACCEPT/DROP/REJECT — terminating, они завершают обход
цепочки. Поэтому LOG ставят перед DROP, чтобы и залогировать, и заблокировать одним набором правил.

**Разбор.** Логи лучше ограничивать по частоте, чтобы не забить journal:
`-m limit --limit 10/min --limit-burst 5 -j LOG --log-prefix "FW-DROP: "`. Префикс помогает грепать
(`dmesg | grep FW-DROP`). В логе видны IN/OUT-интерфейсы, SRC/DST, порты, протокол, флаги TCP.

### 7.8 [С] Что такое defense in depth и зачем firewall И на роутере (FORWARD), И внутри namespace (INPUT)?

**Ответ.** Defense in depth — **многоуровневая** защита. FORWARD на роутере — первая линия (межсегментная
фильтрация). INPUT внутри сервера — вторая (локальная защита самого сервиса). Если атакующий захватил
`app` и стучится в `db` на нестандартном порту, широкое FORWARD-правило может пропустить, но INPUT в `db`
(разрешает 3306 только от app) заблокирует.

**Разбор.** Два уровня снижают риск ошибки в одной конфигурации: одна дыра в правиле FORWARD не приводит
сразу к пробою, потому что есть локальный firewall. Это прямая аналогия с принципом из защиты агентов —
безопасность не должна держаться на одном слое.

### 7.9 [П] Как работает rate limiting? Что такое token bucket и burst?

**Ответ.** Модуль `limit` реализует **token bucket**: «ведро» наполняется токенами со скоростью `--limit`
(например, 3/min = 1 токен каждые 20 с). Каждый совпавший пакет забирает токен; если ведро пусто — правило
не совпадает (и пакет уходит на следующее правило, обычно DROP). `--limit-burst` — ёмкость ведра, сколько
токенов можно «накопить» и потратить разом.

**Разбор.** `--limit 3/min --limit-burst 3`: в начале пройдут 3 пакета подряд (burst), дальше — не чаще
1 каждые 20 с. Для бана по IP используют модуль `recent` (`--set` помечает, `--update --seconds 60
--hitcount 4` ловит превышение). В nftables — динамические sets с `flags timeout` и `limit rate over`.

### 7.10 [П] Почему `iptables-restore` / `nft -f` применяют правила атомарно и почему это важно?

**Ответ.** Они загружают весь набор правил **одной** операцией ядра: либо применяется всё, либо ничего.
При последовательном `iptables -A ...` между командами есть окно неконсистентности. Если скрипт упадёт
после `iptables -P INPUT DROP`, но до добавления `ACCEPT`-правил, хост станет **недоступен**.

**Разбор.** Для production это критично: атомарность исключает промежуточные состояния (полностью закрыт
или полностью открыт). nftables изначально проектировался атомарным; `nft -c -f file` ещё и валидирует
синтаксис **без применения** — обязательный шаг перед раскаткой.

### 7.11 [С] Главные преимущества nftables над iptables.

**Ответ.**
- **Единый инструмент** вместо iptables/ip6tables/arptables/ebtables; семейство `inet` = IPv4+IPv6 сразу.
- **Атомарные** обновления всего ruleset.
- **Sets и maps** (именованные и динамические) — компактно и быстро.
- **Лучшая производительность** и читаемый синтаксис (`tcp dport 80 accept` вместо набора `-m`).
- Встроенная трассировка (`nft monitor trace`).

**Разбор.** Структура: вы сами задаёте имена таблиц/цепочек; цепочка подключается к hook'у с приоритетом
(`type filter hook input priority 0; policy drop;`). Соответствия: `-j ACCEPT`→`accept`,
`-m state --state NEW`→`ct state new`, `-m limit --limit 5/sec`→`limit rate 5/second`.

### 7.12 [П] Чем nftables set лучше сотни правил `-s IP -j DROP`?

**Ответ.** 100 отдельных правил — это **линейный** перебор O(n) на каждый пакет. nftables set хранится в
ядре как **хеш-таблица O(1)** или дерево O(log n). При 10 000 заблокированных IP разница огромна. Плюс
set обновляется **атомарно**: добавить/удалить IP можно без перезагрузки всех правил.

**Разбор.** Динамический set с таймаутом даёт «самоочищающийся» автобан:
`add @ssh_banned { ip saddr }` с `flags timeout; timeout 10m`. Verdict map (`vmap`) позволяет за один
lookup выбрать действие по ключу: `tcp dport vmap { 80 : accept, 22 : drop }`.

### 7.13 [С] Зачем нужен `nft monitor trace` и когда он незаменим?

**Ответ.** Trace показывает **полный путь** конкретного пакета по всем цепочкам и правилам с указанием
вердикта (accept/drop/continue). Незаменим в ситуации «трафик дропается, но непонятно каким правилом» —
особенно в больших конфигурациях с десятками цепочек.

**Разбор.** Включается метой: `meta nftrace set 1` для нужного трафика, затем `nft monitor trace`.
В iptables аналога нет — там пришлось бы вставлять LOG перед каждым правилом, засоряя конфиг и логи.

# Раздел 8. Lab 08 — GCP Cloud NAT

### 8.1 [Б] Зачем приватной VM (без внешнего IP) нужен Cloud NAT?

**Ответ.** Без публичного IP и без NAT исходящий трафic VM не выходит за пределы VPC: у VM есть default
route, пакет уходит, но на границе VPC некому сделать SNAT, и ответ не вернётся. Cloud NAT даёт VM
**исходящий** доступ в интернет, оставляя её недоступной **извне** (нет входящего публичного адреса).

**Разбор.** Это стандартный паттерн безопасности: приватные ноды/поды ходят за обновлениями и к внешним
API, но не светят публичными адресами и не принимают входящие соединения из интернета. Cloud NAT
делает **только source NAT** для исходящего трафика.

### 8.2 [С] Почему `subnet-mode=custom`, а не `auto`?

**Ответ.** В `auto` GCP сам создаёт по подсети в каждом регионе с предопределёнными диапазонами. `custom`
даёт полный контроль: где будут подсети и с какими CIDR. Для продакшна — всегда `custom`, иначе при
VPC peering получите **пересечение** адресных диапазонов.

**Разбор.** Пересечение CIDR — это «тихая бомба»: сети создаются легко, а потом не стыкуются при
объединении/peering, и переделка стоит дорого. Контроль адресации с самого начала — базовая гигиена.

### 8.3 [С] Cloud NAT — software-defined и не создаёт VM. Зачем тогда Cloud Router?

**Ответ.** Cloud Router — это **control plane** маршрутизации в VPC (реализует BGP, управляет
динамическими маршрутами). Cloud NAT использует его инфраструктуру, чтобы внедрить NAT-правила в
SDN-стек GCP (Andromeda). Без Router нет точки управления маршрутизацией, к которой «прикрепляется» NAT.

**Разбор.** Важно понять, что Cloud NAT — **не** отдельная VM-«коробка», через которую идёт трафик
(нет bottleneck и single point of failure). Это распределённая функция SDN; Cloud Router — лишь
управляющий компонент. Поэтому Cloud NAT горизонтально масштабируется без узкого места.

### 8.4 [П] Почему лимит портов привязан к паре destination IP:port, а не просто к числу соединений?

**Ответ.** NAT-маппинг определяется **пятёркой** (src_ip, src_port, dst_ip, dst_port, proto). Внешний
наблюдатель различает соединения по (translated_ip, translated_port). Один и тот же source-порт можно
**переиспользовать** к **разным** destination — для внешней стороны это разные соединения. Поэтому
ограничение «64 порта на VM» — это лимит одновременных соединений к **одному конкретному** dst IP:port.

**Разбор.** Продакшн-проблема: сотни подов, которые все ходят к **одному** внешнему API (один dst IP:port)
→ быстро упираются в порты → **port exhaustion** → дропы новых соединений. Решения: увеличить
`--min-ports-per-vm`, добавить внешних IP в пул NAT, распределять нагрузку по нескольким destination,
кэшировать/пулить соединения на стороне приложения.

### 8.5 [С] Статический IP для NAT: зачем и что будет, если удалить его, не обновив NAT?

**Ответ.** Статический исходящий IP нужен для **IP whitelisting** — когда внешний партнёр/API пускает
только с известного адреса. Если удалить зарезервированный IP, не переключив NAT, трансляция сломается:
исходящие соединения упадут (GCP обычно не даёт удалить используемый IP, но при принудительном удалении
NAT перестанет работать до назначения нового адреса).

**Разбор.** Auto-allocate удобен для разработки, но адрес может меняться — не годится для whitelisting.
Метрики для мониторинга NAT: `allocated_ports` (заполнение портов), `dropped_sent_packets_count`
(дропы = признак port exhaustion), `new_connections_count` (аномалии). Алерт на dropped packets — первый
сигнал проблемы.

### 8.6 [Б] Что такое IAP и почему SSH к приватной VM идёт из диапазона 35.235.240.0/20?

**Ответ.** IAP (Identity-Aware Proxy) позволяет подключаться по SSH к VM **без публичного IP**: трафик
идёт через инфраструктуру Google, а не через интернет. Firewall-правило разрешает порт 22 только из
диапазона `35.235.240.0/20` — это адреса серверов IAP.

**Разбор.** Так совмещаются две цели: VM не имеет публичного IP (безопасно), но администратор всё равно
может зайти по SSH (через аутентифицированный прокси Google). Подключение: `gcloud compute ssh ... 
--tunnel-through-iap`.

# Раздел 9. Lab 09 — Traffic Control (tc / netem)

### 9.1 [Б] Что такое qdisc и какой qdisc эмулирует «плохую» сеть?

**Ответ.** qdisc (queueing discipline) — дисциплина обработки очереди пакетов на интерфейсе. «Плохую»
сеть эмулирует **netem** (Network Emulator): задержки, потери, повреждения, дублирование, переупорядочение
пакетов. Применяется к корневому qdisc интерфейса:
`tc qdisc add dev veth-srv root netem delay 100ms`.

**Разбор.** По умолчанию tc/netem действует на **исходящий** (egress) трафик интерфейса. Чтобы
эмулировать задержку и в обратную сторону, правило вешают на интерфейсы обоих концов. Удаление:
`tc qdisc del dev veth-srv root`.

### 9.2 [С] Как смоделировать задержку 100 мс и потерю 20%? Как проверить?

**Ответ.**
- Задержка: `tc qdisc add dev veth-srv root netem delay 100ms` → в `ping` `time=` стабильно > 100 мс.
- Потеря: `tc qdisc add dev veth-srv root netem loss 20%` → `ping -c 20` покажет ~20% packet loss.

**Разбор.** Это **chaos engineering**: проверяем, как приложение переживает деградацию сети (таймауты,
ретраи, идемпотентность). netem умеет больше: `delay 100ms 20ms` (джиттер), `corrupt 5%` (битые биты),
`duplicate 1%`, `reorder`. Реальные приложения часто ломаются именно на джиттере и reorder, а не на
ровной задержке.

### 9.3 [П] Чем `tbf` отличается от `netem`? Что такое token bucket в shaping?

**Ответ.** `netem` **портит** сеть (вносит задержки/потери). `tbf` (Token Bucket Filter) **ограничивает
полосу** (bandwidth shaping): `tc qdisc add dev veth-srv root tbf rate 1mbit burst 32kbit latency 400ms`
держит скорость на 1 Mbit/s (видно в `iperf3`). Token bucket: токены капают со скоростью `rate`, пакет
тратит токены пропорционально размеру; `burst` — ёмкость ведра (сколько можно отправить разом),
`latency` — сколько пакет ждёт токенов до дропа.

**Разбор.** Это тот же алгоритм token bucket, что в rate limiting iptables (вопрос 7.9), но применённый к
**полосе пропускания**, а не к числу пакетов. Проверка полосы: `iperf3 -s` на сервере и `iperf3 -c` на
клиенте — Bitrate стабильно ~1 Mbit/s.

# Раздел 10. Lab 10 — Динамическая маршрутизация (BGP / BIRD)

### 10.1 [Б] Что такое BGP и что такое AS?

**Ответ.** BGP (Border Gateway Protocol) — основной протокол маршрутизации **интернета**, по которому
автономные системы обмениваются маршрутами. **AS (Autonomous System)** — сеть/набор сетей под единым
административным управлением, со своим номером (в лабе AS 65001/65002/65003 — приватные номера). BGP —
**path-vector** протокол: вместе с маршрутом передаётся путь по AS (AS-PATH).

**Разбор.** В лабе три роутера цепочкой `r1 ↔ r2 ↔ r3`. Изначально r1 не знает сеть `10.3.0.0/24` (за r3),
а r3 не знает `10.1.0.0/24` (за r1). После поднятия BGP они обмениваются маршрутами через транзитный r2,
и в таблице ядра появляется путь с пометкой `proto bird`.

### 10.2 [С] Как проверить, что BGP-сессия поднялась, и что появилось в таблице маршрутизации ядра?

**Ответ.** В консоли BIRD (`birdc -s /tmp/bird-r1.ctl`):
- `show protocols` — статус сессии с соседом; рабочее состояние — **Established**.
- `show route` — полученные и анонсируемые маршруты.

В ядре (`ip route` на r1) появляется маршрут к `10.3.0.0/24` через r2 с меткой `proto bird` — его
**установил демон маршрутизации**, а не администратор вручную.

**Разбор.** Ключевой момент: BIRD (control plane) считает маршруты по BGP и **программирует** таблицу
ядра (data plane). `proto bird` в `ip route` — доказательство, что маршрут динамический. Пинг
`r1 → 10.3.0.1` подтверждает сквозную связность через транзит.

### 10.3 [П] Зачем dummy-интерфейс и что такое состояние Established?

**Ответ.** **dummy-интерфейс** (`dummy0`) — виртуальный интерфейс, который всегда «up» и держит адрес
сети (`10.1.0.0/24`), которую роутер **анонсирует** в BGP. Он имитирует «сеть за роутером», не требуя
реального линка. **Established** — финальное состояние BGP-сессии (после `Idle → Connect → OpenSent →
OpenConfirm → Established`): соседи согласовали параметры и обмениваются маршрутами.

**Разбор.** BGP-сессия идёт поверх **TCP (порт 179)**, поэтому до Established должна быть L3-связность
между соседями. Если сессия «застряла» не в Established — смотрят связность, номера AS, конфиг пиров.
dummy-интерфейс — стандартный приём «дать роутеру сеть для анонса» в лабораторных и в реальных
loopback-анонсах.

# Раздел 11. Lab 11 — mini-docker (контейнерная сеть руками)

### 11.1 [С] Какие пять шагов делает `mini-docker.sh run myweb 8080:80` и какому Docker-механизму
соответствует каждый?

**Ответ.**
1. Создаёт мост `docker-br0` — аналог `docker0`.
2. Создаёт namespace `myweb` — аналог сетевого namespace контейнера.
3. Подключает namespace к мосту veth-парой и выдаёт IP (`10.11.0.2`) — аналог сети контейнера.
4. Настраивает NAT: **SNAT/MASQUERADE** для выхода наружу + **DNAT** `8080 → 10.11.0.2:80` —
   аналог `-p 8080:80`.
5. Запускает сервис внутри namespace.

**Разбор.** После этого `curl http://localhost:8080` работает как в настоящем Docker. Вся «магия» —
это ровно команды из Lab 06 (bridge) и Lab 07 (NAT): `ip netns`, `ip link ... type veth`,
`ip link set ... master docker-br0`, `iptables -t nat -A PREROUTING ... DNAT` и `... POSTROUTING ...
MASQUERADE`. Docker лишь автоматизирует подсчёт IP и формирование правил.

### 11.2 [П] Что именно делает `docker run -p 8080:80 nginx` на уровне ядра?

**Ответ.**
1. Создаёт veth-пару (контейнер ↔ `docker0`).
2. Вешает DNAT: `PREROUTING --dport 8080 → <container_ip>:80` (и аналог в OUTPUT для localhost).
3. Вешает MASQUERADE в POSTROUTING для исходящего трафика контейнера.
4. Разрешает соответствующий FORWARD.

**Разбор.** Понимание этого снимает «магию» контейнерных сетей и даёт навык диагностики: если порт не
пробрасывается, смотрят `iptables -t nat -L -n -v` (есть ли DNAT, растут ли счётчики), FORWARD-политику,
`ip netns`/veth. Это фундамент для Kubernetes (Lab 12), где ту же работу делает CNI-плагин.

# Раздел 12. Lab 12 — CNI (Container Network Interface)

### 12.1 [Б] Что такое CNI и в каком виде поставляются плагины?

**Ответ.** CNI (Container Network Interface) — стандарт настройки сети контейнеров, который используют
Kubernetes (через kubelet), Podman, CRI-O и др. **CNI-плагин — это обычный исполняемый бинарь** (например
`/opt/cni/bin/bridge`), а не «магия внутри Kubernetes». Calico/Flannel/Cilium — это тоже наборы таких
бинарей плюс конфигурация.

**Разбор.** Демистификация: «сеть Kubernetes» в основе — это вызов бинаря, который делает ту же ручную
работу с `ip link`/`ip addr`, что и в Lab 06, но по стандартизированному контракту.

### 12.2 [С] Как kubelet вызывает CNI-плагин? Как передаются параметры?

**Ответ.** При создании пода:
1. Создаётся пустой network namespace.
2. Формируется **JSON-конфигурация** сети.
3. Запускается бинарь плагина, которому JSON передаётся через **stdin**, а параметры — через **переменные
   окружения**: `CNI_COMMAND` (ADD/DEL/CHECK), `CNI_CONTAINERID`, `CNI_NETNS` (путь к namespace),
   `CNI_IFNAME` (имя интерфейса, обычно `eth0`), `CNI_PATH` (где искать бинари).
4. Плагин настраивает сеть и завершается, вернув результат в JSON.

**Разбор.** В лабе это вызывается вручную, без Kubernetes:
```
export CNI_COMMAND=ADD CNI_CONTAINERID=... CNI_NETNS=/var/run/netns/cni-ns CNI_IFNAME=eth0 CNI_PATH=/opt/cni/bin
cat my-cni-config.json | /opt/cni/bin/bridge
```
Плагин `bridge` создаёт мост на хосте, veth-пару, кладёт один конец в namespace как `eth0`, а IPAM-плагин
`host-local` выдаёт IP из `10.22.0.0/24` и прописывает маршруты.

### 12.3 [С] Что такое IPAM и роль `host-local`?

**Ответ.** IPAM (IP Address Management) — компонент CNI, отвечающий за **выдачу и учёт IP-адресов**.
`host-local` хранит выделенные адреса локально на ноде (в файловой БД) и выдаёт следующий свободный из
заданного диапазона. Основной плагин (`bridge`) занимается интерфейсами, IPAM-плагин — адресами; они
работают в связке.

**Разбор.** Разделение ответственности: `bridge` не знает, какой IP свободен, — он спрашивает IPAM.
В кластере используют сетевые IPAM (например, выдача из pod CIDR ноды), чтобы адреса не пересекались между
нодами.

### 12.4 [П] Как «удалить» контейнер в парадигме CNI и что делает плагин по команде DEL?

**Ответ.** Тот же бинарь вызывается с `CNI_COMMAND=DEL` и тем же конфигом/идентификатором. Плагин сам
удаляет созданные интерфейсы (veth) и **освобождает IP** в базе `host-local` IPAM.

**Разбор.** Симметрия ADD/DEL — основа идемпотентного жизненного цикла пода: kubelet вызывает ADD при
создании и DEL при удалении. Если DEL не отработает (утечка), копятся «осиротевшие» veth и занятые IP —
поэтому CNI обязан корректно чистить за собой. CHECK (третья команда) проверяет, что сеть в ожидаемом
состоянии.

# Раздел 13. Сквозные темы (синтез)

### 13.1 [П] Сводка по типам NAT — заполни и объясни.

| Тип | Что меняет | Цепочка | Когда применять |
|-----|-----------|---------|-----------------|
| **DNAT** | destination IP:port | PREROUTING (OUTPUT для локальных) | проброс портов, публикация сервиса |
| **SNAT** | source → фиксированный IP | POSTROUTING | выход наружу со **статическим** IP |
| **MASQUERADE** | source → IP исходящего интерфейса (авто) | POSTROUTING | выход наружу с **динамическим** IP |

**Разбор.** `docker run -p` = DNAT + MASQUERADE. Cloud NAT = распределённый SNAT. DNAT сохраняет source
IP (бэкенд видит клиента → нужен обратный маршрут). MASQUERADE удобнее, но чуть дороже (lookup IP на
пакет). Эти три механизма + conntrack покрывают 90% NAT-задач в инфраструктуре.

### 13.2 [П] Покажи путь пакета `curl localhost:8080 → контейнер` (DNAT) и обратно.

**Ответ.**
- **Туда:** пакет рождён локально (`OUTPUT`) → DNAT в `nat/OUTPUT` меняет dst на `10.0.0.2:80` →
  routing decision (теперь не для хоста) → `FORWARD (filter)` → через bridge в namespace → приложение.
- **Обратно:** ответ из namespace (src `10.0.0.2:80`) → bridge → хост → conntrack узнаёт соединение и
  делает **обратный DNAT** (src снова `localhost:8080`) → отдаётся локальному `curl`.

**Разбор.** Обратную трансляцию делает **conntrack** автоматически (он помнит исходную трансляцию). Именно
поэтому stateful NAT не требует «зеркальных» правил для ответов. На `br0` tcpdump покажет уже `10.0.0.2:80`
(DNAT произошёл раньше), а на внешнем интерфейсе — оригинальный адрес (DNAT для входящих ещё не сработал).

### 13.3 [П] Сопоставь технологии курса с уровнями OSI.

**Ответ.**
- **L2 (канальный):** veth, Linux bridge, MAC/FDB, ARP, VLAN 802.1q, broadcast-домен.
- **L3 (сетевой):** IP-маршрутизация (`ip route`), `ip_forward`, ICMP, NAT (DNAT/SNAT), BGP, inter-VLAN
  routing, WireGuard-туннель (L3-overlay поверх UDP).
- **L4 (транспортный):** TCP/UDP-порты, conntrack-состояния, L4-балансировка, rate limiting по портам,
  Cloud NAT (порты на VM).
- **L7 (прикладной):** DNS, DHCP, HTTP, L7-балансировка (HAProxy `mode http`, `httpchk`).

**Разбор.** Многие «непонятные» баги — это путаница уровней: «один IP-сегмент, но не пингуется» — блок на
L2 (VLAN); «порт открыт, но соединение рвётся» — L4/conntrack; «IP резолвится не туда» — L7 (DNS).
Хороший ответ на экзамене всегда называет уровень.

# Раздел 15. Troubleshooting-сценарии (разбор «боевых» инцидентов)

### И-1. «Маршрут есть, DNAT есть, но `curl` висит по таймауту»
**Причины.** (1) на бэкенде нет обратного маршрута/default gateway; (2) `FORWARD DROP` без разрешения
проброшенного трафика и/или `ESTABLISHED,RELATED`. **Диагностика.** `ip route` на бэкенде;
`iptables -L FORWARD -n -v` (растут ли счётчики DROP); `conntrack -L` (видит ли соединение). (Lab 01, 1.6)

### И-2. «Контейнер не ходит в интернет»
**Причины.** `ip_forward=0`; нет MASQUERADE; FORWARD не разрешает `br0 → ext`; нет default route в
namespace; нет DNS (`resolv.conf`). **Диагностика.** `cat /proc/sys/net/ipv4/ip_forward`;
`iptables -t nat -L POSTROUTING -n -v`; `ip netns exec red ip route`. (Lab 06)

### И-3. «Два хоста в одной подсети не видят друг друга»
**Причина.** Разные VLAN на bridge (L2-изоляция), хотя IP-подсеть одна. **Диагностика.**
`bridge vlan show`, `tcpdump -i br0 -e -n vlan`. **Решение.** один VLAN или inter-VLAN routing. (Lab 04/06)

### И-4. «Соединение к сервису рвётся/не открывается, хотя порт слушает»
**Причины.** stateful-правило не пропускает NEW; conntrack-таблица переполнена
(`nf_conntrack_count` ≈ `max`); INVALID-пакеты дропаются; rate limit/recent забанил источник.
**Диагностика.** `conntrack -S`, `dmesg | grep DROP`, `nft monitor trace`. (Lab 07)

### И-5. «Облачные поды периодически не достукиваются до внешнего API»
**Причина.** Port exhaustion в Cloud NAT (много подов → один dst IP:port). **Диагностика.** метрика
`dropped_sent_packets_count`, `get-nat-mapping-info`. **Решение.** `--min-ports-per-vm`, доп. внешние IP,
пул соединений. (Lab 08, 8.4)

### И-6. «После перезагрузки firewall исчез / хост закрылся»
**Причины.** правила не сохранены (нет `iptables-persistent`/`systemctl enable nftables`); либо скрипт
упал на середине последовательного применения, оставив `policy DROP` без `ACCEPT`. **Решение.**
атомарное применение (`nft -f`, `iptables-restore`) + персистентность. (Lab 07, 7.10)

# Раздел 16. Сводная шпаргалка команд

```bash
# === Namespaces ===
ip netns add red; ip netns list; ip netns exec red <cmd>; ip netns del red

# === veth и bridge ===
ip link add veth-red type veth peer name eth0-red
ip link set veth-red netns red                  # переместить конец в namespace
ip link add br0 type bridge; ip link set br0 up
ip link set veth-red master br0                  # подключить порт к bridge
bridge fdb show br br0; bridge link show; bridge vlan show

# === Адреса, маршруты, форвардинг ===
ip addr add 10.0.0.2/24 dev eth0; ip link set eth0 up
ip route; ip route add default via 10.0.0.1
sysctl -w net.ipv4.ip_forward=1; cat /proc/sys/net/ipv4/ip_forward
ip neigh show; ip neigh flush all                # ARP/neighbor

# === iptables: NAT и фильтрация ===
iptables -t nat -A PREROUTING -d <ext_ip> -p tcp --dport 80 -j DNAT --to-destination 10.0.0.2:80
iptables -t nat -A POSTROUTING -s 10.0.0.0/24 -o eth0 -j MASQUERADE
iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -P FORWARD DROP                          # политика по умолчанию
iptables -L -n -v --line-numbers; iptables -t nat -L -n -v
iptables-save > rules.v4; iptables-restore < rules.v4   # атомарно

# === conntrack ===
conntrack -L; conntrack -E; conntrack -F; conntrack -S
cat /proc/sys/net/netfilter/nf_conntrack_{count,max}

# === nftables ===
nft list ruleset; nft flush ruleset
nft add table inet firewall
nft add chain inet firewall input '{type filter hook input priority 0; policy drop;}'
nft add rule inet firewall input ct state established,related accept
nft add set inet firewall banned '{type ipv4_addr; flags timeout; timeout 10m;}'
nft -c -f file.nft && nft -f file.nft            # валидация + атомарное применение
nft monitor trace                                 # пошаговая трассировка

# === VLAN ===
bridge vlan add vid 10 dev veth-red pvid untagged
ip link add link eth0 name eth0.10 type vlan id 10

# === tc / netem / tbf ===
tc qdisc add dev veth-srv root netem delay 100ms loss 20%
tc qdisc add dev veth-srv root tbf rate 1mbit burst 32kbit latency 400ms
tc qdisc del dev veth-srv root

# === Диагностика ===
ping -c3 <ip>; curl --max-time 3 <url>; ncat -z -v <ip> <port>
tcpdump -i br0 -e -n; tcpdump -i br0 -e -n vlan
iperf3 -s   /   iperf3 -c <ip>
dig +short @<dns> <name>

# === WireGuard ===
wg genkey | tee priv | wg pubkey > pub; wg show

# === BGP (BIRD) ===
birdc -s /tmp/bird-r1.ctl    # затем: show protocols; show route

# === CNI (ручной вызов) ===
export CNI_COMMAND=ADD CNI_NETNS=/var/run/netns/cni-ns CNI_IFNAME=eth0 CNI_PATH=/opt/cni/bin
cat my-cni-config.json | /opt/cni/bin/bridge     # DEL — то же с CNI_COMMAND=DEL
```

# Раздел 17. Критерии оценивания (rubric)

| Оценка | Что должен показать студент |
|--------|------------------------------|
| **Неуд** | Путает L2 и L3; не понимает разницу маршрута и форвардинга; не может прочитать правило iptables. |
| **Удовл** | Отвечает на `[Б]`: настраивает маршруты, DNAT/MASQUERADE, объясняет `ip_forward`, базовый bridge/veth. |
| **Хорошо** | `[Б]`+`[С]`: уверенно про conntrack (NEW/ESTABLISHED/RELATED), VLAN-изоляцию, DROP vs REJECT, defense in depth, путь пакета, диагностику таймаутов. |
| **Отлично** | `[П]`: token bucket, атомарность nftables/sets, cryptokey routing WireGuard, BGP Established, Cloud NAT port exhaustion, собирает контейнерную сеть руками и сводит её к CNI/Docker. |

**Минимальный порог:** студент должен на любом примере (1) назвать механизм, (2) указать уровень OSI,
(3) сказать, что сломается, если механизм убрать, и (4) показать команду диагностики.
