## Часть 1: Теория BGP, AS-path и демона BIRD

### 1.1 Border Gateway Protocol (BGP)
BGP — это основной протокол маршрутизации в Интернете, относящийся к классу **Path-Vector** протоколов. Он оперирует понятием **Автономных систем (AS)**.
* **Автономная система (AS):** Совокупность IP-сетей под единым административным управлением (например, провайдер, университет или крупная IT-компания), имеющая свой уникальный номер (ASN — Autonomous System Number).
* Диапазон номеров `64512 - 65535` зарезервирован для частного использования (Private ASN), аналогично "серым" IP-адресам. Мы используем их в нашей лабе.

### 1.2 Атрибут AS-path и предотвращение петель
BGP анонсирует маршруты в виде векторов путей. Ключевым атрибутом анонса является **AS-path** — упорядоченный список ASN, через которые прошел данный анонс.
1. **Loop Prevention (Предотвращение петель):** Когда роутер получает BGP-анонс, он анализирует список AS-path. Если в списке присутствует его собственный номер AS, роутер отвергает этот анонс. Это гарантирует отсутствие петель на уровне автономных систем.
2. **Выбор пути:** Если до одной сети есть несколько путей BGP, по умолчанию выбирается путь с кратчайшим списком AS-path (наименьшее количество хопов между AS).

### 1.3 BIRD (BIRD Internet Routing Daemon)
**BIRD** — это мощное и популярное в Linux-мире программное обеспечение для динамической маршрутизации. Оно имеет модульную структуру и конфигурируется через декларативный язык описания правил.
* BIRD широко используется в SDN и облачных технологиях. Например, сетевой плагин **Calico** в Kubernetes разворачивает BIRD на каждом узле кластера, чтобы анонсировать IP-адреса подов по BGP на физические коммутаторы сети.

## Часть 2: Настройка BGP и анализ маршрутов

Скрипт развернул три роутера. `r1` соединен с `r2` подсетью `192.168.12.0/30`, `r2` соединен с `r3` подсетью `192.168.23.0/30`. `r2` выполняет роль транзитной AS.

### 2.1 Инспекция состояния сессий через birdc

Для взаимодействия с демоном BIRD используется консольный клиент `birdc`. Укажем путь к сокету управления роутера `r1`:
```bash
sudo birdc -s /tmp/bird-r1.ctl show protocols
```
Вы должны увидеть вывод:
```text
BIRD 2.0.12 ready.
Name     Proto    Table    State  Since       Info
device1  Device   master4  up     06:15:00    
direct1  Direct   master4  up     06:15:00    
kernel1  Kernel   master4  up     06:15:00    
bgp_r2   BGP      ---      up     06:15:02    Established   
```
Строка `bgp_r2 ... Established` подтверждает, что BGP-сессия с `r2` успешно установлена.

### 2.2 Разбор таблицы маршрутизации BIRD

Посмотрим, какие маршруты знает BIRD на `r1`:
```bash
sudo birdc -s /tmp/bird-r1.ctl show route
```
Вы увидите:
```text
BIRD 2.0.12 ready.
10.1.0.0/24          unicast [direct1 06:15:00] * (240)
10.3.0.0/24          unicast [bgp_r2 06:15:02] * (100) [AS65003i]
                     via 192.168.12.2 on veth-r1
192.168.12.0/30      unicast [bgp_r2 06:15:02] * (100) [AS65002i]
                     via 192.168.12.2 on veth-r1
```
Обратите внимание на запись:
* `10.3.0.0/24 ... [bgp_r2 ...] * (100) [AS65003i]`
* Скобки `[AS65003i]` (или список номеров AS) показывают атрибут **AS-path**. Маршрут пришел от автономной системы `65003`. Символ `i` означает происхождение (IGP).

Проверьте пинг с `r1` до фантомной сети `r3`:
```bash
sudo ip netns exec r1 ping -c 3 10.3.0.1
# Пинг должен успешно проходить, так как маршрут импортирован в ядро Linux.
```

## Часть 3: Troubleshooting — боевой инцидент (Сессия не Established / AS Mismatch)

### Симптомы инцидента
Сеть `10.3.0.0/24` недоступна с роутера `r1`. Пинг падает. При проверке таблицы маршрутизации ядра `ip route` маршрут отсутствует.

### Шаг 1: Воспроизведение инцидента
Сымитируем ошибку конфигурации BGP-соседа на `r1`. Изменим ожидаемый номер автономной системы роутера `r2` с `65002` на неверный `65999` в конфигурационном файле `/tmp/bird-r1.conf`, а затем перезапустим BIRD на `r1`:

```bash
# Меняем ASN соседа в конфигурации r1
sudo sed -i 's/neighbor 192.168.12.2 as 65002;/neighbor 192.168.12.2 as 65999;/' /tmp/bird-r1.conf

# Перезапускаем BIRD на r1
sudo kill $(cat /tmp/bird-r1.pid 2>/dev/null) 2>/dev/null || true
sudo ip netns exec r1 bird -c /tmp/bird-r1.conf -P /tmp/bird-r1.pid -s /tmp/bird-r1.ctl

# Дадим время на попытку авторизации
sleep 3
```

### Шаг 2: Систематическая диагностика

1. **Проверяем таблицу маршрутизации ядра:**
   ```bash
   sudo ip netns exec r1 ip route show proto bird
   # Вывод пуст! Динамических маршрутов нет.
   ```
2. **Проверяем статус BGP-протоколов в BIRD:**
   ```bash
   sudo birdc -s /tmp/bird-r1.ctl show protocols
   # Вывод:
   # Name     Proto    Table    State  Since       Info
   # bgp_r2   BGP      ---      start  06:17:34    Active        Socket: Connection closed
   ```
   Статус сессии равен `Active` (что в терминологии BGP означает "активную попытку подключения" или ожидание ответа), либо `Idle`, но **не** `Established`. Сессия лежит.

3. **Смотрим логи BIRD:**
   BIRD отправляет логи в syslog хоста. Проверим логи BIRD:
   ```bash
   sudo journalctl -u bird -n 30 --no-pager || sudo tail -n 50 /var/log/syslog | grep bird
   # Вы увидите сообщения об ошибке:
   # bgp_r2: Peer AS mismatch (expected 65999, got 65002)
   # bgp_r2: Received NOTIFICATION code 2 (Open Message Error) subcode 2 (bad peer AS)
   ```
   Причина ясна: `r1` ожидает соседа с AS `65999`, но `r2` при открытии сессии сообщает свой реальный ASN `65002`. Из-за этого сессия BGP немедленно сбрасывается.

### Шаг 3: Устранение инцидента
Исправьте конфигурационный файл `/tmp/bird-r1.conf`, вернув правильный ASN `65002`, и перезапустите BIRD:

```bash
# Возвращаем правильный ASN соседа
sudo sed -i 's/neighbor 192.168.12.2 as 65999;/neighbor 192.168.12.2 as 65002;/' /tmp/bird-r1.conf

# Перезапускаем BIRD
sudo kill $(cat /tmp/bird-r1.pid 2>/dev/null) 2>/dev/null || true
sudo ip netns exec r1 bird -c /tmp/bird-r1.conf -P /tmp/bird-r1.pid -s /tmp/bird-r1.ctl

sleep 2
```

Убедитесь, что сессия поднялась:
```bash
sudo birdc -s /tmp/bird-r1.ctl show protocols
# bgp_r2 ... Established

# Маршруты снова в ядре
sudo ip netns exec r1 ip route show proto bird
```
Связь восстановлена!

## Как это звучит на собеседовании

1. **Вопрос:** В чем разница между eBGP (External BGP) и iBGP (Internal BGP)? К какому типу относится сессия между роутерами в нашей лабе?
   **Ответ:** eBGP (External BGP) используется для связи между разными автономными системами (разные ASN). iBGP (Internal BGP) используется для обмена маршрутами внутри одной автономной системы (один ASN). В нашей лабораторной работе все сессии связывают разные AS (AS 65001, 65002, 65003), поэтому все они являются eBGP-сессиями.
2. **Вопрос:** Опишите ключевые состояния жизненного цикла BGP-соседства (BGP FSM).
   **Ответ:** 
   * **Idle:** Начальное состояние, BGP-процесс ожидает события старта или повторной попытки.
   * **Connect:** BGP ожидает завершения трехэтапного рукопожатия TCP с пиром.
   * **Active:** TCP-соединение не удалось установить, BGP активно пытается инициировать соединение повторно.
   * **OpenSent:** TCP-соединение установлено, отправлено сообщение BGP OPEN.
   * **OpenConfirm:** Получен ответ BGP OPEN от пира, отправлено сообщение KEEPALIVE.
   * **Established:** Сессия полностью установлена, начинается обмен маршрутами (UPDATE).
3. **Вопрос:** Как атрибут AS-path предотвращает возникновение петель маршрутизации?
   **Ответ:** При передаче анонса маршрута из одной AS в другую роутер дописывает номер своей AS в начало списка AS-path. Если роутер получает BGP-анонс, в списке AS-path которого уже присутствует номер его собственной AS, он понимает, что пакет прошел круг, и отбрасывает этот анонс, предотвращая появление петли.
4. **Вопрос:** Почему Kubernetes-плагины (например, Calico) используют BGP вместо статической маршрутизации?
   **Ответ:** В Kubernetes поды постоянно создаются и удаляются на разных нодах, что требует динамического обновления маршрутов до каждой подсети подов. BGP позволяет нодам кластера автоматически и мгновенно анонсировать адреса подов друг другу без необходимости переписывать статические таблицы маршрутизации вручную или через API, обеспечивая высокую масштабируемость и отказоустойчивость.
