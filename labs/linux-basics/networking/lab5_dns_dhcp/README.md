# Lab 5: Внутренний DNS и DHCP (Dnsmasq)

В этой лабораторной работе мы изучим, как происходит динамическая выдача IP-адресов (DHCP) и настройка внутреннего Service Discovery с помощью кэширующего DNS-сервера. Мы используем легковесный `dnsmasq`.

## Запуск лабы
Для автоматического развертывания стенда выполните:
```bash
cd /root/lern/labs/linux-basics/lab5_dns_dhcp
bash setup.sh
```

## Что происходит под капотом?

1. Создаются два Network Namespaces: `server` и `client`, соединенные напрямую через veth-кабель.
2. В неймспейсе `server` запускается **Dnsmasq** (IP: 10.0.0.1).
3. **DHCP:** Dnsmasq настроен выдавать пустые IP в диапазоне `10.0.0.10-50` и сообщать клиентам, что DNS-сервер находится на `10.0.0.1`.
4. **DNS:** Dnsmasq настроен отдавать кастомные IP-адреса для домена `*.internal.company`.

## Проверка DHCP (Получение IP)
Запросите у сервера свободный IP-адрес для клиента. Мы используем `udhcpc` из BusyBox, чтобы получить адрес безопасно (без перезаписи хостового `/etc/resolv.conf`):
```bash
ip netns exec client busybox udhcpc -i veth-cli -s /tmp/udhcpc.script -n -q
```
Посмотрите, применился ли IP:
```bash
ip netns exec client ip addr show veth-cli
```
*Вы должны увидеть IP-адрес из пула (например, `10.0.0.19`).*

## Проверка DNS (Разрешение имен)
Попробуем разрешить наше внутреннее доменное имя в IP-адрес:
```bash
ip netns exec client dig +short @10.0.0.1 db.internal.company
```
*Ответ должен быть `10.0.0.100`!*

Таким образом работает Service Discovery в простых инфраструктурах — сервисы общаются по красивым именам, а внутренний DNS-сервер (Dnsmasq / CoreDNS / Bind) разруливает их в актуальные IP-адреса.
