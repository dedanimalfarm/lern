## Часть 1: Маршрутизация (Routing) и ip_forward

### Теория для изучения перед частью

- **Таблица маршрутизации (`ip route`)**: Говорит ядру Linux, куда отправлять пакеты для определённых подсетей. Если нужной подсети нет в таблице и нет `default` маршрута, ядро выдаст `Network is unreachable`.
- **Шлюз (Gateway)**: Роутер, который знает путь до нужной сети.
- **Пересылка пакетов (`ip_forward`)**: По умолчанию Linux **игнорирует** пакеты, которые адресованы не ему (если он получил пакет для чужого IP). Чтобы Linux начал работать как роутер и пересылал пакеты между интерфейсами, нужно включить параметр ядра `net.ipv4.ip_forward=1`.

---

**Цель:** Сделать так, чтобы `client_ns` мог пинговать `server_ns`, проходя через `router_ns`.

### 1.1 Настройка маршрутов на клиенте и сервере

Сейчас клиент (IP `10.0.1.2`) знает только про свою сеть `10.0.1.0/24`. Сервер (`10.0.2.2`) знает только про `10.0.2.0/24`.

```bash
# Попробуем пингануть сервер с клиента (Провал)
sudo ip netns exec client_ns ping -c 1 10.0.2.2
# Ответ: ping: connect: Network is unreachable

# Смотрим маршруты клиента
sudo ip netns exec client_ns ip route
# 10.0.1.0/24 dev veth-c proto kernel scope link src 10.0.1.2
```

Добавим маршрут по умолчанию (default gateway) на клиента, указав адресом шлюза `router_ns`:

```bash
sudo ip netns exec client_ns ip route add default via 10.0.1.1
sudo ip netns exec server_ns ip route add default via 10.0.2.1
```

### 1.2 Включение ip_forward на роутере

Попробуем снова пингануть сервер:
```bash
sudo ip netns exec client_ns ping -c 1 10.0.2.2
# Все равно не работает (100% packet loss), хотя маршрут есть!
```

Почему? Потому что пакет доходит до `router_ns`, роутер видит, что IP назначения `10.0.2.2` (не его собственный), и просто "дропает" пакет. Linux по умолчанию не маршрутизирует чужой трафик.

Включим маршрутизацию:
```bash
sudo ip netns exec router_ns sysctl -w net.ipv4.ip_forward=1
```

Проверяем снова:
```bash
sudo ip netns exec client_ns ping -c 1 10.0.2.2
# 64 bytes from 10.0.2.2: icmp_seq=1 ttl=63 time=... УСПЕХ!
```

**Контрольные вопросы:**
1. Что означает ошибка `Network is unreachable`?
2. Зачем нужна настройка `net.ipv4.ip_forward`? Работает ли без неё Docker?
3. Почему мы настраивали маршруты и на клиенте, и на сервере? Что было бы, если бы мы настроили только на клиенте?

## Часть 2: Проброс портов (DNAT)

### Теория для изучения перед частью

- Часто внутренний сервер находится в "серой" сети, к которой клиенты не имеют прямого доступа (например, за NAT провайдера или внутри Docker-сети).
- **DNAT (Destination NAT)**: Механизм iptables, который меняет **IP-адрес и порт назначения** у входящего пакета.
- **Port Forwarding**: Популярный случай DNAT. Когда кто-то стучится на внешний IP роутера (на порт 80), роутер "на лету" подменяет получателя на внутренний IP сервера (на порт 8080).
- Цепочка `PREROUTING`: Правило DNAT применяется **до** принятия решения о маршрутизации, чтобы ядро успело перенаправить пакет на нужный интерфейс.

---

**Цель:** Сделать так, чтобы клиент обращался на IP роутера (порт 80), а попадал на внутренний веб-сервер.

### 2.1 Настройка правила DNAT

В нашем `server_ns` уже запущен Python веб-сервер на порту 8080.
Проверим напрямую:
```bash
sudo ip netns exec client_ns curl http://10.0.2.2:8080
# Hello from Internal Server!
```

Теперь представим, что сеть `10.0.2.0/24` полностью скрыта от клиента. Клиент может обращаться только на внешний адрес роутера `10.0.1.1`.

Настраиваем DNAT на роутере:
```bash
sudo ip netns exec router_ns iptables -t nat -A PREROUTING \
  -d 10.0.1.1 -p tcp --dport 80 \
  -j DNAT --to-destination 10.0.2.2:8080
```
*Перевод правила: "Таблица nat, цепочка PREROUTING. Если пакет летит на IP 10.0.1.1 по TCP на порт 80, сменить Destination на 10.0.2.2:8080".*

### 2.2 Проверка проброса

```bash
sudo ip netns exec client_ns curl http://10.0.1.1:80
# Hello from Internal Server!
```
Магия! Мы обратились на порт 80 роутера, а ответил сервер с порта 8080. Это ровно то, что происходит, когда вы запускаете `docker run -p 80:8080 nginx`.

**Контрольные вопросы:**
1. В какой таблице iptables настраивается DNAT и почему?
2. Почему DNAT делается в цепочке `PREROUTING`, а не `POSTROUTING`?
3. Знает ли внутренний сервер реальный IP-адрес клиента при использовании только DNAT? (Ответ: Да, source IP пакета не менялся).

## Часть 3: Troubleshooting — боевые инциденты

Что может пойти не так? 

### Инцидент 1: Маршрут есть, DNAT есть, но curl висит

Удалим default route с сервера, имитируя частую ошибку (забыли настроить шлюз на бэкенде).
```bash
sudo ip netns exec server_ns ip route del default
```

Проверим `curl` с клиента через проброшенный порт:
```bash
sudo ip netns exec client_ns curl --max-time 3 http://10.0.1.1:80
# curl: (28) Connection timed out
```

**Почему?**
Пакет от клиента (`10.0.1.2`) пришел на роутер. Роутер сделал DNAT и отправил пакет на сервер (`10.0.2.2`). Сервер пакет получил! Но когда сервер хочет ответить, он формирует ответный пакет для `10.0.1.2`. И тут сервер понимает, что в его таблице маршрутизации нет пути до `10.0.1.0/24`, и нет default gateway. Ответный пакет уничтожается.

**Решение:** 
Вернуть маршрут (или настроить SNAT на роутере, чтобы серверу казалось, что запросы шлет сам роутер).
```bash
sudo ip netns exec server_ns ip route add default via 10.0.2.1
```

### Инцидент 2: Фаервол (iptables FORWARD) блокирует трафик

По умолчанию в Linux цепочка `FORWARD` разрешает всё (`ACCEPT`). Но в security-ориентированных системах и Kubernetes дефолтная политика часто `DROP`.

Сымитируем строгий фаервол на роутере:
```bash
sudo ip netns exec router_ns iptables -P FORWARD DROP
```

Проверяем `curl`:
```bash
sudo ip netns exec client_ns curl --max-time 3 http://10.0.1.1:80
# Connection timed out
```

**Решение:**
Нужно явно разрешить транзит (FORWARD) для наших проброшенных соединений.
```bash
sudo ip netns exec router_ns iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
sudo ip netns exec router_ns iptables -A FORWARD -d 10.0.2.2 -p tcp --dport 8080 -j ACCEPT
```

## Шпаргалка

```bash
# === Проверка маршрутов и настроек ===
ip route                         # показать таблицу маршрутизации
sysctl net.ipv4.ip_forward       # проверить, включен ли форвардинг

# === Управление маршрутами ===
ip route add default via 192.168.1.1   # добавить маршрут по умолчанию
ip route add 10.0.2.0/24 via 10.0.1.1  # добавить маршрут к конкретной подсети

# === Проброс портов (DNAT) ===
iptables -t nat -A PREROUTING -d <внешний_ip> -p tcp --dport <внешний_порт> -j DNAT --to-destination <внутренний_ip>:<внутренний_порт>

# === Просмотр правил iptables ===
iptables -t nat -L -n -v         # посмотреть правила таблицы nat (DNAT/SNAT)
iptables -L -n -v                # посмотреть основные правила (FORWARD, INPUT, OUTPUT)
```
