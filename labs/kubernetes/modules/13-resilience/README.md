## Часть 1: topologySpreadConstraints (Равномерное распределение)

### Теория для изучения перед частью 1

Механизм **TopologySpreadConstraints** предназначен для РАВНОМЕРНОГО распределения подов по заданным топологическим доменам. В отличие от `nodeAffinity`, который "притягивает" поды к определенным узлам, `topologySpread` заботится о балансе и выживаемости сервиса при падении целого домена (узла или зоны доступности).

- **Домены топологии (`topologyKey`)**: Это ключи лейблов на узлах. Например, `kubernetes.io/hostname` делает каждый узел отдельным доменом. `topology.kubernetes.io/zone` объединяет узлы в зоны доступности (AZ).
- **Максимальный перекос (`maxSkew`)**: Максимально допустимая разница в количестве подов между любыми двумя доменами. Если `maxSkew: 1`, разница между самым загруженным и самым пустым доменом не может превышать 1 под.
- **Действие при невозможности (`whenUnsatisfiable`)**:
  - `DoNotSchedule`: Жесткое требование. Если планировщик не может удовлетворить `maxSkew`, под останется в состоянии `Pending`.
  - `ScheduleAnyway`: Мягкое требование. Планировщик постарается минимизировать перекос, но если это невозможно, под все равно будет запущен, чтобы избежать простоя.

**Схема: домены топологии и перекос (`topologyKey: zone`, `maxSkew: 1`):**

```text
        zone-a            zone-b            zone-c
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │  [P]  [P]    │  │  [P]  [P]    │  │  [P]         │
   └──────────────┘  └──────────────┘  └──────────────┘
        2 пода            2 пода            1 под

   skew = max(2) − min(1) = 1  ≤  maxSkew(1)   →  баланс допустим
   Следующий под при whenUnsatisfiable=DoNotSchedule сядет ТОЛЬКО в zone-c.
   Размещение в zone-a/-b дало бы 3 vs 1 → skew=2 > maxSkew → под уйдёт в Pending.
```

**Таблица: Примеры расчета skew (maxSkew=1, 2 worker-узла):**

| Количество подов | Раскладка (node1 / node2) | Расчет skew (max - min) | Статус (DoNotSchedule) |
|:---:|:---:|:---:|:---|
| 1 | 1 / 0 | 1 - 0 = 1 | Успешно (skew <= maxSkew) |
| 2 | 1 / 1 | 1 - 1 = 0 | Успешно (идеальный баланс) |
| 3 | 2 / 1 | 2 - 1 = 1 | Успешно (3-й под пойдет на менее загруженный узел) |
| 4 | 2 / 2 | 2 - 2 = 0 | Успешно |
| 5 | 3 / 2 | 3 - 2 = 1 | Успешно |
| 3 | 3 / 0 (принудительно) | 3 - 0 = 3 | Ошибка `Pending` (если попытаемся добавить на node1) |

> `skew = (максимум подов в домене) − (минимум подов в домене)`. Новый под при
> `DoNotSchedule` сядет ТОЛЬКО туда, где итоговый skew ≤ `maxSkew`. Если все домены
> равны и нельзя добавить без нарушения — под перейдет в `Pending`.

#### Глубокий разбор nodeTaintsPolicy

Начиная с Kubernetes 1.26, появилось поле `nodeTaintsPolicy`. Оно критически важно для кластеров, где некоторые узлы имеют `taints` (например, Control Plane).
По умолчанию (`Ignore`), планировщик учитывает ВСЕ узлы при расчете перекоса, даже те, куда под не может быть запланирован из-за taints. Это приводит к тому, что заtaint-ованный узел всегда имеет 0 подов, искусственно увеличивая перекос и вызывая ложные зависания подов в `Pending`.
Использование `nodeTaintsPolicy: Honor` заставляет планировщик исключать узлы с непротолерированными taints из математики `topologySpread`.

**topologySpread vs podAntiAffinity** (часто путают):

| Характеристика | topologySpreadConstraints | podAntiAffinity |
|---|---|---|
| **Главная Цель** | РАВНОМЕРНО (балансировка по доменам) | НЕ ВМЕСТЕ (расталкивание конкретных подов) |
| **Гранулярность** | Количественная (подсчет подов в домене) | Бинарная (можно или нельзя на узел) |
| **Масштабируемость** | Дешево вычисляется даже на 1000+ узлах | Вычислительно дорого (O(N^2) при required) |
| **Применение** | Дефолтный выбор для отказоустойчивых веб-сервисов | Для критичных баз данных или кворумных систем |
| **Комбинирование** | Да, spread для глобального баланса, anti-affinity для точечной изоляции | |

### 1.1 Равномерное распределение по нодам

Применим манифест и понаблюдаем за поведением планировщика:

```bash
kubectl -n lab apply -f manifests/app.yaml
kubectl -n lab rollout status deploy/resilient-app --timeout=120s

# Посмотрим, как распределились поды. Мы ожидаем по одной реплике на каждую worker-ноду:
kubectl -n lab get pods -l app=resilient-app \
  -o custom-columns=POD:.metadata.name,NODE:.spec.nodeName,STATUS:.status.phase
```

Вывод должен показать равномерное распределение. На 2 worker-узлах при 3 репликах вы увидите:
```
POD                                NODE      STATUS
resilient-app-6d8b945c7b-7q2xl     node1     Running
resilient-app-6d8b945c7b-9x8tz     node2     Running
resilient-app-6d8b945c7b-m5kvp     node1     Running
```

### 1.2 Влияние maxSkew и whenUnsatisfiable

Давайте сымитируем ситуацию, когда `maxSkew` нарушается. Если мы запросим 5 реплик с `maxSkew: 1` и `whenUnsatisfiable: DoNotSchedule` на 2 узлах, распределение будет `3` и `2`, перекос `1` — все хорошо.
Но если бы мы заставили один узел принять 4 пода, а другой 1, перекос был бы 3, что заблокировало бы планирование новых подов. Вы можете увидеть это в событиях (`kubectl get events`), где планировщик пожалуется:
`0/3 nodes are available: 1 node(s) had untolerated taint, 2 node(s) didn't match pod topology spread constraints.`

### 1.3 nodeTaintsPolicy: Учет недоступных узлов

Посмотрим на конфигурацию в манифесте:

```yaml
topologySpreadConstraints:
- maxSkew: 1
  topologyKey: kubernetes.io/hostname
  whenUnsatisfiable: DoNotSchedule
  labelSelector:
    matchLabels:
      app: resilient-app
  nodeTaintsPolicy: Honor   # <--- Критичная настройка для нашего кластера
```

Без `Honor`, наш Control Plane узел (с taint) считался бы доменом. У него было бы 0 подов. Worker-узлы имели бы по 1-2 пода. Перекос превысил бы `maxSkew: 1`, и мы бы получили поды в `Pending`. В k8s версии 1.26+ это поведение управляется явно через этот флаг.

### 1.4 Взаимодействие spread с масштабированием

Важно понимать: балансировка `topologySpreadConstraints` происходит **только во время планирования пода**. Если вы меняете правила в запущенном Deployment, старые поды не "разбегутся" по узлам сами. Аналогично, при масштабировании вниз (scale down), контроллер Deployment может удалить поды так, что оставшиеся окажутся на одном узле, нарушая `maxSkew`.
Решение: использовать `Descheduler` в кластере, который автоматически выселяет поды для поддержания баланса, или перезапускать Deployment (`rollout restart`).

**Контрольные вопросы:**
1. Почему `nodeTaintsPolicy: Honor` решает проблему с Control Plane узлами?
2. Что произойдет, если изменить `maxSkew` на 2 в нашем текущем Deployment?
3. В чем фундаментальная разница между доменом-нодой (`kubernetes.io/hostname`) и доменом-зоной (`topology.kubernetes.io/zone`)?
4. Почему при `scale down` может нарушиться равномерность распределения?

## Часть 2: podAntiAffinity (Расталкивание подов)

### Теория для изучения перед частью 2

**PodAntiAffinity** — это механизм "отталкивания". В отличие от `topologySpread`, который работает с математическими перекосами по всему кластеру, anti-affinity работает бинарно: "можно ли разместить этот под на узле, где уже есть под с такими лейблами?".

Существует два типа правил:
- **`requiredDuringSchedulingIgnoredDuringExecution` (Жесткое)**: Под ни при каких условиях не будет запланирован на узел, нарушающий правило. Если свободных "чистых" узлов не осталось, под перейдет в `Pending`.
- **`preferredDuringSchedulingIgnoredDuringExecution` (Мягкое)**: Планировщик будет стараться разнести поды, но если это невозможно, он запустит под вместе с другим. Здесь используется система весов (weight от 1 до 100) для скоринга узлов.

#### Масштабируемость и производительность
Жесткий `podAntiAffinity` является вычислительно дорогой операцией для планировщика (сложность O(N^2) от количества подов). В больших кластерах (1000+ узлов) злоупотребление `required` anti-affinity может значительно замедлить scheduling. В таких случаях предпочтительнее использовать `topologySpreadConstraints`.

---

### 2.1 Жесткое анти-сродство (required)

Если бы мы использовали `required` для нашего приложения из 3 реплик на кластере с 2 worker-узлами:

```yaml
podAntiAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
  - labelSelector:
      matchLabels:
        app: resilient-app
    topologyKey: kubernetes.io/hostname
```

Первый под сел бы на `node1`, второй на `node2`. Третьему поду не осталось бы уникального `hostname` (так как на обоих доступных узлах уже есть поды с лейблом `app: resilient-app`), и он навсегда завис бы в `Pending`. Это частая причина деградации при масштабировании критичных сервисов.

### 2.2 Мягкое анти-сродство (preferred) и веса

В нашем манифесте мы используем `preferred`:

```bash
# Посмотрим на текущую конфигурацию:
kubectl -n lab get deploy resilient-app -o jsonpath='{.spec.template.spec.affinity.podAntiAffinity}' | jq
```

```json
{
  "preferredDuringSchedulingIgnoredDuringExecution": [
    {
      "podAffinityTerm": {
        "labelSelector": {
          "matchLabels": {
            "app": "resilient-app"
          }
        },
        "topologyKey": "kubernetes.io/hostname"
      },
      "weight": 100
    }
  ]
}
```

Именно поэтому 3 реплики успешно работают на 2 узлах. Третий под, несмотря на "предпочтение" быть одному (весом 100), был запланирован на узел с уже существующим подом, так как правило было мягким и альтернатив не было.

### 2.3 Влияние TopologyKey на расталкивание

Поле `topologyKey` определяет границу домена отталкивания. 
Если указать `topologyKey: topology.kubernetes.io/zone`, то `required` anti-affinity запретит размещать два пода в одной зоне доступности, даже если в этой зоне есть сотни свободных физических серверов. Это невероятно мощный инструмент для защиты от потери целого дата-центра, но использовать его нужно с крайней осторожностью.

**Контрольные вопросы:**
1. В каких сценариях `required` anti-affinity абсолютно необходим (например, распределенные базы данных вроде Cassandra или ZooKeeper)?
2. Как планировщик складывает веса (`weight`), если у пода несколько `preferred` правил?
3. Почему anti-affinity считается более ресурсоемким механизмом, чем topologySpread?

## Часть 3: PodDisruptionBudget (PDB) и Eviction API

### Теория для изучения перед частью 3

Даже если ваши поды идеально разнесены по разным узлам и зонам, кластеру необходимо регулярно проводить плановое обслуживание (обновление ОС, апгрейд версии Kubernetes, масштабирование узлов вниз, замена виртуальных машин). Во время таких процедур поды выселяются с узлов.

**PodDisruptionBudget (PDB)** — это ресурс, который ограничивает количество одновременно выселяемых подов приложения во время **добровольных** (voluntary) сбоев. Он является мостом между администратором кластера (который хочет перезагрузить ноду) и владельцем приложения (который хочет 100% uptime).

#### Voluntary vs Involuntary Disruptions
- **Voluntary (Добровольные)**: Инициированы пользователем или контроллерами (например, `kubectl drain`, обновление узлов в облаке, выселение из-за нехватки ресурсов `kubelet eviction`). PDB **ЗАЩИЩАЕТ** от них. Узлы будут ждать, пока приложение не позволит их выселить.
- **Involuntary (Недобровольные)**: Аппаратные сбои, kernel panic, потеря сети, OOMKilled. PDB **НЕ МОЖЕТ** защитить от них (здесь спасают только реплики и правильное распределение topologySpread/antiAffinity).

#### Расчет ALLOWED DISRUPTIONS
PDB оперирует параметрами `minAvailable` (минимум доступных) или `maxUnavailable` (максимум недоступных).

| Поле в PDB | Ожидаемый результат `ALLOWED DISRUPTIONS` (при 3 репликах) |
|------|------------------------------------|
| `minAvailable: 2` | `3 − 2 = 1` (можно безопасно выселить 1 под) |
| `minAvailable: 50%` | `3 − ceil(1.5)=2 → 1` |
| `maxUnavailable: 1` | `1` (напрямую, независимо от числа реплик) |
| `minAvailable: 3` | `0` — нода НЕДРЕНИРУЕМА (drain виснет наглухо) |

**Поток Eviction API (что делает `kubectl drain` под капотом):**

```
drain -> для каждого пода на узле: POST /api/v1/namespaces/lab/pods/pod-name/eviction
            apiserver проверяет PDB приложения:
              Текущие живые поды - 1 >= minAvailable ?
                да  -> 201 Created (под выселен, ALLOWED DISRUPTIONS уменьшается)
                нет -> 429 Too Many Requests -> drain ЖДЁТ, спит и повторяет запрос
            (контроллер Deployment видит удаленный под и пересоздаёт его на другой ноде -> ALLOWED восстанавливается)
```

**Цель:** ограничить одновременные выселения для гарантии доступности сервиса.

**Ресурс:** `manifests/pdb.yaml` (`minAvailable: 2`).

### 3.1 Создание PDB и настройка параметров

Применим манифест PDB для нашего приложения:

```bash
kubectl -n lab apply -f manifests/pdb.yaml

# Проверим статус PDB:
kubectl -n lab get pdb resilient-app-pdb
```

Вывод покажет текущее состояние:
```
NAME                MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
resilient-app-pdb   2               N/A               1                     45s
```
Мы видим, что из 3 реплик мы можем безопасно выселить 1, не нарушая бюджет (гарантируя минимум 2 живых пода для обработки трафика).

### 3.2 Механика drain и защита доступности

Давайте проверим, как PDB защищает приложение при попытке выселения. Мы используем `dry-run`, чтобы не сломать кластер прямо сейчас:

```bash
# Найдем узел, на котором запущен хотя бы один под нашего приложения
NODE=$(kubectl -n lab get pods -l app=resilient-app -o jsonpath='{.items[0].spec.nodeName}')

# Запустим dry-run drain
kubectl drain "$NODE" --ignore-daemonsets --delete-emptydir-data --dry-run=server 2>&1 | head -5
```

Вы увидите вывод, подтверждающий, что поды могут быть выселены. PDB позволяет это сделать, так как выселение одного пода не опустит количество доступных реплик ниже `minAvailable: 2`.

### 3.3 Краевые случаи HPA и unhealthyPodEvictionPolicy

**PDB × HPA (Horizontal Pod Autoscaler)**
PDB считает от ТЕКУЩЕГО числа здоровых подов, а не от `replicas` в Deployment. Если HPA уменьшает масштаб (scale down) во время работы администратора с кластером (`drain`), `ALLOWED DISRUPTIONS` может неожиданно стать 0. Это приведет к зависанию инфраструктурных пайплайнов. Выход — использовать проценты с осторожностью или устанавливать `maxUnavailable`.

**unhealthyPodEvictionPolicy (k8s 1.26+)**
По умолчанию (`IfHealthyBudget`), нездоровые поды (например, застрявшие в CrashLoopBackOff) защищены PDB так же, как и здоровые. Они могут заблокировать `drain`, хотя они уже не обслуживают трафик и являются "мертвым грузом".
Установка `AlwaysAllow` позволяет выселять нездоровые поды всегда (они не считаются в бюджете). Для современных кластеров это предпочтительная практика.

**Контрольные вопросы:**
1. Какова разница между `minAvailable: 50%` и `maxUnavailable: 50%` при нечетном количестве реплик?
2. Что произойдет, если HPA уменьшит количество реплик до 2 в момент, когда PDB требует `minAvailable: 2`, и администратор запустит `drain`?
3. Влияет ли PDB на обычное ручное удаление пода через `kubectl delete pod`?

## Часть 4: Обслуживание ноды (cordon, drain, uncordon)

### Теория для изучения перед частью 4

Процесс вывода узла из эксплуатации или его обслуживания состоит из трех строгих фаз. Это гарантирует отсутствие downtime для приложений, которые правильно настроили PDB и anti-affinity.

1. **Cordon (Маркировка как Unschedulable)**: На узел вешается `taint` `node.kubernetes.io/unschedulable:NoSchedule`. Текущие поды продолжают работать, но новые поды сюда больше не попадут.
2. **Drain (Выселение)**: Kubernetes через Eviction API отправляет сигналы на завершение (`SIGTERM`) всем подам на узле. Контроллеры видят, что поды удаляются, и создают новые реплики. Так как узел "закордонен", новые реплики планируются на другие узлы.
3. **Обслуживание**: Физическая перезагрузка, обновление пакетов, замена железа и т.д.
4. **Uncordon**: Снятие `taint`, узел снова готов принимать нагрузки.

---

### 4.1 Запрет планирования (cordon)

Сначала мы изолируем узел, запрещая планировщику размещать на нем новые нагрузки.

```bash
NODE=$(kubectl -n lab get pods -l app=resilient-app -o jsonpath='{.items[0].spec.nodeName}')

# Запрещаем планирование
kubectl cordon "$NODE"

# Проверяем статус узла
kubectl get node "$NODE"
```
```
NAME    STATUS                     ROLES    AGE    VERSION
node1   Ready,SchedulingDisabled   <none>   120d   v1.28.2
```

### 4.2 Выселение рабочих нагрузок (drain)

Выполняем фактическое выселение. Мы используем флаги:
- `--ignore-daemonsets` (DaemonSet управляются отдельно, игнорируют cordon и не выселяются обычным образом, так как они нужны для системных сервисов типа CNI/CSI/Loggers).
- `--delete-emptydir-data` (соглашаемся с потерей локальных эфемерных данных `emptyDir`, иначе drain откажется выселять такие поды).

```bash
# Выполняем drain
kubectl drain "$NODE" --ignore-daemonsets --delete-emptydir-data

# Проверяем, что подов нашего приложения на этом узле больше нет:
kubectl -n lab get pods -l app=resilient-app -o wide | grep "$NODE"
```
Ожидаемый результат — пустота (grep не найдет строк). Все поды переехали на соседние узлы. При этом PDB гарантировал, что сервис не прерывал работу, так как одновременно выселялся только 1 под, дожидаясь восстановления (Running & Ready) на другом узле.

### 4.3 Возврат ноды в строй (uncordon)

После условного обслуживания возвращаем ноду обратно:

```bash
kubectl uncordon "$NODE"
kubectl get node "$NODE"
# STATUS снова Ready (без SchedulingDisabled)
```

*Обратите внимание: после `uncordon` поды НЕ вернутся на этот узел автоматически. Балансировка происходит только в момент создания новых подов (например, при следующем деплое или масштабировании). Если дисбаланс критичен, необходимо использовать инструменты ребалансировки (Descheduler).*

## Часть 5: Taints и Tolerations в контексте отказоустойчивости

### Теория для изучения перед частью 5

**Taints** (пятна) и **Tolerations** (терпимости) — это еще один слой управления топологией и надежностью. В отличие от `NodeAffinity`, который притягивает поды к узлам, `Taints` отталкивают поды от узлов.

#### 5.1 Изоляция пулов узлов (Dedicated nodes)
Для повышения отказоустойчивости критичных компонентов (например, баз данных, Ingress-контроллеров или узлов мониторинга), их часто размещают на выделенных узлах. На эти узлы вешается `taint` (например, `dedicated=database:NoSchedule`). Только поды с соответствующим `toleration` смогут туда попасть. Это защищает критичные узлы от "шумных соседей" (Noisy Neighbors), которые могут забрать CPU/RAM или вызвать OOM-убийства на узле.

#### 5.2 TaintBasedEvictions (Автоматическое выселение)
Когда узел теряет связь с Control Plane (например, падает сеть коммутатора), контроллер узлов (Node Controller) через `node-monitor-grace-period` (по умолчанию 40 секунд) вешает на него taints:
- `node.kubernetes.io/unreachable:NoExecute`
- `node.kubernetes.io/not-ready:NoExecute`

Поды по умолчанию имеют `toleration` к этим taints с параметром `tolerationSeconds: 300` (5 минут). Это значит, что если узел не вернется в строй через 5 минут, поды будут выселены (Evicted) и пересозданы на здоровых узлах.

#### 5.3 Управление таймаутами при потере сети
Для критичных приложений, требующих быстрого восстановления (например, stateless API), таймаут в 5 минут — это слишком долго. Вы можете переопределить это поведение в манифесте пода:

```yaml
tolerations:
- key: "node.kubernetes.io/unreachable"
  operator: "Exists"
  effect: "NoExecute"
  tolerationSeconds: 30 # Пересоздать под через 30 секунд при потере ноды
```
Это позволяет приложению быстрее реагировать на инфраструктурные сбои (Involuntary disruptions).

## Часть 6: Troubleshooting — боевые инциденты

### Инцидент 1: Под висит в Pending из-за required anti-affinity

**Симптом**: Вы выкатили приложение, но часть реплик зависла в `Pending`.
**Воспроизведение**: Разобран в каталоге `broken/scenario-01/`. Развернуто 3 реплики с `required` anti-affinity по `kubernetes.io/hostname` на кластере с 2 доступными worker-узлами.
**Диагностика**:
```bash
kubectl -n lab describe pod <имя-пода-в-pending>
```
В секции `Events` вы увидите сообщение: `FailedScheduling ... didn't match pod anti-affinity rules`. Это означает, что планировщик не нашел узла, удовлетворяющего жесткому правилу "один под на узел".
**Решение**:
1. Использовать `preferred` вместо `required` (если допустимо снижение изоляции).
2. Увеличить размер кластера (добавить физические узлы).
3. Перейти на `topologySpreadConstraints` с мягким `maxSkew` и `ScheduleAnyway`.

### Инцидент 2: Drain завис из-за нарушения PDB

**Симптом**: Команда `kubectl drain` висит бесконечно (или спамит ошибками) с сообщением "Cannot evict pod as it would violate PDB".
**Причина**: `ALLOWED DISRUPTIONS` равен 0. Это происходит, когда:
- `minAvailable` строго равно количеству запущенных реплик (например, replicas=2, minAvailable=2). Выселение одного пода опустит счетчик до 1, что нарушит правило.
- Приложение не может запуститься на других узлах из-за нехватки ресурсов (CPU/RAM) или жестких правил affinity. Выселенный ранее под зависает в `Pending`, поэтому счетчик `ALLOWED DISRUPTIONS` не восстанавливается.
**Решение**:
- Временно изменить PDB: `kubectl -n lab edit pdb <имя>`.
- Настроить `minAvailable` в процентах (например, 50% или 80%).
- Проверить наличие свободного места и ресурсов на остальных узлах кластера.

### Инцидент 3: Неравномерное распределение из-за старых подов

**Симптом**: Вы добавили `topologySpreadConstraints` в существующий Deployment через `kubectl apply`, но поды все еще сгруппированы на одном узле.
**Причина**: Планировщик (kube-scheduler) принимает решения только в момент создания пода. Изменение параметров Deployment не заставляет Kubernetes перепланировать УЖЕ запущенные поды, если их контейнеры/образы не изменились.
**Решение**:
Инициировать перезапуск подов вручную:
```bash
kubectl -n lab rollout restart deploy <имя>
```

### Инцидент 4: PDB блокирует удаление нездоровых подов

**Симптом**: Под приложения вошел в состояние `CrashLoopBackOff`, но при попытке `drain` узла процесс блокируется PDB.
**Причина**: До версии 1.26 нездоровые поды учитывались в бюджете так же, как здоровые.
**Решение**: Использовать `unhealthyPodEvictionPolicy: AlwaysAllow` в манифесте PDB, чтобы разрешить безусловное выселение "мертвого груза".

### Бонус: Диагностика scheduler и events
Планировщик всегда оставляет следы своих решений в событиях пода (`Events`). Если вы не понимаете, почему под сел именно на этот узел (или почему не сел вообще), `kubectl describe pod` — ваш главный инструмент. Обращайте внимание на конфликты `nodeSelector`, нестыковки `taints/tolerations` и доступные ресурсы (`cpu/memory`), которые могут мешать правильному распределению.

```bash
# Поиск причин зависания в планировщике
kubectl -n lab describe pod -l app=resilient-app | grep -A5 "Topology Spread\|FailedScheduling\|Events"

# === Работа с PDB и Eviction API ===
# Статус всех PDB в namespace
kubectl -n lab get pdb
# Детальная информация и текущий ALLOWED DISRUPTIONS
kubectl -n lab describe pdb resilient-app-pdb
# Безопасная симуляция выселения узла (Dry Run)
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data --dry-run=server

# === Управление узлами (Taints, Cordon, Drain) ===
# Проверка Taints на всех узлах (важно для nodeTaintsPolicy)
kubectl get nodes -o custom-columns=NAME:.metadata.name,TAINTS:.spec.taints
# Запрет планирования новых нагрузок (Cordon)
kubectl cordon <node-name>
# Выселение существующих нагрузок (Drain) с игнорированием DaemonSets
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
# Возврат узла в работу (Uncordon)
kubectl uncordon <node-name>

# === Уборка лаборатории ===
kubectl -n lab delete -k manifests/
```
