## Часть 1: Установка и архитектура Argo Rollouts

### Теория для изучения перед частью

Стандартный Kubernetes `Deployment` умеет только одну стратегию плавного обновления — `RollingUpdate`. Это означает, что новые поды постепенно заменяют старые с фиксированным темпом (регулируется параметрами `maxSurge` и `maxUnavailable`), без пауз, без проверок качества или бизнес-метрик. «Откат» в классическом варианте — это реактивное ручное действие (`kubectl rollout undo`) **после того**, как пользователи уже увидели проблему и начали жаловаться.

**Argo Rollouts** — это специализированный контроллер прогрессивной доставки и набор CRD (Custom Resource Definitions), главным из которых является `Rollout`. `Rollout` — это практически полная (drop-in) замена `Deployment`. У него абсолютно такой же `podTemplate`, но поле `strategy` значительно богаче и мощнее.

Основные стратегии, поддерживаемые Argo Rollouts:

1. **Canary (Канареечный релиз)**: новая версия выкатывается постепенно и получает лишь долю трафика. Шаги задаются явно (например, `setWeight: 25`, затем `pause`, затем `setWeight: 50` и т.д.). На каждом шаге можно остановиться, проверить работоспособность, оценить логи и принять взвешенное решение: откатить (`abort`) или продолжить (`promote`).
2. **Blue/Green**: новая версия приложения (Green) поднимается **полностью** параллельно старой (Blue). Для новой версии создается отдельный `preview-Service`, через который можно гонять автотесты, не затрагивая реальных пользователей. Переключение основного трафика (через `active-Service`) происходит атомарно.
3. **AnalysisTemplate / AnalysisRun**: объекты для автоматической оценки качества релиза. Контроллер ходит во внешние системы (Prometheus, Datadog, New Relic) или запускает Job, измеряет бизнес-метрики (например, долю 500-х ошибок или задержку). Если условия не выполняются — срабатывает **автоматический abort** (откат).

> 💡 **Архитектурная деталь: как делятся проценты трафика без Service Mesh?**
> Если у вас нет Ingress Nginx, Istio или Gateway API с тонкой маршрутизацией, Argo Rollouts делит трафик **масштабированием ReplicaSet'ов**.
> Например, если у вас всего 4 реплики и вы ставите вес канарейки 25%, Rollouts запустит 1 новый под и оставит 3 старых.
> Точность такого подхода жестко ограничена общим числом реплик. Если вам нужно направить ровно 1% трафика или использовать маршрутизацию по HTTP-заголовкам (header-based routing), требуются расширенные интеграции с Service Mesh (так называемый trafficRouting). Об этом подробнее в Части 5.

### 1.2 Архитектурные особенности контроллера

Контроллер Argo Rollouts работает по принципу классического оператора Kubernetes. Он слушает события изменений ресурсов `Rollout` и управляет нижележащими `ReplicaSet`.
Важно понимать:
- `Rollout` **не** управляет `Deployment`. Он полностью его заменяет и напрямую работает с `ReplicaSet`.
- При миграции существующего проекта вам придется изменить `kind: Deployment` на `kind: Rollout` (с указанием правильного `apiVersion: argoproj.io/v1alpha1`).
- Контроллер ведет свою историю ревизий (revisions), аналогично Deployment, что позволяет быстро откатываться.

## Часть 2: Canary-релиз с ручным подтверждением

### Теория для изучения перед частью

В этой части мы рассмотрим базовый Canary-релиз без автоматического метрического анализа.
Весь процесс управляется блоком `steps` в секции `strategy.canary`. Шаги выполняются строго последовательно:
- `setWeight: X` — направляет X% трафика на новую версию (через количество запущенных подов).
- `pause: {duration: 5s}` — ждет указанное время (5 секунд) перед переходом к следующему шагу.
- `pause: {}` — **бесконечная ручная пауза**. Релиз останавливается навсегда и ждет внешнюю команду `promote` от человека (или из CI/CD пайплайна).

Такая гибкость позволяет выкатить 10% трафика, подождать 1 час, затем выкатить 50%, оставить на сутки (ручная пауза), и затем докатить до 100%.

### 2.1 Создание Rollout и первый деплой

Применим базовые манифесты: сам объект Rollout, связанные Services и AnalysisTemplate (он пригодится нам в Части 3, но создадим его сейчас).

```bash
# Применяем все манифесты из директории
kubectl apply -f manifests/

# Проверим, что Rollout создался и успешно развернулся
kubectl argo rollouts get rollout demo-rollout -n lab
```

> **Примечание:** Первоначальный деплой Rollout (когда у него еще нет предыдущей версии) всегда выполняется мгновенно на 100%, игнорируя шаги Canary. Стратегия Progressive Delivery применяется только при **обновлении** существующего Rollout.

Давайте детально посмотрим на стратегию, описанную в файле `manifests/rollout.yaml`:

```yaml
  strategy:
    canary:
      steps:
      - setWeight: 25           # Шаг 1: 25% трафика на новую версию
      - pause: {duration: 5s}   # Шаг 2: пауза 5 секунд
      - setWeight: 50           # Шаг 3: 50% трафика на новую версию
      - pause: {}               # Шаг 4: бесконечная ручная пауза до вмешательства человека!
      - setWeight: 100          # Шаг 5: 100% трафика (полный релиз)
```

### 2.2 Запуск Canary и остановка на ручной паузе

Сымитируем выход новой версии приложения. Мы меняем образ контейнера, чтобы запустить релиз. По умолчанию приложение отдает "цвет", сейчас это `blue` (стабильная версия), мы поменяем на `yellow` (канарейка).

```bash
# Запускаем релиз, императивно меняя образ (в реальности это делает CI/CD через изменение манифеста)
kubectl argo rollouts set image demo-rollout app=argoproj/rollouts-demo:yellow -n lab

# Наблюдаем за процессом в реальном времени.
# Команда покажет живое обновляющееся дерево. Нажмите Ctrl+C, когда статус станет Paused.
kubectl argo rollouts get rollout demo-rollout -n lab --watch
```

Реальное состояние Rollout на ручной паузе (так оно выглядит в консоли):

**Анализируем вывод:**
- Rollout имеет 4 желаемых реплики (`Desired: 4`).
- Мы находимся на шаге 3 из 5 (`Step: 3/5`), где вес равен 50% (`SetWeight: 50`).

- Релиз остановился и статус отображает `॥ Paused`. Контроллер **бесконечно ждет** явной команды от пользователя.

### 2.3 Продолжение (promote) или отмена (abort) релиза

Находясь на паузе, QA-инженеры или дежурные SRE могут проверить логи, метрики, графики, или вручную "пощелкать" приложение через UI.
Если всё работает штатно — мы продолжаем релиз (Promote).
Если выявлены дефекты — отменяем (Abort).

Выполним Promote, чтобы завершить начатое:

```bash
# Продолжить релиз (переход к следующему шагу - setWeight 100)
kubectl argo rollouts promote demo-rollout -n lab

# Посмотрим статус после promote
kubectl argo rollouts get rollout demo-rollout -n lab
```

После выполнения `promote`, Rollout переходит к последнему шагу: масштабирует новый `ReplicaSet` до 4 подов, а старый постепенно уменьшает до 0. Статус Rollout меняется на `✔ Healthy`.

### 2.4 Откат на предыдущую версию (undo)

Если вы поняли, что релиз `yellow` все-таки был неудачным уже после его завершения, вы можете сделать полный откат, как в обычном Deployment:

```bash
# Быстрый откат на стабильную версию
kubectl argo rollouts undo demo-rollout -n lab

# Убедитесь, что образ снова вернулся к blue
kubectl argo rollouts get rollout demo-rollout -n lab
```

## Часть 3: Автоанализ по метрикам и АВТО-откат

### Теория для изучения перед частью

Ручные паузы — это прекрасно для контроля, но в современном CI/CD автоматика ценится гораздо выше. Объект `AnalysisTemplate` позволяет описать, **что именно** нужно измерять в системе мониторинга и **какие значения считаются приемлемыми**.
При старте очередного релиза контроллер Rollout автоматически создает объект `AnalysisRun` на основе этого шаблона.

Ключевые поля объекта `AnalysisTemplate`:
- **interval / count** — как часто и сколько раз выполнять измерение (например, `interval: 20s`, `count: 5` означает 5 проверок каждые 20 секунд).
- **successCondition** — математическое выражение над полученным результатом (например, `result[0] >= 0.9` означает, что метрика должна быть не менее 90%).
- **failureLimit** — сколько **провальных** измерений мы готовы простить (значение из мониторинга получено, но оно меньше требуемого).
- **consecutiveErrorLimit** — сколько **инфраструктурных ошибок** мы терпим (например, Prometheus недоступен, тайм-аут запроса, неверный PromQL).

Если любой из установленных лимитов превышен ⇒ `AnalysisRun` немедленно переходит в статус `Failed` (если метрика плохая) или `Error` (если инфраструктура сломалась) ⇒ срабатывает **автоматический abort** (откат).
Трафик мгновенно возвращается на 100% к stable-версии, а канарейка уничтожается.

Наш анализ в этой лабе (описан в `manifests/analysis.yaml`) — полностью настоящий. Rollout делает запросы к Prometheus кластера и вычисляет долю Ready-подов нашего приложения:

```promql
sum(kube_pod_status_ready{namespace="lab", condition="true", pod=~"demo-rollout-.*"})
/
count(kube_pod_status_ready{namespace="lab", condition="true", pod=~"demo-rollout-.*"})
```
`successCondition: result[0] >= 0.9`

В реальном production вместо (или вместе с) доли `Ready` подов обычно используют метрики HTTP-ошибок (например, 5xx Error Rate < 1%) или времени ответа (99-й перцентиль Latency < 200ms).

### 3.1 Успешный релиз под надзором фонового анализа

В нашем `rollout.yaml` анализ запущен как **фоновый** (указан `startingStep: 0`). Это означает, что он начинается немедленно при старте релиза и "бежит" параллельно с выполнением шагов веса. В Части 2, когда мы делали первый релиз, этот анализ уже отработал в фоне успешно.

Убедимся в этом, посмотрев историю AnalysisRun:

```bash
kubectl -n lab get analysisrun
```
Вывод покажет успешный запуск:
```
NAME                        STATUS       AGE
demo-rollout-676df89f6b-2   Successful   10m
```

### 3.2 Сломанный релиз: автоматический откат (abort)

Давайте сымитируем выкат заведомо нерабочей версии (например, разработчик ошибся в конфигурации или сборке). Мы укажем тег `does-not-exist`, из-за чего поды упадут с ошибкой `ImagePullBackOff`.
Самое главное — мы **НИЧЕГО** не будем делать руками. Посмотрим, как система защитит пользователей сама.

```bash
# Выкатываем битый образ (симуляция критического бага)
kubectl argo rollouts set image demo-rollout app=argoproj/rollouts-demo:does-not-exist -n lab

# Наблюдаем за процессом обновления. Обратите внимание на статус AnalysisRun внизу вывода.
kubectl argo rollouts get rollout demo-rollout -n lab --watch
```

**Хронология событий (что происходит под капотом):**
1. Создается новый ReplicaSet для канарейки, планируется запуск 1-й реплики (25% трафика).
2. Под зависает в `ImagePullBackOff` (или `CrashLoopBackOff`) и никогда не переходит в состояние `Ready`.
3. Запускается `AnalysisRun`. Он делает запрос в Prometheus и видит, что 1 под из 4 не готов. Доля Ready = 0.75 (или 75%).
4. Условие `result[0] >= 0.9` возвращает `false`. Засчитывается Failed-измерение.
5. У нас в шаблоне указано `failureLimit: 1` (мы прощаем максимум 1 провал). На втором провальном замере `AnalysisRun` окончательно переходит в статус Failed.
6. Контроллер Rollouts получает сигнал Failed от анализа и мгновенно инициирует процедуру `Abort`.

Примерно через 1-1.5 минуты вы увидите в консоли следующий вывод:

```
Status:          ✖ Degraded
Message:         RolloutAborted: Rollout aborted update to revision 6:
  Background analysis phase error/failed: Metric "pods-ready-fraction"
  assessed Failed due to failed (2) > failureLimit (1)
Images:          argoproj/rollouts-demo:yellow (stable)
  SetWeight:     0
Replicas:        Desired: 4  Ready: 4

NAME                                       KIND        STATUS        AGE  INFO
⧉ demo-rollout                             Rollout     ✖ Degraded    15m  
├──# revision:6                                                           
│  ├──⧉ demo-rollout-5b8d9c5b6b            ReplicaSet  • ScaledDown  2m   canary,aborted
│  │  └──⚠ demo-rollout-5b8d9c5b6b-xxxxx   Pod         ✖ Failed      2m   ImagePullBackOff
│  └──α demo-rollout-676df89f6b-2          AnalysisRun ✖ Failed      2m   ✔ 0, ✖ 2, ⚠ 0
└──# revision:5                                                           
   └──⧉ demo-rollout-676df89f6b            ReplicaSet  ✔ Healthy     15m  stable
      ├──□ demo-rollout-676df89f6b-aaaaa   Pod         ✔ Running     15m  ready:1/1
      ├──□ demo-rollout-676df89f6b-bbbbb   Pod         ✔ Running     15m  ready:1/1
      ├──□ demo-rollout-676df89f6b-ccccc   Pod         ✔ Running     15m  ready:1/1
      └──□ demo-rollout-676df89f6b-ddddd   Pod         ✔ Running     15m  ready:1/1
```

**Итог:**
Канарейка не смогла стать Ready → анализ по метрикам провалился → Rollout автоматически отменил деплой (`SetWeight: 0`, новый RS ScaledDown).
**Stable-версия (наши 4 пода yellow) не пострадала ни на секунду, они продолжали бесперебойно обслуживать клиентский трафик.** Инженер или дежурный администратор не участвовал в процессе спасения системы.

### 3.3 Разбор метрик Prometheus и работы AnalysisRun

Вы можете детально изучить, почему именно провалился анализ, посмотрев события `AnalysisRun`:

```bash
# Получить имя последнего AnalysisRun
RUN_NAME=$(kubectl -n lab get analysisrun -o jsonpath='{.items[?(@.status.phase=="Failed")].metadata.name}' | awk '{print $1}')

# Посмотреть детали и события
kubectl -n lab describe analysisrun $RUN_NAME
```

В событиях (Events) вы увидите сообщения о том, что метрика вернула значение `0.75`, что меньше требуемого `0.9`, и поэтому измерение было помечено как неудачное.

### 3.4 Возврат в нормальное состояние после Degraded

После автоматического Abort Rollout находится в состоянии `Degraded`. Это сделано специально, чтобы контроллер не пытался бесконечно выкатывать сломанную версию и чтобы привлечь внимание команды.
Чтобы починить Rollout, нужно предоставить ему валидный образ и сбросить состояние ошибки командой `retry`.

```bash
# Возвращаем рабочий образ (мы снова указываем yellow или любой другой рабочий тег)
kubectl argo rollouts set image demo-rollout app=argoproj/rollouts-demo:yellow -n lab

# Говорим контроллеру сбросить состояние ошибки (Degraded) и повторить попытку выката
kubectl argo rollouts retry rollout demo-rollout -n lab

# Убедимся, что статус вернулся в Healthy
kubectl argo rollouts get rollout demo-rollout -n lab
```

> ⚠️ **Важный архитектурный нюанс:** Настройка `startingStep: 0` для фонового анализа критически важна! Если бы анализ начинался только на шаге `setWeight: 25`, то с битым образом (который даже не может запуститься и дойти до состояния Ready) мы бы **никогда не достигли** этого шага! Релиз бы завис в ожидании подов до наступления таймаута `progressDeadlineSeconds`. Фоновый анализ (`startingStep: 0`) позволяет начать оценку системы немедленно и поймать ошибку старта (CrashLoop) почти мгновенно.

1. Чем исход `Failed` концептуально отличается от `Error` в `AnalysisRun` и почему для них задают разные лимиты терпения?
2. Почему ошибка сбора метрик (например, сервер Prometheus упал или сеть недоступна) по умолчанию означает откат релиза, а не его продолжение?
3. Какое выражение PromQL вы бы положили в `successCondition` для мониторинга успешности ответов через Nginx Ingress?

## Часть 4: Blue/Green Deployment

### Теория для изучения перед частью

Стратегия **Blue/Green** кардинально отличается от Canary-подхода:
- Она **не делит** трафик на проценты (нет никаких 10% или 50%).
- Новая версия приложения (Green) масштабируется **сразу до 100%** требуемой емкости (создаются абсолютно все нужные поды) параллельно со старой версией (Blue).
- Для новой версии создается отдельный `preview-Service`. По этому адресу могут ходить автоматические E2E-тесты, скрипты нагрузочного тестирования или QA-инженеры для финальной проверки на "боевой" инфраструктуре.
- Основные пользователи продолжают сидеть исключительно на старой версии через `active-Service`. Они ничего не подозревают о развертывании.
- Команда `promote` **атомарно** переключает селекторы в `active-Service`. Трафик мгновенно и полностью идет на новую версию.
- Расплата за такую максимальную безопасность и изолированность тестирования — **двойная потребность в инфраструктуре** на время релиза (нужно место под x2 подов, памяти и CPU).

### 4.1 Запуск Blue/Green релиза

В директории `manifests/` мы уже применили ресурс `demo-rollout-bg`, который настроен на стратегию Blue/Green.

Посмотрим на его конфигурацию в манифесте:
```yaml
  strategy:
    blueGreen:
      activeService: rollout-active
      previewService: rollout-preview
      autoPromotionEnabled: false      # Требуется ручное подтверждение переключения
      scaleDownDelaySeconds: 30        # Задержка перед удалением старых подов
```

Давайте запустим обновление:

```bash
# Запустим обновление цвета на orange
kubectl argo rollouts set image demo-rollout-bg app=argoproj/rollouts-demo:orange -n lab

# Подождем около 40 секунд, пока новые поды (Green) полностью поднимутся и пройдут readiness probes
sleep 40

# Посмотрим статус Blue/Green Rollout
kubectl argo rollouts get rollout demo-rollout-bg -n lab
```

Реальное состояние паузы (так как `autoPromotionEnabled: false`, автоматическое переключение отключено):

```
Name:            demo-rollout-bg
Namespace:       lab
Status:          ॥ Paused
Message:         BlueGreenPause
Images:          argoproj/rollouts-demo:orange (preview)
                 argoproj/rollouts-demo:purple (stable, active)
Replicas:        
  Desired: 4
  Current: 8   <-- Внимание! Двойная ёмкость инфраструктуры!
  Updated: 4
  Ready:   8
  Available: 8
```
Как мы видим, сейчас запущено 8 подов вместо 4.

### 4.2 Тестирование Preview-окружения

Докажем существование «двух параллельных миров» одновременно. Сделаем прямые HTTP-запросы из кластера.
Сервисы `rollout-active` и `rollout-preview` слушают стандартный порт 80.

```bash
# Запустим временный под с утилитой curl
kubectl -n lab run curl-bg --image=curlimages/curl:8.11.1 --restart=Never --rm -i \
  --overrides='{"spec":{"containers":[{"name":"curl-bg","image":"curlimages/curl:8.11.1","command":["sh","-c","echo -n \"active:  \"; curl -s http://rollout-active/color; echo; echo -n \"preview: \"; curl -s http://rollout-preview/color"],"resources":{"requests":{"cpu":"10m","memory":"16Mi"},"limits":{"cpu":"50m","memory":"32Mi"}}}]}}'
```

Ожидаемый вывод:
```
active:  "purple"
preview: "orange"
```

Обычные пользователи (обращающиеся к `active`) видят стабильный `purple`, а тестировщики на превью-адресе (обращающиеся к `preview`) уже могут взаимодействовать с новым `orange`.

> 💡 **Заметка об инцидентах:** В поде с curl мы явно прописали `resources`. Это не случайность. Namespace `lab` может иметь жесткие `ResourceQuota`, а стратегия Blue/Green в момент перехода уже удвоила потребление ресурсов. Если запустить вспомогательный pod без указания лимитов, он может легко упереться в квоту и не запуститься с ошибкой `Forbidden`. Обязательно учитывайте это при планировании capacity кластера!

### 4.3 Атомарное переключение (Promote)

Если тесты на превью-окружении прошли успешно, мы завершаем релиз, переключая основной трафик:

```bash
# Выполняем Promote
kubectl argo rollouts promote demo-rollout-bg -n lab

# Проверим активный сервис после выполнения команды promote
kubectl -n lab run curl-bg-post --image=curlimages/curl:8.11.1 --restart=Never --rm -i \
  --overrides='{"spec":{"containers":[{"name":"curl-bg-post","image":"curlimages/curl:8.11.1","command":["sh","-c","echo -n \"active after promote:  \"; curl -s http://rollout-active/color; echo;"],"resources":{"requests":{"cpu":"10m","memory":"16Mi"}}}]}}'
```

Ожидаемый вывод:
```
active after promote:  "orange"
```

### 4.4 Завершение работы старой версии

Переключение состоялось мгновенно (на уровне `iptables` / `IPVS` через kube-proxy). Старые поды (purple) не удаляются сиюсекундно. Они будут масштабированы в 0 с небольшой задержкой (которую мы настроили как `scaleDownDelaySeconds: 30`), чтобы старые In-Flight TCP-соединения успели закрыться корректно (Graceful Shutdown). Через 30 секунд количество подов снова вернется к исходным 4.

1. Что именно (какое конкретно поле) контроллер Rollouts меняет в объекте `Service` при выполнении команды promote в стратегии Blue/Green?
2. В каких корпоративных сценариях Blue/Green деплой предпочтительнее Canary, несмотря на его очевидную высокую цену (требование двойной инфраструктуры)?

## Часть 5: Argo Rollouts vs Flux Flagger

На современном рынке Kubernetes есть два безусловных лидера в сфере прогрессивной доставки: **Argo Rollouts** и **Flagger** (проект от создателей Flux). Оба проекта являются частью CNCF, решают одну и ту же задачу, но обладают принципиально разной архитектурой.

Детальное сравнение:

| Характеристика | **Argo Rollouts** | **Flagger (Flux)** |
|---|---|---|
| **Модель ресурса** | Использует собственный CRD `Rollout`, который физически **заменяет** стандартный `Deployment`. | Работает как невидимая надстройка **НАД** стандартным `Deployment`. Flagger сам копирует его и создает `-primary` версию. |
| **Сложность миграции** | Высокая. Требуется переписать все существующие манифесты `Deployment` в `Rollout`. | Низкая. Достаточно повесить дополнительный CRD `Canary` рядом с существующим Deployment без его изменения. |
| **Трафик без Mesh** | Отлично умеет делить трафик "из коробки" обычным пропорциональным масштабированием ReplicaSet. | **НЕ умеет.** Работает исключительно при наличии Service Mesh (Istio/Linkerd) или продвинутого Ingress-контроллера. |
| **Маршрутизация (Mesh)** | Поддерживает интеграции (Istio, Nginx, ALB, Gateway API) для точных весов и заголовков. | Это его родная стихия. Широчайшая и самая глубокая поддержка всех известных Mesh-решений. |
| **Аналитика качества** | Мощные `AnalysisTemplate` с периодическими запросами в метрики и сложным фоновым выполнением. | Базовые встроенные вебхуки (чаще всего используются для запуска load-тестов перед каждым шагом). |
| **Подход к управлению** | Есть мощный CLI-плагин `kubectl argo rollouts`, возможны ручные паузы, есть отличный UI-дашборд. | Полностью строгий декларативный подход (pure GitOps). Ручного promote нет (остановка только если упали gate-вебхуки). |
| **Экосистема** | Отлично интегрируется с Argo CD (визуализация релизов доступна прямо в основном UI Argo CD). | Является нативной частью стека GitOps Toolkit (семейство Flux). |

**Практическое правило архитектора для выбора инструмента:**
- Если вы уже используете в компании экосистему Argo (Argo CD, Argo Workflows), хотите удобный специализированный UI и вам критически важны **ручные паузы** для визуального подтверждения релиза — выбирайте **Argo Rollouts**.
- Если у вас внедрен Flux, есть развитый Service Mesh (например, Istio), и вы хотите построить **строгую 100% автоматизацию** без ручных кнопок — ваш выбор **Flagger**.

Базовые концепции (canary шаги, метрики, авто-откаты) у них полностью идентичны. Выучив работу одного из инструментов, вы без проблем разберетесь с конфигурацией другого.

## Часть 7: Интеграция с Ingress и Service Mesh (Traffic Routing)

### Теория для изучения перед частью

Как мы обсуждали в первой части, базовый Canary (Basic Canary) использует масштабирование `ReplicaSet` для управления весом трафика. Если у нас 4 реплики, то минимальный шаг для разделения трафика — 25%. Что делать, если мы хотим запустить канарейку всего на 1% или 5% пользователей? А если мы хотим пускать на превью-версию только тестировщиков, у которых в браузере установлена специальная cookie или передан заголовок `X-Tester: true`?

В таких случаях Argo Rollouts предоставляет функционал **Traffic Routing** (маршрутизация трафика). Он интегрируется с различными ingress-контроллерами и service mesh, среди которых:
- **NGINX Ingress Controller**
- **ALB (AWS Load Balancer Controller)**
- **Istio**
- **Linkerd**
- **Gateway API (современный стандарт Kubernetes)**

### 7.1 Интеграция с NGINX Ingress

Для настройки интеграции с NGINX, вам необходимо создать Ingress-ресурс и сослаться на него в конфигурации Rollout.

**Пример архитектуры маршрутизации с NGINX:**
1. Вы создаете Ingress для вашего стабильного (stable) сервиса.
2. Argo Rollouts автоматически создает копию этого Ingress, добавляя к нему аннотацию `nginx.ingress.kubernetes.io/canary: "true"` и задавая нужный вес (`nginx.ingress.kubernetes.io/canary-weight`).
3. При каждом шаге (`setWeight`) контроллер обновляет значение веса в аннотации. NGINX Ingress Controller видит изменение аннотаций и динамически перестраивает конфигурацию (reload).

**Пример фрагмента манифеста Rollout для NGINX:**
```yaml
  strategy:
    canary:
      canaryService: rollout-canary
      stableService: rollout-stable
      trafficRouting:
        nginx:
          stableIngress: rollout-ingress
          additionalIngressAnnotations:   # Опционально: кастомные аннотации
            canary-by-header: X-Canary
      steps:
      - setWeight: 5
      - pause: {duration: 10m}
      - setWeight: 20
```

> **Важно:** При таком подходе количество реплик канарейки (`ReplicaSet`) больше не привязано к проценту трафика. Rollout может запустить всего 1 новый под, и направить на него ровно 5% трафика через Ingress, не запуская при этом 19 старых подов. Это существенно экономит ресурсы.

### 7.2 Интеграция с Istio Service Mesh

Если вы используете Service Mesh (например, Istio), Argo Rollouts управляет объектами `VirtualService` и `DestinationRule`.

**Пример архитектуры с Istio:**
1. Вы создаете `VirtualService` с маршрутизацией по умолчанию на `stable` подмножество (subset).
2. Вы указываете этот `VirtualService` в манифесте Rollout.
3. При выполнении шагов `setWeight`, Rollouts динамически обновляет веса в `VirtualService` (секция `route[].weight`). Истинная балансировка происходит на уровне Envoy-прокси (sidecar) в подах.

**Пример фрагмента Rollout для Istio:**
```yaml
  strategy:
    canary:
      canaryService: rollout-canary
      stableService: rollout-stable
      trafficRouting:
        istio:
          virtualService:
            name: rollout-vsvc
            routes:
            - primary
      steps:
      - setWeight: 1
      - pause: {duration: 5m}
      - setWeight: 10
```

### 7.3 Header-based Routing (Маршрутизация по заголовкам)

Самая мощная функция Traffic Routing — это возможность полностью исключить новую версию из публичного доступа, но при этом дать к ней доступ тестировщикам.

Вместо `setWeight`, вы можете использовать `setHeaderRoute`:
```yaml
      steps:
      - setCanaryScale:
          weight: 25  # Поднимаем 25% подов, но не даем им вес трафика
      - setHeaderRoute:
          match:
            - headerName: X-Tester
              headerValue:
                exact: "true"
      - pause: {}     # QA тестируют, передавая заголовок
      - setWeight: 50 # Если QA довольны, пускаем реальных юзеров
```
При такой настройке обычные пользователи получат только старую версию, а любой HTTP запрос с заголовком `X-Tester: true` будет гарантированно направлен на канарейку.

### 7.4 Gateway API

Современным стандартом вместо Ingress становится Kubernetes Gateway API (`HTTPRoute`, `Gateway` и др.). Argo Rollouts поддерживает его нативно. Вместо Ingress вы указываете `HTTPRoute`, и Rollouts меняет веса непосредственно в правилах `backendRefs`.

## Часть 6: Troubleshooting — боевые инциденты

Как и любой сложный контроллер, работающий поверх других абстракций, Rollouts может "застревать" или вести себя непредсказуемо при некорректных настройках инфраструктуры. Вся диагностика сводится к пониманию, на каком этапе находится конечный автомат объекта Rollout.

### Теория для изучения перед частью

- Жизненный цикл статусов Rollout: `Healthy` → `Progressing` → `Paused` → `Healthy` (или `Degraded`).
- Архитектурная связь между объектами `AnalysisRun` и `Rollout`.
- Жесткое влияние неймспейсных квот (`ResourceQuota`) на возможность масштабирования (особенно актуально для двойной емкости в Blue/Green).

### Таблица типовых инцидентов и решений

| Симптом (что мы видим) | Возможная Причина (почему так) | Диагностика / Решение (что делать) |
|---------|---------|--------------------|
| Любой релиз постоянно абортится (Degraded), хотя приложение 100% здорово и обрабатывает запросы | Анализ раз за разом возвращает статус `Error` (он физически не может **ИЗМЕРИТЬ** метрику), а не `Failed`. Исправьте PromQL запрос или доступы к метрикам. |

Почините образ в манифесте, затем выполните `abort` Rollout'а и начните заново. |
| Поды вообще не создаются (в логах контроллера статус ReplicaFailure) | Сработала жесткая `ResourceQuota` в неймспейсе (например, кластерный лимит на CPU). | Выполните `kubectl -n lab describe quota`. Укажите явные, небольшие `resources` для B/G деплоя в манифесте, чтобы удвоенная емкость гарантированно влезла в квоту. |

## Шпаргалка

```bash
# Живой статус Rollout (аналог watch, но с красивым графическим деревом)
kubectl argo rollouts get rollout NAME -n lab --watch

# Запустить новый релиз (императивно поменять образ)
kubectl argo rollouts set image NAME app=IMG -n lab

# Пройти ручную паузу / атомарно переключить Blue/Green
kubectl argo rollouts promote NAME -n lab

# Откатить зависший или неудачный релиз на предыдущую стабильную версию
kubectl argo rollouts abort NAME -n lab

# Повторить запуск после применения Abort или перехода в статус Degraded
kubectl argo rollouts retry rollout NAME -n lab

# Полностью откатиться на прошлую зафиксированную ревизию
kubectl argo rollouts undo NAME -n lab

# Посмотреть историю запусков метрических анализов
kubectl -n lab get analysisrun

# Получить детали конкретного запуска анализа (полезно при дебаге)
kubectl -n lab describe analysisrun RUN_NAME

# Запустить удобный локальный Web UI дашборд (по умолчанию открывается на порту 3100)
kubectl argo rollouts dashboard
```
