## Стартовая проверка

Прежде чем разворачивать новые компоненты, давайте проверим, что сейчас происходит с логами в кластере и какие StorageClass нам доступны.

```bash
# Проверяем StorageClass, который Loki будет использовать для PersistentVolumeClaim
kubectl get sc

# Есть ли уже какие-то поды в lab? (должно быть пусто)
kubectl -n lab get pods
```

**Как мы смотрели логи до этой лабы (базовый функционал Kubernetes):**
```bash
kubectl logs <pod_name>
```

Базовый `kubectl logs` читает логи напрямую с диска ноды (обычно `/var/log/pods/...`), где работает контейнер. Это неудобно по ряду причин:
1. **Эфемерность:** Если Pod удалён или перенесён на другую ноду, его логи на старой ноде безвозвратно исчезают.
2. **Масштаб:** Когда подов десятки, читать их по одному невозможно.
3. **Поиск:** Нельзя сделать агрегацию или глобальный поиск по всем логам кластера.

Именно поэтому необходим централизованный сборщик.

## Часть 1: Развёртывание и базовые концепции

### Теория для изучения перед частью

Большинство классических систем логирования (Elasticsearch, OpenSearch, Splunk) строят **полнотекстовый индекс** (inverted index). Они разбивают каждую строку лога на слова и создают огромный словарь. Это позволяет искать очень быстро по любому слову, но требует огромных ресурсов: индекс часто занимает больше места на диске, чем сами логи, и требует много RAM.

Loki пошёл другим путём: **он вообще не индексирует текст логов.**

```text
┌─────────────────┐       ┌─────────────┐       ┌─────────────────┐
│  Pod (app: A)   │       │             │       │   Grafana UI    │
│  stdout / logs  ├──────►│  Promtail   │       │ (LogQL queries) │
└─────────────────┘       │ (DaemonSet) │       └─────────┬───────┘
                          │             │                 │
┌─────────────────┐       │ - читает    │       ┌─────────▼───────┐
│  Pod (app: B)   ├──────►│   /var/log  ├──────►│   Loki Server   │
│  stdout / logs  │       │ - вешает    │ (HTTP)│  (index: labels │
└─────────────────┘       │   метки     │       │   data: chunks) │
                          └─────────────┘       └─────────────────┘
```

### 1.1 Индексация в Loki: почему он такой быстрый и дешёвый

Loki делит хранилище на две совершенно разные части:
1. **Индекс (Index):** Хранит ТОЛЬКО метаданные (метки, labels), такие как `{namespace="lab", app="payment-api", node="worker-1"}`. Индекс очень маленький, быстрый и легко помещается в памяти или быстром key-value хранилище (в современных версиях используется TSDB).
2. **Чанки (Chunks):** Сами сырые строки логов. Они группируются (сжимаются gzip/snappy) блоками по уникальному набору меток и времени, а затем складываются в дешёвое хранилище (объектное хранилище S3, GCS или просто локальный диск).

Поиск в Loki работает как параллельный распределенный `grep`: сначала Loki использует индекс, чтобы найти нужные чанки, а затем распаковывает эти чанки в памяти и прогоняет по ним регулярные выражения. Это радикально дешевле в хранении, но скорость поиска напрямую зависит от того, насколько точно метки сузили первоначальный объем данных.

### 1.2 Архитектура микросервисов Loki

Хотя в нашей лабе Loki развёрнут как единый бинарник (monolithic mode), внутри он логически разделён на микросервисы:
- **Distributor:** Принимает логи по HTTP, валидирует их (рейт-лимиты) и роутит.
- **Ingester:** Держит свежие логи в памяти, формирует чанки и периодически сбрасывает их на диск.
- **Querier:** Выполняет запросы, обращаясь и к Ingester (за свежими логами), и к хранилищу (за старыми).
- **Compactor:** Оптимизирует индексы и применяет retention.

### 1.3 Жизненный цикл чанка (Chunk)

1. Лог попадает в Ingester.
2. Ingester держит чанк открытым в памяти до достижения лимита размера (например, 1.5MB) или времени (например, 2 часа).
3. Чанк закрывается и сбрасывается в долгосрочное хранилище (S3/Filesystem).
4. Если Ingester падает до сброса чанка, данные могут быть потеряны (поэтому в production используют WAL - Write-Ahead Log).

### 1.4 Развёртывание стека на стенде

Развернём стек: сервер Loki, агенты Promtail и два тестовых генератора логов.

```bash
# Применяем все манифесты из папки
kubectl apply -f manifests/

# Ждём готовности Loki (сервер)
kubectl -n lab rollout status deploy/loki --timeout=120s

# Ждём готовности Promtail (агенты на каждой ноде)
kubectl -n lab rollout status ds/promtail --timeout=120s

# Проверяем, что всё поднялось
kubectl -n lab get pods -o wide
```

В `lab` появятся:
- **loki:** Сервер (1 реплика).
- **promtail:** DaemonSet (по агенту на ноду).
- **log-generator:** Эмулятор текстовых логов `<ts> LEVEL msg id=N`.
- **payment-api:** Эмулятор JSON-логов `{"level":...,"status":...,"path":...,"duration_ms":...}`.

Примеры реальных строк (снято со стенда):

```text
# log-generator
2026-06-11T10:07:05+00:00 INFO user login id=9849

# payment-api
{"level":"error","status":500,"path":"/api/refund","duration_ms":163,"msg":"request handled"}
```

## Часть 2: Promtail — умный агент сборки

### Теория для изучения перед частью

Promtail — это интеллектуальный агент сборки логов. Он не просто читает текстовые файлы, он умеет трансформировать логи на лету с помощью Pipeline Stages.

### 2.1 Механизм Service Discovery

Посмотрим на конфигурацию Promtail:

```bash
kubectl -n lab get configmap promtail-config -o yaml | less
```

Обратите внимание на блоки `kubernetes_sd_configs` и `relabel_configs`. Это копия механизмов из Prometheus. Promtail опрашивает Kubernetes API, находит поды, которые запланированы на его ноду (`field_selectors: spec.nodeName=$HOSTNAME`), и превращает их метаданные в метки.

Например, скрытая мета-метка `__meta_kubernetes_namespace` становится меткой `namespace`. А главное, Promtail формирует скрытую метку `__path__` — физический путь к файлу логов пода на диске ноды.

### 2.2 Pipeline Stages: трансформация логов на лету

Pipeline stages позволяют применять серию трансформаций к каждой строке лога:
- `docker` / `cri`: Удаление префиксов рантайма.
- `json` / `regex`: Парсинг строки и извлечение полей во внутренние переменные агента.
- `labels`: Превращение внутренних переменных в метки Loki.
- `timestamp`: Подмена времени лога.
- `replace`: Замена текста (например, скрытие паролей).

### 2.3 Удаление префикса CRI (Container Runtime Interface)

В современных кластерах (containerd/CRI-O) все логи имеют префикс `...Z stdout F`.

> ⚠️ **Грабля (наступили вживую):** Без стадии `cri` этот префикс улетает в Loki как часть строки. Глазами в Grafana его можно не заметить, но строка `{"error":"crash"}` превращается в `2026-06-11T10:00:00Z stdout F {"error":"crash"}`.
> Все парсеры JSON в Loki ломаются, потому что строка больше не начинается со скобки `{`.

В нашем `promtail.yaml` добавлено:
```yaml
pipeline_stages:
  - cri: {}
```

### 2.4 Манипуляции с метками: как избежать High Cardinality

**Поток (stream)** в Loki — это уникальная комбинация всех меток (например: `{app="payment", level="error"}`).
Если вы сделаете меткой уникальный идентификатор (например, `request_id`, `user_ip` или `trace_id`), каждый новый запрос будет создавать новый поток.
- 1 000 000 запросов = 1 000 000 потоков.

Это называется **High Cardinality**. Loki попытается создать миллионы мелких чанков, индекс распухнет, и OOM убьёт процесс Ingester-а.
**Золотое правило Loki:** Индексируйте только статические вещи. Уникальные данные оставляйте внутри текста лога!

## Часть 3: LogQL — выборка, line-фильтры и парсеры

### Теория для изучения перед частью

Запрос LogQL состоит из двух частей:
1. **Селектор потоков:** Выбор по меткам (`{app="payment-api"}`).
2. **Конвейер:** Опциональные фильтры, парсеры и форматировщики, разделённые пайпом `|`.

| Оператор | Смысл |
|----------|-------|
| `\|= "text"` | строка СОДЕРЖИТ подстроку |
| `!= "text"` | НЕ содержит |
| `\|~ "regex"` | матчится по регулярному выражению |
| `!~ "regex"` | не матчится по регулярному выражению |

### 3.1 Селекторы потоков (Stream Selectors)

Для удобства создадим функцию-хелпер, которая обращается к API Loki:

```bash
lq() {
  local enc; enc=$(python3 -c "import urllib.parse,sys; print(urllib.parse.quote(sys.argv[1]))" "$1")
  kubectl -n lab exec deploy/loki -- wget -qO- \
    "http://localhost:3100/loki/api/v1/query_range?query=${enc}&limit=${2:-5}&since=5m"
}
```

Все логи приложения `log-generator`:
```bash
lq '{app="log-generator"}'
```

Только строки с ошибками:
```bash
lq '{app="log-generator"} |= "ERROR"'
```

Сцепка фильтров:
```logql
lq '{namespace="lab"} |= "ERROR" |~ "(?i)payment" != "queued"'
```

### 3.2 Парсер json и извлечение динамических полей

Парсер `| json` разбирает строку на лету и создаёт временные метки (только на время выполнения запроса). Затем к ним можно применять label-фильтры.

```bash
# Только серверные ошибки (статус 500 и выше):
lq '{app="payment-api"} | json | status >= 500'
```

### 3.3 Парсер pattern для кастомных форматов логов

Если логи не JSON, а структурированный текст, используйте парсер `| pattern`. Это работает быстрее, чем регулярные выражения.

```bash
# Вытаскиваем уровень и id из текстового лога:
lq '{app="log-generator"} | pattern "<ts> <level> <msg> id=<id>" | level="ERROR"'
```

### 3.4 Форматирование вывода через line_format

Читать сырые JSON-строки неудобно. Перепишем вывод с помощью `line_format`.

```bash
lq '{app="payment-api"} | json | status=500 | line_format "{{.path}} -> {{.status}} за {{.duration_ms}}мс"'
```

Вывод преобразится в:
```text
/api/pay -> 500 за 364мс
/api/refund -> 500 за 460мс
```

### 3.5 Использование парсера regexp

Для сложных логов:
```logql
{app="legacy"} | regexp "^(?P<ip>\\S+) - (?P<user>\\S+) \\[(?P<time>.*?)\\] \"(?P<method>\\S+)"
```

## Часть 4: Метрики из логов — агрегация и алертинг

### Теория для изучения перед частью

LogQL позволяет строить полноценные метрики на основе логов. Loki может работать как Prometheus: вычислять рейты, суммы, средние значения и перцентили. Это называется Metric Queries.

### 4.1 Подсчёт частоты логов (rate)

Используем мгновенный эндпоинт `/query` (instant query):

```bash
li() {
  local enc; enc=$(python3 -c "import urllib.parse,sys; print(urllib.parse.quote(sys.argv[1]))" "$1")
  kubectl -n lab exec deploy/loki -- wget -qO- "http://localhost:3100/loki/api/v1/query?query=${enc}"
}
```

Сколько ERROR-строк генерируется в секунду, сгруппировано по уровню лога:
```bash
li 'sum by (level) (rate({app="payment-api"} | json | __error__="" [5m]))'
```

Сколько всего было ERROR-строк за последние 5 минут (count):
```bash
li 'sum(count_over_time({app="log-generator"} |= "ERROR" [5m]))'
# => 63
```

### 4.2 Подсчёт объёма данных (bytes_over_time)

Кто потребляет больше всего места в хранилище логов:
```bash
li 'sum by (app) (bytes_over_time({namespace="lab"}[5m]))'
# log-generator => 39331
# loki          => 364407
# payment-api   => 45371
# promtail      => 182475
```

### 4.3 Извлечение числовых значений (unwrap) и квантили

95-й перцентиль времени ответа API (`duration_ms`):
```bash
li 'quantile_over_time(0.95, {app="payment-api"} | json | __error__="" | unwrap duration_ms [5m]) by (path)'
# Пример ответа:
# {path="/api/balance"} => 475.2
# {path="/api/pay"}     => 455.5
```
> 💡 Без `by (path)` результат сгруппируется по ВСЕМ меткам, и вы получите кашу. Группировка обязательна.

### 4.4 Проблема \_\_error\_\_ и её обход

Что будет, если применить `| json` к потоку логов, в котором случайно проскочила текстовая строка?
Loki присвоит ей скрытую метку `__error__="JSONParserErr"`.

Обычные лог-запросы проигнорируют ошибку. Но метрические запросы очень строгие: если хоть одна строка содержит ошибку парсинга, **весь запрос падает с HTTP 400 Bad Request**.

Вы увидите: `pipeline error: 'JSONParserErr' ... Value looks like object, but can't find closing '}'`

> **Золотое правило:** В метрических запросах ПОСЛЕ парсера всегда добавляйте фильтр `| __error__=""`. Это молча отбросит битые строки.

## Часть 5: Интеграция с Grafana — единый workflow

### Теория для изучения перед частью

Loki максимально раскрывается в связке с Grafana. Это позволяет бесшовно прыгать от метрик к логам и обратно ("Observability Workflow").

### 5.1 Настройка Datasource в Grafana

```bash
# Пароль для admin:

```

Перейдите в **Explore**, выберите Datasource **Loki**.

### 5.2 Корреляция метрик и логов (Split View)

Workflow расследования «всплеска 500-х ошибок»:
1. **Explore → Loki**: Пишем `sum by (status) (rate({app="payment-api"} | json | __error__="" [1m]))`. На графике видим аномалию.
2. **Split view**: Экран делится пополам. Справа пишем лог-запрос: `{app="payment-api"} | json | status=500`.
3. Grafana автоматически синхронизирует время. Вы мгновенно видите конкретные строки логов.

### 5.3 Derived fields: бесшовный переход в трейсинг (Tempo)

В настройках Datasource Loki в Grafana можно добавить Derived fields. Если в логе есть поле `trace_id=123456`, Grafana превратит его в кликабельную ссылку, ведущую в систему трассировки Tempo.

## Часть 6: Troubleshooting — боевые инциденты

### Теория для изучения перед частью

Пайплайн логов хрупок: если Promtail не прочитал файл, логи потеряны; если Loki перегружен, запросы падают.

### Инцидент 1: Логи не появляются в Loki (ошибка Service Discovery)

**Симптом:** Вы деплоите приложение, делаете `{app="my-app"}` в Grafana, а там пусто.
**Диагностика:**
1. Работает ли Promtail: `kubectl -n lab get pods -l app=promtail`.
2. Логи самого Promtail: `kubectl -n lab logs ds/promtail | grep level=error`.
3. Убедитесь, что переменная окружения `HOSTNAME` проброшена в контейнер Promtail и совпадает с именем ноды. Иначе Service Discovery не найдет поды.

### Инцидент 2: Метрические запросы падают с HTTP 400 (JSONParserErr)

**Симптом:** График `rate(...)` выдаёт ошибку `pipeline error: 'JSONParserErr'`.
**Диагностика:** Классическая грабля `__error__`. Кто-то написал `fmt.Println("debug")` в приложении, пишущем JSON.
**Решение:** Обновите запрос, добавив `| __error__=""` после `| json`.

### Инцидент 3: Утечка памяти в Loki (High Cardinality)

**Симптом:** Loki рестартится по OOMKilled. Запросы падают с таймаутом.
**Диагностика:** Кто-то добавил уникальный ID (`user_id`) в метки.
```logql
sum by (app) (count_over_time({namespace="lab"}[1h]))
```
Если миллионы потоков, проверьте метки.
**Решение:** Удалить уникальные ID из меток. Искать через `| json | user_id="123"`.

### Инцидент 4: Ошибка 429 Too Many Requests (Ingestion Rate Limit)

**Симптом:** В логах Promtail массово сыпятся ошибки `429 Too Many Requests`.
**Диагностика:** Loki защищает себя лимитами (MB/s).
**Решение:** Поднять лимиты в `loki-config.yaml`:
```yaml
limits_config:
  ingestion_rate_mb: 15
  ingestion_burst_size_mb: 30
```

### Инцидент 5: "Out of order" при записи логов

**Симптом:** Promtail пишет `entry with timestamp X is older than Y`.
**Решение:** В конфиге Loki включить `unordered_writes: true`.

### Бонус: общая диагностика пайплайна логов

```text
Симптом: Логи отсутствуют в Grafana
│
├─ Пишет ли приложение логи? ─────► kubectl logs <pod> 
│
├─ Есть ли логи на ноде? ─────────► SSH на ноду, ls -l /var/log/pods/
│
├─ Видит ли их Promtail? ─────────► kubectl logs ds/promtail | grep <pod_name>
│
└─ Доходят ли они до Loki? ───────► Проверяем метрики HTTP 429 в Promtail
```
