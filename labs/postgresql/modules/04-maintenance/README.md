## Часть 1: Многоверсионность (MVCC)

В PostgreSQL используется механизм управления конкурентным доступом на основе многоверсионности. Когда вы обновляете строку (UPDATE), база данных не перезаписывает старую строку. Она помечает старую строку как "устаревшую", создает новую версию строки и перенаправляет будущие транзакции на нее.

Это гарантирует, что транзакции чтения никогда не блокируются транзакциями записи.

## Часть 2: Проблема мертвых строк

Создадим большую таблицу и удалим половину данных:
```sql
CREATE TABLE large_table (id SERIAL PRIMARY KEY, data TEXT);
INSERT INTO large_table (data) SELECT md5(random()::text) FROM generate_series(1, 1000000);

-- Проверим размер таблицы
SELECT pg_size_pretty(pg_relation_size('large_table'));

-- Удалим 500 тысяч строк
DELETE FROM large_table WHERE id % 2 = 0;

-- Снова проверим размер
SELECT pg_size_pretty(pg_relation_size('large_table'));
```
Вы увидите, что размер таблицы не уменьшился! Физическое место на диске по-прежнему занято "мертвыми" строками (Dead tuples).

---

## Часть 3: Процесс VACUUM

Для очистки мертвых строк используется процесс `VACUUM`. Он сканирует таблицу, помечает занятое мертвыми строками пространство как свободное (Free Space Map) для последующих `INSERT` и `UPDATE`.

Выполним ручную очистку с подробным выводом:
```sql
VACUUM VERBOSE large_table;
```
Обычный `VACUUM` не отдает свободное место операционной системе (не уменьшает файл), он лишь позволяет базе данных переиспользовать это место. Чтобы вернуть место ОС, необходим `VACUUM FULL`, который полностью блокирует таблицу и пересоздает ее физический файл.

## Часть 4: Обновление статистики (ANALYZE)

Оптимизатор запросов (Query Planner) принимает решения на основе статистики распределения данных в таблицах. Если статистика устарела, оптимизатор может выбрать катастрофически неэффективный план выполнения запроса (например, Sequential Scan вместо Index Scan).

Команда `ANALYZE` собирает актуальную статистику:
```sql
ANALYZE VERBOSE large_table;
```
На практике чаще всего команды совмещают: `VACUUM ANALYZE large_table;`. В современном PostgreSQL эти операции автоматически выполняются фоновым процессом `autovacuum`.

## Часть 5: Troubleshooting

**Сценарий:** Разработчики запустили очень длинную транзакцию чтения (например, экспорт отчета), которая висит уже сутки. В это время в базу непрерывно идут UPDATE и DELETE.
**Последствия:** Autovacuum не может удалить мертвые строки, так как старая транзакция теоретически еще может к ним обратиться. Происходит колоссальное раздувание (Bloat) таблиц. Диск заполняется на 100%.
**Решение:** Для мониторинга таких транзакций необходимо использовать системное представление `pg_stat_activity` и принудительно завершать долгие запросы функцией `pg_terminate_backend(pid)`.
