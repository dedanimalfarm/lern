## Часть 1: Физическое против логического резервного копирования

### Теория: Выбор стратегии
- **Логическое копирование (pg_dump)**: Экспортирует структуру и данные в виде SQL команд или специального бинарного формата. Плюсы: можно восстанавливать отдельные таблицы, совместимость между разными версиями ОС и архитектурами. Минусы: медленное восстановление огромных БД.
- **Физическое копирование (pg_basebackup)**: Копирует бинарные файлы данных непосредственно с диска сервера. Плюсы: экстремально быстрое восстановление, основа для создания реплик. Минусы: восстановить можно только весь кластер целиком.

## Часть 2: Логический дамп утилитой pg_dump

Используем утилиту `pg_dump` для создания резервной копии в формате "custom" (`-Fc`). Этот формат автоматически сжимает данные и позволяет выборочное восстановление.

```bash
pg_dump -Fc -U postgres shop_db -f /tmp/shop_backup.dump
```
Убедитесь, что файл создан и проверьте его размер:
```bash
ls -lh /tmp/shop_backup.dump
```

## Часть 3: Восстановление с помощью pg_restore

Сымитируем катастрофу. Подключитесь к серверу и удалите базу:
```sql
DROP DATABASE shop_db;
CREATE DATABASE shop_db;
```

Восстановите базу данных из дампа:
```bash
pg_restore -U postgres -d shop_db -1 /tmp/shop_backup.dump
```
Флаг `-1` (`--single-transaction`) гарантирует, что если в процессе восстановления произойдет ошибка, все изменения будут отменены (rolled back).

## Часть 4: Потоковая репликация

Для обеспечения отказоустойчивости (High Availability) используется потоковая репликация WAL (Write-Ahead Log) файлов. Ведущий узел (Master) непрерывно передает журналы транзакций на ведомый узел (Replica), который применяет их у себя. Это тема для продвинутого модуля (Advanced Production Track), но знать этот термин необходимо.

## Часть 5: Troubleshooting

**Сценарий:** Во время выполнения дампа закончилось место на диске, и файл дампа был обрезан. Вы пытаетесь восстановить его с помощью `pg_restore` и получаете фатальную ошибку целостности формата.
**Задача:** Как извлечь хотя бы часть данных (в виде текстового SQL) из поврежденного custom архива? Изучите справку `pg_restore --help` и попробуйте восстановить дамп в стандартный вывод (`stdout`) без применения к базе.
