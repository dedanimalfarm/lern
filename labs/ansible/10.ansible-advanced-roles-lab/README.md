## Подготовка окружения
Убедитесь, что у вас установлен Ansible и (опционально) Molecule для выполнения 4-го шага. Создайте рабочую директорию для ваших ролей:

```bash
mkdir -p roles
```

## Шаг 1. Зависимости ролей

Ansible позволяет указывать зависимости одной роли от других. Это гарантирует, что перед выполнением целевой роли будут успешно применены все её зависимости. Зависимости настраиваются в файле `meta/main.yml`.

1. Создайте структуру для двух ролей — `firewall` и `web`:
   ```bash
   ansible-galaxy init roles/firewall
   ansible-galaxy init roles/web
   ```

2. Настройте зависимость роли `web` от роли `firewall`. Для этого откройте файл `roles/web/meta/main.yml` и добавьте в секцию `dependencies`:
   ```yaml
   dependencies:
     - { role: firewall }
   ```

Теперь каждый раз при применении роли `web`, Ansible будет сначала выполнять роль `firewall`.

## Шаг 2. Использование динамических переменных и фактов

Ansible собирает массу информации (фактов) о целевых системах до выполнения задач. Вы можете использовать эти факты, чтобы сделать роли адаптивными и независимыми от конкретной операционной системы.

1. Откройте `roles/web/tasks/main.yml` и добавьте задачу, которая использует динамические факты (`ansible_distribution`):
   ```yaml
   ---
   # tasks file for roles/web
   - name: Печать информации об ОС целевого узла
     debug:
       msg: "Устанавливаем веб-сервер для {{ ansible_distribution }} версии {{ ansible_distribution_version }}"
   ```

2. Вы также можете загружать переменные из отдельных файлов на основе фактов. Например, создав файл `roles/web/vars/Ubuntu.yml` или `roles/web/vars/CentOS.yml`, и импортировав нужный файл динамически:
   ```yaml
   - name: Загрузка специфичных для ОС переменных
     include_vars: "{{ ansible_distribution }}.yml"
   ```

## Шаг 3. Оптимизация сценариев с помощью кеширования фактов

Сбор фактов — довольно медленный процесс. Ansible позволяет сохранять собранные факты в кеш, чтобы ускорить повторное выполнение плейбуков.

1. В текущей директории создайте файл настроек `ansible.cfg` (если его нет) со следующим содержимым:
   ```ini
   [defaults]
   gathering = smart
   fact_caching = jsonfile
   fact_caching_connection = /tmp/ansible_cache
   fact_caching_timeout = 86400
   ```

   **Где:**
   - `gathering = smart` — факты собираются только если они еще не были получены или их время жизни истекло.
   - `fact_caching = jsonfile` — сохраняем кеш в JSON файлы.
   - `fact_caching_connection = /tmp/ansible_cache` — путь для файлов кеша.
   
2. Создайте тестовый плейбук `site.yml` и запустите его дважды. Вы заметите, что второй запуск проходит значительно быстрее, так как стадия "Gathering Facts" будет пропущена.

## Шаг 4. Тестирование ролей с использованием Molecule

Molecule предоставляет средства для разработки и тестирования ролей в изолированном окружении (Docker, Vagrant и др.).

1. Перейдите в директорию созданной роли `web` и инициализируйте Molecule (требуется установленный пакет `molecule`):
   ```bash
   cd roles/web
   molecule init scenario -d docker
   ```
2. Это создаст директорию `molecule/default` внутри вашей роли. В файле `molecule.yml` описано, как поднимать тестовое окружение, а в `verify.yml` — как проверять состояние после отработки роли.
3. Чтобы запустить полный цикл тестов, используйте команду:
   ```bash
   molecule test
   ```
   *(Примечание: для успешного завершения команды у вас должен быть запущен Docker daemon и корректно настроены параметры в molecule.yml)*

## Troubleshooting — частые проблемы

- **Ошибка: `ERROR! The role 'firewall' was not found in...`**
  **Решение:** Убедитесь, что вы находитесь в правильной директории при запуске плейбука, и путь к ролям указан корректно в конфигурации `roles_path` или вы используете директорию `roles/` в корне проекта.

- **Ошибка: `Permission denied: '/tmp/ansible_cache'`**
  **Решение:** Возможно, файлы кеша были созданы от имени другого пользователя (например, при запуске с `sudo`). Попробуйте удалить кеш командой `sudo rm -rf /tmp/ansible_cache` или измените путь `fact_caching_connection`.

- **Molecule: `Docker daemon is not running`**
  **Решение:** Проверьте работу Docker: `systemctl status docker` или убедитесь, что ваш пользователь состоит в группе `docker` (`usermod -aG docker $USER`).
