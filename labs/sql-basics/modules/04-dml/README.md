## Часть 1: Транзакции (ACID)

Любая модификация данных в реляционной СУБД должна подчиняться правилам ACID (Atomicity, Consistency, Isolation, Durability). Команды `BEGIN`, `COMMIT` и `ROLLBACK` управляют границами транзакции. Если в процессе модификации нескольких таблиц произошла ошибка, транзакция должна быть откачена (`ROLLBACK`), чтобы не оставить базу в несогласованном состоянии.

## Часть 2: Вставка записей (INSERT)

Команда `INSERT` добавляет новые строки. Вы можете вставлять несколько строк одним запросом (Bulk Insert), что работает в десятки раз быстрее, чем множество одиночных команд.

```sql
INSERT INTO users (name, email) 
VALUES 
    ('Алексей', 'alex@example.com'),
    ('Елена', 'elena@example.com');
```
Если вы используете PostgreSQL, оператор `RETURNING` позволяет сразу получить значения колонок (например, сгенерированные ID) только что вставленных строк:
```sql
INSERT INTO products (name, price, stock_quantity) 
VALUES ('Новый товар', 1000, 5) RETURNING id;
```

## Часть 3: Модификация и Удаление

`UPDATE` модифицирует существующие строки. `DELETE` удаляет их.
**Обе команды требуют предельной осторожности и обязательного указания фильтрации `WHERE`.**

```sql
UPDATE products 
SET price = price * 0.9, stock_quantity = stock_quantity - 1 
WHERE id = 1;

DELETE FROM orders WHERE status = 'CANCELLED';
```

## Часть 4: Каскадное удаление данных

Если вы попытаетесь удалить пользователя, у которого есть заказы в таблице `orders`, СУБД заблокирует операцию и выдаст ошибку `Foreign key constraint violation`. Это защита целостности данных, предотвращающая появление "осиротевших" записей.
Чтобы дочерние записи удалялись автоматически вместе с родительскими, при создании внешнего ключа в структуре таблицы указывается правило `ON DELETE CASCADE`.

## Часть 5: Troubleshooting

**Сценарий:** На сервере production вы выполняете ручную корректировку цен. Вы пишете:
```sql
UPDATE products SET price = 5000;
```
Нажимаете Enter и понимаете, что забыли `WHERE id = 123`. Все товары в магазине стали стоить 5000. Бизнес остановлен.

**Решение и лучшие практики:** Никогда не выполняйте `UPDATE` или `DELETE` в консоли production напрямую. Всегда оборачивайте их в транзакцию.
```sql
BEGIN;
UPDATE products SET price = 5000 WHERE id = 123;
-- Проверьте, сколько строк было обновлено (UPDATE 1). Если UPDATE 1000 - срочно делайте ROLLBACK.
COMMIT;
```
Если данные уже испорчены и транзакция зафиксирована, единственное спасение — восстановление из резервной копии (или механизм Point-in-Time Recovery, если настроено непрерывное архивирование WAL файлов).

## Вопросы для самопроверки
1. Чем команда `DELETE FROM table;` отличается от команды `TRUNCATE TABLE table;` с точки зрения производительности и внутренних механизмов СУБД?
2. Что произойдет при попытке выполнить `INSERT`, если вставляемое значение нарушает ограничение `UNIQUE`?
3. Гарантирует ли транзакция, что изменения будут сохранены на жестком диске даже в случае внезапного отключения питания сервера (D - Durability в ACID)? Как это достигается технически?
