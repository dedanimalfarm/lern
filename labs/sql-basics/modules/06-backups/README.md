## Часть 1: Логическое резервное копирование (pg_dump)

В PostgreSQL резервные копии делятся на физические (копирование бинарных файлов данных с диска) и логические (выгрузка структуры таблиц и данных в виде команд SQL или сжатых архивов).

Утилита **`pg_dump`** выполняет логическое резервное копирование **одной конкретной базы данных**. Во время работы `pg_dump` база данных остается доступной для чтения и записи пользователями (утилита использует транзакции с уровнем изоляции `Repeatable Read`).

Базовый синтаксис:
```bash
pg_dump -U имя_пользователя -d имя_бд -f путь_к_файлу
```

## Часть 2: Форматы бэкапов (Plain vs Custom vs Directory)

Параметр `-F` (или `--format`) определяет формат резервной копии:

1. **Plain text (`-F p` или `--format=plain`):**
   * *Что это:* Обычный текстовый SQL-файл, содержащий команды `CREATE TABLE`, `INSERT`, `COPY` и т.д.
   * *Плюсы:* Файл можно открыть в любом текстовом редакторе, изменить структуру или запросы вручную.
   * *Минусы:* Медленный импорт, большой размер файла, невозможно восстановить отдельные таблицы выборочно из архива.

2. **Custom (`-F c` или `--format=custom`):**
   * *Что это:* Сжатый бинарный формат PostgreSQL (по умолчанию сжимается с помощью zlib).
   * *Плюсы:* Самый популярный формат. Минимальный размер на диске, высокая скорость, возможность выборочного восстановления таблиц.
   * *Минусы:* Нельзя прочитать обычным текстовым редактором.

3. **Directory (`-F d` или `--format=directory`):**
   * *Что это:* Директория с файлами (каждая таблица выгружается в отдельный файл).
   * *Плюсы:* Поддерживает **многопоточный бэкап** (`-j N`), что критически важно для баз данных размером в сотни гигабайт.

## Часть 3: Восстановление данных (psql и pg_restore)

Метод восстановления зависит от формата резервной копии:

1. **Восстановление текстового бэкапа (Plain SQL):**
   Текстовые бэкапы накатываются стандартным клиентом `psql`:
   ```bash
   psql -U postgres -d shop_db -f /tmp/backup.sql
   ```

2. **Восстановление бинарного бэкапа (Custom / Directory):**
   Бинарные форматы восстанавливаются специальной утилитой **`pg_restore`**:
   ```bash
   pg_restore -U postgres -d shop_db /tmp/backup.dump
   ```

## Часть 4: Выборочное (селективное) восстановление

Одно из ключевых преимуществ формата `Custom` (`-F c`) — возможность восстановить только одну конкретную таблицу из всего бэкапа БД.

Синтаксис:
```bash
pg_restore -U postgres -d shop_db -t имя_таблицы /tmp/backup.dump
```
*Например*, если разработчики случайно очистили таблицу `order_items`, DevOps-инженер может достать только её из бэкапа всей базы данных на 500 ГБ, не тратя часы на полное восстановление.

## Часть 5: Резервное копирование глобальных объектов (pg_dumpall)

Утилита `pg_dump` копирует только одну базу данных и **не сохраняет** глобальные объекты кластера:
* Пользователей и роли (`roles`)
* Права доступа (`grants`)
* Табличные пространства (`tablespaces`)

Чтобы сделать полную резервную копию всего сервера (всех баз данных и ролей), используется утилита **`pg_dumpall`**:
```bash
pg_dumpall -U postgres -f /tmp/all_databases_backup.sql
```
*Примечание:* `pg_dumpall` генерирует бэкап только в текстовом формате (plain text).

## Часть 6: Troubleshooting — блокировки и ошибки при восстановлении

### Сценарий 1: Ошибки «already exists» при повторном восстановлении
Если вы восстанавливаете бэкап в базу данных, где таблицы уже созданы, `pg_restore` упадет с ошибками `relation "users" already exists`.

**Решение:**
Используйте флаг `-c` (или `--clean`). Он заставляет утилиту добавлять команды `DROP` перед созданием объектов:
```bash
-- Для pg_restore:
pg_restore -c -d shop_db /tmp/backup.dump

-- Для pg_dump (команды DROP запишутся внутрь бэкапа):
pg_dump -c -d shop_db -f /tmp/backup.sql
```

### Сценарий 2: Блокировки DDL во время бэкапа
Хотя `pg_dump` не блокирует чтение и запись (`SELECT`/`INSERT`/`UPDATE`/`DELETE`), он накладывает блокировку `AccessShareLock` на таблицы.

**В чем опасность?**
Если во время работы бэкапа на прод-базе запустится миграция приложения (которая делает `ALTER TABLE` или `DROP TABLE`), миграция попытается взять блокировку `AccessExclusiveLock`. Она встанет в очередь ожидания за бэкапом, а все последующие пользовательские запросы встанут в очередь за миграцией. Это приведет к лавинообразному исчерпанию соединений и отказу БД.
**Решение:** Никогда не запускайте тяжелые DDL-миграции одновременно с планируемым резервным копированием.
