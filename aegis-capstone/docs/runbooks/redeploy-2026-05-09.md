## Подводные камни, на которые попали

### 1. Баг суффикса раздела для `/dev/md0`

В роли `01-storage` задача «Обновление списка устройств с учетом разделов» строит имя раздела как `dev + ('p1' if 'nvme' in dev else '1')`. Для `/dev/md0` это даёт `/dev/md01`, которого не существует — kernel создал `/dev/md0p1`. Падало именно `az-storage`.

**Фикс** (применён в этом прогоне):

```yaml
{%- set suffix = 'p1' if item.dev[-1] is match('[0-9]') else '1' -%}
```

Логика: если базовое имя устройства заканчивается цифрой (`md0`, `nvme0n1`), нужен `p1`; иначе (`sda`, `sdb`) — просто `1`.

### 2. Не-идемпотентный `lvol` с `100%FREE`

`community.general.lvol` при повторном прогоне сравнивает «новый» размер (100% от свободного = 0, потому что LV уже занял всё) со старым и решает, что нужно сжать → падает с `Sorry, no shrinking of lv_X to 0 permitted.` Проявилось на ВСЕХ узлах при втором прогоне `site.yml`.

**Фикс:**

```yaml
- name: Создание Logical Volume (100%FREE)
  community.general.lvol:
    vg: "vg_{{ item.name }}"
    lv: "lv_{{ item.name }}"
    size: 100%FREE
    shrink: false   # <-- добавлено
```

### 4. by-path naming различается между VM (Azure storvsc vs scsi)

При повторном raise-from-zero (destroy → packer → apply → ansible) обнаружили, что Azure отдаёт диски через два разных префикса в `/dev/disk/by-path/`:

- На большинстве VM: `acpi-MSFT1000:00-scsi-0:0:0:N` (старый glob `*-scsi-0:0:0:N` матчит).
- На `az-storage` (r3, southeastasia): `acpi-VMBUS:00-vmbus-...-lun-N` (старый glob НЕ матчит → пустой список → `mdadm --create` падает с *"You haven't given enough devices"*).

**Фикс** в `01-storage/tasks/main.yml`, обе задачи «Резолв LUN ...»:

```yaml
shell: |
  set -e
  for f in /dev/disk/by-path/*-scsi-0:0:0:{{ item.lun }} /dev/disk/by-path/*-lun-{{ item.lun }}; do
    [ -e "$f" ] && readlink -f "$f" && exit 0
  done
  exit 1
```

Перебирает оба паттерна, возвращает первый существующий и ненулевой exit при отсутствии (вместо тихого пустого stdout).
